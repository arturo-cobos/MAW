Task: Leverage Existing Assets
Leverage existing assets for a given context by reuse or customization rather than building things from scratch.
Purpose

To ensure the leveraging or reuse off existing assets in the creation of a service-oriented architecture.  At each point within the soma method, where an asset type such as a process called goal, service or component is going to be identified, instead of considering to build this element from scratch, we will endeavor to look up existing assets of that type within a repository or otherwise obtained those assets from third parties or industry models and standards that are made available to us.

This capability pattern is a recurrent one throughout the soma lifecycle.  It is used when ever a new asset type is encountered.  For example when we want to perform process decomposition or process modeling within domain decomposition during service identification, we will not start with the premise that no process model exists.  We will endeavor to leverage the organizations existing process models or attempt to look up process models that we have brought in as assets from our industry models, or other industry models were standards that are available to the client.

        Main Description

        Facilitates the reuse or customization or leveraging of assets rather than building things from scratch.  Ideally an asset will be used as it is in multiple reusable contexts.  However if an asset is not able to be used within a given context that assets should be updated or extended or customized in order to be leveraged to save time and money rather than building some something from scratch.  The decision must be made of whether it is worthwhile to customize an existing asset or to build a new one.

        In general we look for the assets in a repository, seek to incorporate industry models or existing legacy applications or other assets available within the organization before attempting to create them. It is considered a competitive advantage to bring in a set of potentially reusable assets that can be utilized in the context of the project at hand rather than building those assets from scratch. This asset based model of software development is a key competitive advantage that can be leveraged through the use of the SOMA method for building service-oriented architectures.

        Note: This task mirrors Leverage Existing Asset (LEA) capability pattern SOMA. This task is used in the delivery process and other capability patterns instead of LEA capability pattern to reduce number of tasks in work breakdown structure.

        Steps
        Discover and Incorporate Assets

        Description: Identify and discover existing assets that have the potential of being leveraged to realize one or more elements of a service analysis, design or implementation. Rather than building the asset from scratch, consider searching for, discovering and incorporating the asset as is or with modifications.

        Perform an analysis of the existing inventory (might include lookup in a repository) of the specific asset type, for example a service or a service composition and flow. Use it as the basis for extending the asset type, such as a service or a new service composition or flow.

        Purpose: The purpose of this past is to facilitate and expedite the process of building a service-oriented architecture by leveraging existing assets rather than creating them from scratch. This particular step involves the attempt to discover and find a relevant asset that can be a candidate for fulfilling a requirement for a given asset type. For example, services for the banking industry may be searched for in a business service repository for the banking industry. If such a service meets the qualifications required by the project, consider incorporating the asset rather than building it from scratch.

        Key Considerations: Service oriented repositories should be searched first in the preference is for automated search capabilities. Thus the organization's governments should consider the procurement of such an asset repository that will facilitate automated search and find capabilities as well as storing information about the usage of the asset.

        Steps are:

        • Search ecosystem - If you have access to eco-system resources (e.g., outside the enterprise) then access those resources and search for a corresponding asset type.
        • Auto-discover asset type using agents - This is part of the automated discovery of asset types if the capability exists.
        • Search/query repository for industry asset type - For every SOMA activity, task or step, inquire as to whether an asset already exists of that type that can be assessed with a fit/gap analysis for use in the current context. It should be pointed out that the Industry Assets/Models support the identification, analysis and design of requirements’ based solutions, irrespective of the target environment. The technology-based challenges of enterprise integration vary with the selected infrastructure. These aspects should be covered as part of Realization decisions. Example of Industry Assets/Models are: (a) IFW for Banking Industry and (b) IAA for Insurance Industry (c) Industry content packs (WBSF) (d) Industry frameworks from IBM Software Group
        • Query repository for specific assets within the selected industry - Industry specific Business Process Models are an abstraction from the specifics of any one organization, in any one problem domain. The models provide the blueprints and standards upon which the specific process and underlying services may be constructed. It is key to select from existing IBM assets the Process Models that appear to be relevant to the current problem domain being considered.
        • Industry asset, functional area analysis, business process definitions - This is to ensure terms and definitions used by client are similar to those used in IBM assets. If they are different then the variations should be identified and documented. If they are the same then it is important to map these two terms with similar definitions.
        • Customize business process (Activities within a Business Process) to meet organizations' need - As part of customization, each activity within the scoped process should be analyzed for its applicability and validity to the project under analysis. This will result in the addition of new activities, and the removal of unnecessary existing activities from the business process. Note: An activity should always produce consistent output triggers in any process context. If conflicting outputs are required, then a new activity should be specified. Each activity should be checked to determine whether it is a manual activity, a front end application task, or a back-end system task Next key aspect of customization of business processes is to identify control and data flows. Control flows bring together the Initiating (Input) and Completing (Output) Trigger Logic analyzed earlier. Data Flows can be deduced from the detailed description of activities.
        • Review of industry asset use cases and compare with project use cases - In order to leverage the business process model effectively, it is important that use cases that constitute or represent the activities of these processes be examined in detail. This is to find what are the similarities, differences and what is missing with respect to actions and data elements (a s well as the types) and rules. Similar use cases can serve as a basis to build project specific use cases if they do not exist. If the project use case are quite similar to Industry Asset use cases we can now leverage underlying BOM (Business Object Model)

        Perform Asset Fit Gap Analysis
        Description: When an asset is being used it seldom fits exactly what the current context: often it must be modified in some way. The fit gap analysis is done to ensure we are leveraging existing assets and in doing so, determining the amount of reuse we can achieve from the current asset. If there is a gap, then we must identify exactly what the gap is and what the fit is, what aspects are a good fit and what aspects will need to be re-configured, customized or extended (new development)

        In order to determine the suitability of using an asset as is within a project called it is necessary to perform a fit gap analysis. It is highly likely that an asset must be customized for a given project. This gap analysis will shed light on the kinds of customizations potentially needed in order to leverage that asset in a given context.

        Purpose: To facilitate the leveraging of existing assets, a fit-gap analysis is done to determine the extent to which a given asset suits a particular project situation or client environment.

        Note that asset customization is a general term used to denote a set of activities that can be performed on an asset of a given asset type in order to make it usable for a given project context. These activities cover a spectrum spanning intrusive (customization) activities all the way to non-intrusive (extension or configuration) activities.

        Should an asset be reused as is or should it be customized in some way? Note that packaged applications are good examples of where a fit gap analysis is done very early in the project. The result of this fit gap analysis will point to the extent to which configuration, customization or extension needs to occur within a project.

        Finalize and Document Asset Fit Gap Analysis Results
        After carrying out a fit gap analysis, finalize the results of the analysis and document the results.
        Make Realization Decisions

        Description: Making realization decisions is an ongoing activity that takes place in each major juncture in the software development process. Realization decisions are made in order to document the key decisions and rationale behind them.

        Realization decisions are made in order to document the key decisions and rationale behind them and that can lead to those decisions in the context of the project.  These architectural decisions define how the architectural solution is being built.  At each juncture there are options to take into account and based on the current context and realities of the project combo choices are based that must be recorded so that future modifications have access to the rationale used in making those initial decisions.  The documentation of these decisions will facilitate future enhancements by identifying key variation points and key options that were available but perhaps not quite mature at that time and have possibly matured at the present time. 

        Purpose: To arrive at and document architectural decisions on how to implement services after considering various alternatives.

        Specify Asset Customizations or Configurations

        Description: Provide a specification for how the identified gap of the existing assets with the target needs of the project are to be bridged. After a fit gap analysis it is often found that the required asset type, such as a business process will have to be modified or customized in order to be useful in the current context. This modification can take on several forms: customization or configuration are two ends of the spectrum in which configuration is least intrusive and customization requires most intrusive changes. Extension is an additive change made to render the asset useful in the given context.

        Subsequent to fit gap analysis, the customization of an asset may be necessary. This customization can take the form of configuration, extension or true customization which consists of intrusive changes that are less desirable to an asset.

        Key Considerations: Whenever a new asset type is encountered, Leverage Existing Asset is used and this task is included as part of that capability pattern.

        Alternatives: Various design alternatives can exist in how to bridge a gap that has been identified.  Budgetary restrictions strategic and tactical alliances and product directions must be taken into account in a realistic fashion.
        Realize Asset Customizations or Configurations

        Implement and test the customization of an asset in implementation phase as defined by the realization decisions as a result of the fit-gap analysis and the specification of the customization done during the specification phase. Sub steps are:

        1. Review Realization Decisions
        2. Review Fit Gap Analysis
        3. Review Existing Assets and Their Capabilities
        4. Review the Specification of How to Bridge the Gap to make asset meet current needs
        5. Implement the Configuration, Customization(s) or Extension(s)
        Key Considerations
        Use the Leverage Existing Assets capability pattern or this equivalent task any time you encounter a new asset type.  For example when conducting service identification first consult an organization's service repository in order to leverage their current service portfolio rather than attempting to build that portfolio from scratch.
        Note that all asset types can potentially be created or maintained as reusable assets: goals, functional areas, functions, domains, services, flows or processes, components, use cases, etc., are all examples of different asset types that have the great potential for reuse if they are stored and maintained in a repository and are readily accessible to be consulted when there is the need for creating a new instance of their asset type.
