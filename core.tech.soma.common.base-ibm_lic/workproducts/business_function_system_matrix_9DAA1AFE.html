Artifact: Business Function / System Matrix (ART 0578)
The Business Function / System Matrix (BF/SM) artifact addresses the identification of associations between business and software functions, and describes architecturally significant technical aspects of each relevant existing business system.
Domains: Functional
Work Product Kinds: Assessment, Inventory, Model
Purpose

The analysis of assets for possible use in an SOA based solution takes place at several levels of detail. One of the most important objectives of the initial high level analysis can be thought of as coarse-grained mapping of business functions to existing IT functions.  By summarizing of the capabilities of existing application components and aligning them with specific business functions, the BF/SM answers the question “What applications or systems support which business functions?”

The Business Function / System Matrix (BF/SM) artifact documents existing business systems that may be used as a mechanism to realize the exposure of selected services. The BF/SM maps the relationship between business functions and the functionality provided by business system components. This information makes it possible to associate services with business system components that can be used to realize the functional characteristics of the service.

The process of mapping business functions to business systems components can also reveal gaps in the service portfolio, leading to the identification of additional candidate services.

In addition to mapping business functions to business system components, the BF/SM also documents technical characteristics of existing business systems, such as nonfunctional requirements of business functions, and the QoS characteristics of business system components. This allows potential matches between services and business system components to be evaluated for non-functional as well as functional fit, and supports assessment of technical feasibility and making architectural decisions related to service exposure.

Description
Main Description

The Business Function/System Matrix (BF/SM) work product addresses two important aspects of the analysis of existing systems:

  •  Identifying associations between business functions and software functions, and
  •  Describing technical aspects of existing systems that are architecturally significant to an SOA-based solution.

Thus, a Business Function System Matrix instance is subdivided into two parts: Functionality Matrix, and Existing System Profiles.

The Functionality Matrix section documents the mapping of business functions, services, and functions in business systems. It also captures important considerations that require decisions to be made, or that influence the level of effort needed to expose the system function as a service.

Existing System Profiles capture key high-level characteristics of systems, such as operating platform, application environment, process-focused integration capabilities, messaging capabilities, and other aspects that are essential to understanding the technical feasibility of leveraging a system to realize services.

Notation

Functionality Matrix

  • Business Function: Name of a business function. The business function is typically named and described in the Functional Area Descriptions work product and Process Definition work product.
  • System Function: The name of an interface or function provided by a component in the system or application that supports the required functionality.
  • System or Application: The system or application that provides this interface. An existing system profile should be completed for each system or application referenced here. Depending on the level of detail available, this could identify a business system, application, or a specific component.
  • Candidate Service: Name of a (candidate) service that could expose this function. Services are named and described in the Service Model work product. Listing the service in this table creates a cross reference link to the Service Model.
  • Realization considerations: The analysis of business system functions may identify issues that affect the ability to expose a function as a service. These issues should be summarized here.

Existing System Profile

  • Business System: <system name>
  • Operating platform: The operating system and hardware platform
  • Application environment: Summarize the application component runtime environment (J2EE server, CICS, etc.)
  • Process focused integration capabilities: Are application components designed to integrate with other external components (or for example are they only used with a user oriented interface). Is this mode of integration through a standard connector and API, through message oriented middleware, through HTTP get/post, socket, …)
  • Messaging capabilities: Are application components designed to be invoked in a message driven mode. If so, what are the general characteristics of the messages (e.g. are messages XML based, structured field, name value pairs, proprietary message design, …)
  • Data environment: Summarize the data store / database characteristics for data that would be of interest to expose to external systems (i.e. Oracle RDBMS, DB2 RDBMS, non-relational data store, …)
  • Datastores/databases: List datastores used by this business system
  • Data focused integration capabilities: Are these data stores being accessed by other systems, and if so how (e.g. through a database connection, though an application component, through
  • Operational schedule: Are applications and/or data available at all times or on a more limited schedule
  • Availability characteristics: What is the availability objective (i.e. 9n.nnn) during the system’s normal operating window.
  • Workload characteristics: What is workload bearing capacity of the system, what is the current level of capacity utilization (i.e. will the system be able to accept new workload)
  • Security characteristics: Summarize issues related to security (e.g. do some/all functions require that the requester be authenticated in advance of making the request, are their security constraints that limit sources of requests, do functions require that the identity of the originator of a request be provided, or only the identity of an intermediate system, will identify mapping/transformation be required, is identity and access control administered at the level of this system or through a separate security administration system, …)
  • Realization considerations: Summarize key architectural issues for a proposed approach to realizing services using this business system.

Key Considerations
Key consideration questions:
  • Does a choice need to be made for which of multiple potential providers of a function should be used as the single provider of a function
  • Are there multiple potential providers that need to be replaced by a unified provider
  • Are there multiple providers of the function that represent functional variations of the function such that a decision for which provider to use needs to be made at run-time based on business rules.
  • Are there multiple providers of the function that represent non-functional variations of the function such that a decision for which provider to use needs to be made at run-time based on QoS related policy criteria
  • Does the business system expose the function using an interface that is easily exposed as a service
  • Does the function have dependencies on other functions such that the functions must be used in a specific combination or specific sequence (i.e. would the function need to be exposed only as a composition with other functions, rather than as a separate functions)

Business systems characteristics can affect the ability to expose functions as services. Each system that is being analyzed should be profiled at a high level so that it is described in terms that can support decisions for how its functions could be exposed as services. Business system functions are expressed in broad terms rather than at the individual function level. If there are certain characteristics that vary widely for various groups of functions, it may be necessary to classify groups of functions and describing distinguishing characteristics of each function group separately.

Tailoring
Impact of not having

Without this artifact the information needed to determine how services can be realized through existing business system components may not be available. The identification of candidate services may also be hampered by the lack of gap analysis that this artifact allows for.

Reasons for not needing

This artifact may not need to be produced if there is already a similar document available detailing the relationship of business functions to business system components.

Representation Options

An alternative way to capture this information is through the combination of an Activity to Application Matrix in the Process Definition work product, along with the Technical Environment work product.
