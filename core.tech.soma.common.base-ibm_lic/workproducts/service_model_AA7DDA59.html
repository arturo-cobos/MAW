Description
Main Description

This artifact and its sub-artifacts capture details about Services including service hierarchy (functional classification), service exposure (exposure classification), service composition and dependencies, and service specification. As an aggregate work product, this artifact captures various pieces of informaton about services in the portfolio, including a name, a short description, a mapping to functional area, process, existing asset, (with the one that led to first identification in bold or with asterisk), exposure state and decisions (Candidate, Expose), etc.

The Service Model is a core work product in SOMA, updated and used by many activities. Service Model sub-artifacts are updated throughout the Identification, Specification, and Realization phases of the service lifecycle.

Brief Outline

This artifact maintains control information about its various sub-artifacts. It maintains the ownership and versioning information' related to the various sub-artifacts that together capture the service model.

Service Model Control Information

  • Define the scope, owner and version for the service model in this section.

Service Portfolio Control Information

  • Define the ownership and completion information of the Service Portfolio sub-artifact

Service Dependencies

  • For each set of service dependencies define the ownership. Each set of service dependencies will be captured in a sub-artifact.

Service Specification

  • For each set of service specifications define the ownership. Each set of service specificaitons will be captured in a sub-artifact.
Selected RepresentationThis information in this artifact can be captured using SOMA-ME modeling tool. The SOMA-ME also aids in the generation of the artifact from the model created using the tool. An MS-Excel worksheet for the Service Model is also defined for engagement team to use in case of SOMA-ME tool is not available.

Tailoring
Impact of not having

Without this artifact and its sub-artifacts it would be difficult to specify the details of the services that have to be realized and implemented and to document the service realization decisions. This would lead to haphazard construction of Services and a set of Services that are potentially poorly matched with the business requirements.

Reasons for not needing

This artifact and its sub-artifacts are not needed if either the services are not externalized, or if the externalized services are very simple and the non-functional requirements for them are not stringent.
