Work Product Descriptor (Artifact): Impact Analysis
The Impact Analysis work product provides an assessment of the likely effects of requirements on the existing IT applications, infrastructure and supporting documentation.

Main Description

At the first elaboration point, the Impact Analysis work product outlines the requirements high level impact on:

  • The infrastructure - all the physical and associated manufacturer supplied components and vendor utilities
  • The functional application - the descriptive documentation and associated programs (of process and controls) developed by a project team to satisfy client requirements
  • The data - all input and output structures and storage utilized by the application system
  • Interfaces - special units of this application system (includes both data and functional elements) which are used to communicate with other application systems beyond the control of the development team
  • Usage and operation - all the documentation which assist users in utilizing the application system and operators to keep the application system available
  • Security & Privacy - the additional risks to other systems within the organization or to this system from others within the organization

Key Considerations

The availability of a requirement statement (such as project scope and goals, service requests, or problem reports) is a prerequisite to conducting the associated tasks that produce this work product. Initial requirements often contain ambiguities of intent and omissions as to expectations. Therefore, in addition to assessing the stated requirements, areas requiring clarification are also included as an integral part of this output. Where the output is considered as incomplete because of a need for further clarification or incomplete requirements, the Impact Analysis task should be repeated when the information becomes available.

  1. Identify the purpose that the system is currently designed to serve. 
  2. Ensure the availability of system reference documentation and key personnel who can confirm assumptions about and interpretations of the requirements. 
  3. Document the resources that you are unable to obtain and indicate the limitations that this lack of resource can have on your assessment. 
  4. Declare the desirable efficiency and effectiveness attributes (e.g. processing speed, ease of access etc.) that your assessment seeks to preserve or provide. This kind of information may be available in either the non-functional requirements or in service level agreement documentation.
  5. Validation should be done to examine the impact of each requirement against each component (as enumerated in the description, e.g., infrastructure, functional application, data, interfaces, usage and operation.)  Criteria for this examination include existence of the specific impact analysis on a requirement by requirement basis, and reasonability of the analysis.
