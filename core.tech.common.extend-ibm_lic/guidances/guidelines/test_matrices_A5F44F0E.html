Guideline: Test Matrices
Test matrices are tables that are used for a variety of purposes in test design. Any cell, which is checked, indicates that it is used or is applicable.
Relationships
Related Elements
  • Test Specification
Main Description

Functional Requirements Matrix - used to document, at a high level, the functional requirements for the application system.  For each functional requirement, there is a: 

  • Reference number
  • Description
  • Objective
  • Evaluation criteria
  • Business priority (i.e., the impact to the business if defects in the area were to be discovered in production).

Structural Requirements Matrix - used to document at a high level the technical requirements for the application system.  For each structural enabler, there is a: 

  • Reference number
  • Description
  • Objective
  • Evaluation criteria
  • Business, development, or operations priority.

Test Conditions Matrix - Test conditions are identified by decomposing a function/sub-function.  These requirements can often be expressed in the form, “If <test condition> then <outcome>.”  Each test condition has a measurable objective and normally requires the execution of several test cases.  The Test Conditions Matrix is a worksheet used to formulate the test conditions that, if met, will produce the expected result.  It is a tool used to assist in the design of test cases.  Each test condition has a reference number.

Test Sets Matrix - A test set is a collection of test conditions.  Test sets are created for purposes of test execution only.  A test set is created such that its size is manageable to run and its grouping of test conditions facilitates testing.  The grouping reflects the application build strategy.  Typically, the group:

  • Is capable of being executed together
  • Runs in batch or on-line
  • Grouping alternatives include:
    • Valid, invalid or no data
    • System or structure related
    • Processing cycles related

Each test set has a reference number.  The Test Sets Matrix is a worksheet that relates the test conditions to the test set in which the condition is to be tested.  Rows list the test conditions and columns list the test sets.  A checkmark in a cell indicates the test set will be used for the corresponding test condition.

Test Conditions Coverage Matrix - This matrix is used for planning, and for illustrating that all test conditions are covered by one or more test cases.  Each test set has a Test Conditions Coverage Matrix.  The Test Conditions Coverage Matrix has a reference number and a test set reference number.  Rows are used to list the test conditions (condition reference number and description), and columns are used to list all test cases in the test set.  Rows without check marks indicate conditions that are not tested.  Columns without check marks indicate test cases that are not used.

Test Requirements Coverage Matrix - This matrix is used to illustrate that all requirements are covered by one or more test cases.  Each test set has a Test Requirements Coverage Matrix.  This same matrix may be used to illustrate that all functions are covered by one or more test cases.  The Test Requirements Coverage Matrix has a reference number and a test set reference number.  Rows are used to list the requirements (requirement reference number and description), and columns are used to list the test cases in the test set.  This matrix can be used to validate requirements or to illustrate test coverage.  Rows without check marks indicate requirements that are not tested.  Columns without check marks indicate test cases that are not used.  The test requirements coverage achieved by all test sets may be consolidated using the same form.  In this case, rows are used to list the requirements (requirement reference number and description), and columns are used to list all test sets.  Rows without check marks indicate requirements that are not tested.  Columns without check marks indicate test sets that are not used.
