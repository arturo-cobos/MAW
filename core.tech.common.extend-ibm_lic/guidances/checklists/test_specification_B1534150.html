Check Items
Functional and structural requirements have adequate coverage
Use static testing techniques to verify the Test Specification. Development and business specialists should review/inspect the Functional Requirements Matrices to confirm adequacy of coverage, and that the business priority classification applied to each matrix statement is accurate. Development and production support should review/inspect the Structural Requirements Matrix to confirm coverage and accuracy of matrix statements.
Test cases have been prioritized
Check that test cases have been prioritized (high, medium or low) based on the work already done in risk analysis and in identifying the test focus areas before testing is started. Should time run short, this practice will identify the highest priority test cases that must be run.
Tests are logically numbered and cross-referenced
The purpose of test design is also to verify and validate traceability against the original business requirements and technical specifications. Check that a logical numbering scheme and cross-references have been used. This practice will ensure that every functional requirement can be traced to specific test cases or, in reverse, that an individual test case can be traced back to a specific test condition under test.
Tests support the original business requirements and technical specification

Test matrices such as Test Conditions, Test Sets, Test Conditions Coverage, and Test Requirements Coverage can be used with the client to validate the coverage of the tests against the original business requirements and technical specifications. For example, use the Test Sets Matrices to check that test cases exist for all test conditions and the Test Requirements Matrices to check that test cases exist for all functional and non-functional requirements.
