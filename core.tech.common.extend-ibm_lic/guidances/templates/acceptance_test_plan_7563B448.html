Main Description
Introduction This section establishes some project level context or background.

Purpose This section describes the purpose of the Acceptance Test Plan.

Scope A scope paragraph gives a clear explanation of the extent of testing covered by this plan. It specifically addresses items that are in scope and items that are out of scope.

Test Focus Document those attributes of a system that must be tested to assure that the business and structural requirements are satisfied.

Test Objectives Objectives for acceptance testing (or for that matter for all levels of testing) must be clearly stated using unambiguous and quantifiable terms that are appropriate for the level of testing. Test objectives establish what testing is to accomplish, define evaluation criteria to know when testing is completed, and assist in gauging the effectiveness of testing.

In order to be able to assess if an output meets or exceeds a given expectation, the expectation must be stated in testable terms. This means that when the characteristics of an output under test are compared with the expected characteristics, they can be matched in a clear and unambiguous way. The answer to a calculation of (1 + 4) should not be “an appropriate amount.” To be testable, the answer would be a single value like 5, or a range between 0 and 10 if the exact answer were not known. If the result of the test were anything other than 5 or a number within the specified range, the test would unequivocally fail and the variance would be recorded. The example is trivial, but it serves to illustrate the point.

While requirements should be stated in testable terms, it may not always be possible. The necessary level of detail when documenting and testing the requirements may not yet be available. For instance, the process of requirements specification and the High Level Physical Process Model should evolve the functional request from a collection of imprecise statements of user needs to testable user specifications. Even though the root of the word specification is specific, documenting testable requirements specifications is an extremely challenging exercise. At every point in the specification process, the “testability” of the requirement or specification can be evaluated by ensuring it meets the S.M.A.R.T. criteria. “S.M.A.R.T.” is an acronym for:

S - Specific
M - Measurable
A - Attainable
R - Realistic
T - Timely

These specifications will form the basis for the criteria upon which the system is tested and ultimately accepted by the system owner and end-user.

Avoid typical problems; e.g., vague statements, unsupported assertions, difficult or impossible test environment, high problem complexity, and conflicting or confused business objectives and systems objectives, etc. It is useful to remember that the system should be tested the way it is used.

Acceptance Criteria Clear and unambiguous criteria specify what will be tested, what tests will be used, how they will be executed, and how success or failure of the tests will be determined. Defining test objectives is the first step to identifying acceptance criteria for the solution. These acceptance criteria should be signed off early in the development lifecycle, as part of the requirements review process.

Acceptance Process Specific steps outline the actual process for managing deficiencies or failure to meet criteria, for adding, changing or deleting acceptance criteria, for recommending acceptance or not, and for sign-off. A standard set of definitions for classifying defects should be consistently used throughout the different levels of testing, defined in the higher level plans. Every test should have expected results clearly defined that indicate whether a particular test has passed or failed. Defect management and test management processes used at the project level should also be used for acceptance testing. Note that acceptance criteria may allow defects below a defined severity level, or provide that they be corrected in a later phase so that implementation can proceed.

Approach The overall approach to how testing will be conducted is defined here. The approach is described in sufficient detail to permit identification of the major testing tasks and estimation of the time required for completing each one. Significant constraints on testing, such as work item availability, testing resource availability and deadlines, are also identified, as are any special packaging procedures. Any assumptions related to testing are also stated here. Staffing needed to support the approach are identified in terms of number and skill level. Training options for developing required skills are noted here as an alternative to hiring the skills.

The testing approach, test data, and testing priorities are defined here and include:

  • Details of any other areas carrying out testing during the test level; e.g., the service delivery, internal audit, operations, business procedures, etc.
  • The approach to be taken for obtaining test data; e.g., live extract, test packs.
  • The prioritization strategy: prioritize the features and items to test, and identify considerations for not running certain tests should the project run out of time.
  • Any boundaries of testing where the level and/or projects are interdependent.

Functions/Features to be Tested Test Functions/Features appropriate to this test level are identified as well as combinations of such to be tested. These functions and features are derived from the use case models and the non-functional requirements, which describe the functions and features to be tested for the project. The definition of what functions and features will be tested at which level of testing is usually arrived at through a workshop, to ensure consistency and completeness.

Test Functions/Features are defined to a sufficiently low level of detail to support an expansion into test cases. The criteria to be used to determine whether each Test Function/Feature has passed or failed testing are also noted here.

Functions/Features not to be Tested Functions/Features being excluded are noted here to identify out-of-scope items.

Work Items Specific work items are defined, that are required to test the functions/features listed in the section Functions/Features to be Tested. In other words, work items are those systems, software configuration items, system interfaces, and documents that constitute the areas or items to be tested. It may be useful, under some circumstances, to combine this section with the Test Functions/Features section.

  • The extent of the breadth and depth of requirements coverage is also defined here.
  • Function and test matrices are often used to ensure completeness and accuracy of work items.

Entry and Exit Criteria for Level Entry and Exit criteria are usually established during the development of the comprehensive test planning stage. Entry criteria establish the prerequisites that must be met before testing can start, and exit criteria define the steps that must be completed before the test level can be exited.

Testing Tools and Techniques Testing tools or special techniques to be employed during this test level are identified here.

Suspension Criteria and Resumption Requirements The criteria used to suspend all or a portion of the testing activity on the Test Functions/Features associated with this plan are defined here. Testing activities that must be repeated when testing is resumed are identified, to ensure that required testing conditions are reestablished prior to the resumption of testing.

Test Plan and Schedule A schedule is prepared for the testing activities at this test level. A reference is provided to a Plan/Gantt Chart or other planning documents in the appendix.

Testing milestones identified in the Project Plan are identified. The specific test activities to be performed at this test level, the personnel responsible for performing the activities, and the test work products to be produced are also defined.

Test Milestones Identified in the Project Plan Milestones should include completion of the test plan itself, completion of test specification, test execution, etc.

Specific Test Activities for the Test Level The set of tasks necessary to prepare for and perform the testing level is identified. All inter-task dependencies and any special skills required are also noted. The following types of tasks need to be considered:

  • Test Planning
  • Allocation and Assignment of staff for the test level
  • Test Design and Specification
  • Test Execution
  • Test Environment Planning and Set Up
  • Test Management
  • Defect Management

Personnel Responsible for Test Activities Staff responsible for the different tasks included in the plan are identified here. These staff may include the business testers, application programmer, technical support, user representatives, data administration, and quality support personnel.

Test Environment Requirements The necessary and desired properties of the Test Environment are noted here. This section commonly includes:

  • Identification of the physical components, the communications, the system and middleware necessary (hardware and system software).
  • The mode or state of usage (for example, standalone or dedicated to the discrete usage of the application alone).
  • Other software or supplies needed to support testing (e.g., printer forms or specialized stationery).
  • Security and access requirements to the test area and equipment.
  • Test tools and utilities required, including backup/restore requirements.
  • Other testing needs (publications, office space).
  • Additional software licenses to cover test period usage.
  • Support resources and skills required to maintain the test environments.

Risks and Contingencies High project risks need to be addressed in this section. The impact and contingency plans for each are defined. All potential risks identified in section, Features not to be tested, are addressed here.

Common areas of risk are:

  • Changing business requirements
  • Changing end-date
  • Insufficient resources
  • Test environment restrictions or differences relative to production
  • Quality problems
  • A need for special skills
  • New technology
  • The focus is on risks to the testing of the project deliverables, and not to the project itself.

Approvals The names and titles of the people who will read, review, or sign-off on this plan are listed here. Space is provided for signatures and dates.
