Purpose
The purpose of this artifact is to:
  • Provide an appropriate degree of accuracy and precision to the task of calculating the required system resources to support a given workload.
  • Ensure adequate predictions can be made for system configuration requirements and so reducing project risk.
  • To assist in the understanding of the behavior of existing systems, allowing performance bottlenecks to be identified, studied and removed.

Description
Main Description

This artifact is a computational aid used to assist in the estimation of the various computer resources that will be required to support a workload imposed on all or part of a computer system.  By performing some kind of computation on various sets of business and IT related input data, it can provide an assessment of both business related system behavior (such as end-to-end response times and overall workload throughput) and system related behavior (such as CPU utilization and disk I/O rates for specific system components).

Performance models can take many forms, ranging widely in precision and accuracy.  Fundamentally, there are two distinct classes of model static and dynamic:

  • Static Performance Models:  The common characteristic of all static models is that they perform calculations based on a constant rate of input behavior, i.e. with no allowance for any dynamic variability in the behavior of the system being modeled.  This type of model is very effective in estimating overall server resource-usage (i.e. average CPU and/or disk space utilization) as a result of a given workload.  They are also effective in estimating average transaction response times or batch job elapsed times in lightly loaded systems, where the effects of contention are approximately linear.  The models themselves can range from simple models constructed from heuristic rules of thumb, to more sophisticated models custom-built using spreadsheet tools.
  • Dynamic Performance Models:  A dynamic model is built using a sophisticated computer program in which different components of the program emulate the different components of the real system.  These interact with each other in real- (or pseudo real-) time, emulating the behavior of the system under investigation.  Large, sophisticated, or high-risk projects may require the use of complex dynamic models built using tools designed specifically for performance modeling.
Brief Outline

As well as presenting, discussing and analyzing the results taken from its various input sources, this artifact typically includes a description of the approach taken to the analysis, any assumptions and exclusions, and recommendations on the various system configuration options available.

Key Considerations

Consider the following:

  • The level of accuracy achievable from the performance model, along with the level of effort required to deliver that accuracy, varies substantially with the complexity of the model employed.
  • The level of accuracy obtainable from the model is directly related to the accuracy of the input data. 
  • The appropriate level of modeling required for a project must be decided, and hence the technique to use, based on project risk.
Tailoring
Impact of not having

Without a well-constructed performance model, built using good input business, system and technology data, there is an enhanced risk of:

  • Sub-standard performance of an IT system. A failure to deliver the required levels of performance once it is deployed, even though some degree of performance analysis may have been performed.
  • Over-engineering of an IT system. With a corresponding unnecessary expense in providing a system which turns out to be under utilized.
  • Capacity planning errors. For both the initial implementation of the system under development and during any growth of the implemented system, there will be no consistent, accurate or validated means of predicting the required capacity of the various components.
Reasons for not needing

It is nearly always necessary to consider the performance of an IT system and whether it will satisfy the performance requirements placed upon it.  However, there are situations when a formal model may not be required, such as:

  • Adequate existing performance data.  If there is sufficient information from previous projects, reference sites or vendors to make infrastructure design decisions, a systems configuration can be developed in which sufficient confidence can be placed without the need to construct a formal performance model.  For example, a package is to be implemented on an RS/6000 R50 4 way processor to support 1000 users.  The operational characteristics of the package are well understood, and an existing similar implementation supporting 2000 users with the same configuration is known to have no performance issues.
  • Low performance risk.  If there is a very low risk of failing to meet the performance requirements, then a formal model to predict performance may not be necessary.  However, there will still be a need to determine the capacity requirements for the system, which will need a model of some kind â€“ albeit probably not as complex.
