Artifact: Usability Test Plan (ART 0656)
This work product is a description of the participants, setting, materials, procedures, and usability test cases for a usability test.
Domains: Test
Work Product Kinds: Plan
Extends: Test Plan
Purpose
The primary purpose of the Usability Test Plan is to ensure that all essential areas that are important for the successful completion of the usability test have been identified and adequately planned.  When it is time to conduct the usability test, the Usability Test Plan provides the solution team with the specifics for conducting the usability test, including participant selection, test setup, and the capturing of participant’s usability data.

Another purpose of the Usability Test Plan is to ensure that there is full understanding and agreement, regarding the specifics of the planned test, between the client and the IBM project team. The plan sets expectations for the client in terms of the breadth and scope of the testing.

Description
Main Description

A usability test is a controlled evaluation of the usability of a solution by a set of representative users. A formal usability test is usually conducted in a laboratory setting, using workrooms fitted with one-way glass, video cameras, and a recording station. Task completion is timed. Both quantitative and qualitative data are gathered. Prior to conducting the evaluation, a formal usability evaluation plan is written. 

The usability test plan is a written document that describes the plan for conducting the usability testing of a design.

The usability test plan is usually is distributed before the evaluation in order to secure approvals from important personnel.
Brief Outline

This work product contains the following sections:

  • Introduction: The Introduction describes the purpose and test objectives of usability testing.  This section also identifies the scope of testing and identifies those user interface design aspects that will serve as the focus of investigation for the evaluation.
  • Methodology: This section describes how the study will be conducted.  It describes participants, participant training, the usability test cases, test data, test procedure, test roles, data collection, and test location and duration. 
  • Participants: Describes the type and profile of test participants.
  • Participant Training: This subsection describes any training the participants may receive prior to the start of the test.
  • Description of usability test cases: This subsection describes the usability test cases for the test.  Approximately 7-12 test cases are generally created for a usability study, derived from use case scenarios.  A usability test case has the following format:
    Usability test case name The name of the usability test case
    Intended user category This section should contain the name of the user category for which this usability test case was designed.
    Related use cases:

    A usability test case might span more than one use case.  Cross-reference those use case here:

    • <Use case number>:<Use case name>
    • Use case number>:<Use case name>
    Starting state: This section should contain a description of the state of the system before this usability test case is executed.  Furthermore, explicitly mention all the assumptions that the usability test participant may make about this test case.
    End state: This section should contain a description of the tasks that the user is expected to perform.  This description should be made in terms of the user’s work or business goals.  The description should not explicitly tell the user how to do the tasks with the specific user interface (that is, do NOT refer to the specific menu items that have to be used), since the goal is exactly to test this.
  • Test Data: Describes the database where sample test data are kept and organized.
  • Procedure: This subsection describes the step-by-step procedure for conducting a usability test.
  • Test Roles: This subsection describes the tasks involved in the setup and execution of the usability evaluation, and identifies the person responsible for executing the task.
  • Data Collection: This subsection describes how data are collected. If data are collected using automation software, the software and its use are described.
  • Test Location and Duration: Describes the location and duration of usability testing. 
  • Measures: This section identifies the types and amount of test data that will be recorded.  Typically, these data are task completion times, subjective measures of participant satisfaction, and error rates.
  • Criteria: The collected data are compared against usability criteria from usability requirements identified earlier in the solution development effort.  
  • Test Dependencies and Responsibilities: This section identifies dependencies that must be met in order for the usability test to be conducted.  In addition, it names individuals responsible for ensuring that the dependencies are met.
  • Reporting of Results: This section describes when and how usability results will be communicated to the rest of the team.

Key Considerations

The following guidance is suggested:

  • Use test matrices in the development of plans to identify and ensure adequate coverage of key testing areas.  This approach ensures that testing time and resources will be used to the greatest effect.
  • Identify all software and hardware dependencies and when they are needed (for example, before the first iteration, after the second iteration) to avoid schedule delays.  There is often significant lead time required if hardware or software needs to be approved, acquired and installed.
  • Keep testing tools in mind when designing test plans to ensure that the selected approach is aligned with the capabilities and requirements of the tool being used.  Failure to do so may result in significant rework of the plan and possible schedule delays, since the tool cannot be changed to accommodate the plan.
  • Sufficient time should be built into the plan for the tool operators to learn the use of tools selected. 
  • Place time limits on tests.  No test should be allowed to drag on without a time limit.
  • Link tests together by means of prerequisites to save complex test setup instructions. 
  • Recognize that as the project progresses, additional types of testing may be identified or new types of testing may be required.  It is often not possible to completely identify all testing requirements in advance.
  • Ensure that the individuals listed in the signoff section are relevant to the level of testing being done; i.e., there is no need to get customer signoff on a unit test.

  • Involve the client sponsor of the usability test early in the review and revision of the plan.  Remember that sign-off and agreement from the client sponsor is critical for the success of the usability test and the acceptance of its findings.  Leave plenty of time to allow reviews and approvals of the final Usability Test Plan. 
  • Much of what appears in a Usability Test Plan is boilerplate documentation and can be quickly and easily built from a template.  Conserve efforts for the development of the usability test cases, which will be the most time consuming aspect of developing the Usability Test Plan. 
  • The use case scenarios provide a wealth of information for the development of the usability test cases.  If information is missing from the scenarios, make sure to review the entire use case model, and work with the use case modelers in identifying deficiencies in the use case model documentation.  The use case models, if built properly, will serve as the primary basis for the development of the usability test cases. 
  • During the preparation of the usability test cases, do not try to develop an exhaustive set covering all possible scenarios of all possible use cases in the system.  Focus your efforts, especially if there is a limited budget, on usability test cases that represent the most frequently executed use cases; for example, withdrawing cash from a bank account may represent more than 50% of the total number of transactions handled by bank employees.
Tailoring
Impact of not havingBy not writing a Usability Test Plan, a usability test may be conducted that is invalid or focuses on the wrong aspects of the solution. Consequently, key areas of the user interface will not be evaluated, resulting in a difficult to use system and low client satisfaction when the solution is deployed. This could lead to extensive rework for developers on a subsequent release or potentially the termination of the team from the project.
Reasons for not needingA Usability Test Plan is not required if a usability test will not be conducted. This could occur in projects with little or no user interface design or for internal code releases of existing systems.
Representation Options

As an alternative to formal documentation, you might choose to record the elements of this artifact as a set of informal planning notes, possibly maintained on an Intranet Web site or whiteboard readily visible to and accessible by, the test team.

Optionally, some aspects of this artifact can be presented appropriately as enclosures within other plans, rather than as separate discrete artifacts.
