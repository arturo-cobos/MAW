Artifact: Test Findings (ART 0551)
This artifact summarizes the analysis of one or more tests, providing a relatively detailed assessment of the quality of the tested items and the status of the test effort.
Domains: Test
Work Product Kinds: Measurement
Purpose

The purpose of this artifact is to:

  • Record the detailed findings of the test effort and subsequently calculate the different key measures
  • Show status against test and quality targets at any point in time, at specific milestones, and/or at the end of testing
  • Present an objective quality evaluation and assessment to support decision to proceed to the next level of testing or to project implementation by:
    • Summarizing the test results
    • Analysing trends
    • Predicting the level of quality of the test product to date
    • Identifying flaws or gaps in the testing process and plan
  • Explain any anomalies, such as moving to another test level without completing exit criteria, or not moving there when it appears that such criteria were met
  • Determine whether testing at a particular level has been satisfactorily completed

Description
Main Description

This artifact includes information on the following:

  • Scope of testing - any out-of-scope situations are also documented and refer to the risk management process as documented in the test plans
  • What was done and not done versus plan
  • Test results
  • Findings on:
    • Functions working
    • Functions not working
  • Conclusions
  • Quality of software - planned vs. actual
    • Test metrics 
    • Results
  • Issues outstanding
  • Recommendations and action plan

Additionally, this artifact includes analysis of the testing results, any statistically valid conclusions that can be drawn from these analyses are used to predict the quality level achieved by the tested product and compared to the target level established in the test plans.

Some of the factors used to judge the risk are:

  • Number of defects found
  • Severity levels
  • Status of defects - outstanding, investigated, explained, fixed, retested, accepted, etc.
  • Rate of arrival; i.e., defects found per test executed and/or per tested product component, by test period and level
Brief Outline

Executive Summary
This is a concise statement of progress against plan to date with status against quality targets, including risk predictions and recommendations for progression to the next level of testing or implementation. 
 
Test Execution Summary

Testing Overview
This overview is a high level description of the key components of the test strategy, plan and approach.  It will be tailored to the scope of testing being reported; that is, a point in time, a test level, or the entire test project. 

Test Summary
This is a synopsis of tests run to provide a perspective on what quality this set of tests can be expected to prove. 

Test Results Summary

Test Results Overview
This overview is a condensed version of the results.  The objective is to illustrate the completeness of testing to date.

Defect Summary
This summary answers how many variances were found, which are defects, of what severity, and at what state of repair.

Defect Fix Summary
This summary provides information on number of defects that can be considered completely resolved.

Analysis and Conclusion

This section shows what conclusions can be drawn based on the data presented:

  • How many defects are outstanding?
  • What is the arrival rate of new ones?
  • What hot spots of defects have been discovered? 
  • What are the implications of these findings?

Test Summary Report Approval

If approval to proceed or take other action is being requested, identify individuals, process and timing.

Key Considerations

In order to be most effective:

  • Be very specific concerning statements of problems and the rationale behind statements. 
  • Review with the engagement team recommended solutions to problems, before they are included in the test report.
  • Write the report so report recipients can understand the test method, procedures, findings, and recommendations. 
  • Ensure that important members of the engagement team have reviewed the Test Report before it is presented to the client. 
  • Include detailed data in appendices, not the body of the report.  This will make reading easier and more focused. 
  • Prepare separate presentation graphics if the test results are to be shared with the report recipients verbally. 
  • Have client and project team sign-off the report on contentious engagements.
Tailoring
Impact of not having

This artifact is essential to specify the current state of the product quality and to detail what actions are required to meet quality targets. 

This artifact is key to communicating with stakeholders, without its summary of findings and risks, showing the progress against quality measures as defined in the detailed test plans, there are significantly greater opportunities for misunderstandings. These misunderstandings may lead to reduced product quality and/or reduced satisfaction.

If no formal test reporting is to be done, frequent and consistent informal communication is necessary, particularly of defect management activity and its implications for the project. It would be wise to keep an informal record of these communications as well.

Reasons for not needingIn small or very automated testing projects, there may be less need for formalized reporting at any point in time or at milestones, although a conclusive report should not be omitted. In these circumstances, it is assumed that informal communication or automated reporting will provide the client and project team with a clear picture of test status.
Representation Options

If you are using automated test tools, a lot of the determination of the above information can be simplified by the analysis capabilities of the tools. The tools also offer saved and customizable views of the information for reporting purposes.
