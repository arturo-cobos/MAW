Artifact: Deployment Unit Model (ART 0525)
This artifact describes the placement of application components (including data produced by those components) on to nodes of an IT system and their interdependencies.
Domains: Operational
Work Product Kinds: Specification
Purpose

This artifact describes;

  1. The placement of a solution's functional components (hereafter referred to as "components"), as represented by deployment units (DU's), across the nodes of an IT system,
  2. The interdepenencies of the DU's in order to determine the connectivity requirements between the nodes.

The placement of the DU's must be made such that all non functional requirements including those of, but not limited to, security, network and system management can be met.

Description
Main Description

Deployment Units allow for the aggregation of components that have similar service level requirements. The Deployment Units contained in the Deployment Unit Model are defined separately in the Deployment Unit Specification artifact.

In order for an operational model to achieve the solution's functional and non-functional requirements, it is invariably necessary to locate:

  • The various aspects of each component in different locations.  For example, the data aspects of a component may be held centrally, while its execution takes place on a workstation and its executables are stored on a departmental file server.
  • A particular aspect of a component in multiple locations and/or location types.  For example: (a) under normal circumstances a component may execute on a server, but in the event of network failure the same component may be required to execute on the workstation, or (b)  the primary copy of a component's data may be held on a central server, but an operational copy – or replica is available on the workstation.

As a common element linking the component and operational models, placement of the DU's often highlights other needs that must be considered such as;

  • Components may need to be separated by zone boundaries to address risk management concerns e.g., highly trusted components may be separated from low trust level components or separation may be required to address the Privacy implications of storing specified data together.
  • Deployment units for flow control may be required to secure data flows separated across different zones. 
  • Different deployment units may also be needed for different trust environments to fullfill the functional security requirements within the different trust environments.
  • System Management zones of responsibility may dictate that certain components may need to be placed on specific nodes to be managed to a defined SLA.

Generally, these placements decisions relate to all application and technical components and are achieved through the proper placement of the DU's. Therefore for any given solution there may be many DU relationships and interactions to be documented within this artifact.


 

Brief Outline

This artifact typically includes:

  • Component to DU relationship tables
  • DU to DU relationship tables
  • DU to DU interaction requirements
  • Deployment Unit Specifications - see the Deployment Unit Specification artifact
Notation

The Deployment Unit model is represented using tables and text documents. The notation used within those documents must adhere to the standard prescribed by the System Description Standard.

  • For each component the required DU's should be identified in a table.
  • Each DU to DU relationship should be described including the nonfunctional requirements that interconnection.

Key Considerations
  • Deployment Units (DU's) once defined, are placed onto nodes in order to determine the non-functional requirements that node must support. Those nodes are located in locations which are separated by borders.
  • This is a simple yet important concept to understand when developing the DU's. Where as DU's represent the non functional requirements of the various aspects of a component, there will be a need for the DU's of a component to communicate. If those DU's are placed on different nodes in different locations, the communication requirements will define the NFR's of the network connection between those nodes and locations.
  • The identified communication requirements can then be used to validate that the current network can support the required communications or identify potential gaps in the current network design.
  • System Management requirements can also be identified at a DU level which will translate into physical requirements that may impact the solution’s physical design and implementation
Tailoring
Impact of not having

Not documenting a Deployment Unit Model can lead to:

  • A loss of information about the component non functional requirements and the relationship between the DU's that represent these NFR's.and a failure of the operational model in achieving these NFRs
  • Increased complexity during placement due to the lack of understanding of the DU interaction requirements
  • Increased cost of the solution, caused by inappropriate "over engineering" due to inappropriate  placement decisions or lack of appropriate connection requirements.
Reasons for not needing
  • If the operational model is trivial, this work product need not be produced.
  • If the relationships between components and nodes are simple, the allocation of components may be recorded directly on the operational model without using intermediary Deployment Units or a Deployment Unit Model. Great care will be required though to ensure the different aspects of each component are understood and all questions considered.
  • Under certain circumstances, such as when defining certain types of generalized OM, it is valid to focus purely on the location of execution of components.  In these cases, it is valid to place components directly onto the OM’s nodes, since the placement is implicitly related to execution only.
Representation Options

This artifact can be documented in an electronic document, spreadsheet or tool and notated using IBM's System Description Standard (SDS). It is recommended to leverage tooling whenever possible due to the complex task of maintaining the various relationships and connections between the DU's.
