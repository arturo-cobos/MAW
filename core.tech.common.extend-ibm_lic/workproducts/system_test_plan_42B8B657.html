Description
Main Description

This artifact is created for each level of testing identified as necessary based on project life cycle testing requirements.  Each testing level represents a known level of physical integration and quality of the system solution under development.

Project needs - based on size, complexity, risks and costs - determine the levels of testing to be used. The following are commonly used levels of tests:

  • Unit Test
  • Integration Test
  • System Test
  • Acceptance Test
  • Operability Test
  • Systems Integration Test

For each level of testing, types of testing are selected based on the attributes of the application/system to be tested.  Functional types of tests are used to test business functions and structural types of tests are used to test the technical requirements.

Types of functional tests:

  • Audit and Control
  • Error Handling
  • Interface/Intersystem
  • Transaction Flow
  • Usability

Types of structural tests:

  • Backup and Recovery
  • Performance
  • Stress
  • Volume
Brief Outline

The Test Plan for each level of testing is created using a different set of steps and techniques, with the plan content being adjusted accordingly.  Some key sections of the Test Plan are:

  • Test Objective
  • Functions/Features to be Tested
  • Functions/Features not to be Tested
  • Work Items
  • Entry and Exit Criteria for Level (if not defined earlier)
  • Testing Tools and Techniques
  • Test Schedule
  • Test Environment Requirements
  • Risks and Contingencies

Key Considerations

The following guidance is suggested:

  • Use test matrices in the development of plans to identify and ensure adequate coverage of key testing areas.  This approach ensures that testing time and resources will be used to the greatest effect.
  • Identify all software and hardware dependencies and when they are needed (for example, before the first iteration, after the second iteration) to avoid schedule delays.  There is often significant lead time required if hardware or software needs to be approved, acquired and installed.
  • Keep testing tools in mind when designing test plans to ensure that the selected approach is aligned with the capabilities and requirements of the tool being used.  Failure to do so may result in significant rework of the plan and possible schedule delays, since the tool cannot be changed to accommodate the plan.
  • Sufficient time should be built into the plan for the tool operators to learn the use of tools selected. 
  • Place time limits on tests.  No test should be allowed to drag on without a time limit.
  • Link tests together by means of prerequisites to save complex test setup instructions. 
  • Recognize that as the project progresses, additional types of testing may be identified or new types of testing may be required.  It is often not possible to completely identify all testing requirements in advance.
  • Ensure that the individuals listed in the signoff section are relevant to the level of testing being done; i.e., there is no need to get customer signoff on a unit test.
Tailoring
Impact of not having

Higher level plans for testing, such as a master test plan, are not executable, as they do not have detailed level activities or tasks. Without task level details, resources and time schedules to carry out those tasks, it will be difficult, if not impossible, to execute, track, manage and control them. Different levels of tests focus on different areas of functionality and risk, and different organizational units may own responsibility. Thus without detailed test plans organized around your main test cycles or iterations, it will be difficult for different organizational units to plan, prepare, execute and report testing without actually increasing risk. Eliminating detailed test plans, especially for large projects, will virtually guarantee project failure.

The creation of detailed test plans is also the activity that is most often underestimated in projects. It must start as soon as pre-requisites are available to ensure that testing requirements and dependencies are identified early and are addressed appropriately. Delaying the start of this activity often leads to project schedule overruns and increased business or technical risks.

Reasons for not needing

If the main factors driving the granularity of your test plans are test levels and test types, then not all these factors are relevant during the duration of a project and therefore some specialized test plans will not be developed or will be limited to just a few test cycles or iterations. In cases where projects are small, with well-defined scope and structure, and all project members are co-located or work closely together, there may be one joint test plan that covers the high level plan and all detail levels of testing.

Representation Options

As an alternative to formal documentation, you might choose to record the elements of this artifact as a set of informal planning notes, possibly maintained on an Intranet Web site or whiteboard readily visible to and accessible by, the test team.

Optionally, some aspects of this artifact can be presented appropriately as enclosures within other plans, rather than as separate discrete artifacts.
