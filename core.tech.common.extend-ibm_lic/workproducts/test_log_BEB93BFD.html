Artifact: Test Log (ART 0555)
This artifact captures the raw output during a unique execution of one or more tests, and compares the expected results to those of the test.
Domains: Test
Work Product Kinds: Measurement, Record
Purpose

The purpose of this artifact is to:

  • Capture raw data to be analyzed determine the results of some aspect of the test effort
  • Provide verification that a set of tests was executed
  • Provide information relating to the success or failure of those tests
  • Document any variances and their resolution

Description
Main DescriptionThis artifact provides a detailed, typically time-based record that both verifies that a set of tests were run, and provides information that relates to the success of those tests. The focus is typically on providing an accurate audit trail, which enables you to undertake a post-run diagnosis of failures. This raw data is subsequently analyzed to determine the results of an aspect of the test effort.

This artifact also identifies and documents any variances that indicate further analysis is required. Variances are classified according to severity and dealt with accordingly. Analysis of the variance may indicate the cause of the problem to be a defect or may result in a change request. If it is caused by a defect, an impact analysis is required to estimate the effort and time to fix it. The resolution is documented. If the analysis results in a change request, the variance is closed and the resolution is documented.

Test results are usually associated with dynamic testing where the results may be captured and defects logged electronically, often with the aid of test tools. However, test results may also be produced as part of static testing.

Brief Outline

Each Test Log should be made up of a series of entries that present an audit trail for various aspects of the test execution, typical fields would include:

  • Test Level: The possible values for this field are unit, integration, standalone system, networked system, UAT, implementation and operability, etc.
  • Test #: The number of the corresponding test case.
  • Regression Test #: The number of the corresponding regression test case.
  • Application: The name of the application being tested.
  • Category: The type of test.  It is taken from the test case.
  • Reported By:
  • Date Reported: The date and time stamp of when the event occurred
  • Severity: The severity level of the incident. 
  • Causal Analysis: This field indicates where the error was introduced into the development cycle. Some of the possible values for this field are: Code, Design and analysis, Documentation, Ease of use, Environment, New requirements, Process engineering, Scope of work, Testing, or User requirement
  • Problem: A description (usually brief) of the event logged including some indication of the observed status
  • Details: Additional details relating to any anomalous or erroneous condition detected
  • Version: The version of build of the code where the incident was found.
  • Assigned to:
  • Date Assigned:
  • Incident Analysis:
  • Ready for Retest: The date assigned by the developers that this incident will be ready to be re-tested.
  • Retest Status: The values for this field are accepted or rejected.  This indicates whether the incident was found to have been fixed, or not, upon re-testing.  If rejected, the incident report would be placed back into open status and reassigned to the developer.  If accepted, the incident report would be placed into closed status and signed off by the tester.
  • Incident Status: Some values for this field are:
    • Open
    • Closed
    • Ready for re-test: The incident has been fixed but has not yet been re-tested
    • Pending: The incident cannot be completed at this time due to the circumstances outlined in the comments section
    • Deferred: The incident will not be run until another level, as indicated in the comments section
    • Cancelled: The incident will not be fixed at all as, indicated in the comments section
  • Comments: Additional contextual information where relevant
  • Sign Off By:
  • Date:

Key Considerations

The test results should be monitored closely and any variances should be analyzed to identify the cause of the problem.  This might indicate hot spots or problems in specific areas.  For instance, if the same or similar defects are discovered repeatedly, it should be investigated further which might indicate:

  • Poorly defined requirements, specifications, etc.
  • Poorly understood requirements, specifications, etc.
  • A lack of appropriate skills.
  • An improper application of test level entry/exit criteria.
  • A lack of resources to fix the defect the first time.
  • A poor, or poorly understood, defect management process.

Hot spots should be addressed with remedial action as soon as possible.

Contractual and legal requirements should be identified early to ensure that the data collected is adequate to support those requirements. 

Tailoring
Impact of not having

Without this or similar documentation, there is no record of which tests were run, what variances were discovered, and what action was taken. If this information is not available:

  • There is no way to know which tests passed and which failed
  • There is no way to assess the status of testing and the quality of the product at that level of testing
  • It is difficult to know how many tests remain outstanding
  • Contractual and legal issues might arise

If all test results will not be logged, extra care should be taken to document and file test procedures, any outcome files or reports, and at least any variances from expected results.
Reasons for not needing

When you execute automated tests, test logs are automatically produced. Typically, the issue is not whether to produce the test log, but whether to keep a record, and where to keep the records. For manual testing, the issue is whether to keep a separate test log or to summarize the test results in another form.

Without documented test results, testing has little value.  The only situation where this may not be formally done is when ad hoc testing is conducted.  In this case also, any problems that are found will be recorded along with their resolution.
