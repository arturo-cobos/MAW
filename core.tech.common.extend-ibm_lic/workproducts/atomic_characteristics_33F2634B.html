Artifact: Atomic Characteristics (ART 0528)
This artifact documents units of work and associated cost information.
Domains: Operational
Work Product Kinds: Measurement
Purpose
The purpose of this artifact is to provide unit cost information so that the service level characteristics of the target IT system can be estimated.

Description
Main Description

The atomic characteristics captured in this artifact refer to the various units of work, and their associated costs, that are used when describing and measuring the qualities of an IT system.  It is important to know these characteristics when building the various service level characteristic models (e.g. performance model, security model etc) that are used to validate the architecture of the IT system. Some examples of atomic characteristics which may be relevant for different types of service level characteristic are:

  • CPU cost per business transaction – needed for building performance models.
  • Mean time between failure – needed for building availability models.
  • Database rows per entity – needed for building capacity models.
  • User authentications per hour – needed for building security and performance models.

Atomic characteristics will naturally vary by the type of technology used so this must be known in advance if realistic atomic characteristics are to be captured.

Brief Outline

The specific notation used within the artifact will vary depending on the level of granularity to which the cost information is being captured.  For example, the following units may be appropriate:

  • CPU usage per package user
  • Memory usage per package user
  • CPU cost per specific business transaction
  • CPU cost per specific technical transaction
  • Network traffic per specific technical transaction
  • CPU cost per SQL operation
  • Number of I/Os per SQL operation
  • Number of database rows per entity
Key Considerations

The following should be taken into account when creating this artifact: 

  • Where a perfect workload and environment match cannot be obtained talk to product specialists and vendors to understand how different environments may effect the cost information being used.
  • Cost data for various units of work can often be obtained from product developers and vendors however treat these with caution as they are often highly context sensitive and may be completely different outside this context.  Ensure therefore that the costs associated with the various units of work relate both to a workload and environment that are representative of the system being built.  Consider not just the volume of usage, but also the type and spread of usage.
  • The effort required to collect the information for this artifact will vary significantly with the level of modeling that needs to be performed.  If the information can be collected as a result of technology research alone, this activity can be completed in a matter of days.  However, if extensive prototyping is necessary to build this information the elapsed time can extend to weeks or even months depending on the scope of the information required.
Tailoring
Impact of not havingThis artifact is a key input to the service level characteristic modeling that needs to be done when proving an architecture can meet its service level requirements.  Without an understanding of the costs associated with the business system, it is impossible to estimate with any confidence the size and type of system components(s) required to support the given characteristic.
Reasons for not needingThis artifact is always required.  The required granularity of information within this artifact will however depend on the amount of risk associated with the project, whether the system is package or bespoke development based, and at what stage within the overall project lifecycle service level characteristic modeling is being performed.
Representation OptionsThe information that can found in this artifact will depend on the units of work that are being collected and is typically recorded as a spreadsheet or tables in a document.
