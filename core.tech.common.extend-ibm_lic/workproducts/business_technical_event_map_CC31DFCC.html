Description
Main Description

This artifact is used to identify the technical profile of the individual business processes that make up the application.  Examples of the types of information comprising such a technical profile for a specific business process are:

  • An assessment of the activity by system component (e.g. workstation, application server, database server, etc.) associated with the execution of the business process.  This assessment could include:
    • The number of data retrieval requests by type generated by the process.  This could be the number of SQL calls by type (e.g. SELECT, FETCH, UPDATE, etc) issued to the database server, the number of HTML page requests issued to the WEB server, the number of file I/Os by type (e.g. Sequential READ, Random READ, etc.) issued to the file server, etc.
    • The number of data store requests by type generated by the process.  This could be the number of SQL INSERT calls issued to the database server, the number of file Write I/Os issued to the file server, etc.
    • The processing cost involved in composing, and subsequently post processing the result set data from, the data retrieval requests.  This will be estimated from an assessment of the application, middleware, and OS elements of that overall cost.
    • The processing cost involved in composing the source data associated with the data store requests.  This will be estimated from an assessment of the application, middleware, and OS elements of that overall cost.
  • An assessment of the number of message pairs exchanged between the individual system components during execution of the process, along with an estimate of the typical message sizes involved.
  • An assessment of the working set size required by component to support execution of the business process. 

The technical volumetrics for a given business process are then derived from a combination of the business volumetric information captured within as non-functional requirements, along with the technical profile information captured within this artifact for each of those business processes.

Notation

It is recommended that the information within this artifact is presented in tabular form against each associated node and connection.  An example is shown in Figure 1.  For each node and connection, the business process should be broken out by stage, where each stage consists of a specific user - system interaction.  For each stage establish a profile of the use of services (e.g. SQL transaction, message handling etc).  Against each stage record information such as data set size and I/O profile.  This is done for both transactions and batch programs.

Key Considerations

The following considerations apply when creating this artifact:

  • Employ the 80:20 rule.  Given the amount of time that can be required to identify the business and technical events for even a single process, it is important to focus only on the most significant processes.  These transactions/use cases will usually have at least one of the following characteristics:
    • High volume
    • Important to the business and users (for example, large exposure to customer dissatisfaction or financial loss, critical to business operations, high payback, etc).
    • Have critical response time or throughput requirements.
    • Are part of the project Acceptance Test criteria
  • Potentially high resource usage - based on an 'informed' or 'educated' guess (remember that 'resource' can be more than just processor usage - it can include memory, storage, network etc).
  • It is important to identify the level of detail necessary within the business and technical event maps.  This will be driven by the risk associated with the project in not meeting a particular service level requirement.  The required detail will vary with the level of accuracy required from the service level characteristic modeling activity.
Tailoring
Impact of not havingWithout well-understood set of business and technical event maps, the various service level characteristic models are unlikely to provide an accurate estimate of the resources required by the system.  In such cases, there is therefore an increased risk that a system based on those requirements will fail to deliver the service level requirements once deployed.
Reasons for not needing

This artifact is normally only required for systems which involve a significant degree of bespoke development, or in situations where there is significant risk associated with not meeting one or more of the service level requirements for the system.  It is not usually required for pure package based implementations, or in cases where the level of bespoke development is minimal. 

In the latter cases the System Wide Requirements combined with the Atomic Characteristics alone will provide the necessary inputs to building the service level characteristics model.
