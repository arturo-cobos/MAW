Artifact: Test Execution Plan (APP 137)
This work product uses the test specification and design details to lay out what tests will run, in what order, with what infrastructure, environment and data support, and who will run them.
Domains: Test
Work Product Kinds: Plan
Purpose

The Test Execution Plan is used to:

  • Clearly identify the components required to effectively execute the tests
  • Schedule the test runs and resources required for them
  • Set up the defect tracking logs
  • Complete and close off testing activities
  • Provide this detail for each level of testing

Description
Main Description

The Test Execution Plan refers to the test management procedures that will be used to execute, track and report the test results. It initiates the processes and defect logging tools.

Brief Outline

For each level of testing:

  • Introduction: Outlines the activities performed and the resources needed.
  • Environment: Describes the activities performed and resources required to set up the environment uniquely for this level.  Overall test environment detail for all levels is covered in the test environment work product.
  • Execution: For each test set and test case, describes the activities performed and resources needed to perform them.
    • Setup: Describes the tasks and resources needed to setup the tests.
    • Run: Describes the tasks and resources needed to physically run the tests.
  • Logging: Describes the activities and resources needed to record the details of the test runs.
  • Closing: Describes the activities performed and resources needed to shutdown the testing activity.
  • Risk, Issue, and Change Management: Using a project point of view, describes the process of risk, issue, and change management.  These processes are best defined at the project level, and only referred to here.  (The defect management procedure provides specific guidance on rerunning tests and managing changes resulting from non-defect test result variances.)

Key Considerations

The Test Execution Plan is a part of the intellectual capital created during testing, so reuse of part or the whole work product is encouraged.  Experienced testing organizations, CMM SEI maturity levels 2/ 3, will have standard templates and examples that can provide guidance and shortcuts.  However, reuse of this material should be done with caution and only after careful review of any special requirements of this particular test effort. 

As the Test Execution Plan is based on test environment, detailed test plans and test specification and design work products, it is critical that proper configuration management keeps all the documents involved in sync.
Tailoring
Impact of not having

The Test Execution Plan is to test performance what the project plan is to a project. The impact of not having this work product will be qualitative and quantitative.

On the qualitative side, not having a Test Execution Plan could cause incomplete or inconsistent test results by overlooking some tests or duplicating others. Some test cases may be run multiple times, and some others not run at all.

On the quantitative side, lack of a Test Execution Plan will impact testing the same way the lack of a proper project plan will impact a project. Testing activities will be performed ad hoc, without proper preparation, and therefore consume too much project resource.

Testing activities will become unique to each test level and project. There will be no way to duplicate/reuse/recycle them in the future. This condition will cause the testing to be implemented but not institutionalized.

If the Test Execution Plan is not produced, its contents have to be accommodated in some form - in other testing plans or documents.

Reasons for not needingIn case of very simple testing for small, low risk projects, there will always be temptation not to prepare the Test Execution Plan or to run the testing ad hoc. In these situations, two scenarios could be considered. In the first case, an abbreviated Test Execution Plan could be a part of a specific detailed test plan. In the second case, some sort of Test Execution Procedure, even an informal, handwritten one, should be put in place and reviewed to ensure that the team all understand what will be performed, how, and when.
Representation Options

The Test Execution Plan takes the form of a textual document, usually created in a word processing program, and accompanied by a detail project plan (or its part) created using any of the many available project schedulers.
