Artifact: Deployment Unit Specification (ART 0526)
This artifact describes a deployment unit or a set of deployment units.
Purpose

The purpose of this artifact is to simplify the placement of components onto nodes. This simplification is achieved by grouping components (and their different operational facets) into deployment units significantly reducing the number of things that need to be placed and the level of complexity in developing the operational architecture aspect of an IT system.

This artifact assists in mapping the non-functional requirements onto nodes and determining the connections required between nodes to support interactions between components.

This artifact provides a bridge between the functional and the operational aspects of the architecture of an IT system and provides a mechanism for negotiations and discussions between the different roles responsible for developing these two architecture aspects.

Description
Main Description

This artifact’s terminology is defined within the System Description Standard R3 Semantic Specification (see Guidance for link) which includes definitions for:

  • Actors (and workers) – the roles a user or an external system play with respect to the target system
  • Components - modular units of functionality, which make this functionality available through an interface.
  • Deployment Units – a grouping of facets of a component that have similar characteristics.
  • Connectors – enable the exchange of messages (interactions) between resource containers.
  • Interactions – identify the messages exchanged between one or two resource containers in the context of a collaboration, and the sequencing of these messages via their associated send/receive events.
  • Nodes - a collection of components fulfilling a specific responsibility with a certain quality of service within the target system.
  • Connections - supports the required connectivity between connectable model elements.
  • Zones – an aggregation of a number of model elements with a common (sub-) set of values for a particular non-functional requirement and/or non-functional characteristic.

A deployment unit (DU) is used to represent the various operational facets of components including their non-functional requirement’s and service level characteristics.

Deployment units may represent both application and technical components.

A deployment unit is considered as a single unit for requirements analysis and placement considerations.

A deployment unit associates or references additional information with the different facets of a component (e.g., resource requirements, service level requirements, and technical dependencies) that are either relevant or related to the placement of the component onto the nodes of an IT System.

Components defined within the functional aspect of the architecture of an IT system have four different operational facets:

  • Presentation; the presentation between the IT system and its users and/or between the IT system and IT systems, as delivered by the components responsible for presentation to these external actors.
  • Execution; the operational execution of the component within the IT system (i.e where the execution code that implements the component will be loaded as a process and run on an IT platform central processing unit) and the associated resource demands such as “transactions per second” or “required response time”. 
  • Data; the data the component uses together with it’s persistence characteristics such as “size of record”, “number of records”, rate of change”, or “currency”
  • Installation; a component’s installation files, such as its .EXE and .INI files (i.e. where its executables and configuration/context data will be stored), together with its installation characteristics such as “executable size”.

A component may have any one or all of the above facets depending on what the component does.  But, all components must have an Execution facet since all components perform at least one action when implemented in an IT system.

Deployment units (DUs) are groupings of these operational facets. The deployment units aggregate the requirements and characteristics of the grouped component facets.

Groupings are made for a particular facet type and therefore deployment units are associated with a specific facet:

  • Presentation DU; groupings of the presentation facets of components. 
  • Execution DU; groupings of the execution facets of components
  • Data DU; groupings of the data facets of components.
  • Installation DU; groupings of the installation facets of components.

Depending on the development approach adopted and/or the context, it can be very helpful to create levels of this artifact.  The two basic levels of this artifact are:

  • Logical:  This level describes the characteristics and capabilities of deployment units in a technology independent and product neutral manner.
  • Physical:  This level describes characteristics and capabilities of deployment units in a technology and product dependent manner.

Both levels of this artifact may be further described at three different sub-levels relating to the degree to which the level has been sized:

  • Unsized (or unranged) – the technology and products are specified to support the components deployed but not sized and without detailing how the system will achieve the service level characteristics.
  • Ranged – the technology and products are specified to support a bounded range of requirements for a defined range of circumstances.
  • Sized – the technology and products are specified and sized to support the components deployed and details how the system will achieve the service level characteristics. 
Brief Outline

This artifact typically includes:

  • An introduction, describing the overall context and content of the artifact, the nature of the artifact its scope and depth of detail, its level, and its degree of completeness.
  • A list of deployment units
  • A structured description of the deployment unit(s) including
    • Type (e.g. execution, data, installation or presentation),
    • The list of components that have been grouped into the deployment unit
    • The non-functional requirements associated with the deployment unit. The non-functional requirements recorded depend on the type of DU and the specific project circumstances.  In general, service level requirements must be documented.
    • For the physical level of the artifact, the implementation of the deployment unit. Specific technologies or products should be defined, including product name, version number, and pre- and/or co-requisite products.
    • For the physical level of the artifact, the atomic resource requirements of the deployment unit. For example:
      • An execution DU, representing the execution of a component’s operations implemented in a particular technology, will require a certain number of CPU milliseconds for one execution.
      • A data DU, representing the data aspects of a component implemented on a particular database technology, will require a certain number of kB per record.
      • An installation DU, representing the executable of a component implemented in a particular technology, will require a certain number of MB per executable file.

Key Considerations

The non-functional requirements and the characteristics for a deployment unit may not necessarily be the same as for the individual components represented by the deployment unit.  They may be more or less demanding – although it is recommended that an explanation of the difference between requirements and characteristics be provided. 

If there are no differences, and component splitting and/or grouping has been simple, a reference to the appropriate component description will suffice.

Generally, an IT system’s non-functional requirements and, in particular, its service level characteristics of performance, availability and security may be associated with a particular type of deployment unit, representing the facet of the individual components from which the solution’s architecture is constructed.  For example:

  • A required workload, described in terms of “transactions per second” or “visits per hour” is associated with an execution DU, representing the execution facets of the involved components, and helps in deriving the CPU sizing of nodes
  • The need to support a specific number of customer records, each of a certain size, is associated with a data DU, representing the data facets of the involved components, and helps in deriving the disk capacity of nodes
  • The need to provide a mobile user with a graphical user interface is associated with a presentation DU, representing the presentation facets of the involved component(s), and helps in deriving the selection of a hand held terminal (HHT).

The placement of deployment units on nodes is within the operational aspect of the architecture of the IT system.  In order for an operational model to achieve the functional and non-functional requirements for the IT system, it is invariably necessary to locate:

  • The various facets of each component in different locations.  For example, the data facets of a component may be held centrally, while its execution takes place on a workstation and its executables are stored on a departmental file server.
  • A particular facet of a component in multiple locations and/or location types.  For example:
    • under normal circumstances a component may execute on a server, but in the event of network failure the same component may be required to execute on the workstation,  or
    • the primary copy of a component’s data may be held on a central server, but an operational copy – or replica is available on the workstation.

Generally, this placement relates to all (application and technical) components, and therefore all deployment inits.

Deployment units need to address placement to address security requirements. 

  • Components may need to be separated by zone boundaries to address risk management concerns e.g., highly trusted componets may be separated from low trust level components or separation may be required to address the Privacy implications of storing specified data together.
  • Deployment units for flow control may be required to secure data flows separated across different zones.

Different deployment units may also be needed for different trust environments to fullfill the functional security requirements within the different trust environments.

Tailoring
Impact of not having

Not documenting Deployment Units can lead to:

  • A loss of information about the non functional requirements associated with components and a failure of the operational model in achieving the  desired Non Functional Requirements
  • Increased complexity during placement due to the numbers of fine-grained components that need to be taken into account
  • Increased cost of the solution, caused by inappropriate “over engineering” due to the inappropriately coarse grained placement decisions.
Reasons for not needing

If the operational model is trivial, this work product need not be produced.

If the relationships between components and nodes are simple, the allocation of components may be recorded directly on the operational model without using intermediary Deployment Units, although great care will be required to ensure the different aspects of each component are understood and all questions considered.

Under certain circumstances, such as when defining certain types of generalized OM, it is valid to focus purely on the location of execution of components.  In these cases, it is valid to place components directly onto the OM’s nodes, since the placement is implicitly related to execution only.

Representation Options

This artifact can be documented in an electronic document, spreadsheet or tool and notated using IBM’s Architecture Description Standard (ADS) version 2 or above. It is recommended to leverage tooling whenever possible due to the complex task of maintaining the various relationships and connections between the Deployment Units.
