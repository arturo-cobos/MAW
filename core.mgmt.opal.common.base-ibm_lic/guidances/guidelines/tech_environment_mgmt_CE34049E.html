Guideline: Technical Environment Management
Use this guideline to plan or customize, and to start the technical environment, ensure that it provides the expected service throughout the project; and de-install it (or schedule its de-installation) if necessary.
Relationships
Related Elements
  • Establish the technical environment for start-up phase (IMSP391)
  • Outline the technical environment plan (IMSP491)
  • Update other project plans (IMSP391)
Main Description

Purpose


Use this guideline:
To plan or customize, and to start the technical environment, ensure that it provides the expected service throughout the project; and de-install it (or schedule its de-installation) if necessary. Defining the requirements for, estimating the costs of, and tracking the effectiveness of the technical environment is key to the success of the project and to oversee the implementation of the office suite, project management, and product technical environment for the one or more phases of the project.

Scope


This guideline applies to:
All projects that need to start or customize the technical environment.

Overview


This guideline includes:
  • An office suite: spreadsheet, word processor, presentation graphics, and/or drawing tools.
  • Office communications: mail and access to shared office suite files.
  • Project management tools that support:
    • Project management and operational scheduling
    • Time tracking
    • Cost tracking
    • Risk management
    • Project control book (PCB)
    • Access to business systems
  • Tools that support technical activities in all project stages:
    • Conception
    • Development
    • Qualification
    • Launching or deployment

These tools make up that part of the technical environment referred to as the "product technical environment".
Features of the project management and the technical tool set support the configuration management of the technical work products.
The technical environment has a variety of characteristics such as:
  • It may already be in place, may need to be adapted, or may need to be completely built. Project needs, costs, expected benefits, and risks must be carefully balanced before making a decision to build a technical environment or even to adapt one that exists.
  • It may be a deliverable, as the technical environment or parts of it may be left behind or transferred to the sponsor at the end of the project.
  • It may be provided (in its entirety or in part) by the sponsor or another supplier, including a business unit within the delivery organization.

In any case, when planning the project, project management must describe the technical environment required by the particular project and then validate the technical environment architecture and proposed service level with the organization that provides it.
There are four stages of Technical Environment Management (TEM) namely:
  1. Technical environment planning
  2. Technical environment implementation
  3. Technical environment maintenance
  4. Technical environment removal

Details

1. Technical Environment Planning


The purpose of this activity is to establish the overall technical environment plan, as opposed to the detailed plan that is finalized when starting a phase of the project.
This activity includes:
  1. Defining the key options for the technical environment
  2. Identifying required features for the technical environment
  3. Describing the overall architecture of the technical environment
  4. Planning implementation of the technical environment

1.1 Define key options


The technical environment used to create and deliver the solution often has a significant impact on the project.
For example, building a web site, which users navigate as if they were accessing a relational database, requires implementing thousands of hyperlinks and ensuring the integrity of these links. Clearly, the approach, risks, costs, quality of the result, and the duration for building the web site depend entirely on choices made for the technical environment.
In this example, an option may be to simply use a web page editor, to maintain a list of the links to be implemented in a spreadsheet, and/or to implement (manually) the links using this editor. Another option would be to create a relational data base containing the future content of the site, and to find or build an application that generate the web pages and hyperlinks from the data base.
The choice of options is a key decision for shaping the project. It may even be the case that without a proper technical environment the solution will not be feasible.
Additionally, some elements of the technical environment may become part of the solution itself. Unless we develop specific components, the first option may cause problems related to ownership rights, but will be safer than developing specific components.
The key options for the technical environment are determined during project definition and described in the key options in the project definition. During Planning and Starting activities, they will be developed in the key options of the technical environment plan.
When determining key options for the technical environment, project management needs to consider:
  • The state-of-the-art in terms of technologies available in the marketplace as well as proven practices used by similar projects.
  • The sharing of the technical environment with another project or sub-project to save cost and to enhance the return on shared investment.
  • Any pre-existing commitments between the sponsor and the delivery organization for the use of specific standards or tools, and any knowledge about their use acquired during the shaping of the project.

Because this process provides descriptions of quality contributors, it interacts closely with quality planning.
The following represents the steps required to define key options for Technical Environment planning:
  1. Determine the level of automation for the technical work.
    Automating technical work on a project is a significant investment. When making decisions about which aspects of technical work will be automated and which will remain "manual", the levels of activity and quality required by the project to be major considerations when developing a justification for the automation investment.
    Here is the typical approach for determining the level of automation:
    • Identify project activities that could be supported by technical environment resources, given the requested approach or methods, tools and standards, and team organization (for example, consistency checking that could be automated).
    • Consider alternative approaches (more or less automation balanced by more or less centralization or decentralization of assessment activities, tailoring of work, cross reviews, additional procedures and so on).
    • Choose among alternatives.
  2. Define standards and tools.
    Defining standards and tools is necessary to ensure that:
    • Extensions created for the project are compatible with the hardware and software already in place (if any), or other directions given by the sponsor.
    • All participants on the project (team, sponsor, and subcontractors) can communicate to ensure effective collaboration.
    The following is a typical approach:
    • Begin by recognizing obvious choices or existing and reusable standards, tools and/or software for the:
      • Local area network
      • Operating systems for the servers and the workstations
    These operating systems generally are closely linked to requirements for the deliverables themselves
    • Continue by making choices and searching for reusable assets for:
      • Office suite and office communication tools
      • Tools to support project management work, including tools for project scheduling and tools to support business processes
      • Tools to support technical work (for example lower and upper CASE tools in application development and so on )
      • Standards for accessing remote systems
    • Check the compatibility of options that exist among the choices available with a technical expert.
    • Take into account cost and property rights (royalties) concerns when choosing standards and tools such as:
      • A tool may be inexpensive but consume a large amount of resources when it is run
      • A tool may be inexpensive for development purposes but then become mandatory for running the deliverables, resulting in additional and perhaps unacceptable expenses or licensing issues when deployed on final user workstations
  3. For projects that will run for some time, define a policy for upgrading the technical environment as required.
    This policy will be used when considering upgrades to new versions of the tools that are released to the market.
    Benefits, drawbacks and risks related to changing the technical environment during the course of the project to be closely analyzed before making a decision.
  4. Investigate the possibilities of sharing technical environments.
    • Consider sharing the technical environment, or parts of it, with another project. It might be necessary to adapt some of the key options or to accept the vision, and therefore the key options, of another project, if the potential cost savings outweighs the risk of not adapting or updating. An example will be accepting a lower level of automation, knowing that the cost of the additional manual effort is less than the cost of purchasing a new, fully-automated project repository.
    • Invite other projects to consider sharing your project's ideal environment. The cost of everyone's manual efforts might be more than purchasing a new project repository that can be shared.
  5. Identify the risks related to the key options.
    Risks are mainly triggered by the newness and complexity of the technology (although new technologies may be required to support features critical for the client). Some of the key points to be noted here are:
    • New technologies may not be sufficiently stable to support project work.
    • New technologies may not be available on the market when needed.
    • Obtaining support from suppliers on new technologies may be very difficult, if not impossible.
    • Combining technologies in ways that are not supported by previous experience is a major source of risk.

1.2 Identify required features


Required features are identified for each project organizational unit (POU) by considering the work they have to perform, roles and responsibilities, and quality contributors that may be required as described in the project definition.
While identifying the required features, the key options defined during project definition might need expanding or modifying. This more fully developed description of the key options is then added to the technical environment plan.
The following represents the steps required to identify required features for Technical Environment planning:
  1. Agree with the technical environment architect on the form in which required features will be expressed and the metrics that will be provided.
  2. For each phase of the project, identify the features (use cases) required from the technical environment by reviewing the key options for the technical environment, along with work to be performed, roles and responsibilities, and quality contributors that may be required, as described in the project definition.
    Typical features could be as follows:
    • Office suite such as:
      • Spreadsheet
      • Word processor
      • Presentation and/or drawing tools
    • Office communications such as:
      • Mail and access to shared office suite files
    • Project management areas such as:
      • Management and operational scheduling
      • Time tracking
      • Cost tracking
      • Risk management
      • Project control book (PCB)
      • Access to business systems
    • Technical areas such as:
      • Conception
      • Development
      • Qualification
      • Launching or deployment
    • Configuration management associated with project management and technical tools that:
      • Identify features that are in support of or are required by the methodology adopted by the project
      • Indicate which features to be isolated from or integrated with the others.
        For example, upper and lower CASE features may need to be integrated, while planning and tracking functions may need to be interfaced with business systems
  3. For each phase of the project, identify the types of users.
    • Establish a typology of the users according to the sets of features that they need.
      Certain users just need office suite features; others need office suite as well as project management features; still others might need only development or testing features.
    • Count the number of users for each type.
  4. For each phase of the project, identify the sets of files (data or code) that will be handled and estimate their volume.
    • When doing this, take into account the need for maintaining several versions of the files due to work in progress, configuration management, and archiving requirements.
      Sets of files to be considered include:
      • Individual and shared office suite files
      • Pay special attention to the volume of graphical data
      • Messages
      • Individual and shared technical files
      • Indicate the need for isolating certain sets of files from other sets of files. For example, development and test files generally be isolated, which typically means double the volume of files
      • Individual and shared files for project management
  5. For each phase of the project, quantify calls to the key features.
    For example, determine the number of users (team members) who will compile simultaneously (such as call to the compiling feature) for an application development phase.
  6. For each phase of the project, describe the need for group work, information sharing capabilities, accessibility of the systems (local and remote), security at different levels of project organization, and printing capabilities for the group.
  7. For each phase of the project, define the expected level of service.
    This definition typically includes:
    • Days and hours in the day when the technical environment to be available.
    • Response time for batch and on-line processing.
      Take into account special needs such as availability of the technical environment at night to perform specific testing.
  8. At the top level of the project, aggregate required features for all Project Organizational Units (POUs) and document them in the technical environment plan.

1.3 Describe overall architecture


The purpose of this activity is to obtain an overall design showing the different components that might have to be built or bought to create the technical environment in which the project will work.
The technical environment consists not only of tools, but also of processes and methods. Beyond the technical methods that are already provided by your constituency, it may be necessary to create specific processes for the project and these to be included in the technical environment plan.
The following represents the steps required to describe overall architecture for Technical Environment planning:
  1. Monitor the progress of the technical, application, and process architecture schema along with their descriptions. These will be developed to support the required features documented in the technical environment plan.
    Technical and application architecture schemas are based on a classification of workstations and potentially, a classification of servers. These classifications take into account the types of users, feature sets, and sets of files defined when describing required features.
    The descriptions of the technical and application schema should reference specific pieces of hardware or software (with an indication of version numbers), their characteristics (size information, for example), and relationships to other technical environment components. Requirements to be included for components to be built such as:
    • The hardware architecture schema describes elements of equipment, which are the nodes and connections (machines, cards, peripherals, so on) that make up the network of interconnected machines.
    • The software architecture schema describes the basic software installed on the hardware architecture: operating system(s), communication software, database management system, transactional monitors, and so on.
    • The application architecture schema describes the implementation of the tools on the technical architecture from the point of view of the features that are required.
    The processes are specific ways of working to contribute to the expected outcomes of the project.
    • The process architecture schema for the technical environment plan must identify these processes and the components of their descriptions (for example, activities, tasks, work products, guidelines).
      When possible, the architecture schema to be compared with existing intellectual capital to make the best use of the reusable assets.
  2. Ensure that the proposed architecture complies with standards required by key options for the technical environment and request for proposal or project quality plan.
  3. Ensure that the proposed architecture complies with required features.
    For each type of user, think through the way in which the software (specific tools, local area network, communication server, and so on) and hardware layers will meet the required features defined for the technical environment.
    Pay special attention to determining whether features are being effectively integrated on workstations for each type of user.
  4. Ensure that the proposed architecture is optimal with regard to the following:
    • Sizing of computer storage capacities (processors, memory, disks). Sizing of the technical environment must comply with specific requirements of the tools. Decisions made here can greatly influence the productivity of the project in the long run.
    • Minimization of modifications that will have to be made when moving from one project phase to another.
    • Economical sharing of printing resources; for example, assuring easy access to printing resources based on the locations of users.
  5. Check security aspects such as:
    • Back-up features and storage capacities.
    • Recovery features.
    • Accessibility based on roles and responsibilities.
    • Availability of technical environment components, support services, and spare parts.
  6. Check the potential for patenting of innovative components of the technical architecture. If patentable components are identified, handle those components as deliverables, and describe their characteristics in an associated deliverable definition. Also identify them in the work product list.
  7. Check that tools for administration of the technical environment are included in the technical environment.
  8. Ensure that the proposed level of service is satisfactory.
  9. Check overall consistency of the architecture, if it was assembled from required features created by separate POUs.
    Pay special attention to communication and interface capabilities and the economical sharing of resources. Review the architecture schemas of other project organizational units (POUs) in the program or project to validate that your proposed architecture is sharing resources wherever possible.

1.4 Planning Implementation


The following represents the steps required for planning implementation for Technical Environment planning:
  1. Establish or validate, within the planning framework, the work units needed to develop and implement the technical environment.
  2. Refine the work to be done in those work units to such a level that reliable estimates and risk assessments can be done for the technical environment development
  • List or validate the components that to be built, supplied by the sponsor, bought, or reused from existing assets. This creates the product breakdown structure (PBS) for the technical environment.
  • List or validate the activities that should be performed to set up the technical environment such as:
  • Buying off-the-shelf components (if any).
  • Receiving components from a subcontractor (if any).
  • Setting new parameters on existing components.
  • Developing new components.
  • Integrating and testing the technical environment.
  • Writing rules and procedures, including the rules and procedures for configuration management.

This creates the work breakdown structure (WBS) for implementing the technical environment.
  • When completing this step, remember the following:
    • Prototyping and pilot usage activities should be included
    • Technical environment implementation milestones must be synchronized with project management schedule
    • The schedule for release of technical environment components to the project must be carefully checked. Components are not always available when needed from suppliers
  1. Identify needs for education and training, technical support, and technical expertise.
  2. Document the roles and responsibilities of the delivery organization, sponsor and supplier(s) for implementing and maintaining the technical environment.
  3. Validate technical support team organization against the implementation schedule and resources that are necessary to maintain the technical environment throughout the course of the project.
  4. Estimate the cost of the technical environment, including that of any purchased services. The cost might be a percentage of the total cost when the technical environment is shared.
    The result of this calculation is a part of the financial plan.
  5. For each phase of the project, collect the following elements in order to assemble the technical environment plan:
    • Key options for the technical environment.
    • Description of required features. This includes lists of the features that are required, user types, number of users per type, sets of files with their volume, and quantification of the calls to the key features.
    • Technical and application architecture schema.
    • Education and training plan.
    • List of the services that are required. This includes education and training, support, and other areas of expertise.
    • Acquisition plan.
    • Implementation schedule.
    • Roles and responsibilities for implementation and maintenance of the technical environment.
    • Technical support team organization. If this organization is large, it is helpful to identify a focal point, or one contact, within the team.
    • Technical environment cost plan.
      The technical environment plan size may range from one page to over 50 pages, depending on the nature of the work to be performed on the project and responsibilities assigned to the delivery organization.
      If required, update the project quality plan in order to keep it in line with the Technical environment plan. See the QM. Quality management domain for further information.

2. Technical Environment Implementation


This process is used in a simplified way for implementing the project start-up technical environment.
The following represents the steps required for Technical Environment Implementation:
  1. Perform detailed planning for the office suite and project management tools implementation.
  2. Pay special attention to opportunities and threats throughout this effort and adjust plans accordingly.
    For example, the beginning of a new project might provide a good opportunity for implementing a new project management workflow scheduling application, one that has several long-awaited functions. However, if the project duration is short and there is not much time for training, the risk might outweigh the benefit.
  3. Document utilization rules for office suite tools, if any.
  4. Install or check installation of office suite and office communication tools on the workstations.
    • Check conformity with standards.
    • Test communications.
    Beginning to use a new workstation on a project generally requires some effort. Standards are not sufficient to control all the subtleties of setting parameters, particularly when workstations are used for several projects at the same time.
  5. Document utilization rules for project management tools.
    For example:
    • Set up guidelines for access to shared file space so that multiple people can edit the project plan.
    • Set parameters for and initialize project management tools. This includes customizing the tools to the project Organizational breakdown structure.
    • Document, in the asset inventory, the assignment of equipment to the project staff.

3. Technical Environment Maintenance


The purpose of this activity is to maintain expected levels of performance from the technical environment throughout the project.
This is achieved through:
  • Assessing the effectiveness of the technical environment.
  • Maintaining the effectiveness of the technical environment.

3.1 Assessing the effectiveness of the technical environment


An effective technical environment is crucial to the success of the project. The project manager is responsible for ensuring that the environment is performing as planned, and as required, and continues to do so throughout the life of the project. The assessment process helps the project manager evaluate the health of the technical environment.
An assessment might be required because:
It had been planned to coincide with a specific point in the life of the project
The project manager has been receiving complaints, issues, or compliance incidents, and so on, relating to the technical environment
The project manager's perception is that problems with the technical environment have the potential to jeopardize the project.
An assessment could be brief and informal when there are only a few people and components involved, or it could be extensive, requiring the involvement of many people, supporting documentation, and a significant amount of investigation and tracking.
The following represents the steps required for assessing the effectiveness of the technical environment for technical environment maintenance:
  1. Define the scope of the assessment such as:
    • An assessment is required of the entire technical environment.
      This type of assessment might be done when a new technical environment has been installed and the project has reached the point where it is now fully operational and fully staffed. The requirement for doing this might have been documented in the technical environment plan.
      Another option is to do a general assessment when it is suspected that the environment is jeopardizing the project, but it is unclear as to which area is the source of the problem.
      Or
      An assessment is required of only a particular area, such as the project workflow scheduler, or a particular characteristic of the technical environment, such as the response time across many areas.
  2. Define the approach such as:
    • Determine if a formal assessment meeting is required or whether the assessment can be done by less formal means.
    • Decide who will be involved. If there will be a meeting, decide who should be invited. There may be a need to invite people from outside the project team, such as support people who have the technical skills that the project manager might not have.
    • Determine whether additional documentation needs to be gathered before the meeting.
  3. Before the assessment meeting, investigate and gather supporting documentation, if required and available. Examples of documentation might include issue documents, compliance incidents, system outage logs, and notes from project team members.
  4. Conduct the assessment.The steps to be followed are:
    • Define the problem as completely as possible.
      Find the real problem underlying the initial concern. For example, project team members complained about randomly not being able to read certain files sent through the e-mail system or stored in a repository. After discussion, the team realized that they could not read files sent by people using a different operating system.
    • Determine the impact on the project. The schedule or the quality of the deliverable might be in jeopardy because of the inability to read documents critical to the success of the project.
    • Identify the source of the problem, if possible.
      The project team might not be able to do this step without help. In the case of the incompatible files, for example, team at first thought the problem was caused by the mix of operating systems. But after analysis by a technical support person, it was determined that the problem was a down-level version of one of the e-mail products.
    • Design a solution, if possible.
      This might not be in the area of responsibility of the project management team, but if the team has ideas for a solution, develop them, as time allows, and then propose them to the responsible area. For example, the team proposed that technical support develops and distributes a procedure for upgrading the e-mail application.
    • Define a temporary solution, if possible. For example, the project manager sent out and stored the documents in both formats until all the upgrades were done.
    • Identify the person, role, or organization responsible for solving the problem, if possible. When not possible, identify who can further define the problem and its cause so that it can be assigned to the correct function.
  5. Document the findings and subsequent actions, if required, in the technical environment assessment.
  6. Create additional documents, as required, such as risk definition, compliance incident document, issue document, change request, or any documents required by groups outside the project.
  7. Document follow-up activities in the technical environment assessment.
    • Reference any additional documents.
    • Determine if and when another assessment is required to ensure that the improvements have been effective.

3.2 Maintaining the effectiveness of the technical environment


This process addresses the need for continuous improvement to the technical environment. Project team members, including the project manager, and technical support personnel look for ways to improve the environment; project management must then decide if implementation of the improvement is feasible.
The following represents the steps required for maintaining the effectiveness of the technical environment for technical environment maintenance:
  1. Collect or solicit, suggested improvements to the technical environment and its utilization rules, based on requests from the teams and performance statistics.
    • Carefully balance the benefits of changes or additions to the technical environment against related risks.
      Any change that has been determined to be necessary and appropriate should be planned carefully. Pay special attention to communications between users during the migration phase and data conversion necessitated by the change.
  2. Maintain control of developments undertaken by project staff to simplify or enhance the use of the office suite and other tools; for example, "home-grown" macros to automate activities.
    Control the time spent on such activities, the reliability of results, and management of versions disseminated among the teams.
  3. Re-size or re-deploy the technical environment, as required based on resource consumption statistics.
  4. Launch additional support actions such as training, as required based on requests from the teams and performance statistics.
  5. Ask users to change behavior, as required based on analysis of performance statistics.
    Changes in behavior include:
    • Exercising better "clean-up discipline".
    • Making adequate use of batch and on-line features.
    • Thinking through work more carefully before calling for automated checks.
  6. Manage services provided by suppliers to maintain control of the budget.
  7. Update the asset inventory based on arrivals and departures of staff, the start of new phases, office moves and so on.

4. Technical Environment Removal


The purpose of this activity is to oversee the de-installation of technical environment components from workstations when people leave the project, and from the common platform at the end of the project.
This includes the following tasks:
  • De-installing workstations.
  • De-installing the common technical environment.

4.1 De-installing workstations


The following represents the steps required for De-installing workstations for technical environment removal:
  1. Each time a person leaves the project (or it becomes necessary to re-deploy workstations):
    • Review the inventory to identify the components that have been installed by the project on the person's workstation.
    • Review the agreement to understand disposition requirements for these components.
    • Collect reusable office suite developments and utilization ideas in order to provide them to the intellectual capital management focal point.
    • Save or archive project files, as appropriate; transfer files to other team members, as required.
    • Check that the departing person cleans up the common storage areas that he/she has been using.
    • Check that all the data has been deleted from the workstation.
    • De-install software that are returned to the project, following the manufacturer's guidelines.
    • Collect software and hardware components that have been allocated to the person for the duration of the project.
    • Keep components for redeployment or return them to the sponsor or other owner, as appropriate.
    • Record official return to the sponsor or other owner.
    • Delete access authorizations.
    • Update the inventory.

4.2 De-installing the common technical environment


It is important to sequence the de-installation of the common technical environment to allow for any remaining project activities that need to be completed; for example, there may be back-up or printing needs up to the last minute.
The following represents the steps required for De-installing the common technical environment for technical environment removal:
  1. Review the asset inventory and the agreement to understand disposition requirements for technical environment components.
  2. Collect reusable ideas and developments in order to provide them to the intellectual capital management focal point.
  3. Back up files as required such as:
    • Archive the project control book (PCB) as required by constituency policies.
    • Ensure that technical files, such as the product itself, have been archived.
  4. Clean up the common storage space.
    When the technical environment is embedded in a complex environment, this is a tricky activity unless directories have been prepared from the start of the project with this "clean up" in mind.
  5. De-install the software following the manufacturer's guidelines and return it to the sponsor or other owner, as appropriate.
    • Record official return to the sponsor or other owner.
  6. De-install hardware following the manufacturer's guidelines and return it to the sponsor or other owner, as appropriate.
    This may require hardware to be moved.
    • Record official return to the sponsor or other owner.
