Guideline: Project Control Book
This is used to create the Project Control Book, a repository of Project Management System documentation. The PCB is used for storing, accessing, and managing project management documentation, which includes plans, procedures and records.

Main Description

Purpose

The Project Control Book (PCB) is a repository of Project Management System documentation. In any organization, there are benefits to having a common structure to organizing the project management system documentation, which includes plans, procedures and records.

Use this guideline to:

  • Describe the common structure for the Project Control Book, and identify which documents are appropriate for each section. This approach includes identifying a companion Project Work Book (PWB) approach, and clarifying which documents are recommended for the PWB
  • Provide guidance on the refinement and management of a Project Control Book
  • Identify tools, which enhance the efficiency and effectiveness of the Project Control Book approach

Scope

The scope of information stored in the Project Control Book is all the plans, procedures and records to manage and control the project. Project management methods have common applicability across projects, and the PCB structure is common to all projects.
The scope of the PCB information does not include the technical documents and records used to design, develop, test or implement the solution which is the outcome of the project. This technical or solution information is stored in the Project Work Book. While there may be standard Project Work Book structure for types of projects, there is not one single structure common to all projects in your organization. The Project Work Book structures may align with your organization's solution engineering methodology.

This guideline applies to:

  • All proposals and projects
  • All management documents and records considered essential to the Work Request or project

Overview

There are many activities which relate to, or are impacted by, the approach used to implement the Project Control Book. The overview activities describe the key process steps which are referenced in these other activities.

The overview activities supporting the PCB are:

  • Refine the Project Control Book structure
  • Define the Technical Environment supporting the Project Control Book
  • Maintain proposal and project management document and records
  • Archive proposal and project management records

Details

1. Refine the Project Control Book structure

The standard Project Control Book structure is described below. It is recommended that the thirteen sections are not renamed or removed. It is appropriate to add further structure as folders under each major section.

Section 0: Standards and Procedures
  • Summarizes standards and procedures specific to a project or related group of projects
Section 1: Organization People and Resources
  • Includes name, phone, and other contact information for project participants and defines the role of each of those participants
Section 2: Milestones
  • Provides a listing of the sponsor and supplier agreement milestones for the project
Section 3: Plans and Actuals
  • Provides a description of the project plans and schedules, a reference to each plan's location, and storage for actuals vs. plans
Section 4: Meetings
  • Stores meeting histories and provides links to action items
Section 5: Risks
  • Records status of project risks and provides links to action items
Section 6: Changes
  • Provides a place and structure to manage change requests.
Section 7: Issues
  • Provides a record of issues and a history of resolutions
Section 8: Delivery
  • Tracks deliverables and deliverable components with delivery release and acceptance records
Section 9: Sponsor Information
  • Provides access to sponsor agreement(s), to all agreement amendments, and to progress reports
Section 10: Supplier Information
  • Provides access to supplier agreement(s), to all agreement amendments, and to progress reports
Section 11: Requirements
  • Provides a place to store and maintain a list of sponsor requirements
Section 12: Quality
  • Provides a place to maintain a record of project reviews conducted throughout the course of a project, and the results of those reviews

2. Define the Technical Environment supporting the Project Control Book

A logical PCB structure can be physically implemented in many technical environments. The critical issue is that proposal and project management data must be retained, stored, and archived in an organized manner. There are significant efficiency and effectiveness benefits in using a dedicated project management tool to manage the project control book function.

Example technical environments:
The PCB can be a:

  • Binder full of papers with dividers
  • Set of folders or directories on a Server or Local Area Network (LAN)
  • Collaborative data repository
  • Dedicated Project Management tool such as Rational Portfolio Manager
The decision how to manage the Project Control Book is recorded in the Technical Environment Management Plan (ENG361), as described in the Project Start (Project Launch) Procedure (IMSP391).

3. Maintain management documents and records

The Proposal Team Leader or delegate is responsible for ensuring that the PCB is kept up to date and that the proposal team has access to it during the Solution Design phase. During the Solution Delivery phase, the Project Manager or delegate is responsible for ensuring the PCB is kept up to date and that the project team has access to it.

Maintain a PCB Index to reference all documents and records that are stored in the PCB, and is kept in Section 0: Standards and Procedures.
For each document in the PCB, the PCB index records the following information:

  • Title (and file name if an electronic file)
  • Current issue date (date of issue for current version)
  • Current version number
  • Storage location
Update the PCB and PCB index with each new or updated proposal or project management document and record. In general, all proposal or project-related management plans, inputs, outputs, agreements, reports, communication, and records should be kept in the PCB. If in doubt, include it.

4. Archive proposal and project management records

Determine ownership rights and retention requirements for all client deliverables, client supplied materials, and supplier provided materials in accordance with contract terms and conditions.

Once ownership is determined, perform the following:
  • Return and erase all client owned materials and confirm actions to client in writing
  • Return and erase all supplier owned materials and confirm actions to supplier in writing
Determine retention requirements of all other documents and records in accordance with:
  • Contract terms and conditions
  • Information Retrieval and Analysis

If the Project Control Book is hard copy, label it to identify it and associate it with the relevant Work Request Number(s). Download all organization's owned soft copy materials to storage media, carefully packaging and clearly identifying the media content, and marking with the relevant Work Request No(s) and security classification.
Identify any financial documents and records that may require longer retention periods.
Ensure that the content of the Project Control Book is complete, and its content correctly filed without duplication, including new storage media details for downloaded records.
