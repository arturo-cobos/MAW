Guideline: Release Management
Use this guideline to manage the packaging of work items into logical groupings for a one-time release into the target environment, such as testing or production.

Main Description

Purpose


Use this guideline to:
Manage the packaging of work items into logical groupings for a one-time release into the target environment, such as testing or production.
Target audience:
Project Managers, Technical Team Leaders, Test Team Leaders, Project Team Members, Developers, Testers & Client.

Scope


This guideline applies to:
All projects.

Overview

This guideline includes:
  • Verifying Completion of Construction for the Release
  • Designing, Building and Configuring the Release
  • Preparing Test Environment and Planning Migration
  • Migrating to Test Environment for Integrated Testing
  • Verifying Completion of Integrated Testing for the Release
  • Promoting to Production Environment
  • Confirming Changes in Production Environment and Notifying Client

Details

1. Verifying Completion of Construction for the Release


A planned release consists of all new or changed software and hardware required to implement the approved changes. The release plan includes all of the release details, including the completion criteria.
At predetermined times, the Technical Team Leader verifies the completion of the construction for the release, based on the category and definition of the release. Releases are categorized, based on the following:
  • Work involved in construction:
    • Major software releases: Consists of major units of new functionality, some of which may make the preceding problem fixes redundant.
    • Minor software releases: Consists of minor enhancements or fixes, some of which may make the preceding emergency fixes redundant.
    • Emergency software or hardware fixes: Consists of corrections to a small number of known problems.
  • Components included in the release:
    • Full Application Release: All application components are released together for implementation, irrespective of whether they underwent a change since the previous release. Such releases ensure the whole application (changed and unchanged components) is built, tested, distributed and implemented together.
    • Partial Application Release: Only those application components that have actually changed or are new since the last release are released.
    • Integrated Package Release: All changes across application domains are grouped together and released as a 'package'. Such releases are especially useful where the changes to a system or an application may require changes to other systems or applications. Since these changes are made together, they are included in the same package for release.

The Technical Team Leader checks the following to verify the construction planned for the release is complete:
  • The work is constructed as per the service request or project plan for macro design/project plan for micro design.
  • The work is unit tested in the Construction Environment.
  • The data migration plan is created for migrating the work from the construction environment to the test or production environment.
  • The release plan is updated appropriately.

2. Designing, Building and Configuring the Release


After the Technical Team Leader verifies the construction for the release is complete, all the software and/or hardware components, which comprise the release, are required to be assembled in a controlled manner to ensure a reproducible process.
For software, the standard approach is to receive the new or changed source code from the Developers and generate the executables using controlled procedures on dedicated build hardware. This process is called Build Management and is one of the core activities in Release Management. Build management becomes the responsibility of Release Management from the final release entry date onwards.
The instructions to build or to assemble a release are considered part of the definition of the release. It is quite common to write scripts to automate the installation of systems and application software onto servers and workstations. Depending on the implementation plan, this can be performed in advance (for example, if hardware is being replaced) or can be performed directly in the production environment.
The Technical Team Leader plans and documents the procedure for building software releases by reusing standard procedures, wherever possible. A configuration of a particular release of software and hardware may be based upon a set of available components, some of which may be developed in-house and others bought in.
All software, parameters, test data, run-time software, and any other software, which is required for the release, are placed under configuration management control. A complete record of the build results is logged so that a build can be repeated, if necessary.
Design, build, and configuration of the release are done based on the release plan of the project. The outputs of this activity are assembled into the physical packaging and may include the following:
  • Detailed release assembly and build instructions, including the exact sequence of operations.
  • Purchase orders, licenses and warranties for third-party software and hardware.
  • Automated installation scripts and associated detailed test plan(s).
  • Master copies of the installation media and installation instructions.
  • Back-out procedures to document the actions to be taken to restore the service to rollout a release fail, either partially or totally.
  • Updated data migration plan(s).

3. Preparing Test Environment and Planning Migration


The Test Team Leader prepares the test environment for upcoming migrations from the construction environment to the test environment on an ongoing basis, as necessary.
Upon receiving notification from the Technical Team Leader regarding completion of the build for the release, the Test Team Leader creates the appropriate data migration plan based on the approved service request identifying the following:
  • Delivery package and migration category such as daily, weekly or on-request migration.
  • Release information, type of transport (manual or automated), source construction environment, and all target environments and dates.

The Test Team Leader reviews the data migration plan with the relevant stakeholders, obtaining any necessary approvals.

4. Migrating to Test Environment for Integrated Testing


Release Management includes integrated testing of multiple changes at once for early detection of cross-process conflicts and to minimize system disruption. The Test Team Leader performs the following:
  • Migrates all the work from the construction to the prepared test environment to facilitate commencement of integrated testing using either a manual or automated transport mechanism.
  • Updates the appropriate data migration plan with details including the date of migration to the testing environment.

5. Verifying Completion of Integrated Testing for the Release


The Test Team Leader checks the following to verify testing is completed for the planned release:
  • Integrated testing is complete in the test environment and results are documented in the appropriate test report.
  • Regression testing is complete in the test environment and results are documented in the appropriate test report.
  • Critical process testing is complete in the test environment and results are documented in the appropriate test report.
  • Client acceptance testing is complete in the test environment and results are documented in the appropriate test report.
  • The Test Team Leader notifies the Technical Team Leader of the successful completion of the testing effort.

6. Promoting to Production Environment


After receiving the notification from the Test Team Leader regarding the completion of testing for the release, the Technical Team Leader performs the following:
  • Migrates all the work from the construction to the production environment using the management approved manual or automated transport mechanism. Automated transport systems require close monitoring of migrations to prevent incomplete code from migrating.
  • Updates the data migration plan with all of the pertinent details, including the final date of migration to the production environment.

7. Confirming Changes in Production Environment and Notifying Client


After promoting the work from the construction to the production environment, the Technical Team Leader performs the following:
  • Reviews the production changes to confirm all required changes have been successfully migrated into the production environment.
  • Notifies the stakeholders of the successful move into the production environment of the approved changes.
  • Closes the service request, if applicable.
