Activity: Test Integrated Technical Solution
Perform (typically) independent testing of a set of integrated release components to determine progress toward the overall Build objecitves and determine if these have been met. In this case we would proceed to exit the Build iteration for this release.
DescriptionWork Breakdown StructureTeam AllocationWork Product Usage
Purpose
The purpose of this activity is to:
  • Exercise all application functions and verify proper execution of the application with other application interfaces
  • Verify that that the system is functionally correct and operationally sound
  • Perform transaction flow, stress / volume, regression tests, etc.
Relationships
Parent Activities
  • Build (only for Portal & Service Frame Work)
Description
System level tests verify proper execution of the entire application components including interfaces with other applications. Both functional and structural types of tests are performed to verify that the system is functionally and operationally sound.

System test is the first level of testing that is done after the development level testing has been done. System test is the first opportunity when the solution is tested in a production like environment.

This activity is performed at the completion of the programming cycles for this build. To accomplish this activity:
  • Perform functional and structural testing.
    • Execute tests following documented procedures, specifications, and test cases. Verify successful execution of all new, changed, or affected paths.
    • Maintain a log of the tests executed.
    • Fix problems and retest in unit test and integration test (as required).
    • Promote fixes according to configuration management procedures.
    • Run regression tests in the system test environment.
    • Document findings and results. 
  • Update test documentation as necessary.
  • Identify architectural issues.
    • Talk to application developers about any difficulties they have interpreting or implementing architectural templates or policies.
    • Review a sample of the detailed designs, comparing them with the component model and operational model.
    • Review Integration and System Test results for evidence of architectural issues.
  • Raise issues, enforce standards, make architectural decisions, or revise the architecture models, as appropriate
  • Based on the results of the testing, determine whether to exit the current build cycle or add additional programming cycles to the current build cycle. Based on this decision, update the goals of either the current build cycle or the remaining build cycles as a result of the experience gained on this build cycle.
