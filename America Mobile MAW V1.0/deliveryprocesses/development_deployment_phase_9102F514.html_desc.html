Develop Solution

Note that this activity closely aligns with the following activities in other processes: execute and develop.

The AWD Project Team:

1.

Provides AWD Technical Environments.

Obtain, implement, integrate and verify each of the following physical models as described in the AWD Design:

  • Development Environment – hardware, software, database and integrated network used by Developers to develop and unit test the solution
  • Test Environment – hardware, software, database and integrated network used by business customers to perform Acceptance Testing
  • Production Environment – hardware, software, database and integrated network used by end-users to perform their work via the new solution

Ensure the infrastructure supports iterative development, provides consistent use, and uses AWD-dedicated equipment.
2.

Iterates application prototyping.

The AWD Approach uses iterative, release-specific prototyping in place of detailed/micro application design.

For each iteration, this includes:

a) Focusing application prototyping on components, classes and methods at the application level, i.e., independent of the implementation platform

b) Leveraging JAD/JAR techniques

c) Developing mockups/simulations of the look-and-feel, flow and core of an application using paper simulations, whiteboards, graphics packages or development tools

d) Capturing iterations of business customer feedback, allowing walkthroughs of system flow before extensive effort is expended on development and testing, and performing ongoing train-the-trainer training to the business customer

e) Ensuring the prototype serves as the user interface design specification, i.e., describing the sequence of screens (such as windows, dialog boxes, and prompts) the user expects, and describing the objects, widgets, options, menus, buttons, visuals, dialog boxes, button states, screen behavior, user exception handling, error messages, help text, agents, supporting visuals, and wizards

f) Following the AWD Design, technical standards (e.g., coding, usability, branding) and priorities documented in the AWD Accelerator Plan

g) Effectively employing horizontal and vertical prototyping techniques

  • Horizontal prototype (used typically more in the first iteration) represent the customer’s “mental model” of the application, defined as the full breath of business processes represented in look-and-feel, user interface and navigation layers.
  • Vertical prototype (used typically more in later iterations) take software functionality (e.g., transaction, display, report, web page, etc.) seven standard layers (i.e., look-and-feel, user interface, navigation, communications, algorithms, data handling, error handling, and security) for one or more business processes.

h) Utilizing “evolutionary prototyping” instead of “throw away prototyping”

i) Ensuring collective ownership of components by project team members, to allow incorporation of team perspective, fast movement forward and backup for each individual

j) Ensuring traceability to requirements by providing a cross-reference between prototype components and the requirements listed in the Accelerator Plan. Refer to Requirements Traceability Matrix for more information.

Refer to User Interface Design Screens and Reports for more information.

3.

Completes security and audit control review.

This includes the following:

  • Reviewing, applying and resolving the impact of customer-established standards and business controls on the solution approach, e.g., Accelerator Plan, prototype, development
  • Addressing issues where sensitive company data may be exposed unnecessarily
  • Completing the review early so impact to application code can be addressed early
  • Ensuring the Business Customer retains ownership of the review process/results
4.

Delivers User Acceptance Test Plans.

For each iteration, create a plan to ensure the solution addresses all requirements and performs as expected in its intended operational environment.

Specify what regression and new tests will run, in what order, with what infrastructure, environment and data support, and who will run them.

Identify components required to execute the tests, finalize scheduling of the test runs and identify the resources required to run them.

Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.

Document requirements traceability in the AWD Test Plan, indicating for each test case which requirements apply.

Refer to Requirements Traceability Matrix for more information.

5.

Delivers test data.

Identify test data requirements and create/load data into the Test Environment.

6.

Develops and unit tests code.

For each iteration, follow the AWD Design, technical standards (e.g., coding, usability, branding) and priorities documented in the AWD Accelerator Plan and perform the following:

  • Develop the database
  • Develop program specifications, if required
  • Complete application modules through programming – complete prototype components, develop source code and ensure modules fully meet requirements for the given iteration
  • Ensure collective ownership of components by project team members, to allow incorporation of team perspective, fast movement forward and backup for each individual
  • Consider use of test-first methods to receive rapid feedback during development and also for improvement and removal of software defects. This means:
    • Add a test
    • Run all tests and see the new one fail
    • Write only the code required to pass the test
    • Run the automated tests and see them succeed
    • Refactor to remove duplication (see below)
  • Use refactoring methods, where applicable. That is:
    • Document the external behavior of modules, e.g., library routine behavior
    • Create a catalog of common methods for refactoring (e.g., encapsulate field, extract method, generalize type, pull up, push down), each with a description of how to apply it and indicators for when to apply, or not apply, the method
    • Modify the source code of specified modules without changing their external behavior in order to improve the understandability of the code (e.g., change variable name from “H” to “WeeklyHours”); change its structure and design (e.g., replace if-statement with case-statement); improve its internal consistency and clarity; clean-up the code (e.g., remove dead code); make it easier to maintain in the future; or for other reasons
    • Separate processes for adding new functions/tests and fixing bugs from the code refactoring process
    • Use automated unit testing to help ensure refactoring does not make the code stop working
  • Update impacted work products, e.g., modules list, physical model, designs, internal documentation
  • Develop unit test specifications/plans that include documentation of requirements traceability, i.e., indicating for each test case which requirements it addresses. Refer to Requirements Traceability Matrix for more information.
  • Unit test and verify completed modules.
7.

Plans/performs I/T testing .

For each iteration, this includes:

  • Developing I/T AWD Test Plan, e.g., for system and interface testing develop test cases, test order, data needs and documentation of requirements traceability to indicate for each test case which requirements it addresses ( Refer to Requirements Traceability Matrix for more information)
  • Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.
  • Performing work product inspections / peer reviews of the solution and its components:
    • Secure all components
    • Have one or more “buddies” perform a work product peer review of all components
    • Test the solution module-by-module in an integrated manner from a complete build, focusing on unit, integration, usability and system test cases, including interfaces with other applications.
    • Perform both functional and structural types of tests to verify that the system is functionally and operationally sound.
    • Ensure modules satisfy customer requirements and technical standards and perform as expected in the intended operational environment.
    • Document each issue/defect and turn each over to the associated Developer(s) to resolve and document the resolution.
    • Ensure original reviewers verify and approve all resolutions.
  • Fixing problems and resolving defects
  • Keeping supporting test result information, e.g., data values used, screen prints, reports, resulting data files.
8.

Develops documentation and prepares for training and support.

This may include development of tutorials, guides and PowerPoint presentations and scheduling meetings for training and support activities.

Have one or more “buddies” perform a peer review of each work product.

9.

Delivers Solution Code to User Acceptance Test Environment.

For each iteration, this includes:

  • Creating a complete list of components to be placed under software configuration management.
  • Developing an AWD Migration Plan for migration from the Development Environment to the Test Environment, but as a test of the migration from the Test Environment to the Production Environment. This means identifying all tasks, sequence and dependency information.
  • Creating a Change Record with the AWD Migration Plan attached.
  • Performing review of the Change Record and AWD Migration Plan with team members and other affected parties, including professionals from software configuration management, change management, operations, scheduling and database administration.
  • Obtaining release approval prior to release of the components into the Test Environment.
  • Using the AWD Migration Plan to migrate the components into the Test Environment with help from change management, software configuration management and other professionals.
  • Verifying all components are working in the Test Environment.
10.

Iterates activities for transition to maintenance.

The required activities were incorporated into the Project Management System Summary (PMSS) early in the project and may include knowledge transfer/training, documentation and support for building the maintenance environment.

Ensure support of the Maintenance Team (MT) as they acquire, install and configure all of the hardware and software required to provide prescribed application maintenance services, which may include day-to-day production support (problem determination and resolution and end-user support), and minor enhancements. This work must be completed prior to deployment to end-users and includes MT verifying the completeness of the maintenance environment.

Deploy Solution

Note that this activity closely aligns with the following activities in other processes: execute, build, deploy, launch and implement.

The AWD Project Team:

1.

Delivers User Acceptance Test Results.

Leverage the User Acceptance AWD Test Plan developed earlier to test the solution and document the results, including the tester’s initials, date of completion and results received.

Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.

Document each issue/defect and turn it over to the AWD Developer to pursue resolution and documentation of the resolution.

Obtain approval of the results as a whole before proceeding to final deployment.

Ensure supporting test result information is kept, e.g., data values used, screen prints, reports and data files.

2.

Plans/performs stress, performance and load testing.

Develop an AWD Test Plan for stress, performance and load testing, e.g., test cases, test order, data needs and documentation of requirements traceability to indicate for each test case which requirements it addresses ( Refer to Requirements Traceability Matrix for more information)

Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.

Document Test Results.

Document each issue/defect and review it with the AWD Developer to pursue resolution and documentation of the resolution.

Obtain approval of the results as a whole before proceeding to final deployment.

Ensure supporting test result information is kept, e.g., data values used, screen prints, reports and data files.

3.

Supports testing and fixes problems.

Correct and re-release corrected components to the Test Environment. Ensure corrections are documented and that original reviewers verify and approve each correction.

4.

Implements training and support.

This includes:

  • Training trainers, help desk personnel and end users.
  • Preparing the customer’s environment for solution release.
  • Establishing an effective customer management environment.
  • Deploying customer support.
5.

Delivers Solution Code to Production Environment.

This includes:

  • Updating the previously used AWD Migration Plan so it addresses the migration from the Test Environment to the Production Environment.
  • Creating a Change Record with the AWD Migration Plan attached.
  • Performing review of the Change Record and AWD Migration Plan with team members and other affected parties, including professionals from software configuration management, change management, operations, scheduling and database administration.
  • Obtaining release approval prior to release of the components into the Production Environment.
  • Using the AWD Migration Plan to migrate the components into the Production Environment with help from change management, software configuration management and other professionals.
  • Verifying all components are working in the Production Environment.
6.

Completes transition to the Maintenance Team.

This includes turnover of:

Documentation agreed to in the PMSS

Development and test libraries as documented in the AWD Completion Report

Metrics documentation and responsibility, e.g., logging and reporting the number of defects found in the first 30 days of production. It is the responsibility of MT to respond to defect reports and identify defects during this period.

List of remaining prioritized requirements from the Accelerator Plan, for inclusion in a service request backlog

Obtain written acceptance of the system for support and completion of transition activities from the MT Manager either directly or as part of AWD Completion Report signoff.

7.

Provides post-launch support.

Post-launch support is provided for a limited timeframe after the Solution Code has been migrated to the Production Environment, e.g., one week or 30 days.

Once a defect has been raised to Level 1 support and has been escalated to the Maintenance Team (MT), the MT may request Level 2 support from the AWD Project Team by making a request to the AWD Project Manager and providing a defect record for further analysis. Then:

  • Review the defect record and attempt to recreate, validate and clarify the nature and severity of the defect.
  • Estimate resource requirements, timing and availability.
  • Gain authorization from the AWD Center or Program Manager for staffing and funding to make the repair.
  • Assign team members to repair and test the defect, update the defect record and turn the corrected system over to the MT for completion, e.g., regression and acceptance testing, migration to production, and closing the defect record.
  • Capture defect information in AWD Project Metrics.
