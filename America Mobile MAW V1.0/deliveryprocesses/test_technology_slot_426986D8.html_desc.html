Activity: Test Solution
From a system perspective, test and evaluate the developed requirements.
DescriptionWork Breakdown StructureTeam AllocationWork Product Usage
Purpose
Develop and run test scripts to validate that the system satisfies the requirements.

Description

Note that this activity closely aligns with the following activities in other processes: execute, build, deploy, launch and implement.

The AWD Project Team:

1.

Delivers User Acceptance Test Results.

Leverage the User Acceptance AWD Test Plan developed earlier to test the solution and document the results, including the tester’s initials, date of completion and results received.

Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.

Document each issue/defect and turn it over to the AWD Developer to pursue resolution and documentation of the resolution.

Obtain approval of the results as a whole before proceeding to final deployment.

Ensure supporting test result information is kept, e.g., data values used, screen prints, reports and data files.

2.

Plans/performs stress, performance and load testing.

Develop an AWD Test Plan for stress, performance and load testing, e.g., test cases, test order, data needs and documentation of requirements traceability to indicate for each test case which requirements it addresses (Refer to Requirements Traceability Matrix for more information).

Use Testing Specialists, Testing Centers of Excellence, dedicated test environment(s), automated testing tools (e.g., Mercury TestDirector) and a test repository, if possible.

Document Test Results.

Document each issue/defect and review it with the AWD Developer to pursue resolution and documentation of the resolution.

Obtain approval of the results as a whole before proceeding to final deployment.

Ensure supporting test result information is kept, e.g., data values used, screen prints, reports and data files.

3.

Supports testing and fixes problems.

Correct and re-release corrected components to the Test Environment. Ensure corrections are documented and that original reviewers verify and approve each correction.

4.

Implements training and support.

This includes:

  • Training trainers, help desk personnel and end users.
  • Preparing the customer’s environment for solution release.
  • Establishing an effective customer management environment.
  • Deploying customer support.
5.

Delivers Solution Code to Production Environment.

This includes:

  • Updating the previously used AWD Migration Plan so it addresses the migration from the Test Environment to the Production Environment.
  • Creating a Change Record with the AWD Migration Plan.
  • Performing review of the Change Record and AWD Migration Plan with team members and other affected parties, including professionals from software configuration management, change management, operations, scheduling and database administration.
  • Obtaining release approval prior to release of the components into the Production Environment.
  • Using the AWD Migration Plan to migrate the components into the Production Environment with help from change management, software configuration management and other professionals.
  • Verifying all components are working in the Production Environment.
6.

Completes transition to the Maintenance Team.

This includes turnover of:

Documentation agreed to in the Project Management System Summary (PMSS).

Development and test libraries as documented in the AWD Completion Report.

Metrics documentation and responsibility, e.g., logging and reporting the number of defects found in the first 30 days of production. It is the responsibility of MT to respond to defect reports and identify defects during this period.

List of remaining prioritized requirements from the Accelerator Plan, for inclusion in a service request backlog.

Obtain written acceptance of the system for support and completion of transition activities from the MT Manager either directly or as part of AWD Completion Report signoff.

7.

Provides post-launch support.

Post-launch support is provided for a limited timeframe after the Solution Code has been migrated to the Production Environment, e.g., one week or 30 days.

Once a defect has been raised to Level 1 support and has been escalated to the Maintenance Team (MT), the MT may request Level 2 support from the AWD Project Team by making a request to the AWD Project Manager and providing a defect record for further analysis. Then:

  • Review the defect record and attempt to recreate, validate and clarify the nature and severity of the defect.
  • Estimate resource requirements, timing and availability.
  • Gain authorization from the AWD Center or Program Manager for staffing and funding to make the repair.
  • Assign team members to repair and test the defect, update the defect record and turn the corrected system over to the MT for completion, e.g., regression and acceptance testing, migration to production, and closing the defect record.
  • Capture defect information in AWD Project Metrics.
