Description

The Architect takes primary ownership of these activities and deliverables, but includes the other team members to develop the design to the level required to start the first development iteration, The team helps to capture workshop results and translate them into agile requirements and estimates.

Note that this activity closely aligns with the following activities in other processes:  execute, requirements and design.

The AWD Project Team:

Delivers AWD Design.

This is also referred to as high-level or macro design.

The project’s AWD Design may be part of a larger design.

The purpose is to:

  • Specify cross-release technical details
  • Address the global architectural issues which effect all iterations or releases
  • Minimize long term development cost by eliminating as much rework, from release to release, as possible
  • Further, the business marketplace is characterized today by constant change.
  • Ensure flexible systems that can accommodate unforeseen change
  • Present a high-level application-specific design that allows for iteratively prototyping

This includes:

a) Designing the architecture:

  • Architectural Overview Diagram – simple, brief, clear and understandable schematic diagram that includes subsystems, components, nodes, connections, data stores, uses, external systems and supporting text. Note that this is usually based on the client’s reference architecture patterns.
  • Component Model – how the application as a whole is structured across releases, including how major parts of the application communicate with one another and how project releases are consistent with the overall structure of the application.
  • Operational Model – defines the hardware/software/network infrastructure required across releases, including how the components are distributed to the various infrastructure nodes and how project releases share a consistent infrastructure with other releases.
  • Architectural Templates – defines the cross-release application architectural framework that defines the major application design standards, including appropriate approach to common design problems and proper evaluation and selection of products and technologies.
  • Requirements traceability – provides a cross-reference between all components above and the requirements listed in the AWD Accelerator Plan. Refer to Requirements Traceability Matrix for more information.

b) Designing the release-specific application:

  • Logical Data Model in an Entity-Relationship Diagram (ERD) – capture the business entities, attributes and relationships in the system, along with their attributes, keys and fully normalized form for an optimal physical database.
  • System Process Model in a Data Flow Diagram (DFD) – capture the automated system processes required to support the business requirements and the structuring, relationships and flow between processes, including interfaces and data exchanges.
  • Application Model – platform dependent approach that includes the program module invocation model, high-level module specifications, transaction descriptions and the physical database design to enable code development.
  • Requirements traceability – provides a cross-reference between all components above and the requirements listed in the AWD Accelerator Plan. Refer to Requirements Traceability Matrix for more information.

c) Ensuring the infrastructure supports iterative development, provides consistent use, and uses AWD-dedicated equipment.

d) Ensuring design and architectural approaches identify and leverage reuse opportunities.

e) Obtaining standard client certification, if applicable:

  • Required for applications that change standard end-user configurations
  • Certifies the application will not adversely affect other client-side applications
  • Requires application registration and architectural reviews

e) Providing standards, components and assistance to application teams on:

  • User interfaces – address the global, cross-release design decisions regarding the user interface and help establish the user interface and usability design guidelines, e.g., various window layouts and flows within the user interface are consistent from release to release.
  • Solution plans – create/evolve various plans (e.g., test, training and user support, deployment) from outlining general approaches to specific executable plans.
  • Static testing – test work products using techniques such as inspection, walk through, prototyping, reviews, checklists, mapping and structured workshops to ensure defects are found as close to their source as possible and fixed early in the life-cycle before they are passed on.
  • Test specifications – build test specifications that define and describe the test architecture, elements, approach, data and expected results to verify what the system must do and help prioritize the tests to be performed within the available time and resources while mitigating testing risks (also identifies the hardware, software, processes and tools required to set up, operate and maintain the test environment infrastructure).

f) Leveraging JAD/JAR techniques (see description in Step 1 above), where appropriate.

g) Balancing architectural approaches/standards at various levels: company vs. AWD Center or Program vs. AWD Cell vs. AWD Project.

h) Ensuring the customer’s strategic products and architectures are iteratively enabled through project delivery.

Refer to Application and Technical Architecture and Data Flow Diagrams and Systems Interfaces for more information.
