Work Product Descriptor (Artifact): Release Plan
The Release Plan work product is a high-level project document that defines the timing and contents of each solution release, in terms of the functions to be developed or components of the supporting infrastructure to be completed.

Main Description

A release is a functioning version of the solution that is deployed for actual use, at least to a limited number of users. Larger development projects typically are structured as a series of releases to prioritize business needs, deliver early value, mitigate project risk, and to address development interdependencies. The Release Plan, in conjunction with the Increment Goals Work Product, defines the overall delivery approach for the solution. 

The Release Plan may become an appendix to the formal contract signed with the customer.

Key Considerations

To avoid plan 'paralysis,' do not try to plan the entire project in detail up front.  Instead, break the project up into smaller chunks, allocating the high priority requirements to the early releases.  You might put fewer requirements in the early releases to allow room to adjust for learning curves and for other changes to the team productivity 'model.'

To avoid plan 'churn,' do not stop work to change the plan for the current increment.  If for some reason work will not be finished on time, decide whether to:

  • 'Stretch' the phase to allow it to be completed as a planned part of the current release.  Do this when a high-priority requirement is in jeopardy.
  • 'Shift' the work in question to the next release to keep the release date intact.  Do this with lower priority requirements or in early releases where a learning curve (or other form of 'paralysis') may be involved.  Moving work to the next scheduled release can get the team over the hump.
  • 'Share' the work by allocating additional resources to keep both the date and the work products of the current release intact.  Do this for a complex piece of  work that can easily be divided into two or more pieces that can be worked on separately.  Of course, this solution assumes that some resources are available.

Plan some time after each release to adjust the plan for the later releases.  During this 'adjustment' period, use the complexity measurements of the work to revise productivity estimates and facilitate Release Plans, which should become more and more accurate as the project unfolds.

New functionality that is identified during the development of a release cycle should be allocated to a future release cycle, unless its development is certain not to affect the current schedule.  To maintain planning integrity, let new, high-priority functionality shift lower-priority requirements into subsequent release increments, if necessary.

The Release Plan provides key input to both the Deployment Plan and testing plans, and it may be impacted by these plans.  It is important to make sure that these efforts are coordinated.
