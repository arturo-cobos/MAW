Main Description
The time required to produce this plan will increase in relation to several factors:
  • Size and complexity of the testing risks.
  • Skill level of participants in understanding testing concepts and use of testing techniques.
  • Participants’ knowledge of the business, the application, and the technical environment.
  • Number of stakeholder groups.

The following are some guidelines:

  • ½ - 1 day per business process for documenting business process.
  • 1 - 2 days per business function for developing test scenarios.
  • 1 - 2 days per business function for documenting Acceptance Criteria.
  • 1 - 2 days for defining Test Environment Requirements (do not factor this in if it has already been counted elsewhere).
  • ½ - 1 day per business function for defining Test Objectives.

If using Joint Application Development workshops to define the Use Case/Component models, define the test objectives, acceptance criteria, etc. then multiply the estimates by the number of people participating. However, the documentation estimates will remain the same, as only one person will be doing it at any given time.

The estimating considerations that apply to the Detailed Test Plan also apply to the Acceptance Test Plan. However, there is more work involved in the Acceptance Test Plan, as Acceptance Criteria have to be defined and agreed to. As business users and more client personnel would be involved, much depends on the familiarity and skill level of the participants with the methodology, processes, and tools used to gather and share information. Please allow for additional iterations when the method is used for the first few times. This process is really about organization and culture change, as much as defining acceptance criteria.
