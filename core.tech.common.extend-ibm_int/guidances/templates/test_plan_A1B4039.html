Main Description

Introduction

This section establishes some project level context or background.

Purpose

This section describes the purpose of the Detailed Test Plan.

Scope

A scope paragraph gives a clear explanation of the extent of testing covered by this plan.  It specifically addresses items that are in-scope, and items that are out-of-scope.

Test Focus

Document those attributes of a system that must be tested to assure that the business and structural requirements are satisfied.

Test Objective

Document here the tangible goals for assuring that the test focus areas, previously selected as being relevant to a particular business or structural function, are being validated by the test. 

Approach

The overall approach to how testing will be conducted is defined here.  The approach is described in sufficient detail to permit identification of the major testing tasks and estimation of the time required in completing each one.  Different levels of testing will use significantly different approaches to achieve the objectives of their respective test levels.

Significant constraints on testing such as work item availability, testing resource availability, and deadlines are also identified, as are any special packaging procedures.  Any assumptions related to testing are also stated here.  Staffing needed to support the approach are identified in terms of number and skill level.  Training options for developing required skills are noted here as an alternative to hiring the skills.

The testing approach, test data, and testing priorities are defined here, and include:

  • Details of any other areas carrying out testing during the test level, such as the service delivery, internal audit, operations, business procedures, etc.
  • The approach to be taken for obtaining test data; e.g., live extract, test packs.
  • The prioritization strategy: prioritize the features and items to test and identify considerations for not running certain tests should the project run out of time.  
  • Any boundaries of testing where the level and/or projects are inter-dependent.


Functions / Features to be Tested

Test Functions/Features appropriate to this test level are identified, as well as combinations of such to be tested.  These functions and features are derived from the use case models and the non-functional requirements, which describe the functions and features to be tested for the project.  The definition of what features and functions will be tested at which level of testing is usually arrived at through a workshop, to ensure consistency and completeness.

Test Functions/Features are defined to a sufficiently low level of detail to support an expansion into test cases for a given level of testing.

The criteria to be used to determine whether each Test Function/Feature has passed or failed testing is also noted here. 


Functions / Features not to be Tested

Functions/Features being excluded are noted here to identify out of scope items.


Work Items

Specific work items are defined that are required to test the functions/features, listed in section Functions/Features to be Tested.  In other words, work items are those systems, software configuration items, system interfaces, and documents that constitute the areas or items to be tested.  It may be useful under some circumstances to combine this section with the Test Functions/Features section. 

The extent of the breadth and depth of coverage of the tests during the Test Level covered by this plan is also defined here. 

Each testing level addresses a different set of test items, as is shown below:

  • Unit Tests address programs and modules.
  • Integration Tests address programs and sub-systems.
  • System Tests address systems and interfaces.
  • System Integration Tests address external interfaces.
  • Operability Tests include production environment related work items.

Function and test matrices are often used to ensure completeness and accuracy of test items.

Entry and Exit Criteria for Level

Entry and Exit criteria are usually established during the development of the comprehensive test planning stage.  Entry criteria establish the pre-requisites, which must be met before testing can start, and exit criteria define the steps that must be completed before the test level can be exited.

Testing Tools and Techniques

Testing tools or special techniques to be employed during this test level are identified here. 
 

Suspension Criteria and Resumption Requirements

The criteria used to suspend all or a portion of the testing activity on the Test Functions/Features associated with this plan are defined here.  Testing activities that must be repeated when testing is resumed are identified to ensure that required testing conditions are re-established prior to the resumption of testing.  
 

Test Plan and Schedule

A schedule is prepared for the testing activities at this test level.  A reference is provided to a Plan/Gantt Chart or other planning document appendix.

Testing milestones identified in the Project Plan are identified and the specific test activities to be performed at the test level, the personnel responsible for performing the activities, and the test work products to be produced are also defined.

Test Milestones Identified in the Project Plan

Milestones should include completion of the test plan itself, completion of test specification, test execution and so on.

Specific Test Activities for the Test Level

The set of tasks necessary to prepare for and perform the testing level is identified.  All inter-task dependencies and any special skills required are also noted.  The following types of tasks need to be considered:

  • Test planning.
  • Allocation and assignment of staff for the test level.
  • Test design and specification.
  • Test execution.
  • Test environment planning and set up.
  • Test management.
  • Defect management.


Personnel Responsible for Test Activities

Staff responsible for the different tasks included in the plan is identified here.  These personnel may include the developers, testers, operations/service delivery staff, technical support, user representatives, data administration, and quality support staff.

Points of contact and support for particular applications, platforms, systems, business functions (especially for interface testing), and inter-dependent projects are noted here as well.


Test Environments Requirements

The necessary and desired properties of the Test Environment are noted here.  This section commonly includes:

  • Identification of the physical components, the communications, the system and middleware necessary (hardware and system software).
  • The mode or state of usage (for example, standalone or dedicated to the discrete usage of the application alone).
  • Other software or supplies needed to support testing (e.g., printer forms or specialized stationery).
  • Security and access requirements to the test area and equipment.
  • Test tools and utilities required, including backup/restore requirements.
  • Other testing needs (e.g., publications, office space).
  • Additional software licenses to cover test period usage.
  • Support resources and skills required to maintain the test environments.
  • References to any other projects using the same resources as this project.


Risks and Contingencies

High project risks need to be addressed in this section.  The impact and contingency plans for each are defined.  All potential risks identified in section, Features not to be tested, are addressed here.

Common areas of risk are:

  • Changing business requirements.
  • Changing end-date.
  • Insufficient resource.
  • Test environment restrictions or differences relative to production.
  • Quality problems.
  • A need for special skills.
  • New technology.

The focus is on risks to the testing of the project and not to the project itself.

Approvals

The names and titles of the people who will read, review, or sign-off on this plan are listed here.  Space is provided for signatures and dates.
