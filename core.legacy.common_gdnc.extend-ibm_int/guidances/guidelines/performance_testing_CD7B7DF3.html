Guideline: Performance Testing
This paper will provide practical techniques for performance testing. The discussion will cover performance testing on a conceptual level so that the ideas presented here can be utilized for all tests regardless of what is being tested; i.e., different hardware platforms, systems software, applications, and/or databases.

Main Description
Performance testing as described in this paper should be differentiated from other levels and types of testing. The techniques presented here are for performance tests that occur at a single point in time, normally at the end of the development life cycle; for example, after integration or acceptance testing. This kind of performance testing is different from what is sometimes called the "performance engineering" testing level that occurs throughout the entire life cycle of testing.
 
In this paper, performance testing is defined as understanding how a specific IT system, such as hardware, application or database, will perform under specific conditions. It is different from other types of testing, such as stress testing, where a load is placed on the system to find breaking points or bottlenecks. It is also different from capacity or load testing, where the object is to see how much throughput a system is capable of handling.
 
Performance testing is often similar, and sometimes overlaps, other types of testing, but is focused on understanding the performance of the system being tested, whether it is called a business, a piece of hardware, new software levels, a different database design or a larger workload. There are typically two desired results of a performance test. Understanding how a system performs is the first stage; improving that performance is often the second stage. This paper will concentrate only on the techniques surrounding the first stage of understanding performance.
