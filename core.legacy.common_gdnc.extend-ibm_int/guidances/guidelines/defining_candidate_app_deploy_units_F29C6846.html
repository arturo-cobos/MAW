Guideline: Defining Candidate Application Deployment Units
The objective of this technique is to assist in the preliminary definition of a set of application oriented specified Deployment Units (DUs).
Main Description
It combines information from, among others, the Non-Functional Requirements and Component Model work products, documenting the candidate DUs in a DU WP in preparation for their placement across the Conceptual Level of the Operational Model (COM) in the OM WP. As such, it can be seen to have a close link with the COM’s development technique "Defining Conceptual Architecture".
 
In so doing, it focuses on:
  • Defining each application DU - representing the execution, data and presentation aspects of the components in the solution’s Component Model,
  • Describing the Non-functional characteristics of each DU,
  • Identifying the relationships between the DUs.
 
Other technically related DUs are considered later in the design process, since at the conceptual level, the Operational Model does not consider technical components, or the actual hardware and software products, which will be used to realize the design.
 
As the definition of the solution’s application level Deployment Units evolves throughout a project, it is common for different DUs to be at different levels of accuracy, precision and detail, and the technique should be applied flexibly to reflect this.
 
Note that the example presented in this technique paper is rather simplified from those found in actual engagements (please refer to the more complete examples available in the Deployment Units WPD). Also note that the naming convention used in this example is based on that defined in the Operational Model Notation Technique Paper.

It also takes a "solution requirements" driven (not technology) perspective " although at all times it is important to remember these specified DUs will need to be implemented in code or via a package (customised or not!). Therefore experience and planning ahead is required " for example, it is probably necessary at the very beginning to assess the likely difficult areas and the extent of uncertainty and thinking about this system, as well as have in mind the key technologies or products in which the solution will be implemented.
