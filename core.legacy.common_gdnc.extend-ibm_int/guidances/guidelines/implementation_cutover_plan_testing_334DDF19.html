Guideline: Implementation Cutover Plan Testing
Main Description

1 Description

The objective of this technique paper is to identify steps to corroborate assumptions and logistics identified in the Implementation Cutover plan. This document may be considered a quality control checklist for consideration during the period leading up to full deployment.

Deployment, or cutover planning, is a set of tasks which both the client and project team agree are necessary to install and test the new system. Verifying that the cutover plan is workable and complete is crucial to ensuring the plan is built properly, that the installation will be seamless, and ultimately, that the user will not be unduly interrupted.
 
Testing a cutover plan encompasses the following major areas:
  • Ensuring that data conversion and data migration can be implemented securely
  • Verifying the readiness of all hardware, software and infrastructure components
  • Verifying that user needs before, during and following cutover, will be met

The purpose of this technique paper is to provide guidance on the steps in assessing the readiness of all stakeholders to accept and use the new system and to confirm the testing steps which must be completed before cutover commitment takes place.

The steps within each section may be performed in parallel, or in any sequence. The steps may involve large communities within the organization, from purchasing to facilities management, and/or from training departments to help desks. Each of these verification steps must be coordinated by joint teams involving the project manager (IBM and client), systems architect, developers, testers and business users.
 
When verifying cutover plans, some high-level recommendations include:
  • Adhere to a regimen of staged sign-offs or approvals (e.g. "go" / "no go" decisions) as you execute the cutover plan to ensure that tasks within each step do not get missed
  • When qualifying what functions require scrutiny in the cutover plan, consider those functions which match the criteria "high risk, high usage, and /or high visibility"
  • Ensure all contractual obligations will be met by the implementation
  • Validate that all objectives are met as evidenced through system testing

2 Steps

A checklist of verification steps form the basis of testing an Implementation Cutover Plan:

  1. Verification of training, education, and skill levels
  2. Verification of timing of cutover including schedules and milestones
  3. Verification that infrastructure and components are in place
  4. Verification of site surveys, site plans, and specifications
  5. Confirmation of user support readiness prior to, during and following cutover
  6. Confirmation of hardware and software acquisition, delivery and installation logistics
  7. Verification of data backup, conversion and migration procedure 
  8. Review Preparedness and confirm expectations of user community

The following subsections describe these verification steps in more detail:

2.1  Verification of training, education, and skill levels

  • Selected training strategies are consistent or compatible with the current or planned end user technology environment
    • If several methods were used to provide training, verify the suitability of the training method to suit user needs
    • Where a group of users have difficulty completing work in a test environment, re-evaluate whether the training method or level of detailed instruction is at fault
  • Additional training is planned to provide expanded user knowledge and in-depth training for key users
  • Procedure manuals and user instructions have been exercised during Acceptance testing. Errors caught during testing should be re-tested using walkthroughs and pilot install techniques
  • New roles staffed and new responsibilities are documented as verified during User Acceptance
    • Consideration has been made for those who will be involved in pilot tests (advanced training) or parallel testing (additional staff) when staffing
  • Objectives of user training have been met as measured by the user community
    • All usability and operability functions are accepted by the user community
  • Customization of menus required to access new system has been completed and verified
    • Typically, main menu panels must be updated to enable access to new system functions
  • All user ids,  passwords, and network access have been made available to meet the rollout schedule
  • Maintenance team to support new application is staffed and trained
  • Mentoring has been considered as a means to augment training 
    • Responsibility has been assigned and individual acceptance of the role has been obtained
  • A pre-rollout pilot is  scheduled to verify user training, user documentation, and staffing requirements 
  • Cutover rehearsals have been scheduled and conducted
    • Use the experience to refine time estimates, document procedures, and identify gaps
    • After making changes, walk through the schedules and documentation to ensure that the process continues to ‘make sense’

2.2  Verification of timing of cutover including schedules and milestones

  • Verify that schedule does not conflict with highly ‘sensitive’ timeframes such as period end, month end, quarter end, etc.  Also, avoid periods of high business/transaction volumes, if possible
  • Sequencing of activities has been reviewed
  • Schedule allows for reasonable delays in shipping or setup such that equipment can be operational before the cutover or pilot processing begins
  • Schedule allows time for restoration of the old system (fall-back plan) prior to the final “go/no go” decision. Include:
    -  Library reallocation
    -  Disk/file reallocation
    -  Existing database and file conversion
    -  New database and file loading
    -  Network switch over
    -  System parameter switches
    -  Old system backup
  • Cutover plan is built according to milestones (e.g. cutover tasks to be completed 2 weeks prior to going live, 1 day before going live, weekend of going live, first day after going live, etc.)
  • Time is allowed after cutover before clean-up operations commence to ensure system stability
  • Skills have been identified:
    -  Proper resources have been assigned/committed according to the schedule
    -  Each task is assigned to an individual and confirmation of the completed task will be relayed to the cutover coordinator
    -  A clear and complete responsibility matrix is agreed upon and communicated to each team member, including users, quality assurance (QA), technical support, architects, network specialists, and project managers (PMs)
    -  Note that although frequently the test manager and application development manager are not directly involved in the cutover, they should be on site during the cutover for support purposes because of their experience with the system

2.3  Verification that infrastructure and components are in place

  • Project reporting procedures, escalation procedures, and issue management are in place up to the client management and sponsor level until the project is officially closed
  • Change Management is engaged and prepared
  • Problem Management is engaged and prepared
    -  The Project Manager should be prepared for “hard-ball” decisions on whether a problem is really a Severity 1, 2, or 3.  Although it may contradict the original plan, with the stakeholder’s approval, many installations often decide to proceed into production with Severity 2 problems outstanding depending on the risk incurred
  • Space allocation is checked against latest volume predictions
  • Operations and systems management teams are fully advised and have approved desired schedules including staging of systems, and backups, as well as fallback/recovery stages
  • A centralized ‘burn-in’ center has been established to setup workstations, load software and verify consistency and accuracy of installation prior to shipment to the final destination
    -  This is an effective tool to maximize use of resources and improve quality of the workstation configuration
    -  The burn-in center has an additional advantage of providing a single receiving point for components (e.g. UPS, PC, cables and software may come from different locations)
  • Installability testing performed:
    -  Is the installation document complete and correct?
    -  Is machine-readable data in the correct format?
    -  Can the JCL or EXECs be tailored to meet local standards
    -  Are all required components in the installation package?
    -  Does the installation package include test data? Does it run?
  • Are installation tuning parameters provided? Are they documented?
    Environment testing performed:
    -  All required members and files have been transferred to the production libraries
    -  Application works in production environment (e.g. through a pilot test or running the application on the weekend etc.)
    -  All batch jobs run with all job and file dependencies satisfied and can be completed within batch processing window
    -  All situations, such as daily and month-end, tested
  • Metrics are in place for measurement and processes developed to correct deficiencies
    -  Processes in place to collect / assimilate all information
    -  Performance measurement tools enabled
    -  Predicted volumes in place, reporting scheduled and reporting audience prepared
  • Security procedures have been tested during Acceptance testing and all issues resolved
  • Old system has been left in place for backup purposes for an adequate period of time although transition to the production environment has already occurred
  • Software Configuration Management (SCM) is established:
    -  Service level objectives are in place
    -  SCM activities are planned, and the plan is documented and approved by all stakeholders
    -  Selected software work products are identified, controlled and available
    -  Changes to identified software work products are controlled and problems are recorded/reviewed
    -  Installation includes a basic installation verification check to ensure all components are operational
    -  Affected groups and individuals are informed of the status and content of software baselines
    -  Project follows a written organizational policy for implementing SCM
    -  A board having authority for managing the project’s software baselines exists
    -  A group responsible for coordinating and implementing SCM for the project exists
    -  Members of the SCM group are trained in procedures and methods to perform SCM activities
    -  Standard reports on SCM activities are developed and made available to affected groups

2.4  Verification of site surveys, site plans, and specifications

  • Inventory of locations including current addresses, cable identification, and LAN ids is available
  • Updated inventory including cross-reference of changes to current sites and new locations is available
  • Transition plan to accommodate changes is established
  • All outstanding orders are checked:
    -  No backorders exist
    -  All deliveries are scheduled
    -  All receiving location addresses are verified and personnel have been notified
    -  All shippers are confirmed
  • Access to all locations by assigned personnel is authorized, including third party suppliers, and their agents
  • All phone orders, lines, number assignments, deactivations, and reconfigurations are confirmed
  • Circuit diagrams are updated and complete

2.5  Confirmation of user support readiness prior to, during and following cutover

  • Support roles are defined and staffed
  • Support criteria are defined:
    -  Help desk questions
    -  Error reporting processes
    -  Feedback mechanisms
  • User support criteria is met as verified during acceptance tests
  • Requirements compared to final delivery functions are traced, a gap analysis has been performed, and final sign-off from users has been obtained
  • Full staff support is in place during cutover, key processing periods (first business day, weekend, month-end, etc), system catch-up phase (recovering from deferred transactions during planned system outage), and back-out/recovery, if necessary
  • Communications plan is established:
    -  Cutover to a new system utilizes more staff and impacts more people than at any other time in the life of the project.  Although communication occurs throughout cutover, the importance of effectively communicating – progress, problems, updates – is critical to successful deployment. A cutover ‘kick-off’ meeting is one technique that sets the direction for all follow-on tasks.  The cutover kick-off meeting should have, as its purpose, to review all items which are new and potentially unfamiliar to the participants such as:

Objectives of the rollout and the rollout acceptance (exit) criteria

Roles and responsibilities of each participant

Cutover plan and schedule and fallback procedure in case of an outage or failure of the new system

Batch data transmission schedules

Contents of the documentation

Help desk procedures

System support procedures

Production procedures

System startup and shutdown procedures and schedule

Security procedures

Data backup and restore procedures

New report formats and distribution schedules

Manual procedures which are being introduced

Audit procedures

Processes to be followed to phase out the old system

  • Executive communications very important - A status call-in phone number is established for executives/decision makers to hear a voice recording of up to minute progress during cutover.  In practice, this has proven to be a very efficient, effective, and time/cost saving technique to keep executives informed
  • Notification process is prepared and ready to send according to defined schedule identifying each planned disruption
  • A communication plan must also consider what information is pertinent to the general public, especially in the light that the new deployment may suffer setbacks in terms of decreased functionality, decreased performance or total rollback to conditions prior to cutover acceptance.  This part of the plan and its various contingencies must be developed and reviewed very early in the cutover strategy process to enable all stakeholders to consider wording, timing, media, etc.
  • Pagers are available to contact decision-makers during cutover
  • All workflow procedures and processes are consistent with the ‘new’ system and infrastructure. (Cutover will undoubtedly be ineffective if the business process surrounding the new system implementation has not been effectively tailored.)

2.6  Confirmation of hardware and software acquisition, delivery and installation    logistics

  • Hardware / software meets scalability targets for deployment based on expected transaction volumes and rollout targets
  • Risk mitigation process is in place in case volumes exceeded and/or schedules advanced
  • PC software and associated device configuration standards are established, and rollout and installation plans are verified via a walk-through
  • As cutover begins, there will a need to identify, and correct misinformation.  Here are some of the situations which often occur and the plan must address them:
    • The client machine is not the same configuration (software or hardware) as that originally inventoried
    • The client machine is correctly configured but when software is installed, it appears to operate differently than expected (e.g. Perhaps due to additional conflicting software, or a virus)
    • The client machine is correctly configured and the installation goes smoothly, but the user operates the device in a non-conventional manner.  This concept of users of new systems acting in an erratic manner is frequently prevalent and may cause large increases in Help Desk calls, or server transaction volumes.

2.7  Data backup, conversion and migration procedure verification

  • Backup procedures are in place and scheduled
    -  Application must be capable of being restarted after failure (e.g. restoring data sets, backing out changes, restarting or canceling jobs
    -  Copies of critical programs, data, and documents stored off site as part of the vital records program
    -  The following should be tested:

Can a checkpointed program be successfully restarted from a checkpoint?

Can a batch job be successfully restarted at any step marked restartable

Can a batch job be successfully cancelled and rerun another day

If an on-line program fails, are partial data updates correctly backed out?

Can vital record data be used to recreate the application?

  • All tasks that must be performed manually are staffed, backups are assigned, backout procedures are in place, and training has been completed
  • Conversion, migration, and archive strategy and criteria are in place
  • Procedure to access archived data is documented and distributed to all users and has been tested
  • Data conversion specifications may be site specific; thus, instructions for each site have been tailored as necessary
  • Data owners (3rd party, department, individual) have signed off acceptability
  • Data verification reports have been run and checked for accuracy (hash totals)
  • Data initialization verification has been performed
    -  Run jobs with null data as typical with conditions to be expected of a new system (e.g.  new period end, new month-end, etc.)
  • All entry and exit criteria are met during User Acceptance testing under ‘live’ conditions
    -  Each data extraction, conversion, data formatter, copy or archive process is exercised, results are verified under ‘live’ conditions
    -  Data conversion/ migration should be tested immediately before going live and data used as input to start-up cycle to test compatibility with new system being implemented
  • Define and communicate limitations of Database Fallback and Roll Forward
    -  After x number of production hours/days, a database fallback may no longer be possible and/or practical.  The window to update the old database with production transactions since the cutover may be impractical or impossible
  • Test and verify all interfaces to the new system including manual, automated
    -  All systems have interfaces, whether to a human/manual process, a legacy system, or a recently introduced client server application. Frequently, interfaces involve business partners such as EDI applications, web applications or electronic transmission of business data. Each of these interfaces represent points of failure which must be thoroughly tested, results verified and signed off before deployment. 

2.8  Review Preparedness and confirm expectations of user community

  • Degree of schedule attainment during Acceptance testing is an indicator of level of user readiness
  • Think about the unthinkable – develop a fallback plan and establish a formal plan to invoke it with the user community.   The plan must include go-no go decision points, communications steps, in addition to full details.  Be clear in terms of the desired fallback position (eg.  How much to back out, final configuration/ level of software the client machine should end up with, etc)
  • Make precise decision trees with user executives about who calls a fallback
  • Do not assume that users will continue to react the same way immediately after cutover
    • For example, users who were used to getting system responses in a certain time period, knowing there was a new system at work, would do things that were not at all reasonable such as re-booting, canceling and re-entering etc. This drives the transaction load up on parts of the system far beyond the design points
    • As an offset against getting unexpected behavior from users, test their role in cutover by reviewing expectations with them.  Provide users with an understanding of the following considerations:

Value and Function they will obtain right away

Value and Function they will see over a longer term

Limitations or impact they may expect in the short term, such as events that may cause them to feel anxiety in the first 5 minutes, during the first week, etc.  Rehearse the steps to communicate problems so that implementors can fix them

Based on the pilot or user acceptance tests, review changes that have occurred that they may not be expecting

Rehearse the users role in the fallback plan

  • Try to predict the number of Severity 1, 2, and 3 errors that exist before cutover starts and those that will arise during and following cutover.  Attempt to get user agreement to this number
  • When users know they had approved a cutover with X number of Severity 2 errors, this can make the hours/days/weeks/ after cutover much less stressful

2.9  General Hints, Tips, and Considerations

  • The process of thinking, planning and conducting rehearsal cutovers and fallback provides the team with a great deal of confidence in the final cutover.  Cutover rehearsals are done with a view that you will never need to fallback.  However, the process of thinking about fallback often points out areas where partial fallback is possible. This in turn provides insights to cutover activities that can be reorganized to make the staging of the cutover in smaller phases with intermediate checkpoints a possibility (e.g. some place where data stands still for a little while). This can reduce project risk
  • Perform a risk assessment
    -  The cutover plan must be very thorough. Consider using static walkthroughs.  Review what to test. Review schedule and resource plans.  Ensure that any vacation constraints are reflected in the project plan
    -  Focus on high risk, high usage and high visibility functionality and/or applications.  (Note:  sometimes only one application is being implemented and a focus is on the functionality)
    -  Set client expectations at the outset
    -  Ensure that you have client consensus/sign-off
  • Document for each step of conversion process:
    -  What the major points of failure or risk are
    -  Completion criteria
    -  Steps to backout
    -  Steps for manual override
  • Encourage a "Learning Organization" - hold a post-implementation review for “lessons learned”

3 Roles and Responsibilities

  • Systems administration
    -  Operations Test Exit Criteria passed 
    -  New Operational Procedures developed
    -  Implementation steps and tasks defined, documented and understood
    -  Education and/or training completed prior to implementation for key staff
    -  Security authorization requirements defined, documented and understood
    -  Monitor system to validate new capacity plan meets SLA (Service Level Agreement) targets (e.g. users, CPU capacity, channel activity etc.)
  • Database management
    -  Tools identified for implementation and cutover 
    -  Data migration plan documented (and reversible)
    -  Data conversion plan documented
    -  Plan for data maintenance during data conversion timeframe from source production database
    -  Define, document and communicate limitations to database fallback and roll forward timeframes
    -  Familiarity with new database architecture
  • User Help Desk
    -  Included in operations high level training 
    -  All key contacts should be documented and/or an escalation procedure well defined during early production stages
  • Network/LAN/WAN Support group
    -  Included in operations high level training 
    -  Operations network test exit criteria passed
    -  New network operations procedures developed
    -  Implementation steps and tasks defined, documented and understood
    -  Education and/or training completed prior to implementation for key staff
    -  Security authorization requirements defined, documented and understood
    -  Monitor network to validate new capacity plan meets SLA (Service Level Agreements) targets (e.g. transmission time, block sizes etc.)
    -  External client validation of connectivity to new functionality/application etc.
  • HVAC systems engineering
    -  Air cooling capacity planned and verified, and monitor for production use 
    -  Installations coordinated and/or completed before cutoff plan implemented
    -  Update Business Continuity Plan if applicable
    -  When implementation exit criteria has been completed, plan to remove old hardware configuration should be executed
  • Project manager
    -  Creation of Go-Live report
    -  Act as cutover coordinator or oversee those to whom assignment was delegated
    -  Client sign off on all deployment criteria
    -  Part of team to develop final recommendation to executive sponsor
