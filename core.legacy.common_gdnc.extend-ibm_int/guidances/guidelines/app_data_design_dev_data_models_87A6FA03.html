Guideline: Application Data Design - Developing Data Models
The purpose of this paper is to present fundamental concepts to structured data design for use as part of an application development effort. The topics discussed in this paper are done from a broad perspective and hold for most business applications.
Main Description
Topics related to specialized data design considerations such as dimensional data design techniques needed in many data warehouse projects are not covered in this paper. Nor are Enterprise Models (EMs) covered in this paper. Enterprise-wide modeling is a vehicle by which strategic planning for an enterprise can be accomplished. An EM’s purpose and intent are much more germane to aligning processes, information, and technology with the overall business directions of an enterprise rather than supporting a specific set of business processes associated with a given application. An enterprise model can serve as a high-level overlay to which the models discussed in this paper can be mapped, but is not mandatory in being able to create them.
 
While this paper may present data design principles primarily from the context of developing a new, from-scratch application, elements of the discussion equally apply to other types of projects in which data design efforts may be a part. Examples of other projects in which these concepts may apply include but are not limited to data conversion/migration projects and application enhancements/modifications. The extent to which they apply to any given project will vary per each project’s scope definition and the degree to which data is relevant to the activities to be accomplished within that scope.
 
All the topics covered in this paper have strong affinity to development efforts where process and data design are not encapsulated and treated as strongly bound aspects of an application. This separation of process and data design is a core tenet of data design methodologies where they each are operationally viewed as separate but codependent disciplines. It is based on the viewpoint that, in relative comparison to one another, data – if properly designed – is less sensitive to change than processes over time. If the data design is too strongly aligned structurally to the processes it supports, it then too has the same relative degree of vulnerability as the process regarding change. Such was the case with the designs created prior to the emergence of data design methodologies. Consequently, one of the main objectives of data design is to ultimately support applications with data structures that not only support the application’s processes at its inception but continue to do so with nominal impact from changes time will bring,
 
This viewpoint at first glance may seem at odds with object-oriented (OO) development principles. However, OO modelers may benefit from this paper as many core data design concepts transcend physical implementation concerns and have comparable representation in OO class diagrams. In fact, in some OO development projects the OO class diagram serves as a starting point for data designs ultimately targeting non-OO databases. However, any commonality between OO design and data design concepts is tangential and is not a focus of discussion in this paper.
 
This paper’s topics are listed here and are presented in the following order
  1. Data Design Overview – Describes the overall purpose and broad understanding of data design.
  2. Data Models – Introduces the types of data models developed and their relationship to one another.
  3. Entity Relationship Model (ERM): The Foundation Data Model – Discusses the most fundamental building blocks comprising a data model and the notations used in data models to convey information about the data it depicts.
  4. From Entity Relationship Model (ERM) to Conceptual Data Model (CDM) – Presents the analysis applied to resolve many-to-many relationships in an ERM to progress data design to a CDM.
  5. From Conceptual Data Model (CDM) to Logical Data Model (LDM) – Discusses the analysis applied via normalization rules to a CDM to progress data design to a LDM completing Logical Data Design.
  6. From Logical Data Model (LDM) to Physical Data Model (PDM) – Generically overviews items pertaining to the transformation from logical to physical data design.
  7. Standard Entities – These are conceptual entities that appear across many industries and can be used to frame the discussion of data for a specific application.
  8. The Cross Industry Data Model – These are structured, logical depictions of data using the data design concepts and standard entities discussed in this paper.
  9. Data Design Development Summarization
