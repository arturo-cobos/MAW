Guideline: Integration Testing
Main Description

1 Description

An Integration Test is conducted to prove that a module or group of interfacing modules operates as expected and that the program code performs as defined in the module specification. If an existing Integration Test Plan is being reused as a basis for this Integration Test Plan (Form 12) ensure that the Integration Groups accurately reflect this Maintenance Work Item.

When resolving a Problem Log, some of the test conditions will already have been identified as part of "Recreate Problem," and will be available for reuse from the Problem Resolution Report.

1  Integration Group Definition

Programs that use the same data stores are grouped to form units, Integration Groups, that can be discretely tested to a high level.  For instance, programs that create, modify, delete, and report on a particular record type form a good integration group.  If possible, choose the integration groups so that a common logic module is in only one group. 

However, for maintenance on some legacy systems it may not always be possible for clean cut logical Integration Groups to be scoped.  This is due primarily to the way that the system has been constructed where multiple modules access a small number of very large files.  To then create meaningful Integration Groups would be to create Integration Groups that were almost the entire system.

To help define an Integration Group's scope, reference should be made to the impact analysis that was performed in the Analyse System Components task.  By reviewing such reports as where a data element (field) is used by a module it will be possible to build Integration Groups based on this analysis.

The Integration Group dependency should now be established. 

Complete the following steps using the database prepared in the Construct Database subtask.

  1. Prepare Data - To test a module, prepare both JCL and test data. 
  2. Prepare Test Conditions - A rule that must be tested to ensure the correct operation of a procedure step, module, action statement or statement block.

[NOTE: if there are dates involved, consider the implications for 21st Century]

There are several types of conditions:

  • Permitted Range Checks are test conditions on whether or not a value falls within a defined range.  Values outside the range must be rejected.  A test case is developed for valid, invalid, null and boundary values.
  • Algorithms and Computations are test conditions to check correct functioning of algorithms.
  • Integrity Conditions are test conditions to ensure that integrity rules are correctly applied on database updates.
  • Existence Checks are attempts made to retrieve records that should exist and others that should not exist.

The information recorded for a test condition is the procedure/step/action statement/statement block to be validated, and the rule to be tested.

For example, the statement block or program/module associated with the control condition "month less than 13" results in two test conditions whose rules are:

  • If MONTH less than 13 then valid path taken
  • If MONTH not less than 13 then invalid path taken

Note, it may be possible to be more specific about what a valid or invalid path is.  No test data values are specified here.

3.  Prepare Test Case – For each test condition, prepare a test case consisting of input data and anticipated results.  A comparison of anticipated and actual test results will show if the result is correct or erroneous.

Errors will normally require individual test cases.  Whenever possible, verify several correct test conditions in a single test case.

4. Document Test Case – For each test case, document:

  • Test conditions that the test case is verifying
  • Field values for input screens, documents, and records
  • Initial database state
  • Field values for output screens, reports, and records
  • Resulting database state

Test Cases are grouped into Test Cycles.

5. Prepare Test Cycle

A test cycle is a collection of test cases that are applied in sequence to verify the correct operation of all the test conditions tested by the test cases.  It may be performed as a practical on-line testing session, a batch test run or an entirely clerical trial procedure.  The test cases are applied in sequence within a test cycle.

In general, the test data for a test cycle will consist of an initial database, input data for each test case, anticipated results for each test case and a final database of expected results.

For each test case, all activities of the tester are recorded.  Input data consist of the activities required, together with values for the fields that must be entered.  This assumes that the database, files and screens have results based on the previous test cases.

The anticipated results, documented in the test plan, must be detailed enough to show the specific results that are to be validated by testing.  If multiple test cases have been developed for a test condition, the test cases may belong to different test cycles and may be documented accordingly.  The final test case in a test cycle must provide a full description of the anticipated database.

The Integration Test Cycle dependency should now be established.

6. Define Database State – The formats of the test data will vary, for example, with modification to the test database, dialogue inputs, or inputs to a dialogue simulator.  Rather than define the initial database state and the resulting database state for each test case, it may be easier to do it once for the entire test cycle.

7. Define Test Data – If a module is not involved in integration testing, it is not necessary to define test data for that module.  The module will be tested when its calling module is tested.

8. Sample Integration Tests

9. Permitted Range Checks – Test conditions to check acceptance of a value within a range and rejection of values outside a range must be derived.  A field shall be tested for valid, invalid, null and boundary values.

10. Algorithm and Computation – Test conditions to check the correct functioning of algorithms must be derived.

11. Integrity Rule – Test conditions to ensure that integrity rules are correctly applied must be derived, particularly on database updates.

12. Existence Checks – Attempts must be made to:

  • Retrieve records that should exist and others that should not exist
  • Store records with duplicate keys , including both valid and invalid instances

13. Edit Function – Certain fields (e.g., dates) are likely to be stored in a different format from the one in which they are displayed.  Test conditions must be established to ensure that this works, perhaps using white-box techniques.

14. Input-Output – Test conditions are required to ensure that records are created, updated and deleted.  This can usually be performed by multiple test cycles that use the black-box method.  Test conditions must also be created to ensure the working of basic database functions, such as the attempted creation of duplicate key records, the retrieval of non-existent records, and the changing and deletion of key fields.

15. Output Formatting – Processing related to the arrangement of information on a screen or report must be checked.  A higher-level test must take data from being input on a screen to being printed on a report.

16. Audit Trails – Any internal checks, such as record counts and access constraints, should be tested.

17. Security Violation – Test conditions shall be derived to attempt to violate security.

18. Error Handling – Conditions should be established to identify and report expected (e.g., user-created) and unexpected, system-failure-created (e.g., database-call) errors.

19. Regression Test – This is a test that ensures that no new errors have been introduced into existing modules/integration group.
