Task: Specify Input, Output and Error Messages for Service Operations
Specify input, output, error/fault messages for each of the operations of a service. Use common message format to specify the messages.
Disciplines: Technical Architecture
Purpose

To define the messages for input, output and faults for your operations on the service. Specify the its parameters (input and output) and the definition of the fault message of the service operations. The purpose of operational messages is to provide necessary data for a service to be invoked (in case of input message) or carry the output provided by a service after the service is executed.

        Main Description

        Uses enterprise message format as input for defining the message format for the service operation signature. Operation messages are information sent between two services for information exchange. Messages between communicating services are a critical part of a SOA. These include not only the input and output messages of a given service invocation but also the internal message format to be used within the enterprise as the flow of information passes through the layers of the application architecture.

        Create the specifications for the operations on the service. Service operation specification assumes the definition of the input and output messages, the specification of the faults and any other syntactic information that may be associated with the definition of a service operation. Look for verbs and actions while analyzing the use cases or the functional requirements. Verbs usually map to service operations. Analyze the use cases and or functional requirements to identify what are the parameters that may be associated with the service operation. Analysis of the entire use case list provides an exhaustive list of all the parameters that will be used in the scope of the transformation.

        Operation messages are specified to provide necessary input for service invocation or to capture and carry forward outputs from a service.
        As services include input and output messages, this task focuses on:
        Identification and specification of the format and content of input and output messages of a service, its relation to the underlying data models, the internal common message format and considerations and decisions on how to map each of these messages to another .
        The specification of messages for the service model should take into consideration perspectives from enterprise architecture/application architecture, data architecture and integration architecture. This includes:

        • Message standards defined at an enterprise or application level
        • Appropriate meta or data model that are part of a data architecture
        • Message transformation standards that are part of an integration architecture.

Message specifications and data models are closely tied. The data model consists of underlying entities and their relationships, a subset of which may be sent out as part of an output message and received as an input from an incoming message. Thus the mapping between the message formats and the underlying data model or data architecture is a key architectural consideration. In some cases, patterns and their implementation such as the Enterprise Service Bus may handle the transformation (and routing) of messages. In many cases, we may need an explicit handler to transform messages from and to data models.
