        Main Description
        Applications invariably require the management of state. It is merely a question of where we choose (architectural decisions) to assign the responsibility for state management. Individual services are often considered better to be left stateless; therefore the choreographer of the service is often responsible for the management of state. Alternatively, a component that implements and realizes multiple related services or operations on services may need to maintain state between invocations for performance reasons.

        Services may need to maintain different types of state while be participant in a business process. Service and service operations needs to maintain some states. States can be categorized as security, transactional and functional in nature. Transaction state is required to support transactions spanning multiple messages. If an atomic transaction must include the results of multiple business actions performed as a result of multiple messages from a service requestor (client) to a service provider (server), the state of the transaction must be maintained until all the messages involved in the transaction are completed and the transaction is committed or rolled back.

        Security state looks at how the identity of a consumer is verified. In a stateless scenario, the client is authenticated each time a message is received. In a stateful scenario, a token is passed in the message sent by the consumer.

        Functional state refers to state that must be maintained between messages while a business action is accomplished.

        Documenting the state management decisions of a service is a key input to service implementation.

        These considerations are documented in the service model under State Management Decisions.

        State management between service invocations: State Management in a SOA environment can be considered to fall into three main categories: Transaction State, Security State and Functional State
