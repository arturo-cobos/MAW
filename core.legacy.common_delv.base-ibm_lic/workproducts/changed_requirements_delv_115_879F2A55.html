Deliverable: Changed Requirements Deliverable (DCD 115)
This deliverable provides a single, consolidated source of requirements for minor changes to be made to an application. It contains any additional functional and non-functional requirements, the changes to the data model, the matrix showing which processes use the changed data, and the test plan for acceptance of the enhancements to the application.
Purpose
The purpose of this deliverable is to collect and record changes to the application requirements in one single, consolidated document. This will provide easy access to this information when it is needed.
Main Description
This document should consist of the following:
  • A description of any new or changed functional requirements, using graphical symbols and text to specify how users in specific roles will use the system. The textual descriptions describing the use cases are from a user’s point of view; they do not describe how the system works internally or its internal structure or mechanisms.
  • A definition of changes to the service level requirements (SLRs). These are run-time properties that the system as a whole, or parts of the system, must satisfy. SLRs include capacity and performance (volumetrics), availability, security, system management, portability, and maintainability. SLRs may relate to particular parts of the system, e.g., to particular use cases. This section also includes constraints the system must conform to or satisfy, including the business constraints, the technical standards, and the technical ‘givens’ that the system must satisfy.
  • A data model that provides an implementation-free set of changed requirements for the physical database design task, and clearly and uniquely identifies all changes to business entities in the system.
  • A description of changes to the Process/Data Usage work product. This work product provides a bi-directional mapping between the enterprise processes and enterprise information and/or each entity in the data model and all the processes in the function model accessing it. The processes/functions may create new instances, read existing instances, update or modify existing instances, or delete existing instances of enterprise information / data entities. The matrix is usually known as a CRUD matrix because it documents the nature of the action that a particular process has on each entity.
  • Changes to the Acceptance Test Plan work product. This work product describes the steps that the customer will use to verify that changes to the constructed system satisfy the stated requirements. The contract with the customer should include a clause stating that the system will be accepted formally if all acceptance tests are successful. If there is no direct customer, the contract and the acceptance test plan will be informal, but their existence is still relevant.

The concept of completeness of a work product instance depends largely on whether the work product will be finished in this phase of the project, or iterated for additional work in later phases. If the work product is to be iterated later, it will only be partially complete at the end of this phase. Regardless of whether each work product instance is fully or partially completed at this time, the content of the work product should be fully included in the deliverable being produced.

Each work product making up the deliverable will be produced in a reproducible format. The document also may be delivered in hard copy as a single bound document.
