Artifact: Project Definition (ENG 343)
The document that describes the shape of the project and includes the objectives and scope, the stakeholders and proposed organization with responsibilities, and the major risks associated with the project.
Domains: Project Management
Work Product Kinds: Plan
Purpose
The purpose of the Project Definition is to:
  • Formalize the understanding of the Project Charter by the delivery organization.
  • Provide the plan elements in order to control the Defining activities.
  • Give an initial description of the project "shape" used as a framework for the Planning activities.
  • Provide the plan elements in order to control the Planning activities.
  • Gather the fundamental characteristics of the project in a unique document.

Description
Main Description

The Project Definition contains the results of the Defining activities. Those results are used as a starting point to build project plans.

Brief Outline

The Project Definition contains the following components.

  • Project objectives
  • Project background
  • Target solution and overall approach
  • Project scope
  • Planning framework
  • Organization
  • Financial
  • Appendix A, B, and C
Notation
Content Comment
Project identification  
Project / Sub-project  
Project / Sub-project manager  
Project objectives Based on the understanding of the Project Charter by the delivery organization, describe the set of objectives the project must achieve.
Project background  
Business needs and environment Description of the problem to be solved, and of the business areas under consideration. (Market segments, competitive analysis, sponsor activities etc.)
Project general scope and plan constraints Identify the specific needs to be addressed and provide key dates and states.
Solution background Provide a history of previous projects and solutions.
Key requirements List the high-level requirements that the project must meet.
Target solution and overall approach  
General presentation of the target solution Describe how the project will solve the business need.
Major components (high-level PBS) and level of uncertainty Provide a high-level PBS for the major components of the solution. Include information about the level of uncertainty and complexity of the solution.
Key options for the technical environment If the project has special requirements regarding the technical environment then document those key options that influence cost, schedule, and possibly Agreements when subcontractors are involved.
Overall approach Describe, at a high level, the way in which the project will deliver the solution.
Project scope  
Position within the overall approach If necessary, position the project within the overall approach.
Scope Identify the boundaries of the project.
Key points Identify any key points that must be addressed in the Agreement.
Major deliverables Outline the major products or services that will be delivered.
Customer satisfaction criteria (Conditions of satisfaction) Document any informal understanding of customer satisfaction issues that should be transferred, if needed, from the team who shaped the project (before the Agreement) to the team who executes the project.
Planning framework  
High-level project schedule Provide a high-level project schedule for any projects or services and any sub-projects.
Brief description of each work unit Break the project down into its major work units. (High-level WBS)
Brief description of each sub-project If necessary, for each sub-project provide a brief description.
Key assumptions and rough estimates List the key assumptions which, if invalid, would make it necessary to revise the plan. Estimate roughly the amount of effort and duration of each work unit.
Key dates and dependencies List any key dates such as target dates for milestones. List any key dependencies.
Overall risks and risk response List any risk factors associated with the project. Identify planned responses to these.
Organization  
Project management Provide information about the structure that will manage the execution of the project.
Sponsor organization Provide information about the sponsor organization.
Delivery organization Provide information about the delivery organization.
Performing organizations Provide information about the performing organizations.
Initial organization breakdown structure (OBS) Description of how the project will be organized into project organizational units.
Necessary skills (high-level HR plan) Document any necessary skills that are required of project team members.
Initial project decision structure Provide an initial description of the Project Decision Structure.
Financial Costs, revenue and cash, as applicable.

  • Objectives
  • Organization
  • Work products
  • Milestones
  • Schedule
  • Management system
  • Assumptions, dependencies, and risks
  • Costs (if applicable)
Appendix B - Planning activities plan For Planning activities, create a plan that includes:
  • Objectives
  • Organization
  • Work products
  • Milestones
  • Schedule
  • Management system
  • Assumptions, dependencies, and risks
  • Costs (if applicable)
Appendix C - Initial project definitions If necessary, for each sub-project, or project within a program, provide initial Project Definitions for each that includes:
  • Project name
  • Project description
  • Objectives
  • Key milestones
  • Major work units and associated work products
  • Start and finish dates

Key Considerations

An initial Project Definition is created after assessing the project goals and before refining the solution at a level that is appropriate for planning. It contains the reminders and the plan for the Defining activities.

The Project Definition is then developed during the Defining activities, by adding the planning framework and the plan for Planning activities.

The Project Definition is written by the project manager. It is approved according to the document control procedure for the delivery organization.

Tailoring
Impact of not having

Lack of a Project Definition leads to:

  • Possible misunderstanding of the Project Charter by the delivery organization.
  • Unclear view of the feasibility of the solution which can affect the reliability of the plan being defined.
  • Difficulty in ensuring there is a common view of the solution and of the related approach inside the project team and with the sponsor.
  • Lack of clear boundaries for the project scope that leads to:
    • Risks of gaps or overlap between the roles acting on the project.
    • Poorly defined or inadequate plan for the project.
    • Subsequent difficulties in controlling the project.
  • In the shorter term, difficulty in controlling Planning activities.
Reasons for not needing

The Project Definition, or another document with similar content is needed on every project. On a small project, it can be a short document.
