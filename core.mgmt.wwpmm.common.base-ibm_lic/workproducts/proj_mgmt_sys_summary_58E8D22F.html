Artifact: Project Management System Summary (ENG 339)
For each of the plans , procedures and records which document the Project Management System, this work product typically:
- either provides a brief overview of the related work product and specifies the location of the full document
- or (on small projects) may replace the work product entirely.

This plan is a lean, summary of the agile project management system, major project schedule milestones, schedule of the current phase, resources assigned to the project and resource allocated plans. It is written to support agile and lean project delivery.
Domains: Project Management
Work Product Kinds: Plan
Purpose

The purpose of the Project Management System Summary is to:

  • provide a "table of contents" for the Project Management System documentation
  • help ensure that all members of the project team understand: 
    • how the project will be managed, and
    • their role within the management process
  • document the Agreement between the project manager and constituency senior management regarding how the project will be managed
  • document any agreed deviations from constituency policy and practice that have been agreed with senior management and, if appropriate, the quality assurance function
  • provide a starting point for project audits or compliance reviews.

It is used to formally define the project management system. It is the responsibility of the agile project manager to determine the level of detail that is appropriate for a given project. During solution development, detailed activities are managed outside of this plan by the team and project manager using the Accelerator Plan.

For large projects, where the sections of the PMSS become too large, then the supporting stand alone template should be pulled from Method Web.

Description
Main Description

The Project Management System is the way in which the project is managed. It is documented as a collection of plans and procedures, which direct all project management activity, and records, which provide evidence of their implementation.

For each of these plans, procedures and records, the Project Management System Summary typically:

  • either provides a brief overview of the work product and specifies the location of the full document
  • or (particularly on small projects) replaces the work product entirely.

The Project Management System Summary is a relatively short document. The level of detail is such that all the key aspects of the project (i.e. those of which the sponsor and/or functional management should be aware) are fully documented but less important information is relegated to the supporting plans, procedures and records.

Prepare the initial draft of the PMSS for the project.

The PMSS contains the following components:

  • Management System Summary
  • Project Schedule
  • Resource Plan

The Management System Summary:

  • Summarizes the plans, procedures and records required to manage the effort
  • Helps ensure that all members of the team understand how the effort will be managed, and their role within the management process.
  • Documents the agreement between the AWD Manager and constituency senior management regarding how the effort and resources will be managed
  • Documents any agreed deviations from constituency policy and practice that have been agreed with senior management and, if appropriate, the quality assurance function.
  • Provides a starting point for audits or compliance reviews

Typically this summary either:

  • Provides a brief link or reference to other documents, OR
  • Replaces separate work products entirely

The summary, along with corresponding tasks in the Project Schedule, specifically replaces these work products:

  • Communication Plan – Specifies communication activities, information and audiences.
  • Configuration Management Plan – Configuration management identifies the hardware/software configuration at a point in time, and the systematic control of changes to the configuration for traceability throughout the development lifecycle. The plan helps in the execution of CM activities by defining tool(s), steps, roles, responsibilities, frequency and timing of audits, and the software work products to be placed under configuration management. It also helps establish and maintain the integrity of the software work products. Refer to Software Configuration Management Plan and Software Configuration Management Checklist for more information.
  • Quality Plan – This plan describes the strategy and tasks required to ensure deliverables meet acceptance criteria, e.g., quality assurance, process assurance and work product inspection / peer review activities necessary. All quality activities to be performed are defined and include into the Project Schedule. This includes quality-building activities performed by people inside the team and others outside the team.
  • Launch Strategy – Specifies number and names of launch locations, launch group, steps, risks, documentation required, training, infrastructure, system feeds, data conversions, support, help desk, contingency plans, third party hardware/software agreements, online Frequently Asked Questions (FAQs) and timing. Refer to Deployment Plan Checklist for more information.

The Project Schedule and Resource Plan are typically created using a scheduling tool, help from stakeholders and specifying the following:

  • Overall Calendar – Specifies business days and holidays
  • Work Breakdown Structure (WBS) – This is a hierarchical decomposition of the phases, activities and tasks that the AWD Team must complete. Guidelines:
    • Use an AWD-approved WBS from similar past AWD efforts, if possible.
    • Fully represent and conform to the AWD Approach, i.e., the WBS must align with this procedure.
    • Begin all Activities and Tasks with a verb
    • Tightly align the WBS with the Agile Project Definition and Agile Project Definition and Charter, making activities for each deliverable explicit
    • Include standard foundational entries (e.g., contingency of 10%-25%) and management entries (e.g., open, manage and close activities)
    • Ensure all entries contribute directly to the delivery of a specific deliverable and/or represent a step in this procedure
    • Ensure tasks require at least 40 hours of work to minimize tracking and administration overhead
    • For project WBSs, use multiple iteration timeboxes for requirements, prototyping and development.
    • Plan at the highest level possible, (use days if possible), the day-to-day tasks will be managed by the team using the Accelerator Plan.
  • Resources – For each resource, specify name, ID (i.e., last name or equipment type), general availability hours, cost and calendar (i.e., ensure vacations and personal time are fully represented). Ensure alignment with Agile Project Definition and Charter Roles and Responsibilities and facilities/equipment specified in the estimate section of the Agile Project Definition and Charter.
  • Task dependencies – For each task, define predecessors and successors, and for each define its type (e.g., finish-start) and any gaps or overlaps. Systematically tie tasks from the Start Effort task to the final task through a network of dependencies.
  • Task resource assignments – Assign every task to one or more resources and tightly align with the Agile Project Definition and Charter Organization and Responsibility Overview Table
  • Task resource hours and cost – Provide a top-down estimate for all tasks based on timebox dates. Plan resources for a 40-hour week and avoid overworking, since a 40-hour week is a proven accelerator. Ensure a contingency for known tasks of 10%-25% for all core resources. Refer to Estimating Report Checklist for Agile and Estimating Considerations Checklist for more information.

Review and finalize scheduling information and ensure the critical path is clear. Ensure start and end dates and resource levels are reasonable and in alignment with the Agile Project Definition and Charter, and ensure resource weekly hours are level. Ensure the schedule is realistic and achievable.

Integrate the Project Management System Summary components.

Review with stakeholders and update to reflect their feedback

Obtain all approvals, including from Sponsors and other key stakeholders.

Baseline the Project Management System Summary to allow full tracking

Ensure the final version is accessible to the team, password-protected and version controlled.

Prepare the first draft of the Accelerator Plan for the project.

The Project Schedule and Resource Plan typically provide phases, activities and milestones, and the Accelerator Plan provides detail below this structure. In some situations, the Accelerator Plan may serve as the Project Schedule and Resource Plan. In any case, the Accelerator Plan serves as the primary and agile planning tool while defining, developing and deploying the solution.

Use the Accelerator Plan to explicitly plan use of high-return Cross Reference to AWD Accelerators/Best Practices, and empower the AWD Project Team to review and easily adjust functionality within given timeboxes. This includes:

  • Identifying, profiling and prioritizing each user group: a group of users with unique functionality in the resulting system (e.g., separate screens, reports or processes). For example, banking customers utilizing web services may be higher priority in a banking application than an internal bank group assessing utilization of online banking services.
  • Identifying and prioritizing each business function/requirement of the solution
  • Using the Iteration column of the Accelerator Plan to group prioritized-requirements into three equal-sized iterations
  • Defining timebox end dates and ensuring they match the Project Schedule
  • Defining completion criteria for each timebox, i.e., describing work products required by the Timebox End Date
  • Defining the reuse plan by identifying opportunities for reuse of work products from other projects, and work products the project will create that may be reused by other projects. Possible areas of reuse include applets, application services, complete sub-routines, documentation, and test cases. Refer to Candidate Asset List for an alternative approach to listing reuse candidates.
  • Describing the approach for using other Accelerators/Best Practices, including exceptions
  • Reviewing and fine-tuning
  • Obtaining approval from the Business Customer (and Developer if one has been assigned)

Key Considerations
The Project Management System Summary is created during Planning. It may then be expanded during Starting and is kept up to date throughout the project.

Typically, the Project Management System Summary is reviewed by the project team and approved by other stakeholders such as the sponsor, senior management within the delivery organization and the quality assurance function.

It is a key input to all compliance and management reviews.

Project Managers should consider the following items when putting together the Project Management System Summary:

1.  Define the project management processes that are followed when non-WWPMM standards, processes, and/or deliverables are required by project stakeholders.

2.  Include cultural differences and allow for different processes dependent on the receiving country (e.g., how are agreements reached successfully in different cultures?).

  • Cultural differences will need to be factored in if the project team includes global resources or other cross-border organizations.

3.  Consider the possibility that the host country, global resources organizations and the client may all use different processes, standards, etc. that if left unaddressed, may cause significant conflict between various parts of the project team during the course of the project.

  • In order to have control on effort/cost and quality there should be one set of project processes, procedures, etc. that should be agreed to by the host country, global resources, other cross-border participating organizations, and the client.
  • For example, there is the possibility that the project will need to determine which CMMI level to follow. If the global resource organization is at a different level than the managing host country, it may produce more or less documentation than the project requires.
  • Therefore, it is important for the project manager to work with all the project stakeholders to define a single set of processes, procedures, standards, etc. that all can agree to use from the outset.
  • The decisions should be clearly documented, communicated and agreed to by all stakeholders during the early stages of the project life cycle so there is no confusion amongst the project team during project execution.
