Artifact: Technical Environment Plan (ENG 361)
A description of the required technical environment , the plan and schedule used to build and/or acquire it, the plan and schedule to make it operational, and the plan to manage it.
Domains: Environment
Work Product Kinds: Plan
Purpose

The Technical Environment Plan is designed to:

  • Provide a clear description of key options for the technical environment that will contribute to the Project Quality Plan.
  • Determine cost and risks associated with the technical environment, and thereby contribute to the Financial Plan and the Risk Management Plan.
  • Schedule implementation and maintenance of the technical environment.
  • Describe roles and responsibilities for development of architecture, implementation and maintenance of the technical environment.
  • Record utilization rules and descriptions of procedures for administration of the technical environment.

Description
Main Description

The Technical Environment Plan describes the technical environment architecture and collects schedules for implementing it.

The technical environment typically includes:

  • A suite of office tools including spreadsheet, word processor, presentation and other tools.
  • Communication tools including network, telephone, e-mail, collaboration (instant messaging, access to shared office suite files) and other tools. 
  • Project management tools that provide:
    • Project management and operational scheduling
    • Time tracking
    • Cost tracking
    • Risk management
    • Project control book (PCB)
    • Access to business systems
  • Tools that support technical activities in all project phases:
    • Conception
    • Development
    • Qualification
    • Launching or deployment
  • Features of the project management and the technical tool set that support the configuration management of the technical work products.

Technical and application architecture schemas for the technical environment The schemas are based on a classification of workstations and potentially a classification of servers.

These classifications take into account the types of users, function sets, and sets of files defined when describing functional needs.

When it is possible, the architecture should be compared with existing intellectual capital assets.

The hardware architecture schema Describing elements of equipment: nodes (machines with their cards and peripherals...) and connections that make up the network of interconnected machines.
The software architecture schema Describing the basic software installed on the hardware architecture: operating system(s), communication software, DBMS, transactional monitors, etc.
The application architecture schema Describing the implementation of the tools on the technical architecture from the point of view of the required features.

Key Considerations

Most of the Technical Environment Plan must be defined during the shaping and the planning of the project. The feasibility of the solution, the estimates, and the success of the overall project might be strongly dependent on this plan. Costs relating to the implementing and maintenance of the technical environment must also be included in the Financial Plan before the Agreement is prepared.

Development of the Technical Environment Plan is a two-step process:

  • Identifying the key items during defining and planning
  • Adding details when starting the project, sub-project, or new phase

Verify that the technical environment implementation schedule matches the dates in the project management schedule. Verify that the costs are in the Financial Plan.

After the plan is implemented, a Technical Environment Assessment can be used to validate and verify parts of the Technical Environment Plan.

Tailoring
Impact of not having

When the Technical Environment Plan is not established:

  • There is a risk that key options for the technical environment will be revisited and re-discussed several times during the course of the project.
  • Project costs and risks may be underestimated and, consequently, poorly controlled.
  • Work teams might run short of technical resources.
  • There is a risk of a poor collaboration on the project because of a lack of utilization rules.
Reasons for not needing

The following parts of this work product should always be created:

  • Key options for the technical environment that should be agreed with the sponsor before the start of the project.
  • Description of the required features that must be satisfied, whether by the sponsor, delivery organization, or suppliers.
  • Utilization rules.
