Artifact: Delivery Control Documentation (ENG 327)
A paper or electronic record from the delivery organization to the sponsor or to a supplier to furnish evidence of release, receipt, and/or acceptance of a deliverable . This document has up to three components: a delivery notification, a delivery receipt, and an acceptance notification.
Domains: Project Management
Work Product Kinds: Record
Purpose

The purpose of the Delivery Control Documentation is:

  • For the releasing role, to clearly identify authorization to release and to record notification from the accepting party about the deliverable being released, according to the plans and agreement.
  • For the accepting role immediately after the release, to record that the deliverable seems complete so that validation work can start.
  • For the accepting role after the validation work has been performed, to record notification to accept or reject the deliverable.
  • For all the project stakeholders, to be able to track and provide any requesting authority with the evidence that the deliverables have been released / received / accepted.

Description
Main Description

The Delivery Control Documentation accompanies a deliverable from the time it is released until it is accepted.

The Delivery Control Documentation has two sub-types depending on how the releasing and accepting roles are distributed:

  • The project releases the deliverable and it is accepted by a sponsor external or internal to the company. The subcontractor releases the deliverable and it is accepted by the project organization.
Brief Outline

The Delivery Control Documentation includes the following components :

  • Delivery control identification
  • Authorization to release
  • Delivery receipt
  • Acceptance notification
  • Post delivery history

Key Considerations

The components of the Delivery Control Documentation work product are created or updated based on the following events:

  • An authorization to release is sent by the releasing organization to notify the accepting organization about the delivery.
  • A delivery receipt sent back by the accepting organization to the releasing organization to acknowledge the receipt of the deliverable and confirm that validation work can start.
  • An acceptance notification sent by the accepting organization at the end of the validation in order to accept or reject the deliverable.

In simple cases like partial deliveries or when sending a document to another stakeholder only for information, a simplified variant is used that contains only the authorization to release and if necessary, the reference to another Delivery Control Documentation instance.

Tailoring
Impact of not having

Consequences of not using the Delivery Control Documentation are :

  • The project stakeholders will have difficulty in determining which deliverables have been released and/or accepted and which have not.
  • The accepting party will have difficulty in understanding when the validation work should start.
  • In the event of an audit, the authorities requesting evidence that the planned deliveries have actually occurred cannot be provided with appropriate documentation.
  • Management of the configuration of the deliverables will be difficult because their status is unclear. As the composition of the configuration baseline depends on the status of these deliverables, it can also affect the clarity of the configuration baseline.
Reasons for not needingThe projects that deliver only services may not need to use the Delivery Control Documentation.
