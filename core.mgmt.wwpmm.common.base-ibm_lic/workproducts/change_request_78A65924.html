Artifact: Change Request (ENG 319)
A request to change some document or aspect of the project that has been placed under change control. An accepted change request may result in one or more change orders .
Purpose

The prime purpose of change management is to ensure that proposed changes that are of benefit to the project are implemented in a controlled manner and that proposed changes that are of little or no value are not. Change Requests support this objective by providing:

  • A record of the proposed change and the rationale behind it.
  • A record of the results of any analysis of the potential impact of the proposed change.
  • The information required by the change control board to decide how to proceed with the proposed change and a record of those decisions.
  • Together with the Change Orders, the information required to monitor the implementation of the change through to completion.

Description
Main DescriptionA Change Request is a form that contains all the information required to support the decision making processes that are used to manage the unforeseen changes that inevitably arise during the course of all but the simplest of projects.

Action Log
Brief Outline

Change Requests include the following components which map to the stages in the life cycle of a change.

  • Identification, which includes a description of the proposed change and the rationale behind it.
  • Assessment, in which the outcome of the initial assessment of the proposed change is recorded.
  • Analysis, which includes the outcome of the impact analysis.
  • Implementation, which includes the completion criteria and the planned finish date.

Key Considerations

Any stakeholder can raise a Change Request.

After a preliminary assessment of the proposal, the change control board decides whether the Change Request is to be approved for implementation (small changes only), approved for analysis (large changes only), deferred or rejected.

If an impact analysis is performed, the change control board evaluates the results and again decides if the Change Request is to be approved for implementation, deferred or rejected.

If it is decided that the proposed change will be implemented, the required Change Orders are raised.

The Change request is closed when all the corresponding Change Orders have been completed.

Note: The approach outlined here allows for one Change Request to be linked to more than one Change Order and for one Change Order to be linked to more than one Change Request. It is possible to adopt a simpler, but less flexible, approach in which one Change Request is linked to one (notional) Change Order and vice versa. In this case, there is no need for a separate Change Order work product.

Unforeseen change is being managed effectively.

Change Requests are being processed according to the rules documented in the Project Procedures Description.

Note that there may be more than one change management procedure. For example, Change Requests raised by the sponsor may be handled in a different way from those raised by a project team member or changes to the technical work products may be managed in a different way than changes to the project management plans.

Tailoring
Impact of not having

Possible consequences of not using Change Requests include:

  • Poor decision making about which proposed changes should be implemented.
  • Confusion in the project team about what changes are to made.
  • Inadequate analysis of all the activities required to implement a proposed change.
  • In the worst case, the project running out of control with different people working to different plans.
Reasons for not needing

It is company policy that all projects have change management procedures. Consequently, Change Requests are required in all projects.
