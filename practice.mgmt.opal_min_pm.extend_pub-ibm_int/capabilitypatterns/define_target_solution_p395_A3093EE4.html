Task Descriptor: Define target solution (IMSP395)
This task will develop a high-level solution architecture and a high-level product breakdown structure, (which decomposes the solution into components), and will assess the feasibility of the solution, including identifying initial risks.

Main Description

This task is launched when architectural work needs to be performed in order to identify the main components of the target solution and the areas of uncertainty. This might occur when the sponsor does not have a clear vision about the target solution or when the design of the solution that is available needs to be refined to provide a sound basis for the Planning activities.
As a target solution may often be delivered by more than one project, activities specific to a project, such as plan creation, might often address only a subset of the target solution.

Gain an appropriate understanding of the requirements to enable the organization to develop an appropriate response.

  • Team Solution Design should be utilized as the basis to build the technical solution.
  • Consulting projects should use the appropriate method for their type of solution. If an optional pre-bid Consulting Review was conducted, it would be done here.
  • AMS projects may utilize the Requirements Development and Management Procedure (IMSP576). Identifies and document the organization and client acceptance criteria (for deliverables and work products) based on the client requirements and the work request.

Search for existing solution designs, architectures, or Project Definitions and plans that may be used (partially or fully) to fulfill the client's requirements.

Consider whether the product components may be purchased or reused, based on established criteria such as cost, available resources and skills, critical dates, functionality of available products, availability, and so on.

  •  Refer to the Key Decisions Procedure (IMSP445) for information on formal decision-making practices.

The design team will perform one of the following activities:

  • Use existing solution architecture:
    If there was a match of the selected solution architecture against the solution strategy, the design team will simply refine, at a high level, the existing hardware, software and services components of the proposal
  • Customize existing solution architecture: (less than 100% match)
    If there was less than a 100% match of the selected solution architecture against the solution strategy, the design team will customize, at a high level, the existing hardware, software and services components of the proposal
  • Create new solution architecture: (no match)
    If potential reusable solution architectures could not be found based upon the solution strategy, the design team will create or design, at a high level, the hardware, software and services components of the proposal.

Record the high level solution architecture, including alternative solutions and solution engineering methodology used in the target solution outline section of the Project Definition.

Review the solution architectures of any other existing work that may impact on this proposal.
Consider any other work being conducted that may impact on the solution architecture you are proposing. It is possible that there may be multiple work requests or projects that impact the same applications and systems. If so, you must ensure that the solution architectures are compatible.

This activity may also include the following:

  • Create high-level product breakdown structure (PBS)
  • Investigate requirements regarding technical environment
  • Conduct an initial assessment of the viability of the solution, ensuring that the solution is viable based on various risks, assumptions, issues and dependencies.
  • Document risks attached to solution
