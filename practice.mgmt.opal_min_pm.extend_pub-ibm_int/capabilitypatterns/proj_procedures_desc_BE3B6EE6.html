Work Product Descriptor (Artifact): Project Procedures Description
Documentation of the procedures to be used on a project. It includes procedures related to:
- The technical work itself (e.g., how reviews of deliverables must be conducted).
- The project management of the work (e.g., how change requests must be handled).

Main Description

The Project Procedures Description describes the procedures that are used by the project team to complete the work of the project. The Project Procedures Description either actually contains the procedures or contains reference pointers to the delivery organization’s procedures that apply to the project. There are two types of procedures:

  • Technical/work procedures, which define how the technical work of the project is performed to produce technical work products.
  • Project management procedures, which define how the project is planned, tracked and controlled.

Key Considerations

To create the Project Procedures Description, sample constituency procedures from the delivery organization’s business unit and competency are selected and adapted based on applicable policies, sponsor requirements, and the project context.

An initial version is produced during planning. This is then expanded at the start of execution and maintained throughout the remainder of the project.

The Project Procedures Description is created for the top-level project organizational unit within the project organization structure. The top-level project organizational unit uses the policies and procedures of its constituency organization to identify the policies and procedures that apply to the project. These policies and procedures cascade down to the lower project organizational units within the organizational breakdown structure. Each project organizational unit may add to or expand the policies and procedures from above by using the polices and procedures from its constituency to augment the Project Procedures Description to cover work at its level.

  • The sponsor may wish to review the Project Procedures Description. Discussing the content of the Project Procedures Description with the sponsor will help to increase confidence in the project.
  • It is usually necessary to gain the approval of senior management for any deviation from normal business policies.
  • As the project proceeds, it is likely that external factors (e.g., a change in company policy) or internal factors (e.g., a procedure that is delaying progress unnecessarily) will need to be changed. Such changes should be managed using the Issue management procedure and documented as a revision to the Project Procedures Description.
  • Where applicable, the project procedures must conform to ISO9000 and any other relevant standards that have been adopted by the delivery organization.
  • Internal procedural reviews should be conducted on a periodic basis to ensure that the procedures are being followed and are effective.

A managerial health review will typically be performed on the Project Procedures Description to ensure it is complete, clear, and reflective of the work planned in the Agreement. Compliance and health reviews conducted during project execution should ensure that the Project Procedures Description is maintained, effective, and followed throughout the life of the project.

Project Managers should consider the following items when putting together the Project Procedures Description:

1.  Define the procedures that are followed when non-WWPMM standards, processes, and/or deliverables are required by project stakeholders.
2.  Include cross-cultural differences and allow for different processes (for example: how are agreements reached successfully in different cultures?).

  • Cross-cultural differences will need to be factored in if the project team includes global resources or other cross-border organizations.

3.  Consider the possibility that the local country, global resource organizations and the client may all use different processes, standards, etc. that if left unaddressed, may cause significant conflict between various parts of the project team during the course of the project.

  • In order to have control on effort/cost and quality there should be one single set of procedures that should be agreed to by the local country, global resources, other cross-border participating organizations, and the client.
  • An example is to determine which CMMI level to follow. If the global resource organization is at a different level than the managing local country, it may produce more or less documentation than the project requires.
  • Therefore, it is important for the project manager to work with all the project stakeholders to define a single set of processes and procedures that all can agree to use from the outset.
  • The decisions should be clearly documented, communicated and agreed to by all stakeholders during the early stages of the project life cycle so there is no confusion amongst the project team during project execution.
