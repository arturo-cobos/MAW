Deliverable: Requirements Specification (DCD 112)
This document will be used to gain consensus between the client and IBM on the requirements and it will provide the foundation for all future project efforts. This document will further refine the requirements defined in earlier requirements documents.
Purpose

The purpose of this document is to refine and further document the requirements that must be met by a new Solution.

Main Description

This document will contain the following:

  • The Use Case Model which defines the identification-level use cases for the solution and their high level interactions. A subset of the key use cases will be extended and refined to further define the requirements.
  • A system context diagram that identifies the interfaces, data sources, and interface frequencies.
  • The usability requirements and, where possible, the measurement details for specific user profiles, consisting of preconditions and performance.
  • The nonfunctional requirements with defined minimum and maximum acceptable criteria for the system.
  • A catalog of the business rules, policies, and governances that must be addressed by the solution.
Deliverable Properties
External Description

This deliverable expands upon the work outlined in previous requirements by:

  • Adding detailed interface characteristics to the existing system context diagram for number of instances, available access time, type of communication, hardware brand/model, operating system brand, volume and frequencies. 
  • Defining the actor-system interactions for the use cases included in this Release, document the expected usage and behaviors of the system, and restructure and layer the use case model to separate out commonality and avoid redundancy. 
  • Refining the non-functional requirements, which define the service level requirements, system constraints, and other technical properties for the solution, to produce a complete description of the defined minimum and maximum acceptable criteria for the system. 
  • Refining and extending the usability requirements identified during the earlier phase, as needed, to further identify the requirements for the solution and to add measurement details. 
  • Documenting the business rules in order to communicate policy issues relative to the scope of the solution.

The customer will provide input to the IBM team to define/refine the requirements.  This includes providing input regarding use cases, non-functional requirements, usability requirements, and potential business and I/T changes for the IBM project team refining the requirements.  Additional information regarding information exchanges that will be required between the new solution and the external entities that will need to interface with the solution, including the information that must be input and/or output in the exchanges.

Packaging Instructions

This document will be comprised of information formatted for delivery to the client extracted from the following work products:

Business Rules Catalog – provides the detailed business rules that need to be enforced by the system.  Each business rule that is to be implemented by the system needs to be explicitly defined along with any necessary formulas or criteria.  Business rules that are not implemented in code but may affect the system should also be documented here with a brief description explaining how conformance will be achieved.

Change Cases (optional) – defines any additional possible changes to existing requirements.  For each change case it will be necessary to clearly document whether it is or out of scope.  For in scope change cases the likely-hood of each changes case including conditions for the change, as well as it’s expected impact need to be fully documented.  Any assumptions on which this information is based need to also be included in the change case.

Non-functional Requirements – defines the technical requirements and business constraints that will affect the development of the system.  Business constraints need to be clearly defined along with their expected impact to the development effort.  The technical non-functional requirements such as items dealing with performance and capacity need to be defined to a measurable level so that it is possible to judge whether each requirement has been met.

System Context – is a graphical representation of all other systems that will interact with the new solution.   For each system the exact calls that will need to be supported along with the expected data volumes and any limitations on availability need to also be documented.  Any other criteria related to these systems that may have a limiting affect on the new solution or require additional work should also be documented.

Usability Requirements – defines the requirements in relation to the user experience of the new system.  Each usability requirements must be tied to a specific user profile and defined to a measurable state.  Any assumptions or pre-conditions should also be identified along with the expected task, completion time, completion criteria and number of errors.  The goal is to provide a set of requirements that can be measured during usability testing with candidates that match the specified user profile.

Use Case Model – details all necessary functions the system needs to support from an end user or external system perspective.  The use case model should include the following:

  • Full list of actors and definitions for each along with any other required information
  • Full list of use cases including: subject area, actor, use case name, description, triggering event, preconditions and associated usability information
  • Termination outcomes for each use case (these may optionally be represented in a decision table)
  • Use case scenarios (minimum of optimal with alternative paths being shown for all major areas of functionality and typical error conditions) 
  • Flow of events or system sequence diagram

The final document will be delivered to the customer in a reproducible format.  The final report may also be delivered in hard copy format printed in a single, bound document.
