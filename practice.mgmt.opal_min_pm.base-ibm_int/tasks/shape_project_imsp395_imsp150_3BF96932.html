Task: Shape project (IMSP395 and IMSP150)
This task documents the project scope boundaries, initial list of deliverables, high-level schedule or planning outline, identifies organization that can produce the work, ensures documents include the work required to complete the planning activities and finalizes the Project Definition.
Purpose

Document the project scope and the list of deliverables that will be committed to the sponsor, identify the work streams -or work units- and related outcome that should better be assigned to specialized performing organizations to produce them, and outline the decision structure that will likely assure consensus between the parties most interested in the outcome of the project.

As a result, this activity group defines the timeline and the framework within which each project organizational unit should deliver their contribution to the execution of the project, before launching the next activity where these organizational units will be asked to build their plans and confirm that they can deliver within the framework constraints.

        Main Description

        This task is based on the WWPMM work pattern D4: Shape project.

        This task is launched when the target solution (what must be delivered) and the overall approach (major steps towards delivering the target solution, outcome and decision points) are known.
        This may be the case directly upon completion of "Understand objectives and plan defining activities", because the scope and approach information for the project in the sponsor's project charter was clearly defined and looked feasible, or because the delivery organization will satisfy the sponsor's needs with a standard offering.

        In other cases, "Define target solution" and "Describe overall approach" might have been applied also and resulted in delineating the extent of the target solution (part or all) that will be committed for this project.

        This task focuses on the scope that will be committed for the project being considered, and the approach and organization to deliver it.

        The key outputs of this task are the Project Definition; and also the Deliverable Definition, Work Product List, Work Breakdown Structure, and Organizational Breakdown Structure, or the Project Management System Summary. For small projects, the Organizational breakdown structure (OBS) and Project decision structure can be documented in relevant chapters of the Project Management System Summary.

        This task may include the following:

        • Determine and document real project scope boundaries - Document the boundaries of the project being formulated, in terms of sponsor's business areas, locations, IT infrastructure and information systems affected by the project, and in terms of functions, level of service, and deliverables.
        • Define work units and work products - Expand the work units -or candidates for sub projects that were previously outlined when defining the project approach. Expand them to such a level that:
          • broad estimates (effort and duration) can be done,
          • major dependencies are highlighted,
          • critical skills and resources are determined.
          • Organize these work units into a high level schedule -or Planning framework- with intermediate milestones, and identify the work products related to these milestones. Document this Planning framework in the Project Definition and create the initial Work Product List.
        • Create OBS, sub-projects & document decision structure - Based on the planning framework that was defined in the previous task, identify the delivery organizations that have the skills required to produce the work units, and portray the project organization in terms of project Organizational Breakdown Structure (OBS) and decision structure.
          • Finalize the Project Definition - Review all associated project documentation ensuring that the project definition is consistent. 
            • Refer to the Work Product Inspection Procedure (IMSP530) for information about conducting reviews.
          • Store the project definition with all related documentation.
          • Place the project definition under version control.
