Task: Confirm plans
This task will assess the accomplishments of a project with regard to strategic milestones or checkpoints, and make any necessary adjustments or refinements to the tactical plans, reshaping the target solution and approach for the next phase, as necessary.
Purpose
To serve as a checkpoint, ensuring that plans for the next phase of a project are updated, considering, any changes that may impact those plans.

        Main Description

        This task is based on multiple WWPMM work patterns.

        The key outputs of this task are the Project Definition, Project Management System Summary, Organizational Breakdown Structure, Work Breakdown Structure, and the Project Management Schedule.

        The task includes the following:

        • Assess effectiveness of project plans
          • Assess project or program performance
          • Understand context changes and determine impact
        • Shape project (next phase)
          • Refine project approach and planning framework
          • Adjust project organization and future sub-projects
        • Build project organizational unit work plans (next phase)
          • Refine Phase WBS
          • Review phase estimates and planned resources
        • Integrate project organizational unit work plans (next phase)
        • Reconcile work plans and confirm the overall project
        Steps
        Assess effectiveness of project plans

        This task typically applies to programs but may also apply when projects cover a long period of time. Note that WWPMM has adopted the convention of using the word "project" to refer to both projects and programs, and that any instances that do not fit this convention are explicitly defined.

        This task makes an assessment of the project performance, in terms of gaps between accomplishments and project goals and mission, and also to identify areas where the project organization and PM system should be improved or adjusted for the future phases.

        Then, the impact of any changes that might have occurred in the sponsor's business or IT strategy on the project's future work units is determined. These changes may affect the business needs as they were originally defined in the project's scope, and then cause the sponsor to formulate refinements to their initial requirements for the target solution or mission.

        This will potentially result in adjusting, for future periods, the project shape as it was initially defined in the Project Definition.

        Thus, in a nutshell, this task is a condensed version of the whole Defining activity, to be applied at key milestones of the project in order to prepare for updating the project plans, as necessary.
        Shape project

        This is used here in the context of confirming project plans for the next phase, after having assessed both the business value (for the sponsor) provided by previously completed phases, and the business value expected from phases, projects, and services, either not started or under execution.

        This task is launched when the target solution (what must be delivered) and the overall approach (major steps towards delivering the target solution, outcome and decision points) are known.

        This may be because the scope and approach information for the project in the sponsor's charter was clearly defined and looked feasible, or because the delivery organization will satisfy the sponsor's needs with a standard offering. However some minor changes in the sponsors environment may have occurred and need to be addressed. This task focuses on the scope which will be committed for the project's future phase(s), and the approach and organization to deliver it.

        As a result, this task defines the timeline and the framework within which each project organizational unit should deliver their contribution to the execution of the project, before launching the next activity where these organizational units will be asked to build their plans and confirm that they can deliver within the framework constraints.
        Build project organizational unit work plans

        This is used here in the context of confirming plans for the next project phase, after having assessed both the business value (for the sponsor) provided by completed phases, and the business value expected from phases, projects, and services, either not started or under execution.

        It consists of having each of the performing organizations that will be part of the project, develop the plans describing the means that will be necessary to carry out their part of the project (or a project phase, or sub-project).

        This task will be launched as many times as the number of project organizational units (POUs) that have been identified for the project organization in the Project Definition for the next phase(s).

        Note that the goal here is to let each POU verify that what is expected from them, as described in the Project Definition, is realistic. In other words, this activity consists of refining, in terms of activities, tasks, and intermediate work products, the work units that are identified in the Project Definition, to such a level that reliable estimates can be obtained, and to make sure that the work load and time frame are consistent, and that the resources will be available.
        Integrate project organizational unit work plans

        This is used here in the context of confirming plans for the next project phase, after having assessed both the business value (for the sponsor) provided by completed phases, and the business value expected from phases, projects, and services, either not started or under execution.

        Each potential project organizational unit (POU) or performing organization should have received a Project Definition where an overall high level schedule was outlined. They should have assessed the feasibility of their involvement, and designed their own plans for that. Also, conditional agreements should have been signed with planned suppliers.

        This task confirms or adjusts the high level overall schedule, ensures that appropriate slack exists for dependencies between POUs, and may adjust the POU's scope of work and milestones to ensure a proper synchronization between POUs and eliminate possible bottlenecks that may affect shared resources.
