Work Product Descriptor (Artifact): Data Migration Programs
Data Migration Programs are the source and executable files that allow the exchange of data between a source data reference and a target data reference. Data references are components such as files or databases, which can be internal or external to the system context.

Main Description
When data migration occurs only once for a data reference, it is called a conversion.  When data migration occurs on an ongoing basis, it is called an interface.

Depending on the language and environment, there may be several types of code files such as declarations and definitions of data object classes, structured programming code, scripts, Structured Query Language (SQL), and database management system (DBMS) constructs such as tables, files, macros, triggers, and stored procedures.

The source usually resides in text files and comprises entities such as:

  • Code to connect existing data transfer routines or application programming interfaces.
  • Object classes and methods.
  • Structured programming modules or scripts (C, C++, Java, Visual Basic).
  • DBMS constructs (tables, files, macros, triggers, and stored procedures).
  • Structured Query Language (SQL) and procedural variants (PL/SQL, etc.).
  • Lookup or translation tables. 

Documentation of the execution of these programs including manual and automated procedures such as system administration manuals and script-based files is also included as part of this work product.

Key Considerations
  • Ensure that any special administrative and security procedures are established for each program in accordance with the specifications and requirements.
  • Determine the amount and quality of data in the source system at an early stage of the project. Understand the effort involved in cleansing it. Understand the system capacity required to migrate the data.
  • Use real data for the tests. This will eliminate, or substantially reduce, the need for creating test data. It will also test the migration programs and procedures using data that is representative of the operational system.
  • Agree with the sponsor on a date on which the source data will be frozen and the data migration will start. Agree with the sponsor that the source data will not change after that date.
  • Perform the complete migration (both automatic and manual) at least twice prior to the final migration to ensure that the results are as expected.
