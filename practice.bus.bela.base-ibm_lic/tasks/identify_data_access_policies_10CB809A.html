Task: Identify Data Access Policies
Identify data access permissions e.g. read, write, update, delete for different business roles.
Disciplines: business_modeling.bela
Purpose
To document the access policies for business entities and their attributes by business roles.

    Main Description

    For each business role and in each business entity state, specify the subset of data attributes that the role has permission to create, read, update or delete (CRUD). For example, in a Purchase Order processing scenario, the Requester has permission to add/delete Line Items for the Purchase Order in the 'Draft' state. But when Purchase Order has been 'Approved', then the Requester will have read-only access. Of course, each such data access set can be common across many (state, role) tuples (rows).

    The CRUD access specification adds a Service Operation on the Business Entity Service Components (BESC). The CRUD spec, essentially a subset of the data attributes, is mapped to the input/output Message parameter for that operation. For example, the update permission of the Line Items in the 'Draft' state will create a Service Operation called 'addLineItems (UpdateLineItemMessage message)'. The message type UpdateLineItemMessage contains only set of attributes as specified by the access set. Similarly, the read-only permission in the 'Approved' state will create a Service Operation. 'ReadableLineItems getLineItems()', where ReadableLineItems is a message type generated from the read-only access set.

    A conditional element can be added to each CRUD specification to specify conditional access. For example, if the Purchase Order total amount is greater than $10,000, the 'Approver' role will have update access (so that he can delete/merge some of the line items). For PO's less than $10,000, the 'Approver' can only have read access. These conditions will map to rules and will be invoked as pre-condition evaluation before the actual data access service operation is invoked.

    The Access Sets should also be implemented in Security Specifications for runtime enforcement i.e ensuring the invoker of these service operations is properly authenticated and playing the right role.

    One can also use the Access Sets to create the initial set of UI screens. UI Forms can be generated from the CRUD specs, which can then be strung together intelligently as a screen flow to create a set of UI pages.
