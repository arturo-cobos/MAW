Task: Create or Update Business Entity Lifecycle and Information Models
Create a state chart for each key business entity showing its lifecycle. Also update the Logical Data Model by tracing the artifact through the lifecycle and validating that information added and referred is accounted for
Disciplines: business_modeling.bela
Purpose

Once the Business Entities have been identified, this is the next logical step to create its lifecycle model to determine the set of milestones that the business really cares about. For each milestone the business users can now articulate exactly when a milestone will be considered as reached by specifying the “state” of information content for each of the Business Entities.

      Main Description

      This task could potentially have two entry points. One in which there exists a set of process 'models' (either created by business process modeling or scoping a industry reference model like IFW) and the second when we are initiating 'discovery' activity with BELA. We sometimes refer to the process-first approach as using the process-centric on-ramp and the BELA-first approach as using the information-centric on-ramp. Either of these may be more useful to a given design practitioner. Importantly, both of these on-ramps should be used to work towards a holistic design that integrates both the BE types and processes into a robust, unified whole.

      We will cover the execution of the task in detail in the Steps section for both these on-ramps. Follow the appropriate set of steps based on your entry point – PMD or BELA
      Steps
      Steps for Process Model as an entry point:
      Add business entity flow

      Add business entity flow through activities in Process Definition (if not already there). In the Process Definition, just like we have control flow indicating the flow of control across tasks there needs to be an Object/Data flow mentioning the business entities used in various tasks/actions. The business entities needs to be mapped to the input and outputs pins of all tasks/actions.

      Create a state chart for each key business entity

      Show its lifecycle as derived from the Process Definition. To create the state chart select a business entity and trace its path through each task in the Process Definition. Each task becomes a state if (a) the entity appears either in its input or output port and (b) the input port (the entity appears in) is connected to the output port of another task that has already been identified as a state. Also some industry standard object models like eTOM can be used as reference to identify some "must have" states that the business entity needs to have as part of its lifecycle ("must have" states - for regulatory compliance). Some tools like WB Modeler (v6.1 and later) allow 'state' annotations on a Business Entity at task entry and exit. If this support is available then it is definitely preferable over algorithmic derivation of the state name from task names which can be a bit confusing. With this support one can put business legible labels for the state viz. Approved, Ordered, Shipped etc. and derive the state chart from these labels.

      Create the BE-partitioned Conceptual Data Model r

      Do this by tracing the artifact through the lifecycle and validating info added and referred is accounted fo

      • Traverse the BE lifecycle and for each state, note information required in the Business Entity consistent with the state i.e the ‘state’ or ‘milestone’ is inferred based on the content of the BE.
      • If the required information structures are produced, then add the produced business entities to the information model with a “contained” relationship to the key business entity or to a children in the hierarchy, creating a data graph rooted at the key business entity.
      • If the required information is “referred to” only, then add the used information structures as a “reference” relationship to the data graph.
      • Repeat the above steps for each lifecycle state.
      • Add ‘current state' attribute BE to hold the state label, incase such an attribute do not already exist.
      Steps for Information Analysis as an entry point:
      Create the lifecycle for each Business Entity

      Create the lifecycle for each Business Entity as a state-chart with states and transitions. Create a ‘state’ with a descriptive label for each milestone or important executing states reached by the Business Entity (eg. Created, Transferred etc.). Business Entities states define a useful way to understand and track business operations. By default the state names will be in the past-tense indicating that the particular milestone has been reached, but there maybe cases when Business Entity is waiting on something, especially if multiple threads of work are active in parallel. In such cases the labels will be more in present tense (eg. Executing, Waiting for Approval, etc.). Specify valid transitions between the states to complete the lifecycle model.

      Update the BE-partitioned Conceptual Data Model

      Do this by tracing the artifact through the lifecycle and validating info added and referred is accounted for.

      • Traverse the BE lifecycle and for each state, note information required in the Business Entity consistent with the state i.e the ‘state’ or ‘milestone’ is inferred based on the content of the BE.
      • If the required information structures are produced, then add (if not there already) the produced business entities to the information model with a “contained” relationship to the key business entity or to a children in the hierarchy, creating a data graph rooted at the key business entity.
      • If the required information is “referred to” only, then add (if not there already) the used information structures as a “reference” relationship to the data graph.
      • Repeat the above steps for each lifecycle state.
      • Add ‘current state' attribute BE to hold the state label, in case such an attribute do not already exist.
