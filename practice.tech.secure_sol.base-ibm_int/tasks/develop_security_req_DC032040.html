    Main Description

    The requirements should be captured either in a spreadsheet, database or special software, if available and linked to Business Events and Use Cases (if given). Each requirement should be indexed by the appropriate category. These categories should be determined by the project team. Categories could come from models such as ISO 17799, Common Criteria, the 6Aâ€™s, MASS subsystems, or other security bodies of knowledge. Requirements should have customer sign off before moving forward with design. This ensures both the customer and the project team understand and agree on what will be delivered.  Also, the following should be captured: a description of the requirement that fully captures its meaning, the date the requirement was captured, its source, the type of requirement, any Business Event or Use Case reference numbers (including Anti-Use Cases), fit criteria, dependencies on other requirements, and if there are conflicting requirements.

    Key Considerations

    Frequently in projects, security requirements are not explored. Many times, project teams are unaware of what may be needed in terms of security, and are not sure of the right questions to ask. It is imperative that security members of the team allow for the capture and consideration of security requirements in the project. Without security requirements, unintended or unauthorized access to assets could give rise to potential incidents.
