Work Product Descriptor (Artifact): Test Plan
This artifact identifies the scope of testing for a given level of testing, functions/features to be tested, the testing activities to be performed, the personnel responsible for each task, and the business and technical risks that can be addressed through that level of testing.

Main Description

This artifact is created for each level of testing identified as necessary based on project life cycle testing requirements.  Each testing level represents a known level of physical integration and quality of the system solution under development.

Project needs - based on size, complexity, risks and costs - determine the levels of testing to be used. The following are commonly used levels of tests:

  • Unit Test
  • Integration Test
  • System Test
  • Acceptance Test
  • Operability Test
  • Systems Integration Test

For each level of testing, types of testing are selected based on the attributes of the application/system to be tested.  Functional types of tests are used to test business functions and structural types of tests are used to test the technical requirements.

Types of functional tests:

  • Audit and Control
  • Error Handling
  • Interface/Intersystem
  • Transaction Flow
  • Usability

Types of structural tests:

  • Backup and Recovery
  • Performance
  • Stress
  • Volume

Key Considerations

The following guidance is suggested:

  • Use test matrices in the development of plans to identify and ensure adequate coverage of key testing areas.  This approach ensures that testing time and resources will be used to the greatest effect.
  • Identify all software and hardware dependencies and when they are needed (for example, before the first iteration, after the second iteration) to avoid schedule delays.  There is often significant lead time required if hardware or software needs to be approved, acquired and installed.
  • Keep testing tools in mind when designing test plans to ensure that the selected approach is aligned with the capabilities and requirements of the tool being used.  Failure to do so may result in significant rework of the plan and possible schedule delays, since the tool cannot be changed to accommodate the plan.
  • Sufficient time should be built into the plan for the tool operators to learn the use of tools selected. 
  • Place time limits on tests.  No test should be allowed to drag on without a time limit.
  • Link tests together by means of prerequisites to save complex test setup instructions. 
  • Recognize that as the project progresses, additional types of testing may be identified or new types of testing may be required.  It is often not possible to completely identify all testing requirements in advance.
  • Ensure that the individuals listed in the signoff section are relevant to the level of testing being done; i.e., there is no need to get customer signoff on a unit test.
