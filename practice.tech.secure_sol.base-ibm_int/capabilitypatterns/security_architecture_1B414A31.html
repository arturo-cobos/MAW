Main Description

A Security Architecture work product documents the structure, capabilities and components of the solution in a manner that is independent of specific vendor products and services. The appropriate architecture is directly related to understanding the business requirements and translating them into an architecture that will provide for the appropriate security services to help ensure the overall enterprise security requirements are met once the solution is implemented.

A security architecture work product documents the design of either security solutions (such as enterprise-wide security administration, intrusion detection or single sign-on) or secure solutions (such as SAP, e-business, ERP, etc.). Within e-business, secure solutions typically relate to business needs in one or more of the following areas: Simple Access, External Publishing, Private Publishing, Production Access, E-commerce, Collaboration, and Hosting.

A Security Architecture work product is produces a design at three levels:

  • Conceptual Architecture
  • Logical Architecture (previously known as Functional Architecture)
  • Physical Architecture (previously Component Architecture)

The conceptual architecture provides a basic schematic of the different elements (process and technology) of the solution and their relationship to one another. The logical architecture defines the required security building blocks (process and technology), their functions, protocols and interfaces. The physical architecture documents the hardware and software elements that will make up each of the security building blocks. Note that it is not until the physical level that any hardware or software products (see Product Selection work product) are considered.

Key Considerations

Security architectures must address both technology and process requirements.  With security, an incomplete architecture can function, but it could also leave the client exposed to the same risks that a client without a security architecture has.  But, a client with no security architecture tends to have a realistic view of their risks (i.e. very high!), while a client with an incomplete architecture often assumes, incorrectly, that their risks have been reduced to an acceptable business level.  A complete security architecture, then, is one that meets all the requirements of the client, based on their business needs.

The tasks to produce this work product should only be performed by Methodology-trained Security professionals.  Additional resources available in the IT Security ICAP Database.

The Security Architecture must be validated against the following work products: 

  • Information Asset Profile
  • Security and privacy requirements
    • Administration: Defining and managing the rules.
    • Asset Protection: Controlling data confidentiality, privacy, and integrity.
    • Authorization: Verifying identities and controlling access.
    • Accountability: Tracking actions and events to unique individuals or entities.
    • Availability: Providing information and services where and when needed.
    • Assurance: Validating that controls are working as designed, efficiently and effectively.
  • Principles, Policies and Guidelines, particularly
    • Security and Privacy Policies
    • Security Architecture Principles
  • Standards (particularly Security Standards)
