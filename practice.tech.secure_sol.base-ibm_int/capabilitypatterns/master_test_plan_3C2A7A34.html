Main Description

The Master Test Plan documents the "what" (scope and objectives), the "why" (purpose), the "when" (key milestones), and the "who" (organizational roles and responsibilities) for all levels of dynamic testing. Full lifecycle testing is an iterative process. The development of the master test plan is started early in the development lifecycle, and is refined, as more information becomes available. It ties together all the detailed test plans and points to them. Each testing level represents a known level of physical integration and quality of the system solution under development.

The master test plan evolves from the test strategy and is driven by the quality goals contained in the quality plan. It uses the concepts or organization level standards, and applies them to the project to identify specific test focus areas, types and levels of testing as they apply to the project. As the master test plan is developed it is possible that the levels of testing, as prescribed by the standards, may not be sufficient to meet the test objectives or the quality goals set forth by the quality plan. In this situation, the base set of test levels may be modified or additional levels of testing defined to accommodate the project needs. For example, Performance Testing, which is a type of testing as per IBMâ€™s standard testing methodology, could also occur over a number of different levels of integration (i.e., at unit, integration, systems integration, operability, etc). It is possible that there could be a separate detailed (and/or master) test plan for Performance Testing and that Performance Testing could occur, as a stand-alone test, at different levels of integration. It is also true for Usability testing that spans the full lifecycle and runs throughout the development phases. Thus the master test plan needs to be scalable and tailorable, and must be flexible enough to be used for the small project or the very large projects.

For smaller size projects the master test plan may combine the detailed plans for different levels of testing into one plan.

Some of the various detailed test plans that may be referenced are:

  • Unit Test Plan
  • Integration Test Plan
  • System Test Plan
  • Systems Integration Test Plan
  • Acceptance Test Plan
  • Operability Test Plan

Key Considerations

The Master Test Plan is gaining acceptance as a standard deliverable in most major projects.  In fact it is perceived to be a major catalyst and facilitator to achieving the test objectives in a cost-effective manner.  It is therefore very important that this is verified and validated before embarking on any construction activities, such as coding and unit testing.

The quality plan for development and testing would have defined the quality targets for static and dynamic testing.  Using these quality goals, the master test plan is developed that balances the level of quality against the cost of quality, so as to define cost effective testing that reduces the project and testing risks. 

If the engagement involves a number of projects that correspond to portfolios of applications, it may be prudent to create a template for the master test plan (similar to the one done for test strategy).  After review by the client and refinement, it can be used as a standard template for creating the master test plans for subsequent projects. 

In situations where the project is very large, and multiple releases and multiple implementations may be running concurrently, there are a couple of suggested approaches.  Please refer to the test strategy work product for examples.
