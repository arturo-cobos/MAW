Artifact: Data Quality Plan (APP 424)
The Data Quality Plan is intended to provide guidance to identify data quality issues and establish the processes and approach to eliminate or mitigate these issues. These guidelines should be modified for the specific needs of the project.
Domains: Data
Work Product Kinds: Plan
Purpose
The main purpose of the data quality plan is to provide guidance to numerous aspects of data quality that should be included in development effort.

A Data Quality Plan is required in order to attain and maintain the required level of data quality in the system. Building data quality is typically a result of solid adherence to the definition of data quality criteria from both a business process and data design perspective. As a result there are preventative data quality best practices that focus on the development of new data sources and integration processes and there are detective data quality best practices that focus on identification and remediation of poor data quality.

Description
Main Description

The Data Quality Plan provides guidelines to build and maintain data quality processes required to achieve the necessary levels of data quality for a target environment.

There may be two levels of the Data Quality Plan – strategic or tactical. The difference between these is in the scope and detail of the contents of the work product. A strategic Data Quality Plan would be at some level of the business organization, addressing how the organization interacts with it’s customers and suppliers through data interchange activities resulting in a given level of data quality within the boundary of the organization, whereas the tactical Data Quality Plan would be at the project level showing the interaction of the data stores within the project scope with other components outside the project’s boundaries.. These interactions may occur either during application system initiation time by way of extract, transform and load (ETL) programs or they may occur throughout the life of the system through recurring interface programs. This document could be used with both tactical and strategic data quality plan. But, the review was conducted with tactical view.

The document describes different aspects of data quality to be included at various phases of the project in order to achieve data quality.

The data quality plan:

  • Defines the scope of data quality,
  • Defines data quality requirements,
  • Identifies critical data,
  • Assesses quality of existing data,
  • Designs data quality checkpoints and controls,
  • Defines how to resolve issues,
  • Defines how to cleanse the data,
  • Defines how to measure data quality,
  • Defines how to ensure data quality is achieved after applying the recommendations
  • Defines the roles and their responsibilities in data quality plan.
A Data Quality Plan defines the data quality requirements for planning purposes for the proposed phase or project. The scope would differ from project to project depending on whether data quality is being considered at strategic level or tactical level and the level of data quality to be achieved.
Brief Outline

A brief description of the various sections required to be in the Data Quality Plan work product is provided below.

  1. Introduction - This provides an introduction to the Data Quality Plan of the proposed project. It describes the purpose and scope of the Data Quality Plan. The scope would depend on whether the Data Quality Plan is being looked at the strategic or tactical level.
    1. Purpose - This should describe the purpose of having a Data Quality Plan for the specific project. This information would assist in determining the feasibility of the intended project by providing levels of risk, scope and time.
    2. Scope of the Data Quality Plan - This section defines the data quality requirements for planning purposes for the proposed project. The scope would differ from project to project depending on the level of data quality to be achieved.  This would also address post production continuous improvement and is based on the data source analysis and business rules developed for a project. The scope would depend on whether the data quality plan is being looked at the strategic level or tactical level. The below sections explain the scope at the strategic level.
  2. Data Quality Context
    1. Data Quality Objectives and Goals  - The objectives would set the direction for how to achieve the data quality goals defined by the project. Objectives would describe the plan to prevent data defects from occurring, detect the defects as early as possible in the development/test cycles and identify the ways to implement continuous improvement to maintain the data quality once the project moves to production.
    2. Data Quality Approach and Procedure - Describes the tasks and approach for evaluating the data quality of data structures and interface specifications. Although not a common focus area during the development lifecycle, this plan also includes the approach to implement continuous improvement after a project is put into production.  The three dimensions of data quality which need to be considered as part of the Data Quality Plan are:
      1. Data elements
      2. Technical data quality
      3. Business process
  3. Roles and Responsibilities - This section identifies the individuals and responsibilities required to execute the Data Quality Plan, as well as data ownership of the required data. Identify any additional resources required to maintain the required data quality, like resources to fix the data issues that are found once the project is moved to production or resources required for data cleansing activities.
  4. Data Quality Activities
    1. Identification of Major Data Quality Issues - This would include the major data quality issues identified by assessing and analyzing the existing data.
    2. Trusted Source and Interface Specifications - This section provides a detailed description of the trusted sources and interface specifications required for the project as well as the criteria that each source must meet before being considered as trusted. This section will establish the validation process for trusted sources and provide a top level schedule.
      1. Interface Specification Analysis - This subsection describes the steps to be taken to ensure the quality of data through the data movement process. The objective is to make sure that no data is lost or corrupted as it passes from one application data source to another. The validation of all the interface specification material can be considered a data quality function.
        1. Review Interface Specifications and Analyze Application Interface Points - This subsection outlines what is needed to review each interface specification. This process requires that the interface specification has completed the project’s approval cycle. The Data quality issue identified earlier (4.1) need to be addressed by the Interface Specifications.
        2. Review the UI Specifications - This UI should be reviewed to ensure that all compatible data is being entered into the system.
        3. Review the Batch Interface Specifications  - There could be a number of configuration parameters that might be involved in an interface and have to be set up in order to allow data to flow between systems. These parameters should be checked to ensure they are set to allow the flow of the data required by downstream system.
        4. Review the Inbound Interface Specification - The interface specifications for the inbound feed should be reviewed.
        5. Review the Outbound Interface Specification - The interface specifications for the outbound feed should be reviewed
    3. Data Analysis and Tools - This section provides the details on the critical data elements to be analyzed for the project, along with the tools used to assist with the analysis. Typical data analysis is performed on source data prior to loading into the test environments.
    4. Data Migration Plan for Test - This section provides the details on the data migration plans for the test environments.
      1. Defect Detection - This subsection defines the approach that will be taken to detect and isolate data quality defects during the test cycle.
      2. Unit Test - This subsection defines the minimum data set required to support unit test. This section will define the data types required, the attributes of each data type and a top level schedule for migration to support unit test.
      3. Functional Verification Test Feeds - This subsection when applicable, defines the minimum data set required to support System Integration Testing (SIT). This subsection will define the data types required, the attributes of each data type and a top level schedule for migration to support SIT.
      4. System Test / UAT - This subsection defines the minimum data set required to support System Test / UAT. This subsection will define the data types required, the attributes of each data type and a top level schedule for migration to support System Test / UAT. The objective of this phase of testing is to test all functional requirements (includes all requirements tested in FVT and System). End to End testing data requirements should be derived from the Test Scenarios documentation.
    5. Data Migration Plan for Move to Production - This section defines the minimum data set required for Move to Production (MTP). This section will define the data types required, the attributes of each data type and a top level schedule for migration to support MTP.
    6. Data Timeline Synchronization - This section defines the timeline from data origin to project end point for each data feed including interdependencies.
    7. Implementation of Business Rules, Data Requirements and Data Standards - This section provides detailed descriptions of all the business rules and data standards for the required data.
  5. Measurement Plan - This section defines the metrics/scorecard to be used and the ongoing plan to measure the
    quality of the data post-production.
  6. Risk Assessment - This section defines the risk associated with failure to execute the Data Quality Plan and the failure to
    resolve any known data issues prior to deployment.
  7. Issue Management and Escalation - This section defines the process to be used to manage data issues and the escalation procedure for
    addressing unresolved data issues.
  8. On-Going Data Quality Strategy - This section defines the overall approach for managing the on-going data quality for the project and continuous improvements. While data quality focuses on prevention and inspection during the project development process, focus should also be made on continuous improvement.
  9. Defect Prevention - This section provides the details on the process established to prevent data quality defects. Defect prevention procedures would include improved data entry, data creation processes, monthly or
    quarterly data audits of critical data elements at the trusted sources to eliminate propagating poor data quality to downstream applications.

Key Considerations

A major aspect of Data Quality Management is the collection and management of Data Quality Issues. Centrally tracking and managing data quality issues will allow an organization to:

  • Understand, prioritize, and approve data quality issue remediation at an enterprise level,
  • Reduce the number of poor decisions resulting from inaccurate data, 
  • Focus deployment of resources to remediate data quality issues quickly, 
  • Remediate data quality issues with knowledge of its impacts to the enterprise, 
  • Identify and quickly remediate critical data quality issues,
  • Facilitate better decisions with high quality data and 
  • Fully leverage the benefits of its technology investments. 
  • The approach needs to be tailored depending upon the situation and verified that only the required elements are executed.
Tailoring
Impact of not having
  • Not having a Data Quality Plan may result in not attaining the required level of data quality. This would later result in additional efforts like fixing the defects, cleansing data and rework.
  • Data quality relates to the accuracy with which the data reflects reality. An organization’s actions, if based on a ‘flawed reality’ may create costly mistakes. This would negatively impact the overall satisfaction.  For example, financial systems require a high degree of quality data due to the importance and usage of the data, but a marketing system may have the latitude to operate with a lower level of data quality without significantly impacting the realization of the granular strategic objectives attributed to its stakeholders. Since the “purpose” varies, so does the bar that is used to measure fitness to purpose.
  • Not having a Data Quality Plan may result in not achieving the data quality required by the internal control system. Strategic system development control points may be required to make sure data quality is taken care of.
  • If the checks are not in place the compliance defined for data may not be met.
Reasons for not needing
  • A Data Quality Plan may not be required if the impact of bad data is negligible on the system. The level of data quality required to effectively support operations will vary by information system or business unit, depending upon the information needs to conduct that business unit’s operations.
  • A Data Quality Plan may not be required if the data is coming from a trusted source where the data quality is already in place.
  • A Data Quality Plan may not be required when the size or duration of the project is small (such as a few months with a staff of 10 or less).
  • A Data Quality Plan may not be required in the unusual case where the data is self contained and there is no data implication.
