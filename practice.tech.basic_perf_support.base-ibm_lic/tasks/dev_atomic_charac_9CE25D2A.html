Task: Develop Atomic Characteristics
Develop the required atomic characteristics by research or prototyping.
Disciplines: Technical Architecture
Purpose

The purpose of this task is to provide data which defines the unit or atomic ‘cost’ of performing some specified business functionality.  This unit cost, when multiplied by the frequency of occurrence of the function, will give the total cost.  ‘Cost’ per atomic unit may be expressed in terms such as:

  • Seconds of CPU per order processed
  • Number of I/O operations per stock movement
  • Elapsed time for an I/O
  • Message length per cash withdrawal
  • Bytes of memory per logged-on user
  • Disk space required per customer entity

This data is needed when building a service level characteristic model (e.g. a performance model, availability model etc).

        Steps
        Establish Units of Work
        Decide on the appropriate unit of work for sizing the business system component(s) under investigation.  For example, is this at the level of package user, business transaction, specific technical transaction, or SQL primitive?  The answer will depend specifically on both the performance risk associated with the project (and hence the level of modeling detail required) and where you are within the overall project lifecycle (e.g. initial contract acceptance, detailed design).   
        Capture Atomic Characteristics

        A number of different approaches may be necessary to capture the required set of atomic characteristics.

        • Data may be available already from previous projects in which case this can be reused.
        • Perform technology research by reading published information  regarding the performance of a specific software or hardware component.  For example, a specific application package or a specific server model.
        • Perform technical prototyping using prototype code to obtain performance information regarding the behavior of the business system, or one or more specific components within that business system.
           
        Document and Review Findings
        Once all necessary information has been obtained it should be documented and reviewed by peers or product specialists.
