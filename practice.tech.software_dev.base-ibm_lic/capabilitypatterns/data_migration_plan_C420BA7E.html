Main Description

Data migration is defined as the movement or transfer of data from one data reference (file, database, etc.) in one system to another data reference (file, database, etc) in the same or another system. The data reference from which the transfer occurs is called the source and the reference that receives the data is called the target. When data migration occurs only once for a data reference, it is called a conversion. When it occurs on an ongoing basis, it is called an interface.

The manner in which the transfer takes place can be described as one of the following:

  • Initial Load - the target has been defined but contains no data. Initial load can be a complete one-time transfer or portions of the complete initial set of data can be loaded in increments. This is typically viewed as a one-time operation
  • Refresh - target currently contains a complete set of data. Current target data will be deleted and replaced with a complete set of new data. Once the current target data is deleted, the load operation is similar to an Initial Load. This is typically viewed as an ongoing operation
  • Incremental Update - target currently contains a complete set of data. Updates will be applied one row at a time and only the changed rows will be affected, this is typically viewed as an ongoing operation. In this approach, the data received from the source can be supplied is a variety of ways:
    • Source operational database made available for processing
    • Database unload / image copy operation produced a copy of the database
    • Application program operated on the source to produce a file of only changes
    • Database system produced a file of only the changes

Key Considerations

The following guidance is recommended:

  • A previous project's documentation/plan could be used as a starting point for the development of this project's Data Migration Plan. Care must be taken to ensure that this is systematically reviewed before acceptance, as each project will have its own idiosyncrasies.
  • In the situation of Unit Testing, it will initially be acceptable to independently test each unit without the existence of the full Data Migration plan. However, any unexpected adverse experiences should be communicated with the project team responsible for this data transfer.
  • The Data Migration Plan for an operational system should be done in a timely fashion so that it can be used to extract the other plans. However, if this is not the case, the other Data Migration Plans will have to be developed separately, thereby forfeiting the re-use opportunity, and more importantly, the assurance of a complete and integrated plan.
  • The timing of data migration can be crucial, especially for dynamic data, e.g., invoices, orders, etc. Less critical is the extraction and loading of static data. However, loading materials two weeks prior to a ‘go live’ situation, for example, may require a later manual load of any new materials created on the legacy system during this period, or a second extraction and load after ‘go live’ to fill in the gaps.
