Purpose
The purpose of this practice is to turn software designs into running software. In the planning stage, the team creates coding guidelines and build procedures that define how the source code should be written and structured as well as the integration of the various subsystems (also referred to as components) that will be conducted to achieve progressively larger subsystems, which culminates into the final, integrated system.

To develop (implement) services, flow, messages, service components and the supporting functional and technical components.

Main Description

We typically develop hybrid solutions for our clients today and these solutions are composed of part custom development, part legacy integration, and part package application integration. Therefore, the implementation of a solution in the new software development paradigm for building solutions and application based on service oriented architecture, principles and best practices of service orientation must not only include programming and code generation related business-as-usual activities but also include assembly and integration related activities. Moreover, it must also includes activities to build services and related artifacts for service development.

Software Development may consist of:

  • Custom application development of components and functionality
  • Configuration of existing packages and legacy systems
  • Modifications to legacy systems to support services (ie. Exposure of legacy system functionality)
  • Constructing and configuring composite business services

The Softeare Development practice in SOMA encapsulates set of activities and tasks to implement services, their composition and flows to build and support SOA. First, we develop or verify the necessary XML schema definition for service message specification. Usually, a tool is used to develop or define XSDs for service message specification. Then we develop WSDL for service interfaces. Again, a tool is leveraged to define the WSDL for the service interfaces. Then we develop BPEL, or Business Process Execution Language, flow for service choreography supporting business processes. Again, a tool is leveraged to define the BPEL flows.

Then we develop the components realizing the services, either by custom development or integrating with existing assets and systems and applications and packaged applications. Then we implement functionalities supposed to be provided by candidate services that failed service litmus test and were decided not to be exposed. These functionalities are essential for the overall solution and hence needs to be developed and implemented.

Service realization is performed either by custom development or by integrating with existing asset, systems, and applications and packaged applications. Recall that we make different kinds of realization decisions for realizing services. Services may be realized by integrating with legacy systems and applications (i.e. Integrate option), services may be realized by building from scratch (i.e. Build option) , and services may be realized by transforming existing assets and components to enable them for service realization (i.e. Transform option). SOA Development practice has set of activities and tasks to implement these different kinds of realization decisions such as build versus integrated versus transform. 

There are tasks in Software Development practice to enable existing components as services where we create necessary wrappers, adapters, and other mechanism by which existing components can be integrated in realizing a service and to implement services where services are realized by integrating with legacy systems and applications (i.e. Integrate option).

Next, there are tasks to implement components where we generate, for example, using model driven transformation and/or code components such as service components as realizes the services and associated functional and technical components. These set of tasks allows implementation of services to be realized by building from scratch (i.e. Build option)

The third set of tasks are there to refactor existing component where we refactor and refine one or more existing components so that they can be transformed and enabled to participate in realizing a service. These tasks allows us to implement services where realization decisions made was to transform existing assets and components (i.e. Transform option)

So as you see, Software Development has a set of activities and tasks to allow us to implement services based on different kinds of realization decisions such as build versus integrate versus transform.
