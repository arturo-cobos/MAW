Purpose
To define the data sharing rules that will be implemented between those components in the solution which do not conform directly to the logical data model and canonical message model. To identify inconsistencies in data structure or data meaning across components that could prevent successful realization of the solution. To validate that the business process models, service model, logical data model and canonical message model have been designed appropriately to support all components in the solution. To provide design specifications for implementing run-time data mapping and transformation components in the solution.

    Main Description
    This mapping specifies how services and the related message model map ultimately to the underlying existing (or new) assets. The mapping can take into consideration existing mappings between the conceptual and logical data model and the underlying data stores and refine that mapping. It can also map services directly to existing APIs.
    Steps
    For each S3 layer boundary crossing identify from-and-to data elements
    Establish sufficiency of data stores to support data requirements
    Provide direct traceability from a service to underlying data stores
    Identify and capture any data transformation required at each layer transitioned
