An example of a corporate policy for a measurement implementation is: CLAIM Tool: All IBM employees and subcontractors will be required to use the CLAIM Tool to report their labor hours on a weekly basis.

An example of a contract policy for a measurement implementation is: All projects will track defect data at the fault and failure levels for reporting to the customer.

Main Description

The following is a sample set of policies supporting measurement implementation:

Human Resources Category

  • Percent Subcontractor Labor Hours: All subcontractors will be required to claim their labor hours.  For any fixed price contracts with contractors, dollars are to be converted to labor hours.  ($65 = 1 labor hour)

Quality Category

  • Project Defects per Function Point: Root cause analysis must be performed to determine if the defect was introduced by the development project against which the defect is being logged.  Analysis must also be performed to ensure the defect is valid and unique.
  • System Availability: Root cause analysis must be performed to determine the cause of interrupted service.  Only interruptions caused by application defects should be counted for this measurement.  

Cost and Schedule Category

  • Project Completion Date Variance: A project is only considered complete if both the IBM Business office and <Client>’s Business office have both signed-off on the project.   The completion date specified on the sign-off sheet will be used for purposes of this measurement.

Customer Category

  • Percent Transition Milestones Met: IBM and <Client> will jointly select the key milestones to be met, and the quality standards around these milestones.  These milestones and their quality attributes will be documented as part of the quality review process.

Process Category

  • Project Manager Capability: This metric includes any personnel classified by CERIS as a Project Manager.  A project manager is considered certified if they are PMP certified or IBM certified.  It does not include subcontractors.

Productivity and Output Category

  • Function Points Maintained per Labor Hour: Function Point counting is to be conducted using IFPUG 4.1 standards.  Only “active” applications will be considered part of this measurement where “active” is defined as having greater than 1 person month of effort applied to the maintenance of the application.
  • Function Points Developed per Labor Hour: Function Point counting is to be conducted using IFPUG 4.1 standards.  This measure should capture the total effort in labor hours starting with design through project closure.

Measurement Implementation Policy (Governance) Plan

The example below shows a process to be followed for collecting the required data for the scorecard in an automated fashion from CLAIM, a Help Desk system, or any other data collection facilities. It is important that the days and months are specified, so that each month the process is repeatable and stable.

In this example, the data required for the scorecard is collected in an automated fashion from Claim, and Help desk system.   In addition, some data is collected manually by a business office on spreadsheets.  All of the data for the performance period needs to be input into the appropriate system or spreadsheet by the end of the “agreed to” business day of the following month.   On the “agreed to” business day of the month following the performance period being measured, the data is pulled into the measurement data repository and the scorecard is produced and reviewed for calculation related errors.  Once the scorecard has passed the technical review it is passed to the governance team to perform a business review of the data.  This process is depicted in the chart below.


Audit Considerations

A documented change request process and form must be used in order to ensure the integrity of the data in the scorecard and in the scorecard reports.  Work items may not be randomly added unless this process is followed and documented.  This results in the creation of an audit trail.
