Main Description

Executive Summary

The potential of using the Online Buying Reference Architecture as a basis for the LiUS e-commerce system was assessed. It was determined that the reference architecture provides a good basis on which to proceed but needs to be enhanced to support the volumes and performance requirements of the LiUS e-commerce system.

Introduction


The purpose of this document is to provide a common understanding of the high-level functional and non-functional requirements to be supported by the LiUS e-commerce system and to evaluate whether the Online Buying Reference Architecture could be used as a basis for the LiUS system.

Key Drivers
The key business drivers behind the development of the LiUS e-commerce system are:

  • Complement not replace LiUS stores with the LiUS e-commerce site. Customers should be able to reserve items on-line and pick them up from stores.
  • Support effective cross selling ("customers who bought this item also bought these items").
  • Provide an e-commerce site in all geographies in which LiUS operates with a single data center.
  • Significantly improve on the down-time experience by the current site.
  • Leverage existing product and stock control systems.


Architecture Requirements Checklist
The LiUS e-commerce system has the following key functional requirements:

  • Browsing and searching the site, viewing products and the associated information about those products.
  • Checking the availability of products at a local LiUS store.
  • Purchasing products on-line in which case the site visitor must log-on and register if he has not already done so. The status of orders can then subsequently be checked.
  • Applying for a LiUS Family card on-line and subsequently checking the status of the application.
  • Applying for a job at LiUS.
  • Contacting LiUS to ask a general question or place an e-mail order
  • Subscribing to regular updates, news articles etc about LiUS.
  •  Interfaces to all relevant LiUS back-end systems (currently LiUS Stock Control System, LiUS Product System,, LiUS e-mail, a search engine and a directory server).
  • Managing the site content. This includes creating, reviewing and publishing content and managing templates and product structures. This functionality will be provided by a 3rd party content management system.
  • Site analysis tools.
  • Site personalization tools.
  • Site management tools.
  • Indexing the site content to allow an external search engine to search that content.

Statement of Fit
The following problems were not addressed by the Online Buying Reference Architecture:

  • A back-end integration problem was encountered associated with LiUS's requirements for integration with its stock control system.  Consequently, some modifications will have to be made to the integration server.  The risk involved is rather limited given that LiUS has considerable experience in back-end integration with various systems, and it is expected that a large part of the existing adapter design will be reused.
  • There is a requirements for the use of e-mail to advise customers of order status.  Although the Online Buying Reference Architecture provides some support for e-mail, it is not as fully integrated into the transactional systems as desired.
  • The requirements is that outbound communications (delivery notes etc) be created using MS-Word, but the Online Buying Reference Architecture provides a server-based mass-processing document composition paradigm to achieve performance and integrity.  LiUS has agreed to alter its requirement to fit the recommended pattern.
  • The performance characteristics required by LiUS, in particular those for peak hour transaction rates, are capable of being met by the logical reference architecture pattern.  However, currently the physical reference architecture instantiation described by the Online Buying Reference Architecture proposes use of Distributed Object Transaction Monitors, which limit the transaction rate to a value below that specified by LiUS.  Additional architectural work will be needed to modify the current physical reference architecture to support these higher transaction rates.
