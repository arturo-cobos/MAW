Deliverable: Component Requirements Specifications (DCD 197)
The Component Requirements Specification (CRS) contains all of the component requirements for a project.
Purpose
The purpose of the Component Requirements Specification (CRS) is to specify the component level requirements for a project. The component level requirements will be developed from the system requirements (which in turn were developed from the customer / business requirements that were provided by the PDTL). The Component Requirements will be reviewed at the Preliminary Design review (PDR). When the PDTL and the SPM approve and baseline the requirements at the PDR, then the document and the requirements will be placed under change control.

Main Description

The Component Requirements Specification (CRS) contains all of the component requirements for a project. The CRS, together with the Component Level Architecture and the Requirements Traceability and Verification Matrix, form the component technical baseline. This baseline will be reviewed and approved at the Preliminary Design Review (PDR).

The system baseline is used as the starting point for creating the Component Requirements Specification. This information may be documented in many places, including:

  • System Requirements Specification
  • System Level Architecture
  • System Level Requirements Traceability and Verification Matrix (RVTM)

Packaging InstructionsThis document should be compiled and included in the Project Control book. The component requirements may be documented as described in the Component Requirements Specification, they may be added to the System Requirements Specification, or they may be documented in other formats. There are many Requirements Management tools available (for example, Rational RequisitePro and DOORS). The project’s requirements may be documented within one of these tools rather than using the format presented here. The Project Control book should describe the documentation method used by the project.
