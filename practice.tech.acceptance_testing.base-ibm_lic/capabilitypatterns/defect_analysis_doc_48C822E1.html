Work Product Descriptor (Artifact): Defect Analysis Document
This work product is a detailed description and record of a defect identified during the project lifecycle, including its analysis and recommendations to rectify it.

Main Description
The Defect Analysis Document describes a defect found during the execution of test management, static or dynamic testing activities, or other project activities. It includes a thorough description of the defect and how it was caused, brief description of the steps followed/techniques used to analyze the defect, the findings and results of the analysis, and the prescribed action to be taken to address the defect. Also documented is a listing of all the project domain components or program modules that are impacted by the defect that may have to be examined and/or corrected in the resolution process. The work product may also include the approval of the recommendations, details of the resolution activity and a formal sign off of the defect resolution. The Defect Analysis Document can be used throughout the project lifecycle, although a document is required for each defect for which analysis is deemed necessary.

Key Considerations

Complete all sections of the Defect Analysis Document as thoroughly as possible.  The description of the defect may appear simple and obvious to the author but confusing and complex to the recipient who will be doing the analysis, and similarly the analysis and estimate may appear simple, but be misinterpreted by the resolution team.  Avoid ambiguous terminology, abbreviations and messy diagrams. 

Be cautious when capturing information on the defect.  Focus first on describing the defect, not what the possible causes may be; this will be the purpose of the analysis step.   It is considerate to attach supporting documentation in an organized manner, explaining its purpose, e.g. this core dump was taken when program XXX3363, module aaa-bb-cd abended on an OC4 whilst running Unit Test #4 using the Customer and Product databases at 14:15:23 on 23/09/2000. 

The Test Manager should facilitate the efficient flow of defect analysis documents and activity.  Any originator of Defect Analysis Documents should complete them as the defects are discovered and then batch them for each test execution cycle and submit them to the Test Manager for prioritization.  For iterations around the same test, do not hand over one document at a time for analysis as one defect may trigger another and it is then more advantageous to analyze the thread of a series of defects to derive a resolution. 

In order to ensure that adequate resources are scheduled to resolve defects according to priority, it is beneficial to have regular defect review meetings daily or weekly.  At this meeting, development management, test management, and business and technical test specialists would

  • review all new defects, and assign them or reclassify them as issues, enhancements, etc
  • review all closed defects and their resolutions
  • discuss how defects should be batched or prioritized;  and
  • review the metrics such as number of open/fixed/closed defects, and trends in debug/fix time and project impact.

Any critical impact on project deadlines or budget, other dependent projects and high-risk items are escalated to the appropriate stakeholders such as the project sponsor for resolution. 

At the end of the project, the Defect Analysis Documents should be collated into logical batches (e.g. by project/sub-project/test levels/test cycles/test types/defect severity/application impacted) and handed over (with the Defect Log)  

  • the test organization for trends analysis on defect management effectiveness;  and 
  • the application maintenance team (a) for reference in case the same error arises in production or during testing of enhancements and (b) for root cause analysis in preventive or perfective maintenance. 

In a mature test organization, where an automated defect management toolset is used, there may be some overlaps in information captured on the Defect Log and the Defect Analysis Document and the tool will dictate where the data is captured.
