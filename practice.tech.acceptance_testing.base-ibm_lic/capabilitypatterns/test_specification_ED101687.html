Main Description

The Test Specification work product uses the various functional and non-functional requirement documents along with the quality and test plans. It provides the complete set of test scenarios and all supporting detail to achieve the objectives documented in the detailed test plan.  In short, the test specification is to testing what application design is to development. As in application design, modularization to facilitate standardization and reuse is a goal. All testware (i.e., testing assets) has the potential for use in other test levels and, beyond the immediate project, in maintenance and regression testing of the tested product.

Key Considerations

User management should commit user resources to developing test scenarios and test cases, and to participating in testing with the test team as early as integration testing.

Test design must focus on the testing techniques, tools, and integration strategies, and the basic characteristics of the system to be tested.  The testing techniques that are to be used to develop test cases are specified in the detailed test plans for each level of testing defined for the product. 

Well-documented testware will serve as system documentation for future testing.  It is, therefore, essential that all test matrices, test scenarios, test conditions, test cases, test scripts, and data created during test specification are described and stored such that they can be easily retrieved and reused. 

All testware developed as part of test specification should be cross-referenced.  Cross-reference any test matrices and test conditions to test cases and test data.  Cross-referencing will allow the test case designer to:

  • Demonstrate the traceability and coverage of the tests against the original business requirements and technical specifications.
  • Cascade deletions to test cases and then to test data when test conditions are no longer valid.  This practice will eliminate examining the data every time to determine whether or not they are required.

All test cases should be prioritized (high, medium or low) before testing is started.  The assigned priority should be based on the work already done in risk analysis and in identifying the test focus areas.  When the inevitable test crunch happens, the test cases that have to be run first can be quickly identified, thereby contributing to the success of the testing. 

Software configuration management should be used to manage soft copy versions of all testware.

The test case constructor should exploit the capabilities of automated test tools to demonstrate that the application under test meets technical requirements in the areas of coverage analysis, source code analysis, and load testing.  Test script development tools assist the test case constructor in the preparation of executable test scripts.

  • Coverage analysis tools measure the extent to which test cases validate a program's functions, parameters, inputs, paths, branches, statements, conditions, modules or data flow paths. This measurement is usually expressed as a percentage; e.g., 98% coverage.  They also provide useful information about the dynamic behavior of the application under test. 
  • Source code analysis tools generate metrics (e.g., code complexity), detect violations of coding rules, and test for compliance with defined coding standards based on an analysis of the source code. 
  • Tools are available to support the development of load tests, simulating large numbers of virtual users using the application under test, in a controlled and repeatable way.
  • The creation of executable test scripts is supported by test development tools that are capable of capturing keystrokes and mouse movements while the tester interacts with the application.

Test case development will identify deficiencies in the input design deliverables, and as such, should be considered a verification, or inspection activity, on these deliverables. Defects normally become apparent while attempting to describe a test case to be applied to a specific requirement. A difficulty in developing a specific test case indicates that the requirement is ambiguous or incomplete, or that another related requirement may be missing or was not specified clearly.
