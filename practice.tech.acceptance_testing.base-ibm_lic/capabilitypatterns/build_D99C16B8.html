Main Description

This working version of the system or part of the system is the result of putting the implementation through a build process (typically an automated build script) that creates an executable version, or one that runs. This executable version will typically have a number of supporting files that are also considered part of this artifact.

Key Considerations


 

For all but very small projects, it is advisable to develop a packaging plan that identifies every deliverable Build and describes how it will be delivered to the customer. Do not just say: “Yes, we have executables.”  Name and track them from first identification through delivery and installation.

Keep track of all deliverable Builds either in:

  • Configuration management documents
  • An extension to the work product list or product breakdown structure
  • A separate list or table

Most Builds are built directly as the result of executing build procedures. Some Builds might come from sources other than the development team. For example:

  • Standard code resources, such as device drivers, decompression facilities, and multi-media players.
  • Multi-media resources, such as icons, images, video, and audio.
  • Technology or infrastructure code .

The build procedures indicate how the source files will be used to produce the Build work product. The build procedures not only produces most of the executable files it also identifies them, names them, and places them in the right location.
