Task: Plan User Acceptance Testing
Plan for user acceptance testing to determine whether or not a system satisfies its acceptance criteria and to enable the client to determine whether or not to accept the system and to enable a user, client, or other authorized entity to determine whether to accept a system or component.
Disciplines: Technical Testing
Purpose
To ensure that the Acceptance Testing activities appropriately address the testing required to demonstrate that the system meets the business and technical requirements specified for it and is in an acceptable state by the business for operational use.

        Main Description

        Define the Business Scenario Matrices to ensure all the "end to end" functions which will constitute the Acceptance Test activities are identified. As part of the planning activity, define the test objectives, the acceptance criteria, tests to be performed, the testing methods to be used, roles and responsibilities for the team members as well as the test schedule.

        Steps
        Define business functions
        Define the functions, at a high level, features and items to be tested. This is done by identifying all the features of the system under test and those of the systems it impacts; assessing which of the features actually need to be tested. Verify the functions identified with the users to ensure the completeness and correctness. Then prioritize based on risks the functions that must be tested and the extent to which they must be tested.
        Develop business scenario matrices
        Using the functional requirements matrix as input, develop business scenario for each of the "actors" i.e. players in the business situation. Then decompose the business scenarios into its component sub-functional activities and document these in matrix format. These more detailed business scenarios must identify all the "end to end" functions which will constitute the acceptance test activities.
        Identify document of understanding requirements
        Identify the parties responsible for systems, internal and external, which will interface with the system being developed. Define their high level interfacing requirements and develop a plan for the subsequent creation and sign-off of DOU's with each of these parties. These DOU's will cover timing, roles and responsibilities, objectives and scope.
        Define integration approach

        Define how the system will be integrated for Acceptance testing. This depends on the "build strategy" used and the availability of drivers and scaffolding. The integration approach will depend on several factors such as:

        • In-house development
        • Vendor package
        • Prototyping etc.


        Other considerations are:

        • What common modules are needed?
        • Is driver or scaffolding needed?


        And data components will be integrated into testable user acceptance units and ultimately into a final system/release. To ensure that resources and plans are in place to address the integration of:

        • Conversion
        • Full volume data
        • System components (perhaps from multiple system testing projects)
        • Specific testing facilities (e.g. model office)
        Document detailed acceptance test plan
        The acceptance test plan identifies the functions/features to be tested, the testing tasks to be performed, the personnel responsible for each task, and the risks associated with this plan.
        Review acceptance test plan
        • Review the acceptance test plan
        • Walkthrough the test focus areas
        • Verify that the acceptance criteria and tests are clear and unambiguous
        • Verify that all sections of the acceptance test plan are complete and accurate
        • Obtain the sign off from the approved signatories
