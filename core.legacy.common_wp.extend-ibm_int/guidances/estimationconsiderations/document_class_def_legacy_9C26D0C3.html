Main Description
The size of the task of document analysis is based on a number of factors that measure the complexity of the system and the data the system uses. The complexity of the system also affects the number of classes that must be defined to the system. Here are the key factors when assessing the complexity of the system.
  • Complexity of the functional requirements may lead to more classes that identify information needed to support the functions of the system.
  • Degree to which information will be reused in the system since reuse requires that the definitions be general and flexible enough for each context in which they will be used.
  • Number of different document types to be considered with reuse of information classes between document types and within document types in different contexts.
  • Types and number of systems or technologies used to maintain the source data (tagged source, document management attributes, relational database).
  • Number of system outputs or processing options.
  • Requirements for compatibility with existing systems or applications.
  • Absence or existence of SGML DTD or other analysis.
  • Need to anticipate and provide for future use of the data.
  • Number of analysts required.
These factors can lead to a wide range in the number of classes found during document analysis. We have seen applications that range from simple HTML-like applications that support a single output type and that have on the order of 100 classes defined to very large applications with multiple, large document types sharing many classes and several output types that have more than 600 classes. The factors that lead to larger class counts are complex functional requirements that necessitate the definition of many different classes to support the functions; this includes sophisticated style specifications for printed output. Multiple output options also lead to higher class counts because information must often be differentiated more carefully for different handling in different output processes.
The time required to complete each class definition gets larger with more complex systems. We have seen systems that required from less than .5 hours/class on small projects to up to 1 hour/class. These times include the document reviews, the data capture and report generation, the review and rework. The key factors that drive longer analysis time are reuse of classes in different document types and multiple system outputs generated from the same class.
Analysis time can also be affected by the existence of industry standard or previous document analysis. If well done, these can dramatically reduce the amount of time needed. If poorly done and compatibility with this prior work is necessary, this can dramatically increase the amount of time needed.
As usually, having more than one analyst will reduce productivity because of the time spent in communications.
