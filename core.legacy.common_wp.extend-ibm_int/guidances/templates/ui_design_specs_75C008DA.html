Main Description

Design Overview

  • Background: Provides an introduction of the underlying requirements to be addressed as documented in the Statement of Work (SOW). This description should provide information for a reader who is not familiar with the project and its scope, so that s/he may obtain a basic understanding of the functionality to be delivered. This section should not repeat all the requirements; it should be more of an introductory narrative.
  • Prototype: Describes where to find, load, and run the prototype code. Also lists any system requirements/assumptions such as Internet Explorer 6 is required.
  • UI Guidelines and Coding Standards: Describes where to find the User Interface Design Guidelines work product (if it exists or produced on the project) that provides a consistent set of standards for the UI look and feel. Also describes the coding standards that are to be used by the development team.
  • Accessibility considerations: Describes the relevant accessibility standards and/or legislation to which the application design must adhere. This may include:
    • IBM CI-162 for products produced, marketed, sold or used internally by IBM and its subsidiaries, even if the product uses components not developed by IBM
    • W3C Web Content Authoring Guidelines for web-based content
    • Section 508
    • UK Disability Discrimination Act
    • Note any assistive technologies (screen readers, screen magnifiers, on screen and alternative keyboards, single switches, and a wide variety of input and output devices that meet the needs of people with disabilities) the application is expected to support, and any special considerations required (e.g., Java Swing applications must implement the Java Accessibility API, and use the Java Access Bridge).
  • Screen Transition Diagram/Site Map:
    • For smaller applications (<25 screens), this section shows relationships between screens. Each screen is represented by a rectangle labeled with the screen’s name. Whenever you can open one window from inside another, indicate this by an arrow from the departure window to the destination window. Note: If the user interface design specification is incorporated into a live prototype, screen transitions should actually be coded in the prototype. In the case where not all screens are prototyped, user interface design notes should describe screen transitions that have not been coded in the prototype.
    • For larger applications, it is recommended that this section be moved to the “Detail Design” section and only those screens that are inputs/outputs be detailed using a screen transition diagram because it may be difficult to fit all the screen transitions in one view.
  • Usage: Describes the overall usage of the user interface. This description should be a summary and will correspond closely to the flow of events communicated in its accompanying use cases. If process flows are available, then this section should also link users to these documents or describe where to find the documents.
  • Entitlements: A matrix that describes the page level access level by the user/role. Includes a short description of each of the users/roles as well as a reference to the user profiles work product (if available).
  • Common/Reusable elements: Descriptions of views that are common to all screens such as the banners, footers, menus, global navigations and widgets (calendar widgets, for example). More detail on reusable elements may be provided in a UI Design Standards document. If so, refer to the document and provide instructions for finding it.
  • Framework specifications: Describes the dimensions of areas within a page.

Detailed Design

  • Overview: This section gives an overview/objective of the view, its name, the type of window, and other general information regarding the window.
  • Traceability: Lists related use cases, reference documents and/or additional related business requirements.
  • Screen Captures: Includes wireframes or screen captures of the user interface and if available includes variations based on user entitlements and/or screen states. Note: If the User Interface Specification is embedded in a live prototype, then the prototype itself replaces screen captures.
  • Tabbing Order: This section shows the keyboard tabbing order for the view, including the default starting position or input focus, such as the default button in a dialog or default insert cursor in a form. Note: If the User Interface Specification is embedded in a live prototype, the desired tabbing order should be defined in the prototype code itself.
  • Control Specification: This section provides detailed visual design and behavior information for each UI element contained in the view. The name or label, role, description, and relationship of each control needs to be specified so they will be programmatically usable by such systems as Help, assistive technologies (e.g., screen readers, magnifiers, etc.), and automated testing tools. Also describes enabling, visibility and masking behaviors, as well as how widget-specific events are handled. See User Interface Specification Development Guidelines for control specification templates.
  • Programming interface: Describes the programming resource (e.g., JavaBean attribute or operation) for each function and variable content element surfaced to the user.
  • System Event Handling: This section describes the page events such as initialization and page refresh.
  • Help Information: This section identifies help panels and other help information details that are associated with the view. The accessibility features need to be included along with alternative means of navigating the user interface, such a keyboard tabbing order, mnemonics, and shortcut keys.
  • Open and Closed Design Issues: This section provides a description of issues that arose in the design of this view. It helps the design team maintain a permanent paper trail documenting the reasoning for design decisions.
