Main Description

Development of DTDs for XML/SGML applications requires thorough document analysis to insure success. A trained document analyst should determine the Document Class Definitions before the actual DTD is coded.

The simplest way to create the DTD from the Document Class Definitions is to use Near & Far Designer. This tool allows you to create the element hierarchy structure and define the attributes for these elements without having to know the SGML/XML DTD syntax. Near & Far also has a very robust report generation engine.

However, creating a DTD using Near & Far produces a different format than the one outlined above. Near & Far separates the elements, attributes and entities differently. A DTD created using Near & Far is easiest to read using Near & Far. The Near & Far DTD ASCII file may be difficult to read for XML/SGML novices. Also, the Near & Far DTD ASCII file will require manual editing to include some of the required header information (e.g., Copyright statement).

It is possible to create SGML/XML DTDs directly using a flat file editor. DTDs created using a flat file editor may be easier to read and will have a more hierarchical appearance to a novice XML/SGML practitioner. To create a DTD using just a flat file editor start with the Document Class Definitions. A high degree of XML/SGML experience is desirable for this approach.

The practitioner should make the approach decision based on their preferences and abilities however, in either case; DTDs should be coded so that the following information is included and available. All of this information is found in the Document Class Definitions description of the document types or information objects (the document type is an information object).

  • Copyright statement.
  • Security classification, generally, a confidential classification is inconsistent with any copyright notice.
  • Document type name, which matches the document element type.
  • Document type summary.
  • Applicability of the DTD, a short statement of when to use the DTD.
  • The public identifiers used to reference the DTD.
  • Document type version, including a parameter entity definition that allows subsequent declarations to define a fixed attribute with the version so that processing systems may interrogate the version.
  • Document type change history.

A DTD coding template has been provided in the Notation section above that shows how these fields may be attractively and effectively coded.

The implementation of the document management system objects and attributes is dependent upon the document management system package chosen and the documentation for those systems should be consulted to determine the appropriate forms, formats and procedures to be used.
