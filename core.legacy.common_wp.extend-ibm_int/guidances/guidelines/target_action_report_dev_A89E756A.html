Guideline: Developing Target Action Report

Main Description

The steps for developing the Target Action Report are:

  • Document the business goals,
  • Analyze maintenance log and backlog of service requests to determine high cost areas.
  • Add Perfective Maintenance Actions from recommendations in the Candidate Report.
  • Add Perfective Maintenance Actions based on business goals.
  • Define Exit Criteria.
  • Estimate Effort.
  • Set priority for implementing actions.

Document the business goals.

Produce a statement of the objectives of the perfective maintenance exercise.
These could be:

  • Reduce cost, of maintenance and enhancements.
  • Reduce defects causing adverse impact on customer satisfaction.
  • Improve performance to ensure service level agreements are met.
  • Ensure responsiveness to change requests critical to client business.

Responses in the Application Information Questionnaire can provide assistance here.

Perfective maintenance actions are selected to support the business goals.

Analyze maintenance log and backlog of service requests to determine high cost areas.

The maintenance log can provide information on what functions or software components in the application are being worked on and which are most costly. Future costs are reflected in the backlog of service requests. Actions targeted at high cost areas are more likely to be cost effective.

Add Perfective Maintenance Actions from recommendations in the Candidate Report.

Review the recommendations in the Candidate Report taking into account the business goals and the high cost areas. If a main business goal is to reduce defects, then add actions to correct serious code flaws indicated in the Candidate Report. If decreasing maintenance costs is a goal and recommendations for restructuring and remodularizing are made in the Candidate Report, add actions to do this in the high cost areas (for software components with high levels of activity).

Refer to Identifying Perfective Maintenance Actions Technique Paper for more details on how to select appropriate actions.

Add Perfective Maintenance Actions based on business goals.

Add actions beyond recommendations in the Candidate Report that address the business goals. Software components with high costs or defects associated with them may be improved through manual code improvement or redesign. Responsiveness to service requests may be improved by creating work products, such as design documents. Consider the overall structure of work products in SIMethod as depicted in the work product dependency diagrams.

Define Exit Criteria.

The exit criteria are a condition to be met for the action to be finished. The exit criteria for actions recommended in the Candidate Report are related to the selection criteria. For example, if the action is to remodularize a software component included in the Candidate Report based on selection criteria of cyclomatic complexity > 10, then the exit criteria, Cyclomatic Complexity to be < 11 follows from this.

Exit criteria for actions creating documentation may be to complete successful review.

Estimate Effort.
Estimate the effort required to implement each of the Perfective Maintenance actions, that is, to meet the exit criteria.
The estimation of perfective maintenance actions is similar to estimating other code change actions. It can be useful to review specific instances of code to evaluate complexity of change.

The availability of code understanding tool which enables the code structure to be seen graphically, can accelerate the code understanding, understanding the required code change, and the code change review.

The effort estimation should take into account that actions have the following steps:

  • Receive action definition
  • Obtain the current code (or document) version
  • Run code evaluation tool (or Review document informally), to confirm the basis for change.
  • Understand code (or document)
  • Change code (or document)
  • Run code evaluation tool (or review document informally) to establish exit criteria are met
  • Prepare documentation to enable review.

For the “Change code” step, consider the following perfective maintenance change types:

Code flaw changes - Code flaws tend to be fairly straight forward, taking only an hour or two, depending on how complex the code is. One difficult code flaw to track down without a tool is recursion, especially in complex code or when the recursion is through many code modules. [For example A calls D, D calls G, G calls H, and H calls A]. A difficult recursion may take a day to determine its nature.

Structuring code - The effort to improve code structure is dependent on the complexity of the code. Typically this is not a difficult task but made much easier with a tool. Single occurrences of ‘unstructuredness’ take an hour or two to correct. Highly unstructured software components having a high essential complexity may take days to fully structure. Estimating should be based on potential gains in the future and set to determine how much time to spend on structuring, that is, the goal may not be to fully structure the code, but to improve the structure.

Modularizing code - Again the effort to modularize code depends on the code complexity and can be made easier using a tool. Also, estimating should be based on potential gains in the future and set to determine how much time to spend on modularization, that is, the goal may not be to achieve optimum modularization, but to improve the modularization. Breaking out a single module to simplify a complex module can take an hour or two, bringing a module from a cyclomatic complexity of 18 to two modules with cyclomatic complexity 10 and 8. Remodularizing code of cyclomatic complexity 70 to 170 in a meaningful way may take a day to a week resulting code more understandable and easier to maintain and improve.

Redesigning code - The effort in redesigning code is likely to take at least a day, often several days, and a complex suite of code modules may require months to achieve. If the change is expected to be complex, estimation should be done in consultation with someone who is experienced with the code. Otherwise, allocate enough effort for an analyst application developer to become familiar with the requirement.

Prioritize Actions.

Assign implementation priority to each Perfective Maintenance action. The priority can either be Urgent, High, Medium, or Low based as follows.

Urgent - urgent fix, such as serious code flaws.

High – Meet business objective directly and is cost effective.

Medium - average benefit can be gained.

Low – action can wait until it is convenient to implement.

The priority is based on business goals. If the business goal is to minimize defects, then actions that can remove defects have a higher priority. If the business goal is to reduce maintenance cost, then actions that can improve code understanding and reduce maintenance cost have a higher priority.
