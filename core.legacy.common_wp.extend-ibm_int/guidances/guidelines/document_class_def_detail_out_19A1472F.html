Implementation-Specific Information


Title of field
Description
Example

Name as implemented
This field contains the name of the class as implemented. This could be a column name in a database, an element name in a DTD, etc. It must conform to the naming conventions of the implementation technology chosen for the class. If a class is instantiated in more than one type of implementation (it is in the DTD and the database schema), the name used in both should be the same (this will have implications for the naming strategy employed on projects).
CUST_NAME

Methodology
This field indicates how the class is implemented. This may be as an SGML/XML element or attribute, as a RDB column or table, as an object or an attribute in the document management system or as an object in customization code. This field may take the values: SGML, XML, RDB, DMS, and PGM. This field may be multi-valued since a class may exist as an SGML/XML attribute and a document management system attribute, for example. In these cases, it may be important for the system to automatically keep the values in the document and the document management system synchronized.

Implementation type
This field identifies the type of implementation used in the specified methodologies, this field may take the following values: SGML/XML element, SGML/XML attribute, table column, folder, document, data attribute, object. These values must be paired with the values found in the Methodology field defined above.

Data type
This field contains the implementation-specific type of field: SGML/XML element, SGML/XML CDATA attribute, RDB Long Text, etc.

Definition
This field contains the technology-specific definition for the class. This may be SGML/XML declarations, SQL statements, a CASE model, etc.

Attribute list
This field contains a list of classes that are implemented as attributes to this class. This may occur in an SGML/XML design, the document management system design or in an OO design.

Methods list
This field is used to record the methods which are to be associated with the class. This description should include a reference to the Use Case Model or Business Case Model that necessitates the method, if this is possible.

Future extensions
This field is used to capture any ideas from the designer or the customer regarding future extensions or changes to the class.
