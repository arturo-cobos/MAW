Main Description

DFD modeling enables both developer and user to determine and understand what requirements and functions are actually required by the application on a common basis of “concise and clear graphical representation”.

As a result, this is an effective means for the management of development in terms of step-by-step analysis, study, and evaluation of an application system.

The system needs are a combination of two things: needs for certain desired information (logical aspect) and needs related to the methods for implementation of that needs (physical aspect):

System requirements = Logical aspect + Physical aspect.

  • Logical requirements

Definition of information required by an application for the accomplishment of its objectives.

  • Physical requirements
  • Devices to implement those objectives (such as, persons, organizations, locations, cycles, timing, media, means, performance).

The person in charge of development must organize and execute the work involved in requirements definition based on a sufficient understanding of these two aspects while gaining an understanding of the user requirements. 

Procedures for Drawing DFD-Application Model:

1. Draw a context diagram

Draw a context diagram that clearly specifies the scope of analysis, while paying special attention to the input and output data flows. The use of a context diagram is not a must, but it is helpful in clarifying the scope of the work before starting DFD development.




2. Draw diagram 0

Diagram 0 is the highest-level DFD. It is used to present the overall data flow of the application. Diagram 0 consists of several process boxes to describe whole applications processes, major data flow, data stores, and external entities. By diagram 0, overall picture of the target application can be grasped.



3. Explode each process of Diagram 0

Check the integrities between upper level DFD and lower level DFD, which means input and output data from or to the external entities on the lower level diagram must be logically same as those of the upper level process.

 

4. Further break down each process box sequentially. The number of breakdown levels depends on the objective.

- When using the current physical model, break it down up to the level from which all the current application forms, screens, files, and databases can be extracted.
- When using the new logical model, break it down to the level from which all the data elements can be extracted and the contents of the processes can be described.
- When using the new physical model, break it down to the level from which the physical media can be shown, the functional specifications can be described, and the capacity can be assessed.
- Explode processes only up to the extent required.

In external design, in the case of an on-line process, at least all the transactions and the panels (name) must be extractable from the DFD. In the case of a batch process, breakdown has to be carried out to the EXEC level of the job.

- The detail level (depth of hierarchy) of each process box can vary. The same level of breakdown need not be applied in all cases.
