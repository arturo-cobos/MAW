Main Description
The development of a Program Module Specification is highly iterative. Many of the steps enumerated below are concurrent and not strictly sequential. The process of understanding module responsibilities, determining what responsibilities it assigns to the modules it invokes, and determining the data needs of all the modules involved proceed simultaneously. Furthermore, as one part of the specification becomes more concrete, it may affect the others.
The current module name is descriptive, giving indication of its intended function. Using its name, refer to the processes in the process model from which it derives. This module may map directly to a process of similar name or to a group of processes. In the latter case, its name will reflect the functional sum of which the individual group processes are separate steps.
  • Review the process documentation and relevant system requirements to determine the details of the required system function.
  • Inspect the program invocation model.
    - Determine the current module’s formal parameter interface with modules that invoke it, and develop a first draft interface section.
    - Determine the modules that are invoked by the current module:

- Ensure that the current module’s interface supports the information needs of the modules it calls.
- Establish an understanding of the intended functions of the invoked modules.

  • Review any database transactions that must occur within the scope of the module. Ensure that any data needed by the transaction are provided by an interface. Ensure that any data provided by the transaction are used in an interface.
  • Using the invoked module names and any transactions, develop a structured English description of the steps to implement the requirements as enumerated in the process and system documentation.
  • Review the logic of the structured English description considering the following:
    - Does any logic actually belong in an invoked module?
    - Does any of the logic have enough internal cohesion to be made a module itself?
    - Is any logic used in more than one place? If so, it should be a module.
    - Do the data interfaces support the needs of the logic processing?
    - Does the logic support the efferent data interface requirements?
  • Formalize the interface definition of the current module so that data type, structure, units, domains, and error handling are established. Ensure that the structured English logic is compatible with this formalization.
  • Ensure that the logic supports any data interfaces that have already been defined for invoked modules.
  • Examine the structured English logic for any pre-conditions that need to be documented for correct program execution. Convert as many pre-conditions into interfaces that are checked at compile or link time as possible.
  • Examine the structured English logic for any post-conditions that are not manifested in the interfaces and that represent side effects of program execution. Ensure that these side effects are really necessary.
  • Document any module performance requirements that may have been extracted from the process model or system requirements.
  • Estimate the module’s lines of code size from the structured English description of its processing. If productivity estimates have been provided, estimate the labor hours required to implement the module based on its complexity.
