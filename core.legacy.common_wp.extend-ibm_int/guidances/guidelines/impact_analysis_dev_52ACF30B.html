Main Description

The form that the analysis takes will depend on the prevalent view held by the project team about the system that the client requirements have been identified. This view may be development process, data or functional oriented. Notwithstanding the view held, it must be remembered that the system exists to serve its users for a particular purpose, and this should be the fundamental focus of the assessment exercise.

The essential issue for the assessment to tackle is how the new requirements can be incorporated in the existing IT environment without degrading users ability to realize the purpose that the system is expected to serve. If the new requirements are likely to alter the identified purpose, then the nature of the changes must be documented and their implications assessed.

An assessment of the new requirements would usually reveal a need for changes in the structure (or architecture) of the current system to accommodate the needs of the requirement. These changes must also be documented and the components in the system where the changes are to be effected, referenced.

Irrespective of the system view taken, the requirements should be assessed with respect to:

  • Inputs and Outputs (e.g. Data files, Copybooks, Screens, Maps, Variables, Constants etc.)
  • Data Structures (e.g. Tables, Index Sequential files, Flat files, Objects etc.)
  • System controls (e.g. Data Confidentiality, Data Availability, Data integrity, Access Controls, Data ranges, Valid options, Error messages etc.)
  • Operational controls (Delivery and support documentation, Help facilities)
  • Processing models (e.g. Algorithms, utilities, sub routines, system functions, interfaces etc.)
  • Technology (e.g. Support tools, implementation language, infrastructure etc.)
