Main Description

Place the package extensions under change control as early as possible in the project’s life. Ensure that the specifications and the implementation, including test cases, remain synchronized through strict version control in the build process. Use a source repository that supports versioning to assist with the management of the constant changes to the code files. Differences between versions of a code file can be captured without maintaining multiple physical version files of the same code file. A versioning tool that also supports release information is most desirable. A release can be frozen when the package extensions are shipped. This allows the development team to respond to customer problems by testing on a known code base.

Document the package extensions carefully. Place the documents under change control and make them accessible to the whole of the project team. This can conveniently be done within the version management control system for the project. This will assist in rectifying errors found in testing and provide a solid basis for the approval of the package extensions by the client.

Package Extensions may be created from source by using:

  • Text editors
  • Design tools or languages such as the Interface Definition Language (IDL) or Structure Query Language (SQL). The output of the tools may be augmented by source written using a text editor.
  • Visual programming techniques that generate source from graphic and text representations of the interface.

However the package extensions are created, they should be based on approved specifications defining the operations of the intended application. They should use approved standards and procedures, such as coding standards and build procedures.  Standard extensions such as plug-ins may already exist for a program. They should be treated like those that require development and should be tested in conjunction with the client’s application as a whole.

Intermediary files may be required during the operation of the extension, particularly if the package and extensions reside on different platforms. The format of these files should conform to the specifications governing the interface and the project’s programming standards.
