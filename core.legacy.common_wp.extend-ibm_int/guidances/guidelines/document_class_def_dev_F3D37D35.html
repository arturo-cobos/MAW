Main Description

On the surface, document classes may seem to be very similar to programming objects used in implementation of object-oriented systems.  In fact, the same type of thinking is used in the analysis: identification of classes of objects, encapsulation, abstraction, sub-classing and finding other relationships between the classes.  But in the analysis of document classes, the emphasis is somewhat different.  First of all, the analyst is examining information and finding patterns and discrete and useful divisions within information.  The classes may be quite abstract, not the more tangible objects of a physical world that serve as the basis for object-oriented analysis when designing a program.  In evaluating the classes, the analyst is focusing on the hierarchical relationships, or containment relationships, between instances of information classes rather than the ‘is-a-kind-of’ relationships that are usually the focus of object-oriented analysis, although ‘is-a-kind-of’ relationships can be useful to know later during implementation.  The analyst is also focusing meta-information (information about information) that is necessary to implement the system functionality.  This meta-information may be inherited down the instance tree rather than the class hierarchy.  The analyst will want to capture information about how classes will be processed to implement the function of the system but during the initial analysis, identification of the methods that specific classes must support is not a priority.  One of the objectives of a good document class design is that as additional uses for the information are found, additional processing may be introduced without upsetting the design of the information itself.  The processes of abstraction and encapsulation, properly performed, will facilitate this. 

At a high level, document analysis is comprised of the following activities:

1. Examine the source material, the system requirements and other specifications.

2. Find and define document classes and their boundaries.

3. Build basic models of common structures in the data remembering the principles of abstraction and encapsulation.

4. Classify, group, and define the relationships between document classes.
