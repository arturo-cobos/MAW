Main Description

To define an Architectural Template on a project:

1. Identify Required Templates

Decide what topics are to be addressed by Architectural Templates. Candidates are:

  • Integration of the user interface with the information system (e.g., MVC)
  • Multi-user synchronization
  • Client/Server communication
  • Persistency
  • Security
  • Error handling
  • Recovery
  • Workflow
  • Event notification
  • Transaction handling

Compose a list of areas for which a generic approach can be outlined. Check for which one of these areas a combined abstract collaboration can be defined.

2. Decide Rules and Policies

For each Architectural Template, define principles that govern the general features and behavior as well as the interface of a service. These principles form the "heart" of an Architectural Template.

For example, a transaction strategy must define how resources are shared and how locking is handled:

  • Decide how to share resources. One of the key decisions is defining the strategy for working with objects that are used concurrently in different transactions. Specify whether there a separate instance of the customer object for each transaction, a single instance that is shared between transactions. Each approach may lead to a very different design and even to different infrastructure requirements. In both cases, a notification mechanism should inform dependent objects at the end of the transaction.
  • Decide how to handle locking. On behalf of transactions, a locking strategy has to be defined. A transaction service can drive the release of locks as the transaction commits or aborts. There are different possible lock modes â€“ read, write, intention read, intention write, and upgrade. Write locks can be optimistic or pessimistic. Note that in a distributed object environment, locking is not necessarily handled only by the database.

3. Choose Implementation Approach and Products

Decide whether the Architectural Template will be implemented by commercially available middleware, by adapting an asset, or by project-specific code. For example, concrete products that will provide the basis for the transaction handling have to be chosen. This may include a decision for a DBMS, a standard TP monitor, or an implementation of a CORBA OTS.

The chosen rules and policies can influence the product choice profoundly, i.e., in such a way that it imposes another decision (e.g., to favor a database that supports row-level locking or an implementation of a CORBA transaction service that supports nested transactions). This choice is interdependent with other architectural choices.

4. Define Detailed Collaborations

In defining these collaborations:

  • Identify abstract use cases out of concrete use cases.
  • Define the informal representation of the architecture (potentially based on the identified abstract collaborations).
  • Determine the required roles and interactions in order to realize the abstract use cases. Document the implementation of these roles in terms of the classes that will take on these roles. As a consequence, a class diagram can be drawn relating the base classes of the framework to one another.

Even when projects use a large-grained asset such as the San Francisco frameworks or CORBA based middleware, the particular approach used to tackle the above areas needs to be decided. Many of these large-grained assets still leave open a number of possibilities for constructing a particular solution.

5. Validate the Adopted Approach

Assess if needed transaction rates and usability requirements are met. This can be achieved by setting up prototypes (see Section 6, Validation and Verification, for more detail).

6. Define the Programming Interface

According to the rules and policies, define the programming interface that is used by the application developer.
