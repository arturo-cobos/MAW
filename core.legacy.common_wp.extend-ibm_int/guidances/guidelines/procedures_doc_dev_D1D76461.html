Main Description

Procedures for an existing process

These are best developed by the procedure author conducting workshops and interviews with the Business Process Owner, subject matter experts – and the people who actually perform:

  • The worksteps as part of their regular job. The purpose of the exercise is to document what they actually do, and the information can be gathered by asking a:
    • Series of questions; What do you do? How do you do it? What systems, applications, and reference documents do you use? How do you verify your work?  The template in the Notation Section may be used to capture this information.

These procedures are typically developed when improvements are being made to an existing process where the operational level procedures are not well understood or documented.  The level of detail documented for an existing process is determined by the need to sufficiently understand the process.

Documentation for processes that will be replaced need to capture the key business activities. 

Documentation for processes that are to be improved needs to be completed to a level that identifies all the worksteps required and unnecessary.

Procedures for a newly designed process

Unlike current process definition, new processes are developed iteratively. 

  • First the activity level flow needs to be defined and agreed to at a general level. 
  • Then a template can be created for each Activity.  The name and description should be entered first. Spreadsheets with a tab for each Activity provide a means of keeping all of the activities for a given process organized during the design process.
  • The Workstep column can be completed with the most important worksteps in each activity.  This is a good stage in the documentation process to have a process walkthrough to gain understanding and consensus.
  • The Workstep column can then be completed in more detail to capture all the necessary worksteps and any decision logic if necessary.  Technology worksteps can also be indicated at this time in the IT reference column.
  • At this point the application configurators or developers are critical to the development of the documentation.  All IT references need to be completed.  These references can take the form of an Application Transaction type, a Screen name or number, a reference to standalone Database or the name of a spreadsheet or document to use.  This is another good point to have another process walkthrough especially if actual or prototype IT references are available.
  • Finally the inputs/sources and outputs/destinations are finalized to aid the performer during training and actual performance of the activity.
  • This documentation is continually modified along with the IT support functions until the development is complete and the process is ready for training and production.

Drafts need to be reviewed by:

  • user representatives (using walk-through and simulations where possible)
  • development and training teams
  • Senior management.

The process owner must approve the completed procedure description. Once approved, copies can be made and distributed. Industry best practices can be used as input to design the procedures in the most effective manner.

Note: What level of detail is necessary?

For existing processes the level of detail should allow those that don’t perform the process to understand the processes and be able to identify good and bad points in the process.

For new processes the question is more difficult to answer. Ultimately the documentation has to describe how the work has to be done. The level of detail in the Activity templates depends on other supporting information available.  For example, Do you describe all the steps in logging on to a system or is one workstep “Log on to the system” sufficient?  If this is a new system and there is no other documentation, then the all of the details are probably necessary.   If this is an existing system or there is other basic training available and the performer is expected to know how to log on to a system then a single workstep is sufficient.  There is one guiding question, Can the performer perform the work once they have attended the training and have their process documentation?
