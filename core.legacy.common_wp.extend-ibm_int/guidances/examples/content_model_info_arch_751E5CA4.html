Common Vocabulary
The following definitions represent the types of things which must be defined and consolidated into a Common Vocabulary, and includes general content terms, as well as definitions of Metadata. Some of these terms may be in general use, but each organization will have unique terms and definitions that must be addressed.
DEFINITIONS
The following terms may be used in this document, and are temporarily defined here until a full glossary has been created.
  • Derived Element - A derived element is one that was created by modifying another element. For example, some HiRes images only contain the cropped portion of another image, i.e. only that portion that appeared in the usage. Metadata associated with a derived element may be the same as, a subset of (some styles cropped out), or different than (colors modified), the metadata associated with the original element.
  • Drop Code – See Media/Drop Code in next section.
  • Element – An element is the lowest level object in the data store that is described by the metadata. An element may be image, text, etc.
  • Headline – A headline is a specific type of text element.
  • Layout – A layout is a sketch, or rough drawing, showing the placement of elements on a page or spread.
  • Navbar – The navigation bar on an Internet page.
  • Page – A page is a single page, i.e. half a spread. A page contains elements from a single category.
  • Photo sample – Garment samples that are created and maintained specifically for use in photos.
  • Sidebar – A sidebar is a specific type of element. A sidebar may be a complex element that contains text, art, or both.
  • Splash – A splash is an Internet feature page.
  • Spread – Two pages that are related to each other. A spread contains elements from a single category, or related categories, e.g., 09 Outerwear and 80 Cold Weather Accessories.
  • Theme – A theme is an abstract concept that identifies the overall ‘look and feel’ of a catalog.

METADATA
Requirements have been identified for the following metadata
  • Catalog Number – The eight-digit, unique number for an item in a catalog. The Catalog Number identifies the Style Number, the Media/Drop Code, and a Check Digit as described in Appendix A. An item will have the same Catalog Number when it appears in the same Media in different years if the Style Number has not changed.
  • Category Number – The Category Number is a unique identifier for the category, or group, a product belongs in. Categories include groups such as Luggage, Footwear, etc. Refer to Appendix B for the list of catalog Category Numbers. The Category Numbers for the Internet are different from those of the paper catalogs.
  • Check Digit – A Check Digit appears as the last character of the Catalog Number. The Check Digit is used to verify that the Catalog Number was entered correctly.
  • Color – The Color attribute identifies the color of a product in an image. The client has a cumulative list of colors and their encoding. (Refer to Appendix C for examples of the color encoding.) New colors may be added to the list, but the encoding of existing colors should not change.
  • Element Type – The type of object an element entry describes, e.g., image, text, product specification, etc., as described in Appendix D.
  • HiRes – The identification / location of the HiRes form of an image.
  • Image Identifier – A unique identifier for an image. The Lands’ End Naming Convention is described in Appendix E.
  • Last Retrieved – Contains the date an element was last retrieved. This will be used for creating aging reports. The specific events that trigger the updating of the Last Retrieved date need to be defined.
  • LowRes – The identification / location of the LowRes form of an image.
  • Master Style Name – The name or description associated with the Master Style Number.
  • Master Style Number – The Master Style Number links all items of the same style even though they may have different style numbers, i.e. regular size, tall size, petit size, etc. In some cases, e.g., Coming Home, there may only be the Master Style; sub-styles, e.g. king size, queen size, etc. do not have individual Style Numbers. In other cases, regular size, tall size, etc. will have individual Style Numbers and one of the individual Style Numbers will be used as the Master Style.
  • Media/Drop Code – The Media/Drop Code in the sixth and seventh characters of the Catalog Number identifies the media, e.g., catalog, Internet, Business Reply Card (BRC), etc. An element may have multiple Media/Drop Codes associated with it. Media/Drop Codes repeat each year and need the Year metadata attribute to be unique. Examples of the Media/Drop Code are shown in Appendix F.
  • Page Number – The page number in a catalog where an element appeared.
  • Parent Element – The parent element is the element from which the current element was derived.
  • Persons – The persons attribute identifies the type(s) of people in an image. For example, man, woman, child, men, etc. Refer to the list of Persons Values in Appendix G.
  • Pick-or-Out – Identifies whether the image described by an element is a Pick, an Out, etc. Refer to the list of Pick-or-Out Values in Appendix H.
  • Pose – Identifies the pose, e.g., standing, sitting, running, etc., of the model in the image. Refer to the list of Pose Values in Appendix I.
  • Scan Code – The Scan Code identifies the type of scan of an image as shown in Appendix J.
  • Scene – The Scene attribute identifies the type of scene in an image. For example, beach, school, none, etc. Refer to the list of Scene Values in Appendix K.
  • Shoot Date – The date of the photo shoot that created the image.
  • Shot Identifier – The Shot Identifier is the identifier given the image (shot) by the Art Director.
  • Spread Identifier – The identifier of a spread in a media as described in Appendix L.
  • Stock Keeping Unit – A 5- or 6-digit number used to identify a specific size and color for a style. There does not appear to be a correlation between the Style Number and the Stock Keeping Unit.
  • Style Name – The name or description associated with a Style Number.
  • Style Number – Identifies the style of a product. A Style Number does not identify a specific size or color (see Stock Keeping Unit). Multiple styles may be included in an element. Non-image, e.g., text, editorial, sidebar, etc., elements may also be related to style numbers, e.g., the text describing the product.
The Style Number will change if a pocket is altered or the silhouette is altered, and the other ‘matching’ styles need to be identified. The ability to link all elements of the same style that have different style numbers, e.g., regular size, tall size, petit size, etc., is required. See Master Style.
Note: A style may change categories over time without the Style Number changing.
  • Thumbnail – The identification / location of the Thumbnail form of an image.
  • Time Value – The run-time value that is associated with some types of elements, e.g., video clips, audio clips, etc.
  • Version – The client’s Naming Convention includes a Version character that does NOT have a predefined meaning, but is simply used to differentiate between different versions of an image.
  • View – The type of view of an image as identified in Appendix M. Examples of views include on figure, laydown, etc.
  • Year – The year the media was ‘published’. The Year attribute is needed to uniquely identify a specific catalog (or other media) because the Media/Drop Codes repeat each year.
The following metadata represent placeholders for a ‘group’ of metadata, where the actual contents of the ‘group’, i.e. what information is actually required, is unknown at this time.
  • Media Rights – Identifies any copyright, royalties, or other legal issues related to the use of the element. For example, an element may have been purchased for use in print, on the Internet, etc. Refer to Appendix N for potential Media Rights attributes.
  • Model Information – Identifies the model in an image. Multiple Models may appear in an image.
  • Photographer Information – Identifies the photographer (or author) of the element.
  • Workflow Information - Need to be able to route to (1) standard list of predefined people with (2) flexibility to delete and add to the list prior to routing. Need confirmation that a file has been opened and a record of by whom. Need to use roles (not names) as the routing designator. But must also know WHO is registered in that role. Need delegate function (allow to either be assigned by normal person in role or taken by person who will perform duties for role.
Note that not all the identified requirements will result in actual metadata attributes, but may actually be satisfied by a retrieval from the database.
  • Catalogs Used In – Identify all the catalogs where an element appeared. International requests should also include the page number.
  • Images Containing a Style – Identify all the images that include a particular style.
  • Images Used by Catalog – Identify all the images used in a catalog.
  • Styles in an Image – Identify all the styles contained in an image.
  • Styles Referenced by Text – Identify all styles referenced by text
  • Text References to a Style – Identify all text that references a style.

EXAMPLE 2 - A DOCUMENT-ORIENTED EXAMPLE
The following example is representative of the steps required to enter the classification and indexing values on a document-by-document basis. Each of the values in each of the fields, the classification, usability, etc., are the responsibility of the Information Architect.
This level of classification and indexing is a pre-requisite for navigation, effective search, etc. Over time, the elements of the content may be broken down and defined at a lower level, i.e., element within each defined structure.
For example, the style for a FAQ may define the headline for the question, the answer, the summary section and submission information separately. However, migration from a document level to an element level is something that requires a robust Information Architecture in place, and both the need and resources to (manual, or automated tool) break the documents apart, tagging each element specifically.

Structures and Description
Whether initially working at the document or content element level, the content must be fully defined in terms of specific attributes and metadata.
Both the values for the attributes, metadata fields, and metadata value will vary widely according to the organization.
This example has 2 primary segments: A visual ‘Logical Content Model’ which establishes the content elements, the associated metadata labels, and the Content Element Relationships. The second portions of this example are “Link Tables” which fully define element attributes and relationship to cross-indices that support alternate navigational constructs.

Element Table
The Element table contains information about a single element in the database. An element may be an image, a piece of text, an audio or video clip, etc. The Element entry may actually contain the identification and location of several different variations of an element. For example, an element for an image might contain the identification and location of a Thumbnail image, a LowRes image, and a HiRes image.
The Element Table needed by the client will map directly to the MediaBank Element Table using existing fields where available and the 25 existing KEY1-5 and UFLD1-20 fields
Primary key: The Element Table attributes map into the MediaBank Element Table and use the same primary key, i.e.’Element_ID’.
Alternate keys:
  • ‘Image Identifier’ is unique.
Foreign keys: none
# of rows: There will be one row per stored element.
Insert rules:
  • Permit insertion of an Element entry at any time (when ingested).
  • Permit insertion of a derived Element entry only when the referenced Element entry already exists.
Delete rule:
  • Permit deletion of an Element entry only under the client’s defined policies, and
  • Nullify the Parent Entry attributes of any Element entries referencing the deleted Element entry.
  • Cascade the deletion to all ElementModelLink entries referencing the deleted Element entry.
  • Cascade the deletion to all WhereUsedInfo entries referencing the deleted Element entry.
  • Cascade the deletion to all ElementStyleLink entries referencing the deleted Element entry.
Origin of the information: Most of the attribute information in the Element Table will originate when the element is entered (ingested) into the Digital Library. Other attributes, e.g., Media Rights, will be specifically entered at a later time.

ElementModelLink Table
The ElementModelLink table is used to resolve the many-to-many relationship between an image Element and the Model Information describing the models in an image.
Not all Element entries will be associated with an ElementModelInfo entry. For example, an image of a stack or a swatch will not have an associated model.
Primary key: The composite key, ‘Shot Identifier’ + ‘Model Information’, is unique.
Alternate keys: none
Foreign keys:
  • ‘Shot Identifier’ refers to the Element Table.
  • ‘Model Information’ refers to the ModelInfo Table. The exact attributes of the Model Information that make up the needed key are not known at this time.
# of rows: There will be one row for each model that appears in each image, i.e. (the average number of models per image) times (number of images in the Digital Library).
Insert rule:
  • Permit insertion of an ElementModelLink entry only when
  • the referenced Element entry already exists, and
  • the referenced ModelInfo entry already exists.
Origin of the information: The information in the ElementModelLink Table might originate from the post-shoot information on the Shot Card.

Functional Requirements
Include the identification and definition of core common functions, such as: printing, search, personalization, virus checking, language translation, currency conversion, alternate destination routing to a FAX, email, distribution list, etc), subscription, alternate formatting, advertising support, etc.)
The Information Architect will need to understand the application requirements, and work with the developers of those common functions, to assure that the underlying information structure will correctly support the functions.
In some cases, there may be additional classification and storage issues. For example, in order to maintain print integrity, some content may require storage and access in separate formats, e.g., one for browser display, and one for printing (such as a PDF file).
Technology Requirements
The requirements to support the site as designed must at some point be translated into specific technologies. For example, if the site objective is dynamic catalog publishing, showing current prices and available items only, then the underlying tools must be selected and implemented by the organization. In today’s environment, that would represent XML and SSL constructs, relying on DTD’s.The content requirements must be passed to the actual technology staff responsible for the development and deployment of the underlying technical structure and content coding. The responsibility of the Information Architect is to supply the definitions, not to do the actual coding of the supporting content constructs.
