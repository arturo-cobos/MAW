Main Description

Rules and Policies

In the San Francisco programming model, all code that accesses or modifies business objects must be executing as part of a transaction, known as the "current" transaction. When shared, persistent objects are accessed, locked as specified, and added to the current transaction context. The exception to this is the NO_LOCK lock mode, which can be used to get non-transactional access to the data from an entity outside or across multiple transactions. Changes made to an entity with NO_LOCK access are not reflected in the real object.

As an extension to the San Francisco programming model, the transactional scope has been defined implicitly by integrating the transactions into controller objects. That means whenever a controller object is started or deleted, a transaction is started or committed, respectively. Nesting controller objects mean nesting transactions.

Interface

Classes where transactional behavior is going to be used must be derived from the abstract base class "Base." When the transactional scope needs to be independent of controller objects, the following BaseFactory methods can be used:

void begin() // start a transaction

void commit() // commit changes and end a transaction

void rollback() // rollback changes and end a transaction

void rollbackOnly() // mark a transaction to require rollback

Control suspend() // temporarily exit this transaction

void resume(Control) // resume a suspended transaction

getTransactionStatus() // indicate the status of the current transaction

Client code must initiate a transaction by using the BaseFactory begin() method before accessing any persistent San Francisco object. Otherwise, inconsistent object state may be read, changes to business objects will be lost, and other failures may occur.

When a commit() method is executed on the BaseFactory, a traditional commit protocol is executed either to persist all updates made to objects, or to invoke recovery procedures.

At any time during the transaction, rollback() can be called to release the locks on all the objects accessed so far in the transaction and set the state of all these objects back to their state at the time of the begin() call. A rollback() ends the current transaction, so it must be followed with another begin() before any business objects are used.
