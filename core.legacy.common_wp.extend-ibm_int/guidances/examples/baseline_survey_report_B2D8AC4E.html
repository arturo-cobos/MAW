Main Description

This example assumes that the same project team originally developed the application system being supported. The project team is now broadly divided into a development sub-group who focus on enhancements and new functionality, and a maintenance sub-group who focus on corrections and production support. The project team has recently undergone substantial staff turnover and about half of the team members are relatively new. In addition, the application was developed using a localized approach therefore its deliverables do not neatly fit into the IBM work products.

INTRODUCTION

Purpose

The purpose of the Baseline Survey Report is about discovering and reporting on the state of the relevant areas of the application system that are likely to be affected.

Scope

The requirements suggest a need to survey the following areas of the system. Note that the system is categorized to align with the development process.

  • System context
  • Logical design components
  • Physical design components
  • Operating processes and procedures
  • Training and support materials

The survey does not include:

  • IT Standards because they only form the minimum practice requirements. They will be extended wherever there is a need
  • Operating environment because the requirements do not prompt any changes

BACKGROUND

System context

The application provides telecommunication services activation functionality that supports the business processes of rapidly connecting customers to the company’s network and also providing them with network services and rental equipment. Refer to System Description for an elaborate account.

Requirements context

The project team has received a set of client requirements summarized below which has been bundled into one release. Refer to the Client Request Package for requirements detail.

  • Reduce the current elapse time between service activation completion and network usage recognition from two days to 2 hours.
  • Correct the display of customer credit check warnings to ‘red and flashing’ on the screen instead of the current ‘black and no flash’.
  • Introduce the following two new service products (call number display and ‘red eye’ special rates for off peak network usage) to the application’s capability.

SYSTEM DOCUMENTATION SURVEY

The deliverables, although not IBM work products, generally represent the expected output of the IBM work products.

  • The Conceptual System Design document provides an overview, which describes the purpose of the system, and how it supports the business. This provides a useful introduction to the circumstances that guided the development of the application.
  • The Logical System Design document defines the logical composition of the application into functions rather than business process steps, however it is heavily data focused. It consists of a number of diagrams with very little text to explain them; therefore it is difficult to follow.
  • Program module specifications exist, however they are out of date.
  • There is a limited set of operating processes and procedures, perhaps due to the lack of an application turnover package and the fact that the same project team developed and supports the application. New operating procedures to reflect the operational changes that the solution is likely to effect will have to be developed from scratch.
  • Training and support materials are not updated. Release notes are sent to inform users of new changes therefore the base documentation may be an unreliable source to target.

SYSTEM FUNCTIONS SURVEY

The Logical System Design document shows that the following functional areas are likely to be relevant to this release:

  • Service status update
  • Service completion
  • Interface to Billing
  • Product availability

Service order entry will not be affected because the processes and data will remain the same. All other areas are similarly unaffected.
Up-to-date comments exist in the application source code; however, they vary considerably in their descriptiveness and elegance of structure. Comparatively, the billing interface programs are well structured and are relatively easy to follow. Product availability also has shorter programs and relatively simple algorithms.
With respect to requirement 2, there is no documentation that facilitates the location of affected screens. Therefore, each screen will need to be examined individually.

CONCLUSION

Business processes can be deduced from the logical models but it is difficult to know which controls have been incorporated.
The logical level documentation is considered incomplete, because of the lack of process documentation at this level. The project team would need supplementary information to facilitate the design decisions they will have to make in satisfying the requirements.
Extra care should be taken when analyzing the service creation status at which to trigger the billing interface because of the insufficient and out-of-date documentation as well as the complex algorithms present in the associated programs.
