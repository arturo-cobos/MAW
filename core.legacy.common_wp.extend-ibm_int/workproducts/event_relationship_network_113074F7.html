Artifact: Event Relationship Network (OPS 401)
The Event Relationship Network work product documents the logical associations of specific sets of event messages issued within an individual IT resource.
Domains: IT Management & Monitoring
Work Product Kinds: Model
Purpose
The ERN is a design document. Its purpose is to describe event relationships at the level of detail necessary for customization of systems and network management event correlation software. The ERN provides the detailed information for the IBM services team, the client team, or any subcontracted team to customize the client’s event correlation software for effective correlation of the events addressed within the ERN.
Description
Main Description

Event Relationship Networks (ERNs) provide a conceptual framework for description of event correlation logic. One ERN is created for each source of event messages present in the IT environment targeted within the scope of the project. Example event sources include network resources such as routers and switches, server hardware, server operating systems, middle-ware such as MQ Series, databases, and applications.   

ERNs can grow to include a large number of events. For this reason the ERN framework includes the concept of the ERN subnet. ERN subnets contain an arbitrary subset of the events contained in an ERN. ERN subnets are cross-referenced (linked) within an ERN.   

There exists no single standard for protocols used by IT resources when issuing event messages. Different categories of IT resources adopt differing event protocols. Examples include ASCII messages written to system or application logs, Simple Network Management Protocol (SNMP) traps issued by IP network devices, and Common Management Information Protocol (CMIP) notifications issued by telecommunications equipment. The Event Relationship Network provides an abstracted view of event correlation relationships which transcends the specifics of any individual event protocol and is thus capable of describing event correlation logic spanning all resource types and event protocols present in the target IT environment.   

This abstracted perspective of event correlation logic also enables the designer to create ERNs for any client regardless of the event correlation software used by the client. The ERN work product is a totally open design tool, independent of both the client’s event protocols and event management tools. This enables use of ERNs across a broad spectrum of client environments and IBM engagements.

Brief Outline

ERNs are defined using a graphical syntax and notated as a directed acyclic graph. Events are represented as nodes and are labeled with the role of the event within the ERN.  Arcs from one node (event) to the next indicate that the latter is associated with the former. These graphs do not depict causal relationships, nor are they temporal relationships.

An ERN must contain, at minimum:

  • An ERN Table of Contents
  • One ERN subnet

Key Considerations
IT Specialists, IT Architects, and Consultants would normally use Event Relationship Networks as a tool for analysis and documentation of IT event correlation requirements.  However, the syntax of Event Relationship Networking may be applied to event relationships that exist outside of the IT environment.  For example, ERNs may be used to indicate the relationships amongst non-IT business events issued by business processes, independent of the IT applications which may be implemented to support these processes.  As the client’s management focus shifts from management of the IT infrastructure to management of end-to-end services and business processes ERNs may be utilized to address yet other classes of events and other event correlation applications.
Tailoring
Impact of not havingAny event management design and implementation project, which addresses event correlation, should document its event correlation requirements with ERNs. No other standard exists for documentation of event correlation logic. Not utilizing this work product would result in incomplete and ineffective event correlation documentation. As a result, when the design is implemented the event correlation software is unlikely to deliver the correlation results required by the client.
Reasons for not needingThe project does not include event correlation design and/or implementation.
