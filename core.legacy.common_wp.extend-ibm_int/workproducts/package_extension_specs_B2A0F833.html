Artifact: Package Extension Specifications (APP 024)
Package Extension Specifications describe the extensions to an existing package. Package Extensions Specifications gather together in one place all the information necessary for developing and unit testing extensions to a standard package.
Domains: Requirements
Work Product Kinds: Specification
Purpose

The primary purpose of the Package Extensions Specifications work product is to:

  • Ensure that the programs comprising a client’s package and its extensions operate together as required
  • Document the specifications for each extension according to the specified standards
  • Provide the development teams with the details to develop the package extensions or modify the package itself

The work product is also used to:

  • Provide a consistent communications vehicle among the development teams that have responsibility for developing the various programs comprising the client’s package and its extensions
  • Act as documentation for the system
  • Develop unit test cases

Description
Main Description

A package is a program or set of programs provided by IBM or a vendor. It may or may not be already installed in a client’s organization. The client may require additional functions that the package does not provide. Package extensions are programs and activities that provide these functions such as the addition of radio terminals and bar coding. The cleanest way of providing package extensions is to confine development to the extensions themselves and leave the package unchanged, however, this is not always possible. Changes to the package itself should be kept to a minimum.

The Package Extension Specifications work product documents the specifications for package extensions such as:

  • Add-on or bolt-on solutions.
  • Configuration of the package in a technical area where the standard functionality is not configured appropriately for the client.
  • Enhancements to the functionality of a standard package.

It includes the specifications of any external object required by new developments or modifications to the standards package.

The work product also documents:

  • The format and use of any intermediary file used by the package extensions.
  • How each extension will be unit tested.
  • How security will be maintained for each extension.

In essence, Package Extension Specifications are particularly important:

  • In large or complex projects where several teams may be developing different parts of a client’s application.
  • When the specifications are written by one team and the package extensions are developed by others.
Brief Outline

Package Extensions Specifications are text documents that contain the following information for each interface to be implemented:

  • Extension Name: a name defining the extension to be specified.
  • Extension Definition: a description of the extension including what information and entities are passed between the extension and the package, semantic meanings, structures peculiar to this particular extension, intermediary files, and error handing mechanisms.
  • Pre-Conditions: a detailed description of such items as global data requirements, hardware and software requirements (including memory), operating system requirements, API’s, database views used, and the system state prior to invocation.
  • Post-Conditions: a description of the extension after use including the state of the system and the possible error handling conditions and their effects.
  • Tracability: identifying the client’s requirements implemented by the extension.
  • Functional Specifications: a description of the logic of the extension including details of any data transformations, database transaction definitions, signals, and error conditions.
  • Performance Requirements: a description of any execution time or invocation frequency requirements that must be considered while developing the extension.
  • Testing Considerations: a description of the unit testing required to verify that the extension performs as expected including boundary conditions.
  • Security Considerations: a description of the security requirements for each extension.

Key Considerations

Once the client has approved a specification, it should be placed under strict change control. Changes to specifications can cause significant effects in the project, particularly in schedule and cost estimates.

Any development should be kept to a minimum especially where this involves modifying standard objects. Future upgrades may overwrite or corrupt such development work. Resolving problems with modified objects is usually more difficult than with the standard.

The level of detail for the package extension specifications should suit the project, its staff, and the application. Project size, schedule, cost, staff turnover, designer and programmer experience with both the application and the implementation language, program complexity, and level of formality required by the contract all affect how package extensions interfaces are specified.

Review the extension from both the package and extension points of view. Ensure that the information requirements of package and extension are satisfied. That is, ensure that all the information sent from the package to the extension is used, that the extension supplies that information required by the package, and that all information required is available.

Ensure that the extensions support of the logic of the client’s application, external systems, and business processes.

The package extension specifications form part of the basis for the test plans for the application. The testing considerations should be well documented so that verification can be completed as thoroughly and expeditiously as possible.

Ensure that error handling is specified and tested for each extension.

Ensure that performance and security testing is included in the application test plans.

Consult the client to ensure that the business processes to be used by the intended application are supported by the extension specifications. In other words, do the extensions support the intended business processes?

While it is important to stabilize package extensions as early as possible to allow separate teams to work independently, the package extension specifications’ level of detail should match the needs of the project. It should tell the programmer what he/she needs to know, not what he/she already knows. The designer’s judgement must be applied here. Sometimes the details of an algorithm must be specified; more often a simple, short text description of what the interface is supposed to accomplish is appropriate.

Tailoring
Impact of not having

Not producing this work product presents several risks:

  • The project may fail
  • The client’s application (package plus extensions) may not function as expected
  • Package information used by the extension may be corrupted or lost
  • The teams developing the extensions may make erroneous assumptions
  • Schedules and costs may be exposed due to the possibility of having to rework the extensions
Reasons for not needing
  • There may be no package extensions to develop.
  • The package extensions may be standard or have been specified already through other activities.
  • Diagrams, detailed requirements documents, or use cases may already exist. These, supplemented by reference to APIs, may be sufficient to document the package extensions.
  • If standard extensions are used or if the extensions are described by means other than the Package Extensions Specifications, the project team must be informed of the nature and location of these entities.
