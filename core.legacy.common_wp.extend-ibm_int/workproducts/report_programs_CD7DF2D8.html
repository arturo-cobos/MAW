Artifact: Report Programs (APP 036)
Report Programs are the source and executable files that produce new and enhanced reports for the client’s application.
Domains: Implementation
Work Product Kinds: Realization
Purpose
The purpose of the Report Programs is to generate reports using information contained in the client’s application and associated systems using pre-determined formats.
Description
Main Description

Depending on the language and environment, there may be several types of code files, the source usually resides in text files and comprises entities such as:

  • Structured Query Language (SQL)
  • Report generation programs such as RPG
  • OO classes and methods
  • Structured programming modules
  • Scripts

The reports that Report Programs generate may be printed, displayed, or stored depending on their intended use.

Notation

The notation of the Report Programs depends on the nature of the client’s application and associated systems. The notation of the report program source should be in accordance with the programming standards defined for the project.  The source should contain comments on its operation and, through the build process, should generate executables that allow the reports to be produced in the operational system.

Key Considerations

Ensure that any special administrative and security procedures are established for each report in accordance with the specifications and requirements.
Tailoring
Impact of not having

If no report programs were developed, the client must rely on standard reports supplied with the various programs that comprise his/her application and its associated systems.

Not producing this work product presents several risks:

  • The client’s application may not function as expected
  • Users may not be able to extract information from the application and its associated systems
  • Schedules and costs may be exposed due to the possibility of having to implement the report programs
  • Client dissatisfaction
Reasons for not needing

There may be no reports programs required. The engagement may not involve programming. For example, the engagement might involve only architecture, project management, code reviews, or the development of methodologies or courseware.

The required reports may be standard or have been implemented already through other activities.

If standard reports programs are used or if the reports are described by other means, the project team must be informed of the nature and location of these entities.
