Artifact: Cabinet Folder Hierarchy (APP 102)
This artifact typically represents a virtual view of the documents within a structured organization or may simulate the existing physical cabinet/folder organization. It also includes location descriptions of documents, folders, and cabinets within the paradigm of the file system of the tool or software package being used.
Domains: Design
Work Product Kinds: Specification
Purpose
The purpose of the Cabinet/Folder Hierarchy is to define the cabinet/folder views of the document management system being implemented so that developers and customers have a clear understanding of the organization of documents within the system. This work product directly drives the implementation in many document management systems.
Description
Brief Outline

If possible, use the cabinet and folder icons (see below) with text descriptions. Use indentation to indicate objects contained within other objects. Another option would be to use a table format (such as a spreadsheet) with indentation to indicate the hierarchy.

Cabinet/Folder Hierarchy 

Cabinet name
Description of the cabinet and what it contains. 

Folder 1 Name
Description of the folder and what it contains. 

Folder 2 Name
Description of the folder and what it contains.

Also include tables providing location information for the different document types and cabinets/folders in the system. Possible headings for the document locations tables are:

  • Document Type Name – The type of document identified in the Document Inventory.
  • When Received/Created – Details of how the document comes to reside in the given location.
  • Location – Cabinet and/or Folder where the document resides.
  • Lifecycle Phase – Point in the lifecycle when it resides in the given location. Useful when documents move to different locations during their lifetime.
  • Format – Format of the stored document, e.g., Lotus WordPro.
  • Document Management System Document Type – The type used for the document within the document management system, e.g., Documentum doctype.
  • Notes – Any other relevant information worth capturing.

For the Cabinet/Folder Locations table, possible headings are:

  • Cabinet/Folder Name – The name of the cabinet or folder.
  • When/How Created – Describe when and how the cabinet/folder is created.
  • When/How Destroyed – Describe when and how the cabinet/folder is removed from the system.
  • Location – Where the cabinet/folder resides in the hierarchy.
  • Document Management System Type – The type used for the document within the document management system, e.g., dm_folder in Documentum.
  • Notes – Any other relevant information worth capturing.

Key Considerations

If it is envisioned that using the Cabinet/Folder hierarchy will attempt to solve problems related to searching for documents, this is not recommended.  Potentially drilling down too many hierarchical folders in order to locate a document is rarely a good solution.  In this case it is usually recommended to design the solution to make use of predefined queries (such as Documentum Smartlists).  You could design the Cabinet/Folder hierarchy that allows for all documents to be physically placed under a single cabinet and use predefined queries to represent the drilling down of folders.

Using Documentum, be aware that Documentum SmartLists only work on Documentum document types and their attributes and not on Documentum registered tables.  The solution to creating predefined queries that can access Documentum registered tables and Documentum document types is to create a dm_query object.  Note however, that the dm_query icon represents a query object and does not look like a folder icon.  If you want to use the out-of-the-box capabilities of Documentum SmartLists, and your client wants to be able to search on attributes that are not related to documents (i.e., attributes designed to be stored in registered tables), you may have to duplicate or move these attributes onto the Documentum document type.

If you are organizing the Cabinet/Folder hierarchy to support “life-cycle phase” cabinets, and you are not using predefined queries, the documents will have to be programmatically moved to the correct folder or linked/unlinked based upon their status change.  One solution would be to store all documents under a single cabinet and provide predefined queries that represent the different life-cycle phases.

Assign permissions or access control lists (ACLs) on the cabinet/folder objects to control that is authorized to perform operations on the contents.  In addition, use ACLs to control the visibility of cabinets and folders.

Where multiple options may be presented to the client, present the pros and cons of each option as well as any recommendations.


 

Tailoring
Impact of not havingIf you do not have this work product, the client may not comprehend the cabinet/folder hierarchy to be used on the new system. There will also be confusion among developers on the structure of the system being designed and built.   
As a result, allow more time for the project team to perform rework as a result of not having a clear understanding of what is being built.
Reasons for not needingThis work product should always be produced for document management systems that use a cabinet/folder paradigm. It is not necessary otherwise.
Representation Options

This work product usually takes the form of a bullet list or table using indentation to show containment within cabinets and folders. In addition, it uses tables to capture the location information for documents and cabinets/folders.
