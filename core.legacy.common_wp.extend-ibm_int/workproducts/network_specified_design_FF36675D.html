Artifact: Network Specified Design (ARC 323)
A Network Specified Design Work Product Description contains information about the topology, sizing, and functionality of the network, routing versus switching, connections and nodes.
Domains: Networking
Work Product Kinds: Specification
Purpose

The purpose of the Network Specified Design Work Product is to provide additional information further refining and defining the network design characteristics, namely link capacity, performance, and availability as well as node functionality (security, management, connectivity, etc.).

The additional level of detail is typically added onto a conceptual (‘high-level’) design that was created earlier as part of a conceptual design phase.

Network service providers or vendors may use this work product to offer their ‘best’ products or services as recommendations to the implementation of the network node design.
It is also used to develop a network implementation plan.

Description
Main Description

The Network Specified Design is where the decisions are made concerning OSI Layer implementation on specific nodes.

Specifically, sizing will be included for the connections and nodes to reflect the requirements (e.g. capacity, performance) and the nodes ability to handle the expected traffic. Functionality will be included for both nodes and connections to reflect basic connectivity, protocols, capabilities, management, operations, and security characteristics.

Brief Outline

 

Notation

The notation used in expressing a Network Specified Design is typically a topology diagram.

For nodes, it is best to use generic or non-vendor specific images. For example, it is best to use generic symbols for routers, hubs, switches, etc. rather than specific vendor product images that will be used in the more detailed physical design step – once the vendor product is chosen then more specific images may be used. These diagrams may have a mixture of specified items and ‘givens’, the later are known, non-negotiable items that are part of the client’s environment or imposed through standards or regulations. As long as the diagram does not become overly complex, it is appropriate to annotate the nodes with certain functional characteristic references (type of network management support, protocols supported, etc.). To manage complexity, different views of the network environment may be developed, for example, a Store specification, a Home Office specification, or a Data Center specification. However, to keep the diagrams ‘clean’ and more easily readable, it is usually more commonplace to provide an attached text description of required functionality by node types.

For connection, simple lines or arrows are appropriate to show connectivity between nodes or devices. These links can be annotated with speeds (i.e., 56Kb line, 10Mb Ethernet LAN, etc.) to reflect the sizing characteristics. Again, a separate connection text description may be needed to support a detailed specification to support a product selection.

Key Considerations

Three key points should be kept in mind when creating a Network Specified Design:

First, the terminology used when selling any network strategy/architecture/design engagement MUST be clearly articulated and defined - initially to the client and then also in the proposal (and to the engagement team). These terms can very easily be misrepresented, misunderstood, or misconstrued, and it is very difficult to complete a successful engagement if the client thinks they are getting a ‘Specified Design’, when in the network consultants’ minds they are only planning to deliver a ‘Conceptual Design’ with no intent of getting to a deeper level of design. In particular, be very careful when using terms such as ‘high level design’ or ‘low level design’ without an additional level of qualification on what deliverables the client actually will be receiving. An effective technique is to actually show the client sample sections of prior engagement deliverables that help distinguish between the terms ‘high level’ and ‘low level.’ The use of definitions and examples found in the Networking Methodology and in the Network Conceptual, Specified, and Node Design WPDs.

Second, include an experienced network architect at this point of the engagement. The networking industry is changing at such a rapid rate that to attempt to do a network design without this level of experience and skill in networking standards, protocols, products, industry directions, etc. will surely result in a less than satisfactory engagement.

Finally, validate the recommended network design is in alignment with the client networking strategy and conceptual design. It obviously does not make sense to create a network design that either can’t or won’t be implemented in the client’s environment.

Tailoring
Impact of not having
  • A critical phase of the design process will be omitted, which may lead to a flawed overall network design.
  • It would be almost impossible to design an implementation plan (and choose vendor products) without the Network Specified Design work product.
  • Vendors and telecommunications services providers will be unable to provide recommendations or solutions to the design specifications without this additional level of detail.
Reasons for not needingThe Network Specified Design Work Product Description would not be needed if this was purely a network assessment or if the scope of the engagement only called for a network strategy and conceptual design.
