Artifact: Information Security Assessment (OPS 311)
This artifact defines and documents gaps against best practices and client’s business capabilities found in the client’s information security controls across more than 150 items for an entire enterprise or subset, an individual security process, a platform or network component, or a particular application.
Domains: Security & Privacy
Work Product Kinds: Assessment
Purpose

This work product helps client identify and understand their security risks so that they can put corrective actions in place based on industry "best practices" balanced against their unique business needs.

Description
Main Description

This artifact is typically focused on one aspect of a client’s environment:

  • A "Health Check" is high level and enterprise wide assessment using a common Security or Privacy standard (e.g. ISO/IEC 17799:2005 or NIST SP 800-53)
  • A Process Assessment is an in-depth review of one of 36 possible process areas based on ISO 17799:2005;
  • A Host Assessment is focused on a particular type of system platform; system or environment.
  • A Network and Internet Assessments review internal and external connections respectively;
  • A Security Software Review is a review of source code for vulnerabilities and coding flaws, based primarily on the Open Web Application Security Project (OWASP) and Common Vulnerabilities and Exposures (CVE), a standard for information security names.
  • An Application Assessment focuses on a single application, end-to-end.

The final report of each assessment includes strengths and weaknesses found with recommendations for short and long-term improvements to reduce risks to an acceptable level based on client’s business needs.

Key Considerations

This work should only be performed by Methodology-trained Security professionals.  Additional resources available in the IT Security ICAP Database.

This work product is validated against the client’s enterprise level security policies and security standards, established IBM procedures and guidelines and a standards-based "best practices" baseline using ISO/IEC 17799:2005.

Tailoring
Impact of not having

The Information Security Assessment reveals existing or potential security exposures in an organization’s environment. Insiders or outsiders who get through existing controls may exploit these exposures unnecessarily jeopardizing valuable information assets. The assessment recommendations allow an organization to reduce risks to an acceptable level based on business needs.

Examples of projects that are particularly at risk are re-engineering a client’s infrastructure, implementing any enterprise-wide solution (SCM, ERP, Intranet, etc.), a new e-business application that provides connectivity to the client’s production environment from the Internet or other public network, a new data mining or information warehouse.

Reasons for not needing
  • Client has recent audit results and, thus, is aware of their security risks.
  • Client indicates that security considerations are out of the engagement scope (e.g., when engagement is a proof of concept or rapid development of a prototype). In this case, consider adding the approved Security Disclaimer to the SOW (see References below).
  • Clients may (implicitly) believe that security should not be considered because a new application has the same characteristics as other deployed applications. In effect, why bother now? This may be a reason for not doing this work product, but you should still protect yourself appropriately in the SOW using the approved Security Disclaimer. Alternatively if there are several vulnerable deployed applications, this may be a clear further opportunity for IBM. If you omit the work product from this project (with SOW protection), consider bidding a separate security study.
