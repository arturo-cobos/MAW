Artifact: Interface Programs (APP 039)
Interface Programs are the source and executable files that allow the various programs of a client’s application to communicate amongst themselves.
Domains: Implementation
Work Product Kinds: Realization
Purpose

The purpose of Interface Programs is to provide communication paths between the various programs of a client’s application and between the client’s application and external systems.

Interface programs allow the disparate functional elements an application to communicate effectively and efficiently. They are the glue that holds the application as a whole together.

Description
Main Description

Interface programs also allow the client’s application to communicate with external systems. Depending on the language and environment, there may be several types of code files such declarations and definitions of object classes, structured programming code, and customization tables.

The source usually resides in text files and comprises entities such as:

  • Code to connect existing application programming interfaces.
  • OO classes and methods.
  • Structured programming modules.
  • Scripts.
  • Lookup or translation tables.

An interface is a shared boundary between two functional units defined by functional, signal, or other characteristics as appropriate. The concept of an interface includes the specifications of the connection between two entities having different functions. Information passes across the interface between two programs in both directions. Interfaces have an extract side, where the data is extracted from one program, and a load side, where the extracted data is loaded into the other program. Depending on the direction of data transfer either side of the interface can be the extract or load side.

Notation

The notation of the Interface Programs depends on the nature of the interfaces between programs comprising the client’s application and its external interfaces. The notation of the interface program source should be in accordance with the programming standards defined for the project.  The source should contain comments on its operation and, through the build process, should generate executables that allow the operation of the interfaces among the various programs of the client’s application and external systems.

Key Considerations

Interfaces are boundaries between functional units but they also possess boundaries within themselves. These boundaries are the edge conditions between success and failure, the latter meaning that an error condition has occurred.  It is essential to document and test the boundary conditions of the interfaces to ensure that they are correctly interpreted.  This, in turn, means that error handling must be tested carefully so that a robust application is delivered to the client.

Ensure that any special administrative and security procedures are established for each interface in accordance with the specifications and requirements.
Tailoring
Impact of not having

In projects where applications comprise disparate elements that must communicate, interface programs are essential. If no interface programs were available, data transfer or communication between application elements would have to take place manually.

Not producing this work product presents several risks:

  • The client’s application may fail or not function as expected
  • The programs comprising the client’s application may not be able to communicate amongst themselves or with external systems
  • Schedules and costs may be exposed due to the possibility of having to create the interfaces programs
  • Client dissatisfaction
Reasons for not needing

There may be no internal or external interface programs required. The engagement may not involve programming. For example, the engagement might involve only architecture, project management, code reviews, or the development of methodologies or courseware.

The interfaces programs may be standard or have been implemented already through other activities.

If standard interfaces are used or if the interfaces are described by other means, the project team must be informed of the nature and location of these entities.
