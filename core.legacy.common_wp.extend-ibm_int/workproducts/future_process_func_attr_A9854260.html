Artifact: Future Process/Function Attributes (BUS 010)
The Future Process/Function Attributes work product describes the detailed application/system functionality needed to support the business process being designed and developed. It is described in terms of the functional attributes of the process such as events, inputs, methods, features and outputs.
Domains: Business Process
Work Product Kinds: Model
Purpose

This work product is produced to document for each process activity, the application/system functions and the detailed attributes for each function. It focuses on documenting the desired functionality that is required from the technical solution to support and perform the process activities.

The Process/Function Attribute work product supports the development of subsequent work products for both package-based and custom built solutions. For package-based solution designs, it provides the detail functional requirements that can be used to define the software requirements and selection criteria to identify, select, and assess the fit of package solutions. In custom-built implementations, the Process/Function Attributes help in the definition of the use case model that describes the users interaction with the system. These subsequent work products drive out the detail system specifications on how the application will actually achieve the desired functionality to enable the process.

This work product is typically used to support the package selection and package validation activities.

Description
Brief Outline

Documentation of the Process/Function Attributes is captured in a table matrix with the following format:

  • Function Attribute Name:

The name given to the logical function that is needed in the system to support the process and activity.

  • Function Attribute Detail:

A brief statement that describes the characteristics of the Function Attribute. It describes the desired functionality that is required from the solution to perform the process activities. It can provide the details on its format, usage, etc. to help define the required attributes in the technical solution.

It may include any of the following details:

  • Event triggers: something that causes the process to begin its work, such as receipt of some information, or a scheduled daily time, etc.
  • Input data requirements or other resources that are used in the completion of the process.
  • Methods, techniques, rules, procedures, features, etc. that might be used by the user in completing the process.
  • Output data requirements or other elements (such as printed reports, as an example), which are produced or changed by the process.
  • Process Activity Name:
    The related activity level of the process that the function attribute is supporting. The process activity names are defined and notated in the Process Identification and/or Process Definition work product(s).

Sample format of a Process/Function Attribute table:

Function Attribute Function Attribute Detail Process/Activity

Key Considerations

There is no "right" level of abstraction for documenting these attributes. The real test of usefulness is when even the part-time project team members can easily recognize the concepts defined by the Function Attribute.

A major dependency exists on the development of the Process Identification and Process Definition model prior to defining the Process/Function Attributes. These process models provide the needed context and structure to organize the Process/Function Attributes.

The Process/Function Attributes can be used as the basis for the development of Product Demonstration Scripts in a package selection effort. This dependency is particularly important, since the detailed Process/Function Attributes will lower the effort required to develop these scripts.
Tailoring
Impact of not having

If this work product is not produced, then:

  • The function requirements may be defined at too high level and may not be measurable. Even when the requirement is met, it will not be obvious to the evaluator
  • The requirements are defined at too low level, and become a "laundry list", difficult to place a business value on, the raw number of requirements becomes difficult to manage
  • The requirements are not placed in a business context, i.e. as part of a process; they become difficult for the evaluator to remember "why" the requirement is important
  • The requirements cannot be easily "configured" into an efficient real-life scenario, the basis of a script for the demonstration of alternative solutions
  • The requirements are not understood at a level that is useful once a package has been selected, i.e. for detail package gap analysis, release planning, and configuration
Reasons for not needing

This work product will not be required when:

  • The requirements have been previously defined at a granular enough level
  • A technology or package is being selected on some other basis, e.g. technical criteria or vendor qualifications alone
