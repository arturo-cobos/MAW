Domains: Design
Work Product Kinds: Specification
Purpose
The purpose of this work product is to provide the information needed to make design decisions about the document information stored or used in the system. By providing a consistent format that includes all the document classes implemented in the system, it will be easier to make consistent design decisions and to communicate those decisions to other members of the implementation team. It will also provide documentation for subsequent phases of the project when the analyst may have moved on to other projects or other tasks on the same project.
Description
Main Description

The Document Class Definitions work product contains the information gathered and the design decisions made about the project-specific set of document classes. Document classes do not define just classes of documents, although some classes will represent documents. In fact, document classes are defined down to the lowest level of granularity that is needed to support the requirements of the project. In most cases, this level of granularity is lower than the level of a ‘document’. Typically, most document classes are combined to define a hierarchy that comprises a ‘document’. This work product also includes relationships between document classes. For example, some document classes are used to describe documents or further describe other document classes; these document classes are typically attributes.   

These document classes may be instantiated in SGML/XML designs, in document management system attributes, as database tables or columns or they may not be represented in the final design at all, depending on the needs of the system, the costs of supporting a specific class, etc.   

This work product usually takes the form of a document where the different sections describe the document classes. See the Notation section of this work product description for details of the information captured as well as for a template to be used.

  • Design Decisions

This section contains documentation of document class level design decisions. These are design decisions that do not significantly affect other classes or system components. This section provides a place for recording the rationale behind the decision-making process. An example might be why an RDB field was used for a specific document class rather than a document management system attribute or SGML element. Not all such decisions would be documented here, just the significant decisions where there was significant discussion or disagreement that should be captured.

Key Considerations

Be sure to remember that the process of analyzing information, while superficially similar to object analysis for building object-oriented software systems, is different with a focus on hierarchical instance structure and information content and much less emphasis on methods and the ’is-a-kind-of’ relationships between the classes.

When document analysis is complete, the requirements of the system should be reviewed to verify that the information needed to implement the functions of the system has been identified and accounted for.
Tailoring
Impact of not having

Design decisions will not be as well informed or consistent. It will be difficult to track, and in some cases, to understand why design decisions were made. The documentation of the project will be incomplete and it will be more difficult for new members of the team to understand the design of the information without constant consultation with the SGML and other analysts who did the document class design.

Without this work product, the project team should count on extra time to revisit design decisions (possibly multiple times) as well as to rework designs based on false assumptions about the document classes.

Reasons for not needingIf document analysis does not need to be performed, then this work product is not needed; otherwise this work product should not be omitted.
