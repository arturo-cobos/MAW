Artifact: Recommended IT Management Solution Path (OPS 310)
The Recommended IT Management Solution Path work product describes the path or approach that is recommended for the design, development and implementation of the required IT services and capabilities.
Domains: IT Management & Monitoring
Work Product Kinds: Strategy
Purpose

The purpose of this work product is to describe the path that should be to be taken through the high level design, detailed design, development, and deployment modules of the solution life cycle. This work product will directly influence the construction and sequencing of the transition initiatives that will put in place the required IT services.   

Additionally, this work product will provide the results of the solution path evaluation and selection in a structured, logical and justifiable package. It will be used to communicate the results of the selection process and the resulting recommendations to the client, and to generate action. Once accepted, the recommended solution path forms the basis for defining the implementation initiatives and completing the solution design and integration.

Description
Main Description

A solution path in Recommended IT Management is a combination and sequencing of design and implementation activities that create an IT solution. This work product will describe the sequencing and prioritization of subsequent activities in the solution design and implementation life cycle. It will:

  • Identify the major elements of the required solution.
  • Highlight the primary drivers and constraints for the subsequent design stage.
  • Describe how each of the solution elements will be adjusted and sequenced to accommodate the primary drivers for the design.

At this stage in the solution life cycle this recommendation will usually not name specific products or technologies.

The Recommended IT Management Solution Path should include:

A description of the recommended solution approach.

A summary of the inputs used to make the final recommendation. These inputs may include:

  • Relevant decision support studies.
  • Analysis of decision-maker preferences and pre-dispositions.
    • Analysis of the client’s past performance with alternative solution approaches.
    • Assessment of the client’s change readiness and ability to make the required transformations.
    • A formal evaluation of each solution path alternative against agreed criteria and weightings.
  • The documented rationale for the recommendation and details supporting rejection of the other options.
  • Initial identification of any gaps between the recommended solution and the requirements.
Brief Outline

This work product will generally take the form of a text document detailing the recommended solution path and any supporting information as outlined earlier.  Where a combination of solution paths is to be used the output may include a matrix or text document identifying which generic solution approaches will be used to develop each set of IT services.

Where the solution path or approach entails alternate sourcing of IT services, this work product should also include a sourcing matrix.  This is a matrix describing where and how each IT service or group of services is to be sourced.

Tailoring
Impact of not having

If there is no structured consideration and analysis of the alternative solution approaches there is an increased risk that the chosen solution path will be sub-optimal for the client’s specific situation. The impact of a sub-optimal solution approach could be a more expensive solution, one that is slower to implement, or one that the client does not have the capability to implement and maintain.   

If no solution path is defined at all, then there is a further and more serious risk that major elements of the solution will be overlooked. For example, there are many instances where clients have taken a tool-driven approach to providing IT services, however that approach has failed because it did not also include the necessary process and organization changes.   

Without a clearly recommended solution path and a supporting rationale it may be extremely difficult to justify the recommendations and convince the client to act on them.

Reasons for not needingThis work product may not be necessary if the client team has been very closely involved in the evaluation and recommendation of the solution approach, and has seen all the work products that have led to this conclusion. In this case, and if the recommendations and their justification do not need to be communicated outside the team, then the formal work product may not be needed.
