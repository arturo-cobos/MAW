Artifact: Package Extensions (APP 033)
Package Extensions are changes or additions to an existing software package that provide the functionality to support a client’s business processes.
Domains: Implementation
Work Product Kinds: Realization
Purpose
The purpose of Package Extensions is to provide the additional functions required for a package to support a client’s business processes.

Description
Main Description

Depending on the package environment, the Package Extensions could take the form of new or changed programs, scripts, or data files.  A package is a program or set of programs provided by IBM or a vendor. It may or may not be already installed in a client’s organization. The client may require additional functions that the package does not provide. Package extensions are programs and activities that provide these functions.  The source usually resides in text files and comprises entities such as:

  • Code to connect existing application programming interfaces
  • OO classes and methods
  • Structured programming modules
  • Scripts
  • Lookup or translation tables
  • Structured Query Language (SQL) files
  • Plug-ins
Notation

The notation of the Package Extensions depends on the nature of the package and the extensions themselves. The notation of the package extension source should be in accordance with the programming standards defined for the project. The source should contain comments on its operation and, through the build process, should generate executables that allow the operation of the interfaces among the various programs of the client’s application and external systems.

Key Considerations

The cleanest way of providing package extensions is to confine development to the extensions themselves and leave the package unchanged; however, this is not always possible. Changes to the package itself should be kept to a minimum. Changing the base package can cause significant problems in the future. Maintenance and version to version upgrades of the base package can become extremely difficult. Future upgrades may overwrite or corrupt such development work. Resolving problems with modified objects is usually more difficult than with the standard.

Ensure that any special administrative and security procedures are established for each extension in accordance with the specifications and requirements.

Tailoring
Impact of not having

Providing package extensions is the reason for using this work product. If there are no package extensions, there is no impact in not having this work product. If there are package extensions and this work product is not produced, the project has failed.

Not producing this work product presents several risks:

  • The client’s application may fail or not function as expected
  • Schedules and costs may be exposed due to the possibility of having to implement the package extensions
  • Client dissatisfaction
Reasons for not needingThis work product is not needed when there are no package extensions required. The engagement may not involve programming. For example, the engagement might involve only architecture, project management, code reviews, or the development of methodologies or courseware.
