Artifact: External Data Reference Assessment (APP 308)
This work product describes the results of analyzing an External Data Reference(s) to determine its acceptability to participate in data transfer operations with the system.
Domains: Data
Work Product Kinds: Assessment
Purpose

The purpose of this work product is to verify / certify the external data references that exchange information with the system. Valid definitions and quality data content are essential for successful development of the information flows into and out of the system.

  • Perform a qualitative assessment of the capability of the External Data Reference to support the information flows defined for the system
  • Develop an understanding of the potential data anomalies
  • This information should be developed to sufficient detail that a reasonably sound effort estimate could be developed for data migration and/or data conversion activities
Description
Main Description

Each External Data Reference is analyzed from the following perspectives:

  • Environment - data describing the environment has been verified
  • Definition - the format and content definitions describing the reference are accurate and consistent, business rules are documented and correct
  • Content - the content of the data is accurate and consistent and has been certified according to prescribed tests (data analytics)
    • Data structures and database design
    • Application edit checks and logical unit of work (LUW)
    • Business rules enforcement
    • Data availability and reliability
    • Current data (12 months)
    • Near-current data (13 - 24 months)
    • Historical data (25 months plus)
  • Deployment
    • Availability - whether the data element exists in the source systems
    • Level of detail - whether the data element is at the required level
    • Accuracy - whether the data element has the correct data values
    • Completeness - whether the data element has ALL of the required data values
    • Consistency - whether the data values are of the same format across all records
    • Integrity - the stability of the data element over time
    • Validity - whether the data is the right data. This metric is a measure of the assumption of the relevancy of the data element
    • Data violation
    • Problem identification
    • Problem resolution
    • Data protection
    • Backup and recovery of data
    • Disaster recovery of data
    • Manual operations of business transactions
  • Source Environment Constraints for Extracting Data
    • Method for detecting net change
    • Extract processing requirements
    • Source environment impacts
  • Data Usage
    • Report generation
    • Internal report dissemination
    • External report dissemination
  • Timeliness - data can be delivered / made available according to the schedule documented in the External Data reference definition; whether the availability of data is restricted over certain time periods
  • Volumetrics - the volume statistics have been verified and documented

Results of analysis and recommendations for certifying the external data references. Results should include accept / reject status for each reference. Deployment.

Brief Outline

This work product should contain four sections that cover the data references from the table/file level and also an inspection of the contents of the element level. The four sections describe:

  • Section One: Verification of External Data Reference Definition
    • The definitions of each of the External Data References are verified and any discrepancies are noted.
  • Section Two: Data Deployment Findings
    • The results of reviewing how each of the External Data References is used in an operational environment are documented here.
  • Section Three: Data Quality Analysis
  • The results of conducting a data quality for each External Data Reference are documented in this section. The results highlight:
    • The environment used for analysis
    • The analytical software used
    • The tests that were performed
    • The results of the test
  • Section Four: External Data Reference Assessment Summary
    • This section summarizes the results of the previous sections and presents recommendations for use of the External Data References in the project.

Key Considerations
  • Project team and client need to define and agree what the quality criteria are before the analysis process starts.
  • There are numerous tools available to conduct a data quality analysis.  The extent of the quality criteria will dictate what tools are suitable for the analysis.
  • Client application architects that support the External Data References should be involved in the data quality analysis.  Their knowledge and experience will minimize the analysis effort.
Tailoring
Impact of not havingPotential problems will arise when developers attempt to design and develop data transfer components and discover that the reference definitions are not correct or that the content of the data is not consistent or of good quality. Assessing these references before data transfer design and development begins, will provide insight as to what efforts need to be expended to correct / circumvent the problems.
Reasons for not needingThe only situation where this work product is not needed is where the system does not exchange information with any external data references or the content has been developed by the client or another third party vendor and the results have been shared with the current project team.
