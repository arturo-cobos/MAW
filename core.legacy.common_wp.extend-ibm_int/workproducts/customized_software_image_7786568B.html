Artifact: Customized Software Image (OPS 410)
A customized software image is a client specific equipment image that contains all relevant software and settings that are common to their situation.
Domains: Deployment
Work Product Kinds: Realization
Purpose
The purpose for a custom software image is to reduce the time, and/or the resource interaction it takes to configure the equipment prior to or during deployment.
Description
Main DescriptionDepending on the situation, and the other technology employed by the client, customer software images may contain one, two, or tier three applications. A client may require multiple images in order to match department, local, or user group profiles. A software image is a "snapshot" of a piece of equipment and as such is hardware specific. Multiple images may be required by the client, if more than one machine type or configuration is being deployed. The custom software image may be stored on Compact Discs, servers, or other storage medium.
Brief OutlineThere is no standard notation for this work product.  However, the end result of the creation of this work product is one or several computer files, specific to the imaging software used, which can be transferred to a CD or server for deployment.

Key Considerations
  • Imaging technology is changing rapidly.  Before making a decision on how to create, store, or deploy images you should check the IBM web sites for up to date information.  Options exist that break from the norm of a customized image being a static hardware-specific item.
  • A Customized Software Image Configuration Checklist should be used whenever possible.  Such items as the version numbers of applications, encryption level, drive partition sizes, and desktop icons should be included. This will record what was important and tested by the client prior to the delivery of their image.  It will also give the client an opportunity to review what will be included on all deployed machines.  Creating this list is usually more difficult than creating the custom image, and as such elicits closer scrutiny by the client.  The more detailed the list the less likely an oversight will occur.  Have the client project manager sign off on this list.
  • Make certain that all models that will be deployed have been tested with their images under the same conditions that will exist at the integration center and at the client deployment site.
  • Be aware of slight model or type changes to the received equipment.  A bios or CD drive change could cause the custom image to fail.  Suppliers may not notify you of changes with the order.  Plan and budget for alterations to occur.
  • Establish with the client what types and degrees of equipment changes are acceptable and what types need authorization.  Factories can alter the configuration of the ordered equipment within certain parameters in order to keep up with demand, because of upgrades, and because of changes in suppliers.  They may not notify you of some of these alterations.  Be clear with the client and the supplier what is acceptable and what is not.  As the middleman in this transaction the risk is all yours unless you have set up clear parameters and transferred the risk to other parties.
  • Certain types of file formats have been developed specifically for duplication and may provide some value when deploying CD based images.
  • Test every version of the image prior to acceptance, and do not release to the field or the integration center images that have not been tested. 
  • If using CD or other portable medium, test some percentage of the image copies prior to deployment.  A random selection of 10-15% of the copies would most likely expose any duplication errors.  If any of the copies fail, a larger percentage should be tested, up to 100%.  Reduplication may be necessary should significant errors be exposed.
  • If using CD or other portable medium, request multiple copies of the image.  You will need replacements for the integration center, as images will get damaged during use.  You will need copies for the deployment team in case re-imaging is required at the site. 
  • If using CD or other portable medium, maintain two images at the project office.  One image should be kept as an archive copy and only used for verification purposes.  The second copy should only be used as an emergency master to make copies from.  Neither of these images should be released to the field nor handled extensively.

Issues to be aware of that may impact the usability of the Customized Software Image:

  • Customer proprietary software (and incomplete information on same).
  • Other rollouts occurring during your project that may impact the correctness of the image (e.g. Token Ring to Ethernet architecture).
  • Removal of machine or user specific identification, created on the equipment during application loading, which should not be copied during image creation.
  • The correct level of user rights for the deployed equipment (often administrative rights are established in order to load applications prior to image creation but lesser rights are intended for the user).
  • Consideration of the chosen image medium and the ability to use that medium at an Integration Center or on the client site during deployment (e.g. CD images deployed where no CD drives are installed, images delivered over a server where intranet traffic is already at a critical level).
  • Images for all of the affected rollout equipment and all of the equipment configurations.
  • Any other relevant issues, settings, or policies that are to be deployed via the image.

The Project Manager may also wish to be certain,

  • That the appropriate number of software licenses have been obtained for the rollout equipment.
  • That the correct user rights are established on the image.
Tailoring
Impact of not havingIf a custom software image is not used, the operating system, each custom application, and all client settings will need to be installed manually. This increases the time required to configure each machine and increases the risk of human error.
Reasons for not needing

If the client configuration has only minor differences from the standard image that is being used at the factory a custom image may not be required. It should be determined how long a manual configuration based on the factory image will take as compared to a custom image load.   

If the client has numerous models or configurations of equipment to be deployed, creating a custom image for each machine may be cost prohibitive. It should be determined how much manual configurations will cost, as compared to creating and testing multiple custom images and then using those images to configure the equipment.   

If existing client equipment is to be imaged and re-deployed, and information about that equipment is insufficient or untrustworthy, creating and using a custom software image is not suggested.
