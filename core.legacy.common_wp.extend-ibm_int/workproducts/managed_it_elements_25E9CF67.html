Artifact: Managed IT Elements (OPS 309)
The Managed IT Elements work product describes the list of IT infrastructure elements which are to be managed.
Domains: IT Management & Monitoring
Work Product Kinds: Inventory
Purpose
The managed element list is used as the basis for analyzing management roles and responsibilities and tool requirements. This activity, critical to the manageability and success of the engagement, involves reviewing the target environment and creating a list of elements - the items that make up the IT infrastructure.   
This list can also be used to validate the scope of the engagement with the client. Often this list will point out gaps in the scope as defined.
Description
Main DescriptionA managed IT element is an IT resource that is operated upon by infrastructure management functions or people. An element represents the maximum level of detail to which an operational environment is decomposed in order to determine which management functions to apply to it.
This list will clearly define which of the technology elements (applications, platforms and data) will require management and which will not. The list will also be used to differentiate management requirements for elements on the list. For example, the client may want to completely manage a LAN server, but manage the workstations to a lesser degree.
The list consists of the significant IT managed elements at a level of detail which:
  • Accurately interprets the target environment to a more granular level than is represented in the IT infrastructure.
  • Reflects the necessity of constraining the number of elements to a quantity that is manageable within the context of the design engagement.
  • Includes all elements that are essential to manage at an individual level in order to meet IT service quality expectations.

Views

There are many influences that determine the infrastructure management style most appropriate for the client's specific environment. A view can act as a pattern or template to structure the groups of objects to be managed. Views define sets of element groups. These groups give us a method for pulling the actual elements in the infrastructure together based on the customer's systems management style, and reflect how the client’s want to manage their environments.

Key Considerations
  • The completion of this work product requires the use of the client’s ‘architect.’  If no such role at the client exists, then seek the advice of a person who best understands the client’s IT architecture and structure.  It may require the use of subject matter experts to complete the list. 
  • Don’t allow the list to become too large.  This will result in a lengthy discussion of subsequent management requirements with little value. 
  •  Remember that this list is being built in order to identify requirements for managing the technology.  These requirements are at the skill, tool, and data levels.  If two elements on the list have the exact same management requirements, then combine them into one element.  This will expedite subsequent requirements analysis.
Tailoring
Impact of not havingRequirements for the management of specific IT elements may be overlooked. Analyzing requirements at the specific technology level ensures that all tool and role requirements are complete.
Reasons for not needingIf an engagement includes management processes that are largely independent of the specific technology being managed (for example, call management) then this work product may not be required.
Representation Options

The infrastructure management style most appropriate for the client's specific environment can vary significantly based on the customer’s systems management style.  The views described here are not to be considered a definitive list. There are certainly other perspectives and categories that could be used. Note that the SMFD-TE templates are structured based upon the Component View.

Component View

This view represents the traditional IT way of describing an operational infrastructure; it is a view from the inside looking outward toward the end user. The elements in the target environment are grouped together according to the data processing functions they perform within the infrastructure. Such basic categories as system or computer nodes, networks and applications serve as a starting point. They are broken down further based on organizational responsibility and ownership; a particular group within the IT organization may be responsible for managing the remote LAN networks, while another organization is responsible for central site hardware. Another factor in defining categories in the component view is the technical skills required to operate and manage a particular group of elements; in a heterogeneous environment, one group of people may be responsible for mainframe operating systems, while an entirely different one is responsible for distributed UNIX operating systems.

End User View

Much has been written about a re-orientation of the IT world resulting from the "PC Revolution". For many large customers, the primary thrust of this change has been to drive the focus of data processing priorities away from the central site and toward the empowered end user. This view starts at the user's desktop and works inwards. The user becomes a virtual application. The user assembles and links various pieces (objects) of application logic together based on the particular business function she is performing. When another business function is started, a different set of objects will be assembled dynamically and kept active only for the length of time required to perform the function. This "outside-in" view demands services from different systems that may have very little in common with the nominal target one.

Application View

As the name implies, this view utilizes an application centric perspective. Where the component view is "inside-out" and the end-user view is "outside-in", the application view starts with the application as the stable center and brings the user and the technical infrastructure together around it. This view is often encountered where the customer's application development department has a strong influence and where extensive use of application development tools is present.

This view takes the application as a representative of a group of homogeneous, stable users. The set of IT infrastructure resources which support the application are identified and grouped accordingly. Since different types of applications require different sets of resources, application classes play an important role in defining the element group taxonomy of this view.
The distributed client/server application is an example of how a particular application type defines a set of element groups. There are standard ways of designing client/server applications so that their functions can be distributed. The most common is to divide the application into the following components:

  • Presentation Logic
  • Business Specific Logic
  • Logic to access System Resource Managers

Each of these logical components requires a set of IT resources to execute; the required resources are defined as element groups for this class of applications.

The practitioner can document the managed IT elements in several ways. The best way is to start with a pictorial overview.

The high-level overview should be followed by a list of the managed elements in a hierarchical format showing the relationships of the elements.
