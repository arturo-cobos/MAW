Artifact: Document Inventory (APP 104)
The Document Inventory describes the set of document types that will be processed by the system.
Domains: Design
Work Product Kinds: Specification
Purpose

The purpose of the Document Inventory is to define the set of document types that the system will handle and provide the entire basis for Document Analysis activities. With concurrence from the customer, this work product defines the baseline for document analysis activities and limits the scope of the project relative to documents, and any life-cycle phase unique characteristics, which will be analyzed and be part of the solution.

Description
Main Description

The Document Inventory consists of key information about each document type (such as format, number of pages, number created per week, etc.) as well as actual samples of each document type. The samples need to be representative of all documents in the system since they will form the basis of the Document Analysis activities. Additional materials may also be part of the Document Inventory such as user documents, specifications, and training materials.   

This work product usually takes the form of a simple table listing each document type and related, relevant information. In addition, it consists of a list detailing the sample documents and other documentation that are gathered from the client and form the basis of document analysis activities.

Brief Outline

A table listing the document types and related information should be created. This table is usually included in a deliverable to the customer. The related information is project dependent, but possible columns to consider for the table are:

  • Document Name.
  • Type/Format – for example, Microsoft Word.
  • Current Type/Format.
  • Future Type/Format.
  • Process – Identify business processes that are relevant to the document. Also describe how the processes are relevant. For example, identify where created, modified, or destroyed.
  • Tasks Supported – Business tasks that are supported by the document.
  • Quantity/Period – Number of documents per period of time.
  • Volume – Total number of documents.
  • Number of Pages.
  • Size (Bytes).
  • Scan Resolution (dpi).
  • Retention Period.
  • OCR/ICR/OMR.
  • Language (English, Hebrew, Chinese, etc.).
  • Life-cycle Milestones – Important milestones in the life of a document where something needs to change about how the document is handled by the system, processing, security, or presentation.
  • Change Frequency – How often is the document modified?
  • Retrieval Frequency – How often is the document retrieved?

Any given project will typically use only a small subset of the items above in the final work product. It is also possible that some situations will require other fields not listed above. The intent of this work product description is to be flexible for the needs of the project and yet provide some items to consider.

For the samples and additional documentation, obtain both hardcopy and softcopy, as available and produce a list or table of contents for this reference information.

Key Considerations
The Program Manager should keep the original set of sample documents and other reference material for the life of the project.
Tailoring
Impact of not having

The set of document types that are to be part of the system solution will not be crisply defined. This can lead to scope questions later on and customer satisfaction issues.   

In addition, it may be difficult to determine what sample(s) and other documentation were used to develop the design. This may lead to confusion or degenerative rework later in the project if changes need to be made to the original design. On large projects, it may help the Analyst to remember the line of reasoning used in making evaluations and design decisions.   

Without this work product, the project will need to allow for extra time in the testing phases to rework the document design for aspects of the system that are overlooked or forgotten because the scope of the document analysis was not defined up front.

Reasons for not needingThis is an essential work product for EDM projects and should not be omitted. If the solution does not deal with the management of documents, then this work product can be omitted. It is conceivable that a particular project may be dealing with only one or two document types with trivial structure so that everyone on the project team can keep this information in their heads, but even then, it is in everyone’s best interest to document the scope of what is being done.
