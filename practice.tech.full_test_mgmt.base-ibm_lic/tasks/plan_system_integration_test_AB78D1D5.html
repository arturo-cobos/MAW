Task: Plan System Integration Testing
Plan for system integration testing in which the integration of application subsystems, integration of applications with the infrastructure, and impact of change on the current live environment are tested.
Disciplines: Technical Testing
Purpose
To ensure that the Systems Integration Testing activities appropriately address the end-to-end systems testing of a networked set of applications and systems, including application subsystems, applications, infrastructure, and platforms. If this phase is being done as part of a standalone evaluation and assessment of performance and capacity, the full set of tasks may not be required, but the context usually found in a projectâ€™s master test plan must be provided here.

        Main Description

        Define the interfaces that must be tested to ensure that business and technical requirements have been met. Determine the business functions and technical functions that must be tested. Define the environment components, applications, infrastructure, and platforms to be tested. Plan the environment for testing implementation. Define the scenarios that will exercise the elements to be tested. Identify the workload requirements and how they can be met. Plan the contacts that will be required to confirm the test requirements and facilitate their execution. Document the overall detailed systems integration test plan. Ensure that both business and technical categories of tests are covered.

        Consult your stakeholders throughout this activity to ensure client expectations are aligned to build buy-in, consensus and identify miscommunication or misunderstandings as soon as possible. This is especially important for performance and capacity evaluation and assessments because test facilities are expensive and rework is very undesirable.
        Steps
        Define systems integration test scope and objectives

        Systems integration level tests verify the integration of all applications including external internal interfaces and the integration of applications with the hardware, software and infrastructure components in a production-like environment.

        The scope of the testing for this level defines:

        • Business functional requirements to be addressed
        • Technical (structural) requirements to be addressed
        • Quality and schedule targets appropriate for this level

        This step clarifies key objectives such as, do we expect to merely receive a file through the interface, do we need to prove that it can be processed by the receiving system. Systems integration often tests a subset of the tests used for system and integration test levels. This is where the criteria for selecting such a subset are applied. What are the risks that this level must address? For instance, in Y2K systems integration tests, the focus may have been on the correct manipulation of the new four-digit date from one system, by a receiving system, after the date flowed through a changed interface.

        Test functions/features appropriate to this test level are identified as well as combinations of such to be tested. These are derived from the use case models and the non-functional requirements, which describe the functions and features to be tested for the project. The definition of what features and functions will be tested at which level of testing is usually arrived at via a workshop to ensure consistency and completeness. Test functions/features are defined to a sufficiently low level of detail to support an expansion into test cases for a given level of testing. The criteria to be used to determine whether each test function/feature has passed or failed testing are also noted here. Functions/features being excluded are noted here to identify out of scope items.

        This is a good opportunity to clarify expectations which may have changed during the project or never been completely understood. This is particularly important performance and capacity evaluation and assessment standalone testing, where expensive test facilities on tight schedules may be wasted when reset efforts are required.

        Develop test approach

        The components necessary to support this scope are identified:

        • Type of testing which will demonstrate compliance to requirements and targets
        • Approach (top-down, bottom up, critical first, functional grouping, phased, big bang, rapid, etc.)

        The testing approach, test data requirements and testing priorities are defined here and include:

        • Details of any other areas carrying out testing during the test level, such as the service delivery, internal audit, operations, business procedures etc.
        • The approach to be taken for obtaining test data e.g. live extract, test packs
        • The prioritization strategy; prioritize the features and items to test and identify considerations for not running certain tests should the project run out of time
        • Any boundaries of testing where the level and/or projects are inter-dependent
        Entry and exit criteria are usually established during the development of the comprehensive test planning stage. Entry criteria establish the pre-requisites which must be met before testing can start and exit criteria define the steps that must be completed before the test level can be exited. If they have not been sufficiently defined in the master test plan, they should be addressed here.
        Define test scenarios

        Systems Integration testing is achieved by executing scenarios that are supported by the application suite. The scenarios are exploded into more detailed transactions. These transactions specify the actions that need to be performed to complete the transactions and specify which data to perform these actions on. These are required for both business and technical (structural) tests.

        It is important to understand the logical flow of the transactions that make up the scenarios:

        • The screens are used and the sequence.
        • The screen input and the possible variations of input.
        • The resulting output from the input and possible variations of input:
          • Return code
          • Another screen
          • Error message
          • An unusual message that will require special processing 
        • The interfaces that are exercised.
        • The applications that input, receive and return transactions, data, and codes.
        • The network components that are invoked.
        • Any infrastructure participants.

        The goal is to translate the transactions from the logical flow to the physical flow defining the transaction protocols and decomposing the records into fields. The databases and type of access is also defined. Once this task is completed scripts can be developed, using the protocol and data from the component gathering.

        Defining the components of the scenarios allows us to simulate real transaction flows and loads. This task can be difficult or time consuming to do as:

        • It requires high business community involvement
        • The translation from requirements to scenarios may be difficult, particularly for performance and capacity test objectives
        • It may be difficult to identify the significant metrics or to capture them
        • The data may be very difficult to obtain for net new application suites or business ventures

        Some appropriate volumetrics expressed in business terms are:

        • Numbers of users and geographic locations.
        • End-user characteristics -- number of office tasks, business processes.
        • Key business entities - e.g. number of customers, accounts, products.
        • Business processes - e.g. numbers of accounts opened, stock movements, sales per day.
        • Usage profiles with time: e.g., Day, Week, Month, Seasonal peaks, Year.

        Some other challenges lie in understanding what is meant by, or the impact of, these aspects on the test objectives:

        • Peak period - the shortest time period for which the response time (average or percentile) will be "guaranteed" to meet the SLA. Within this period there may be times when the response time will be unacceptable.
          • When peaks occur?
          • Which level of peak this system should support (there is a trade-off with cost)?
          • What about "peaks within peaks", e.g., within the peak month, there may be one day?
        • Which has substantially higher volumes, or within each day there may be one or more busier periods such as opening the mail at 09:00.
        • Business growth plans.
        • Inputs from / to linked systems, e.g., 10,000 transactions arrive each day via the interbank S.W.I.F.T. network.
        • Scenarios relating to different user types; e.g., use case actors may have variations.
        • Scenarios when the mix of work may be very different, e.g.. Saturday morning, lunch hour.
        • System used in ways not foreseen, e.g., broadcast messages, or paying refunds very quickly via "no-sale" when customer queues build generating higher message rates, or using workstations from other departments.
        • Scenarios for end-to-end business transactions will generate these results:
          • Data
          • Response times
          • Utilization and performance rates
          • Network traffic rates
