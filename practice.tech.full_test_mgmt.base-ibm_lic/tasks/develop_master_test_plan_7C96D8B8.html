Task: Develop Master Test Plan
Define and develop test plans from a high level system view point and to describe what will be tested, how testing will be performed, and what resources will be needed and when. We need to make sure testing and development environment be consistent.
Disciplines: Test Management
Purpose

To define testing from a high level system view point and to describe what will be tested, how testing will be performed, and what resources will be needed and when. The purpose of the Master Test Plan is to address testing from an overall system viewpoint. It aims to prevent missed or duplicated testing during the full lifecycle testing process and to provide a big picture view, or a view of the forest, whereas the detailed test plans provide views of the trees in the forest. Master Test Plan helps ensure that the test plans are built systematically and completely and that there is no more and no less overlap between detailed test plans than is necessary.

    Main Description

    Master Test Plan is a comprehensive test plan that includes the objectives and scope, test methods and procedures, starting and completion criteria, test organization, and high-level test schedule for all levels of testing. A master test plan provides planning information, ahead of time, to alert the organizations to get ready for testing by allocating the right set of resources at the right time. By providing high-level requirements for hardware, software, and people resources as early in the lifecycle as possible, there is more likelihood that the resources will be planned, prepared, acquired, and trained in a timely manner. It also provides an early definition of the test environment requirements.

    The Master Test Plan is a document of understanding on testing between the various stakeholders of the project. It documents the "what" (scope and objectives), the "why" (purpose), the "when" (key milestones), and the "who" (organizational roles and responsibilities) for all levels of dynamic testing. Full lifecycle testing is an iterative process. The development of the master test plan is started early in the development lifecycle, and is refined, as more information becomes available. It describes all the testing levels such as Unit, Integration, System, Systems Integration, Acceptance, and Operability testing and ties together the respective detailed test plans. Each testing level represents a known level of physical integration and quality of the system solution under development.

    The master test plan evolves from the test strategy and is driven by the quality goals contained in the quality plan. It uses the concepts or organization level standards, and applies them to the project to identify specific test focus areas, types and levels of testing as they apply to the project. As the master test plan is developed it is possible that the levels of testing, as prescribed by the standards, may not be sufficient to meet the test objectives or the quality goals set forth by the quality plan. In this situation, the base set of test levels may be modified or additional levels of testing defined to accommodate the project needs. For example, Performance Testing, which is a type of testing as per IBM’s standard testing methodology, could also occur over a number of different levels of integration (i.e., at unit, integration, systems integration, operability, etc). It is possible that there could be a separate detailed (and/or master) test plan for Performance Testing and that Performance Testing could occur, as a stand-alone test, at different levels of integration. It is also true for Usability testing that spans the full lifecycle and runs throughout the development phases. Thus the master test plan needs to be scalable and tailorable, and must be flexible enough to be used for the small project or the very large projects.

    If the engagement involves a number of projects that correspond to portfolios of applications, it may be prudent to create a template for the master test plan (similar to the one done for test strategy). After review by the client and refinement, it can be used as a standard template for creating the master test plans for subsequent projects.

    In situations where the project is very large, and multiple releases and multiple implementations may be running concurrently, there are a couple of suggested approaches. Please refer to the Test Strategy Work Product for examples.

    For smaller size projects the master test plan may combine the detailed plans for different levels of testing into one plan.

    Determine build strategy

    Develop a build strategy to be used by development to construct system components (builds), which are compatible with the function priorities and with the project delivery schedule, taking into account the incremental testing opportunities, which can result.

    A set of testing plans will include the high level list of items for which testing is needed to be performed. These are the lists of business functions and structural functions that will be covered by testing activities. You may consider that these are testing blueprints and use them in the way that architects use blueprints as plans to guide them in their design and construction processes. The blueprints are not like resource and schedule plans - they serve very different purposes. Blueprints are more like a functional road map for building test cases. Both of these types of plans will exist on a development project, and both will share dependencies.

    The build strategy is the approach to how the physical system components will be built, assembled and tested. A build strategy is driven by overall project requirements such as user priorities, development schedules, earlier delivery of specific functions, resource constraints or any factor where there is a need to consider building the system in small pieces. Each build can be assembled and tested as a stand-alone unit although there may be dependencies on previous builds.

    Develop data strategy
    The subject of test data can be very complex and is specific to each individual project, necessitating the preparation and documentation of a detailed strategy.

    The data strategy for a project is a fundamental component of the Master Test Plan (MTP). Depending upon the size and complexity of the project, the data strategy may either be a sub section within the MTP document, or a separate strategy document, which is cross-referenced within the project MTP.

    Test data needs to fulfill the requirements of business users, service delivery and systems development staff.

    Key characteristics for test data are as follows:
    • Functional accuracy - supporting the required business test process requirements, to include:
      • Positive functional testing, i.e. results meet design/business requirements.
      • Negative functional testing, e.g. error recovery routines.
      • Continuity of processing.
    • Appropriate volumes:
      • Testing phase dependent.
      • Testing type considerations.
      • Cost effectiveness factor.
    • Levels of data integration:
      • Unit to unit
      • Function to function
      • Application to application
      • System to system (internal and external)
      • End-to-end business cycles
    • Data acquisition and maintenance:
      • Sources
      • Security and sensitivity
      • Data inventory
      • Ownership
      • Version control
      • Tools usage

    Test Data Management

    Test Data Management deals with requirements relating to the creation, sampling and storage of test data.

    • Support for test data preparation
    It should be possible to create, amend, display, print, replicate and delete test data for unit testing without recourse to the application under test. Using an editor and then using batch jobs to load the test data into test environment files and databases should create test data for unit testing.

    Files of test data loaded this way can contain descriptive comments relating it to documented test conditions and cases. The comment is stripped out when the data is loaded into the test environment. It is important, where possible; to support annotation of test data as it allows testers to find particular data and to understand why each record is present in the test bed. The aim is to ensure that tests are reusable by making them understandable to testers who may not have used them before.

    Once a capture/replay tool is available the technique should be extended to testing on-line transactions by writing procedures to enable the capture/replay tool to read files of test data and apply them to the application's user interface.
    • Support for randomization and sampling of production data
    Software should be written to randomize personal information contained within test files and databases. This is needed to ensure that personal information cannot be obtained from test data and makes production data available as a legitimate source of test data. Note: There is data protection legislation concerning the accuracy, use and availability of personal data that effectively makes its use as test data unlawful. For this reason test data containing personal data must be scrambled so as to conceal the identity of the people to whom it originally related.

    Software should be written to copy a specified number of data records matching specified criteria from one file into a similar file. If necessary the software should be further refined to ensure that referential integrity is maintained within the test data sub-sets. The data extraction method should ensure that at least one instance of every condition present in production should be represented in the extract. To do this is necessary to have performed domain analysis so as to define the set of test conditions for which examples are to be extracted. Domain analysis is, in any case, the first step in defining the test condition inventory and is a prerequisite to common testing techniques such as equivalence partitioning and boundary value analysis. A spin-off benefit is that the extraction process will identify invalid data values in production files. This technique will ensure that a representative sample of production data is available for system and acceptance testing.

    The data can then be enriched using application software or otherwise to add any test conditions that did not occur ‘naturally’ in production.
    • Provision of procedures for storage, retrieval and cataloguing of test data
    A considerable investment of time, effort and money may be made in the creation of test data. It is absolutely essential that this investment be not lost by failing to take a secure copy of the test data or by recreating the test unnecessarily every time a test is run. Likewise, previously achieved results should be treated as a valuable asset as they can provide a benchmark against which test results can be compared.

    There should be procedures to copy test data to systematically named back ups held on low cost storage media and to maintain a catalogue of the data stored. There should be similar procedures to restore specified data to the test environment. These procedures should be developed and controlled by a central testing support function.

    Develop test management systems
    When these procedures are available at the project level, they should be used as much as possible. If required these project level procedures may be tailored or adapted to suit the testing project needs.

    Typically, progress tracking and status reporting procedures are developed specifically for the testing project. Communication within the test project is a critical success factor for the project outcome. Ensure that regular status meetings and other communication related tasks are included in the test project plan. The guideline on Test Management Procedures provides a lot of guidance on how to set up these procedures and how to run the status meetings.

    The reporting management system consists of a series of regular meetings involving representatives from all the key participant teams with the intention of ensuring that the interlocked plan objectives are achieved according to schedule.

    On large projects, during the testing execution phases, daily operations meetings will be required, to monitor and set direction/prioritization on the progress of the testing activities and phases, and the project defect report status.
