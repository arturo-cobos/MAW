Task: Plan System Testing
Plan for system testing in which all the components that comprise a system are tested to verify that the system functions together as a whole.
Disciplines: Technical Testing
Purpose
To ensure that the System Testing activities appropriately address both functional and structural aspects of the system. To ensure that the environment is planned for adequately and that the System Test Plan clearly documents the activities to be performed.

      Main Description

      Define the system-level groupings of modules and programs that must be tested to ensure that Business and Technical Requirements have been met. Determine the structural functions that must be System Tested. Define the System Testing environment and plan for its implementation. Define the software release migration approach for the software being tested and document it. Document the overall detailed System Test Plan that includes Regression Testing. The System Testing activities are divided into two categories of testing:

      • Functional tests (e.g., error-handling testing, audit and controls testing, installation testing) and
      • Technical tests (structural testing , e.g., backup and recovery testing, performance).
      Steps
      Refine technical (structural) functions
      Refine the technical functions that must be tested for the system being built. These were defined at a high level during the test strategy activities. During this process, internal design of the application is examined using structural test focus areas to identify any additional structural functions that need to be tested. Also the preliminary criteria developed earlier are refined to complete the criteria for the acceptance of the system from a technical view point.
      Define software release library migration approach

      Define how the business software to be system tested will be migrated, controlled and synchronized into the overall planned software release libraries. Define an approach to control access, update ability and act as a basis for control of changes to the business software to be system tested. Define an approach to allow previous versions of business software in System Test to be recovered if necessary for other testing purposes.

      Note: The interrelationships between business software components makes changing system test software difficult. It is however important that they should be properly understood and considered if underestimated and inadequate changes are to be avoided.
