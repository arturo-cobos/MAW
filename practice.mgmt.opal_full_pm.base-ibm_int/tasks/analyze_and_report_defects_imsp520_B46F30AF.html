Task: Analyze and report defects (IMSP520)
Analyze the defect data and report the status in accordance with the project's communications plan.
Purpose
To analyze and summarize the defect data and report in the status report.
Relationships
RolesPrimary Performer:
  • Project Manager
Additional Performers:
InputsMandatory:
  • Communications Management Plan
  • Defect Analysis Document
  • Defect Log
  • Organization Level Measurement Program
  • Project Quality Plan
    		Optional:
    • None
    Outputs
    • Defect Analysis Document
    • Defect Log

      Main Description
      Analyze and summarize the defect data and report the status in accordance with the project's communications plan. Identify those defects to be analyzed further through a formal Root Cause Analysis. Update the defect documentation as necessary.
      Steps
      Analyze and summarize the defect data

      Analyze and summarize the defect data from the defect log.

      Analysis data includes the preliminary root cause of each of the defects, which is documented in task "Identify, raise and log defect".

      Report the status

      Report the status, summarized analysis data and measurements data for defects, in the project status report, in accordance with the project's communications management plan. 

      Analysis data includes the preliminary root cause of each of the defects, which is documented in Task "Identify, raise and log defect". 

      •  Refer to the Project Monitoring and Control Procedure (IMSP421) for information on producing the project status report.
      Identify those defects to be analyzed further

      Identify those defects to be analyzed further through a formal Root Cause Analysis.  A formal Root Cause Analysis may be required when a serious defect has an adverse effect on the project (for example, it impacts the schedule) or where trends in defect data indicate that there may be continuing, underlying problems with the conduct of the project.
       
      While determining the defects to be analyzed further through the formal Root Cause Analysis process, consider the following:

      • Impact of the defects
      • Frequency of occurrence of defects
      • Similarity between various defects (for example in terms of similar description, phase or cause)
      • Cost of analysis
      • Time and resources required for the analysis
      • Safety considerations

      A formal Root Cause Analysis is typically required when:

      • A severe defect occurs. For example:
        • If actuals exceed organization specific threshold (such as more defects were found than expected)
        • If performance did not meet expectations (such as actuals vs. estimates are further apart than expected)
        • If measurements are outside established parameters
      • A number of instances of the same type of defect or a pattern of similar defects are detected.
      • An increase in the number of defects is detected over time.

        Refer to the project quality plan for the project's planned quality activities.

        Refer to your organization's organization level measurement program for details of your organization's measurement program, such as the organization specific thresholds.

      Initiate the Root Cause Analysis
      If a formal Root Cause Analysis is required then, initiate the Root Cause Analysis in accordance with the Root Cause Analysis Procedure (IMSP525).
      Update the defect log

      Update the defect log as necessary.

      This ensures sufficient management visibility and control of the status and measurements for defects.

      Report defects

      Report defects in accordance with the organization level measurement program.

      Refer to your organization's Organization Reference Document (ORD), located in the OPAL Annex for details on your organization level measurement program, such as the organization-specific thresholds.
