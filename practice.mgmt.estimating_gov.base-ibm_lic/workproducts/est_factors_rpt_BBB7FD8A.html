Artifact: Estimation Factors Report (ART 0535)
This artifact assesses the current count of the estimation factors at each periodic estimation checkpoint review during project delivery, calculates the actual productivity experienced on the project, and charts the estimation factors progress and the project’s assumed and actual productivity.
Domains: Project Management
Work Product Kinds: Record
Purpose

This artifact is the main input for the estimation checkpoint review during project delivery. It has two primary purposes:

  1. To record the progress of the estimation factors at each estimation checkpoint review.
  2. To manage risk within the project during delivery by:
    • Measuring the productivity of each estimation factor over time as the project progresses; and
    • Measuring the progress towards completing the work associated with each estimation factor used in the baseline estimate.

Estimation Factors can be used to indicate whether the initial sizing or estimate was wrong. As the project progresses, is the size materializing as estimated or was the size misunderstood? You can also measure whether or not you are achieving the assumed productivity. These two measurements are the primary purpose during project delivery.

To indicate whether the initial sizing or estimate is wrong, the Estimation Factors Report measures the Actual Estimating Factors Completed and the Estimated Number at Completion at every checkpoint, and compares these to the baseline number. Deviations from the baseline become immediately apparent and any lack of progress is highlighted.

At Project Close, the primary purpose is to document project information that can be used to compare projects of similar type and size, and refine estimation models.

Estimation Factors can be used to quantify scope in some way – leading to the ability to compare projects of similar type and size. The Project Details will provide information on type, and the Estimation Factors will provide information on size. The data collected on the Closed worksheet and in the final Estimation Checkpoint Review, after applying some analysis, will be used to validate and calibrate further refinements of parametric style estimates. For example, further calibration of tools such as the Use Case Estimation Worksheet (UCEW) is possible if we know the total number of use cases and the overall effort of the project. This is because you can associate the overall size/effort and the key Estimation Factor(s) used in the particular tool to revalidate the tool. This is only possible at project close.

Description
Main Description

This artifact measures the project’s on-going progress towards completing each estimation factor, and the productivity trend.

At project close, the counts of estimation factors and the productivity experienced on the project become important historical facts useful in refining future estimates, creating estimation models, and estimating future enhancements in Application Maintenance.

It also documents the results and records the values of the estimation factors at each estimation checkpoint review. The report is updated with the current actual values just prior to the review.

Brief Outline

The Estimation Factors Report:

  • Records:
    • Basic project information, such as the project name, client name, and project dates, category, business sector, and industry
    • The current baseline and actual counts per estimation factor for the project as of the checkpoint date
    • The assumed and actual effort per estimation factor as of the checkpoint date

  • Calculates the current productivity of the project measured from the project start date to the current checkpoint date

  • Charts:
    • Progress towards completing estimating factors
    • A comparison of the actual productivity of the project against the assumed productivity used in the baseline

Key Considerations

The actual productivity (actual effort per factor) shown on the Comparison of Actual versus Assumed Effort per Estimation Factor chart should fall close to the assumed productivity (assumed effort per factor). Differences from the assumed productivity need to be examined further.

  • If the graph for actual productivity is below the graph for assumed productivity, then more work is currently being accomplished for less effort than originally assumed. This might be due to incorrect assumptions, a greater than assumed average skill level, or easy tasks being performed first with more difficult tasks delayed until later in the project.
  • If the graph for actual productivity is above the graph for assumed productivity, then less work is currently being accomplished for more effort than originally assumed. This may be due to incorrect assumptions, a lower than expected assumed skill level, or difficult tasks being performed first with easier tasks being delayed until later in the project.

Each of the Estimation Factors to be tracked is associated to some particular role and phase. For example, Use Cases would be tracked by the lead BA in the early phases of the project. The desire would be to validate both the count of use cases, and the specific effort required to deliver them. Similarly, the lead developer could track the number of components/classes delivered (pieces of source code that are easily counted and effort allocated), and the lead Tester could track the number of test scripts written and executed. Greater detail is available during project execution than at project close and it would be good to collect that detail data. If a bottom up estimate was originally produced, then these values can validate the bottom up estimate. If not, then the values become useful in future estimates as validation of rules of thumb such as 2 - 5 days to produce a use case.

There are multiple ways to collect the data amongst the various team leads:

  • a single shared workbook on a shared drive updated by all team leads
  • one workbook per subject area or team lead
  • an interview process to collect values from appropriate individuals into a single workbook containing all estimation factors used on the project
Tailoring
Impact of not having

Consequences of not having this work product are:

  • Absence of confirmation that risk mitigation tasks are effective in reducing risk
  • Absence of confirmation that the productivity being experienced on the project is adequate to complete the project within the effort budget
  • Absence of confirmation that the effort expended on the project is resulting in completed work
Reasons for not needingIf the project has not been estimated based on estimation factors, then this work product is not required.
