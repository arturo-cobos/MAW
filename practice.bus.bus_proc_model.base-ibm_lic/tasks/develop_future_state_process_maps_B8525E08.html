Task: Develop Future State Process Maps
Develop process maps for future state business process maps. If future state process maps are already developed, refine existing process maps.
Disciplines: Business Modeling
Purpose
To document future state business processes.

      Main Description

      Develop a future state or "to-be" high level business process map if none is available, or refine an existing one to capture the to-be state of the business process as the basis for identifying candidate services. Update high level process maps by adding and revising key milestones of the business process based on the business goals.

      Re-engineer existing process models so as to make improvements.

      Develop or refine the to-be process model and use this as the basis for identification of services by selecting a subset of the activities and processes presented in the to be process model as candidate services. If an activity is a low-level IT activity, that pertains to or describes an action with an IT system it is probably a candidate for service operation rather than a service itself. Services are described by the service interface which contains a number of service operations.

      The creation of the to be process model based on the as is process model must follow best practices in business process modeling and business transformation and engineering. As such, we encourage the practitioner to look up best practices for business process modeling which is prevalent in the industry.

      However, as a set of generic guiding principles, we suggest the following heuristic formula:

      To-be Process Model = As-is Process Model + Industry Model + Goal-Service Model focus area + Refactoring.

      This suggests that the to be process model should be created based on the problems encountered and the enhancements required to be applied to the as is process model. In addition standard industry models that contain standard processes such as that found in, for example, eTOM in the telecommunications world; or IAA (insurance application architecture), etc. These industry models provide a basis for typical processes found in particular industries and reflect best -- practices for that particular industry. The next input into creating it to be process model is to focus on areas specified by the goal service model. The goal service model further constrains project scope by pinpointing areas that are focal points of interest for the business and the need to undergo transformation or enhancement or alleviate pain points within the previously defined scope of the project. The process of re-factoring consists of identifying commonality and variability. The redundant processes or activities provide opportunities to refactor and to extract a common set of activities and to identify a set of variations for different processes, data types etc.
