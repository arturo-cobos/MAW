Concept: Subsystem Bridging Business and IT
Describes the concept and importance of Subsystem.

Related Elements

Main Description

Service Orientation is about business-to-IT alignment. Hence in SOMA we have highlighted the importance of subsystem which links business partition to IT components.



 

Subsystem bridges business and IT levels of abstraction. The partitioning of the Business Domain results in a set of Functional Areas that contain aggregates of cohesive functionality representing subsystems. The relationship between functional areas and subsystems may be one-to-one or one-to-many. Subsystems bridge the gap between business, as defined by functional areas, and IT, as realized by service components, allowing a seamless transition from identification of business functionality to their implementation.

Each subsystem is a conceptual mechanism by which the boundaries of the potentially cohesive set of service components that expose services are defined. Subsystems are logical groupings of functionality to enable parallel workstreams. The subsystems become a blueprint for reuse. They provide an abstract behavior of not only the internal functioning but also contracts by which subsystems collaborate and depend upon one another. Subsystems have no interfaces and the contracts are defined by Services and their description/specification. Subsytems are realized by Service Components realizing the Services.

It is the Service Components of a subsystem which have interfaces that can be exposed as services. A Service Component as defined in Service Component Architecture, or SCA, acts as a controller or facade and has an internal component flow, just like an EJB session facade which exposes coarse-grained methods and talks to many different other EJBs and objects in the background. If staying with the Java analogy, a subsystem is like a Java package. As each Java package may contain multiple session facades, similarly each subsystem may contain multiple Service Components.

In SOMA, we map Functional Areas to the Subsystems in Functional Area Analysis. This mapping provides the bridge between the business domain and the IT domain. Then we apply the principles of Object Oriented Analysis and Design, OOAD, within the boundaries of subsystem in SOMA, to define the composition.of the Subsystems in Subsystem Analysis. Subsystems are composed of a set of service components that realizes services. A service component is an enterprise-scale asset that needs to be managed and governed. A service component is realized by a set of functional and technical components. Functional components provide business functionality; whereas technical components provide abstraction of infrastructure to encapsulate technical capabilities and non-functional aspects of the subsystem.
