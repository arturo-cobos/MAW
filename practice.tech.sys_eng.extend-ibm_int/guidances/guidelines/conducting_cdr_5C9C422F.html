Guideline: Conducting a Critical Design Review
The Critical Design Review is held to consolidate component design documentation and review it to develop a design baseline.

Main Description

1. Description

The objectives of the Critical Design Review (CDR) are:

  1. Establish component design baseline,
  2. Review the components' design for completeness,
  3. Review the components' test plans,
  4. Verify that the end-to-end test strategy validates system requirements,
  5. Verify production infrastructure capacity supports the system requirements,
  6. Establish traceability from component requirements to component design,
  7. Identify/review technical risks and mitigation plans.

2. Details

Review Participants

Team members to be part of the review should include:

  1. Customer(s) / Stakeholder(s) 
  2. Solution Project Manager (SPM) 
  3. Systems Engineering / Architecture team 
  4. System Test Team (system and performance test) 
  5. Service Delivery Center representative - architect and/or Move To Production (MTP) lead 
  6. Application Development teams.

Review Information

The systems engineer (SE) leads the Critical Design Review, thus he/she must work with other team members to prepare the CDR package. Copy the CDR template and update it as described below. The template has been used for other CDRs, but it may be tailored as necessary for your and your customer's needs.

The CDR should be a REVIEW of the system design and not a working session to define the design. The systems engineer should define have defined the baseline in the System Requirements Review (SRR) and the Preliminary Requirements Review (PDR). The SRR defines the system baseline and the PDR (if held) defines the component baseline. If the PDR is not held, then the component baseline should have been reviewed as part of the SRR.

The CDR is a review of the following information:

  • Component designs - to verify that the designs meet the component requirements
  • Component and e2e test plans - to verify that the test plans are sufficient to test the requirements
  • Production infrastructure capacities - to verify that the production infrastructure will support the requirements

Steps for Conducting the Review

The Systems Engineer leads the CDR by following his CDR agenda. The goal of the CDR is to verify that the component designs meet the component requirements, to verify the component and e2e test plans, and to verify the necessary infrastructure needs.

Experience from past reviews has shown that the following process is best for the CDR:

  1. Review all system and component requirement changes since the requirements baselines were defined. Or, if possible, provide a list of all requirements, showing traceability, to define the review requirements baseline. Review any Business Process changes since it was baselined. 
  2. For each system requirement, review all component requirements and designs that are needed to satisfy the system requirement (which shows traceability to the system requirement). Gain approval that the sum of the component designs fully satisfies the system requirement. 
  3. Review the Service Delivery Center (SDC) material
    Production infrastructure architecture
    Monitoring, Measurement, Management (M3) should have probe architectures documented and probe design scenarios documented if any business requirements include "real time" transaction availability and transaction timing reporting requirements.
    Any systems monitoring processes over and above the standard offering (i.e., World Wide Commend Center, Centralized monitoring and support, application monitoring, business volumes measurement and forecasting, etc.) should be documented here.
    Production capacity projections and resources 
  4. For each of the test types to be performed (Systems Integration Testing, Performance / Stress Testing, User Acceptance Testing, Pre-Production Testing, etc), review
    • Test scope and requirements
    • Test environment 
    • Test Data 
    • Test Ground rules
  5. Review any Data Load for the Tests or for the Production System. 
  6. Note defects, issues, risks, and dependencies for all sections as you review the sections 
  7. Gain signoffs for each section, as you review the section 
  8. Review the signoff sheet, the defects, the risks, the issues, and the action items so that everyone agrees with the review results. 
  9. Update the scorecard at the back of the review template to score the effectiveness of the review.

The CDR should be aligned with the SDC Kickoff meeting, since the SDC representative will prepare much of the same materials for the two meetings.
