Guideline: Performance Engineering TRR Criteria Checklist
The Test Readiness Review (TRR) ensures that test plans and test cases fully support the requirements for the system being tested, after which a baseline is established for both future reference and change control.
Relationships
Related Elements
  • Perform the Test Readiness Review
Main Description

1 Description

In an Enterprise SE&A project, the Test Baseline encompasses the test plans and supporting documentation for the system described in the preceding baselines.

This technique paper contains the checklists which the PE team uses during the TRR to ensure that the PE goals and objectives have been adhered to during the Test Readiness Phase, It is important that the performance and capacity aspects of the test plan are well-conceived and focused on addressing the key performance requirements and the areas of greatest performance risk. Deliverables, which are deemed deficient from a performance and capacity standpoint, may lead to the entry of a defect, which is then taken into account during the TRR Scorecard process.

2 Usage Context

This technique paper should be used as guidance while conducting a Test Readiness Review in an enterprise SE&A project.

3 Content

The following PE specific items should be verified during TRR:

TRR Preparation

  • PE work products / deliverables from earlier phases are revisited based on any new information from development 
    • Performance estimates 
    • Performance models 
    • Resulting performance reports (SLCA, Operational Model updates, sizing) 
  • Key performance risks areas in developed solution were evaluated using selected estimation / modeling techniques 
  • Results of estimation forwarded to appropriate areas (e.g. development, test team, etc.) 
  • Component baseline materials reviewed by PE team, especially 
    • Test plans and cases 
    • RTVM

TRR Performance Questions

  • Is the performance test set up so that it can assess the solution against the business and technical volumetrics, the SLRs and the capacity plan? 
  • Have business volumetrics changed? If so, does the test plan take into account the latest information? 
  • Has the prior estimation/modeling work identified areas of performance risk that the performance testing will address? Will the performance test team be sharing their results so that existing models can be calibrated against actual results? 
  • Do the development team findings show any areas of performance risk that the performance test should be attentive to? 
  • Is the performance test team prepared to engage the appropriate analysis and tuning skills should they be needed during the test? 
  • Are all performance test deliverables (test strategy, test plan, test cases, test environment) ready to go?
  • Are the development and systems support teams ready to support the test effort and respond to performance problems should they be found? 
  • Will the performance test team be rehearsing the live monitoring and performance/capacity management tools and processes that will be used by the production support team? 
  • Has the performance risk been reassessed since the CDR and are those risks being adequately addressed? 
  • Will the performance-centric items satisfy the applicable TRR scorecard criteria, i.e. 
    • Test Planning 
      • Test scope, objectives and assumptions have been documented, reviewed by stakeholders and agreed upon. 
      • Test entrance and exit criteria has been documented, reviewed by stakeholders and agreed upon. 
      • Test automation strategy has been defined. 
      • Problem/defect management procedures, including turnaround time by severity, have been established and agreed upon. 
    • Requirements Traceability Matrix [RTVM] 
      • All requirements that will be tested are included in the RTVM. 
      • Test types and methods to be performed for each requirement are documented. 
    • Application Readiness Verification 
      • All applications/components to be tested meet test entrance criteria. 
      • Results from previous testing have been reported and are available (i.e. unit, integration testing).

TRR Actions

  • TRR issues are raised for concerns which may affect TRR scorecard findings.
