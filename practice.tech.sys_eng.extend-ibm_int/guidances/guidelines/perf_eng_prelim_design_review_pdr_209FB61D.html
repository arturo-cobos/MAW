Guideline: Performance Engineering Preliminary Design Review (PDR) Checklist
The Preliminary Design Review (PDR) ensures that overall system design & component requirements aptly address system (and customer) requirements selected for implementation, are well understood and are agreed to by all parties, and then a baseline is established for future reference, change control.
Relationships
Related Elements
  • Perform the Preliminary Design Review
Main Description

1 Description

In an Enterprise SE&A project, the Component Baseline contains the component requirements[see footnote1] for the system described in the System Baseline.

This technique paper contains the checklists which the PE team uses during the PDR to ensure that the PE goals and objectives have been adhered to during the Component Baseline Phase, It is important that the performance and capacity aspects of the system design are well-conceived and capable of meeting requirements. This proves helpful to the later SE&A baselines as the components are designed, implemented, tested and deployed, which in turn enhances the likelihood that the overall project will satisfy the customer’s requirements. Deliverables, which are deemed deficient from a performance and capacity standpoint, may lead to the entry of a defect, which is then taken into account during the PDR Scorecard process.

2 Usage Context

This technique paper should be used as guidance while conducting a Preliminary Design Review in an enterprise SE&A project.

3 Content

The following PE specific items should be verified during PDR:

PDR Preparation

  • PE work products / deliverables from SRR phase are revisited based on any new information 
    • Initial Sizing developed using appropriate level of performance estimation or modeling 
    • System viability / feasibility / risk assessed from performance perspective 
  • Key performance risks areas were evaluated using selected estimation / modeling techniques 
  • Results of estimation forwarded to appropriate areas (e.g. architecture, development, deployment / operations, etc.) 
  • Component baseline materials reviewed by PE team, especially 
    • Architectural model (to gain an understanding of the components) 
    • Application model (to gain an understanding of the functionality and data) 
    • Non-Functional Requirements pertaining to performance 
    • Technical volumetric assumptions 
    • RTVM

PDR Performance Questions

  • Are the component performance requirements understood?
  • Are the component acceptance criteria for performance understood?
  • Has an initial sizing of the component solution been performed and fed into the overall system sizing?
  • Has the performance feasibility and risk for the component been accessed?
  • Have requirements for performance tools which will be required for development and for live monitoring of the solution been defined.
  • Have business volumetrics changed? Will they change our technical volumetrics or other performance-critical assumptions? 
  • Are both system and component technical volumetrics available?
  • What implementation specifics will require technology research with respect to their performance characteristics?
  • What results are available for the system performance estimation and modeling activities done to date?
  • What component areas will require performance estimation and/or modeling to address areas of performance risk?
  • Has the capacity plan been updated to take into account the system estimation/modeling done so far? 
  • Are the Service Level Requirements (SLRs) complete? Does the system estimation/modeling work done so far give us an idea of whether these SLRs can be met with the proposed solution? 
  • What performance techniques (performance budgeting, profiling tools, etc.) will be used by the component design/development teams?
  • Has a performance test strategy been defined?
  • Has the performance risk been reassessed since the SRR?
  • Will the performance-centric items satisfy the applicable PDR scorecard criteria, i.e.
    • Static Architecture Definition
      • Business monitoring tools (volume and volume projections) identified and agreed to.
    • Test Architecture Definition
      • Test architecture, approach, and the test data will sufficiently facilitate the verification of the system acceptance criteria and the system requirements, including the to-be business processes.
  • PDR Actions 
    • PDR issues are raised for concerns which may affect PDR scorecard findings.
    • PE Strategy and Plan documents are updated as needed to reflect any new information coming out of the PDR, with stakeholders being informed.
Footnote:

[1] From an architecture perspective, the design of a higher level of the system will create a requirement against the next lower level component of the system. This means that the “Preliminary Design” of the overall system will include the requirements against each of that system’s components.
