Guideline: Creating Requirements
Creating good requirements is essential for project success. Requirements provide the technical basis for project design, development, test, and acceptance.
Main Description

1 Description

Good requirements provide the project foundation to enable the project to meet its cost, schedule, and technical objectives.

Depending on the scope or complexity of the project, requirements may be developed at several levels. Examples of requirement levels are:
  • Business, System, and Component (as defined in the Systems Engineering Capability Pattern) or
  • Customer, Product, and Product-Component (as defined by the Software Engineering Institute (SEI) Capability Maturity Model Integration (CMMI)
For a specific project, there may be many levels of requirements (including several levels of the component or product-component requirements that are defined above) or there may be as few as one level of requirements (project requirements for a small, simple project).
 
Each level of requirements should contain the following categories of requirements. Some projects may not have any requirements within a certain category, but the project team should verify whether there are any requirements to be met within each category. The requirements categories, as defined within Global Services Method, are:
  • Functional requirements
  • Non-functional requirements
  • Interface requirements
  • Usability requirements
  • Other requirements
Also, requirements may be defined in many formats. Requirements may be defined within a Use Case Model, within a picture, as a ‘shall’ statement, or using free-form text.
 
For all requirement levels, requirement categories of requirements, and requirement formats, ‘good’ requirements have certain characteristics. The characteristics of ‘good’ requirements include:
  • Requirements must have certain ‘quality attributes’
  • Requirements must be baselined and changes to the baselines must be controlled
  • Appropriate stakeholders must help develop and must approve the requirements
  • Requirements must be prioritized
  • Appropriate tools must be used to develop and control requirements
  • Requirements must be traceable

2 Details

Details for each of the characteristics of ‘good’ requirements defined above are provided below:

Requirements must have certain ‘quality attributes’

Individual Requirement Attributes
  1. Unambiguous
    • Has only one interpretation
    • Clearly stated in the words of the audience
    • Consistent with other requirements
  2. Understandable
    • The interpretation is clear
    • Easy to read and understand
  3. Correct
    • Defines the conditions and limitations that are desired
    • No errors
    • Complete
    • Necessary – no gold-plating
  4. Concise
    • Unnecessary information is not included in the requirement
  5. Traceable
    • Up through higher level requirements to a business requirement
    • Down to lower level requirements
    • Down to design documentation
    • To test documentation (including test cases and acceptance criteria)
    • Via some unique name or number
  6. Design independent
    • At the current level
  7. Feasible
    • Can be implemented within project constraints
  8. Verifiable
    • A finite, cost-effective process has been defined to check that the requirement has been attained
Attributes of the Set of Requirements
  1. Unique
    • Requirements are not overlapping or redundant with other requirements
  2. Complete
    • Everything the system is required to do throughout the system’s life cycle is included
    • Responses to all possible inputs throughout the system’s life cycle are defined
    • Document is defined clearly and self-contained
    • There are no to be defined (TBD) or to be reviewed (TBR) statements
  3. Consistent
    • Internal, no two subsets of requirements conflict
    • External, no subset of requirements conflicts with external documents from which the requirements are traced
  4. Comparable
    • The relative necessity of the requirements is included
  5. Modifiable
    • Changes to the requirements can be made easily, consistently (free of redundancy) and completely
  6. Attainable
    • Solutions exist within performance, cost and schedule constraints

Requirements must be baselined and changes to the baselines must be controlled

As defined in the Systems Engineering Capability Pattern, technical baselines are created for each level of requirements and architecture documents (customer business, system and component, as appropriate). The baselines are then placed under configuration control, as each baseline is approved in the appropriate review.

The System Engineering Capability Pattern creates a set of six technical baselines. These baselines can be tailored or combined based on the needs of the project. Each baseline contains a set of activities, tasks, and work products to create a set of deliverables. The project requirements are baselined in the first three technical baselines. These baselines are outlined below. For more information on the technical baselines and tailoring of the technical baselines, see the Systems Engineering Capability Pattern.

The technical baselines provided by the capability pattern which baseline the requirements are:

Customer Baseline – baselined at the Business Requirements Review (BRR)

Objectives:

  • Convey a clear understanding of the business scope, objectives, and requirements that pertain to this project
  • Understand the business acceptance / success criteria for the project
  • Identify "as is" business processes that will be affected (if available)

Deliverables Created:

  • Business Requirements Specification – a customer requirements and business process DCD

System Baseline – baselined at the System Requirements Review (SRR)

Objectives:

  • Establish the system technical baseline and review the customer baseline
  • Review changes to the customer
  • Review the system requirements - both functional and nonfunctional
  • Review the system architecture
  • Review requirements acceptance criteria

Deliverables Created:

  • System Requirements Specification – a system requirements DCD
  • System / Component Architecture (at system level)
  • Technical Solution Decisions
  • Requirements Traceability and Verification Matrix (with system information)

Component Baseline – baselined at the Preliminary Design Review (PDR)

Objectives:

  • Establish the architecture and component baselines
  • Review system and component architectures
  • Verify system and component architectures against allocated system requirements
  • Review test architecture (software and hardware)
  • Review and approve the allocated component requirements and business scenarios

Deliverables Created/Updated:

  • Component Requirements Specification – a component requirements DCD
  • System / Component Architecture (at component level)
  • System Test Architecture
  • Technical Solution Decisions
  • Requirements Traceability and Verification Matrix (with component information)

The baselined documents, which have been formally reviewed and agreed upon, can now be changed only through formal change control procedures. The change control process will be defined by each project team.

Baselined requirements provide control and stability when interacting with the customer and serve as the basis for:

  • Further refinement of the project solution
  • Design and delivery work products and activities
  • Testing  work products and activities
    Appropriate stakeholders must help develop and must approve the requirements
    The appropriate stakeholders should aid in the development of the project requirements. The stakeholders may include:
  • Client
  • Customer
  • User
  • Business owner
  • Executive sponsor
  • Project team
    • Systems Engineer
    •  Architect
    • Project Manager
    •  Development team members
    •  Test team members

For optimal requirements understanding and agreement, it is usually best to have both the requirements ‘owner’ and ‘receiver’ be part of the requirements development process. For example, business requirements may usual owned by the customer, client, user, or business team and received by the project team. In this case, both teams should be part of the requirements development process. In a similar fashion, component requirements are usually owned by a Systems Engineering and Architecture team and provided to the development teams, and thus it is best if both teams are part of the development process.

Joint Application Development (JAD) Sessions or Joint Application Requirements (JAR) sessions can be used to facilitate the ability / need for the appropriate stakeholders (the requirements owning team and the requirements receiving team) to develop the project requirements. For more information on these sessions, see the ‘Conducting Joint Application Development (JAD) Sessions’ technique paper. 

The ‘Process Redesign Workshop’ technique paper also provides information on developing project solutions.

Requirements must be prioritized

Why Prioritize Requirements?

To ensure that the highest value requirements are satisfied first, it is important to prioritize requirements as a first step towards selecting which are included in upcoming projects. Prioritizing requirements additionally provides developers a mechanism for prioritizing their work to ensure that high priority work is completed, even when some scope must be sacrificed to meet delivery dates. 

Benefits of prioritizing requirements:

  • When performed with the customer, improves communication between customer and supplier, and facilitates separating essential from desirable, needs from wants, in a way not possible from only one side’s perspective.
  • May enable early delivery of functionality, translating into improved value to the customer.
  • Facilitates tradeoffs during Analysis and Design activities.
  • Facilitates managing requests for new requirements as well as change requests.
  • Facilitates delivery of a useful product in spite of schedule and resource limitations. Lower priority requirements can be deferred to later increments/releases. 
  • Priorities can be used to guide the work effort (e.g. ensuring high priority work is completed first), avoiding bailing important work at the end of a project to enable delivery of critical components.
  • Helps resolve requirement conflicts. Tradeoffs can be considered towards achieving the most customer-pleasing solution. For example, in the absence of requirements priorities, developers tend to resolve trade-offs in terms of ease of development and maintainability.

When to prioritize requirements?

 Prioritization of requirements is performed throughout a project.

1) Pre-Project

The stakeholders will prioritize requirements to verify that the correct requirements are selected for the project.

Whenever there is a backlog of requirements, prioritizing them assists selecting high priority requirements for upcoming projects. In order to select which requirements should be included in upcoming projects/releases, it is recommended that the requirements (including functional and nonfunctional) first be prioritized using a predefined scorecard.

Scoring schemes can be relatively simple and still provide a good estimation of the requirement’s value (e.g. score range of 0-100, five questions weighted for importance, with range of 0-5 points depending on the answer). Each requirement should have the Scorecard questions answered to enable prioritization. Simpler schemes avoid user frustration and excess overhead. It is also appropriate to record the rationale for Scorecard responses (e.g. what is the baseline and measurement for expected productivity gain).

Requirements that are near the top of the Scorecard list (e.g. those that are likely candidates for upcoming projects) qualify for further investigation. The  investigation, typically done as part of developing the business case, should include:

  • Validation of scorecard responses and further enhancement of the value statement (e.g. a review with Finance and Marketing)
  • Risk assessment (High Level)
  • Cost/Sizing (High Level)

Note that it can be appropriate to include lower priority requirements when forming a project. A requirement may be justifiable because it is of reasonable value, and fits logically with the other higher priority work (e.g. it hits the same components).

This prioritization feeds into, and does not replace, the business case. The prioritization scorecard helps select which requirements will give high value. The business case considers not only value, but also cost and risk.

2) Prioritize Business, System, and Component Requirements as part of creating the project baselines

Prioritization of detailed requirements within a project is often done using a classification scheme and is done with close cooperation from the customer. Prioritization should include all requirements (e.g. functional and nonfunctional).

Business, system and component requirements should be prioritized, and the priorities should be reviewed at the BRR, SRR, and PDR reviews. By working with the stakeholders in JAD/JAR sessions, the SE&A team can define the priorities of these requirements. The priorities will be inherited from the parent requirements and then updated to reflect the true value to the stakeholders.

For example, if 3 system requirements are defined based on a single business requirement (the parent requirement), then the initial priority of the 3 system requirements will be the same as the business requirement. If the business requirement was prioritized as an 'essential' requirement, then the 3 system requirements will be initially prioritized as 'essential' requirements. Through working sessions with the stakeholders, it may be determined that one or more of the system requirements are less important than the others. Thus, these requirements may be given a lower priority. The same relationship holds true for component requirements and the parent system requirement.

Prioritization of requirements may be necessary at even lower levels than Component Requirements. For example, if a component requirement indicates that a report is to be provided with certain fields in the report, it may be useful to prioritize the fields in the report (some fields may be more important to the customer than other fields, and prioritization of the fields would allow the customer to receive the most important report information in an earlier delivery of the functionality).

3) Prioritize requirements changes versus current requirements baselines - reviewed when the changes are assessed / approved

Most Plan Change Requests attempt to add requirements to a project. It is important that  requirements included in change requests be prioritized against all other outstanding requirements. Organizations and customer relationships may affect how value prioritization is implemented for Change Requests.

Prioritization Definitions

The following definitions (sourced from the IEEE Standards Collection, Std 830-1998) may be used for priority:

  1. Essential - This implies that the system will not be acceptable unless these requirements are provided in an agreed manner.
  2. Conditional - This implies that these are requirements that would enhance the system, but would not make the system unacceptable if they are absent.
  3. Optional - This implies a class of functions that may or may not be worthwhile.

Other prioritization definitions include (from highest to lowest priority):

  • Musts, Shoulds, Coulds
  • Essential, Useful, Desirable

The definition and meaning of each priority must be agreed to with the customer.

Requirements must be traceable

The Requirements Traceability and Verification Matrix demonstrates how each requirement must be tracked through the project life cycle. Traceability of requirements must be performed until each requirement is delivered, withdrawn or transferred to another project.

Requirements traceability is bidirectional and includes:

  • Tracking each requirement to the previous levels of requirements
  • Tracking each requirement to design and build documentation
  • Tracking each requirement to test documentation
  • Tracking each requirement to acceptance criteria
