Guideline: Performance Engineering PRR Criteria Checklist
The Production Readiness Review (PRR) ensures that the implemented, tested system has been able to meet its requirements and is ready to be deployed.

Related Elements
  • Perform the Production Readiness Review
Main Description

1 Description

In an Enterprise SE&A project, the Test Baseline encompasses the release contents, test plans and implementation for the system described in the preceding baselines.

This technique paper contains the checklists which the PE team uses during the PRR to ensure that the PE goals and objectives have been adhered to during the Test Readiness Phase, It is important that the that performance testing has confirmed the ability of the system to meet its performance requirements, and that there are no performance- or capacity-related obstacles to the systemâ€™s deployment in the production environment. Deliverables, which are deemed deficient from a performance and capacity standpoint, may lead to the entry of a defect, which is then taken into account during the PRR Scorecard process.

2 Usage Context

This technique paper should be used as guidance while conducting a Production Readiness Review in an enterprise SE&A project.

3 Content

The following PE specific items should be verified during PRR:

PRR Preparation

  • PE work products / deliverables from earlier phases are revisited based on any new information from development 
    • Performance estimates 
    • Performance models 
    • Resulting performance reports (SLCA, Operational Model updates, sizing) 
  • Key performance risks areas in developed solution were evaluated using selected estimation / modeling techniques 
  • Results of estimation forwarded to appropriate areas (e.g. development, test team, stakeholders, etc.) 
  • Component baseline materials reviewed by PE team, especially 
  • Test results 
  • Deployment / production management plans

PRR Performance Questions

  • Do the performance test results indicate that the solution will meet the performance requirements, SLRs within the constraints of the capacity plan? 
  • Is there (or are there plans to develop) a Service Level Agreement (SLA) for the deployed solution? 
  • Have the tuning recommendations from test been communicated to the deployment team and planned for implementation? 
  • Have all tests been completed and all performance requirements been met within acceptable tolerances? 
  • Is there documented evidence of the performance acceptance criteria having been met? If not, will the stakeholder sign off on any exceptions to allow deployment to continue? 
  • Did the performance test team rehearse the live monitoring and performance/capacity management tools and processes that will be used by the production support team? If so, have the results of this rehearsal been communicated to production support, or were they involved in the rehearsal process? 
  • Has the production support implemented the live monitoring and reporting tools and processes that will be needed to support the deployed solution with respect to performance and capacity? 
  • Has the performance risk been reassessed since the TRR and are those risks being adequately addressed? 
  • Are there plans that will allow for a backout staged implementation or extra capacity should unforeseen critical performance or capacity issues occur? 
  • Will the performance-centric items satisfy the applicable TRR scorecard criteria, i.e. 
    • Requirements and Architecture (Content) 
      • Requirements in the baseline versus requirements being delivered - Non functional requirements defined, enumerated, traced 
      • Gaps or risks identified and mitigated or transferred (Exclusions) 
    • Test Results 
      • Non-functional testing complete or projected 
      • Gaps or risks identified and mitigated or transferred (Exclusions) 
    • Expected Availability, Performance, and Response Times 
      • Expected end-to-end availability in production 
      • Expected system response times and other performance criteria (if not covered in performance or stress test results) 
      • Infrastructure changes made to the production systems to meet requirements 
      • Gaps or risks identified and mitigated or transferred (Exclusions) 
    • Deployment / Production Readiness 
      • Install Plans defined 
      • Dependencies, Issues and Risks defined

PRR Actions

  • PRR issues are raised for concerns which may affect PRR scorecard findings.
