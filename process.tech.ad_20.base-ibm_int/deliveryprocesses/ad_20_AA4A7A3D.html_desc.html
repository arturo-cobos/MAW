Delivery Process: Application Development 2.0
This delivery process (AD 2.0) provides the standard method for delivering the major work required for an application development engagement.

Purpose
This delivery process is intended for small to medium application development projects.

Description

The Application Development 2.0 (AD 2.0) delivery process consists of the following five technical phases, plus three standard PM phases serving as bookends to the process:

  • Start the Project: This PM phase covers the required project management start activities, plus running a Method Adoption Workshop (MAW) 
  • Solution Outline: Focuses on providing clients with the necessary costs, schedule and risk information to make an informed investment decision regarding a potential new solution. This phase is typically managed on a time boxed basis, so identification of the key areas of challenge for this phase to emphasize is important. 
  • Macro Design: Develops a robust architectural framework upon which to build incremental releases, and establishes detailed plans for the release(s).
  • Release: The phases in this cycle are repeated for each iteration of the solution to be developed and deployed. This phase is a logical loop around these next phases rather than a physical phase element.
    • Micro Design: Prepares for the build cycle of a specific release of the solution by driving the architecture and design to a release specific and implementation platform view.
    • Build: Incrementally develops and tests the solution until the objectives of the release are achieved.
    • Deploy: Deploys the solution and prepares for the next release.
  • Manage the Project: This PM phase covers the required event driven project management activities.
  • Close the Project: This PM phase covers the required activities to close out the project including gathering metrics and posting intellectual capital.

Solution Outline is conducted once. Macro Design is conducted in it's entirety once, but selected activities may be revisited during later releases as necessary. The three phases that make up the Release iteration are performed many times, once for each release of the solution. There is potential for the various iterations of the Releases to overlap, especially in the time frame allocated for Deploy. The Start the Project and Close the Project phases bracket the beginning and end of the project and may be duplicated depending on how the project is sold, i.e. where the breaks between engagements may occur. The Manage the Project phase should be active in parallel with all other phases in the project schedule throughout the duration of the project.
