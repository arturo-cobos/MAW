Description

While Solution Outline "scopes" the engagement and provides the client with the information needed to make an informed investment decision. and the Release phases (Macro Design, Micro Design, Build, and Deploy) deliver a specific release, Macro Design is also the phase that prepares for the releases. This phase addresses the architectural issues which effect all the releases. If the client situation can accommodate it, it is far better to spend the time in this phase to address the design issues that impact every release (global decisions) rather than rushing off to deliver a release. The standards, guidelines and architectural work products developed in this phase  will minimize long term development cost by eliminating as much rework, from release to release, as possible.

Further, the business marketplace is characterized today by constant change. Our clients require systems agile enough to accommodate this change and agility is accomplished through proper architecture. Before developing the releases, Macro Design lays the groundwork in preparing for the releases by developing a robust architecture, capable of withstanding unforeseen business change through the delivery of the releases and beyond.

This phase typically requires 1 to 4 months to complete, depending on the size and complexity of the system, and consist of the following major activities:

  • Develop Business Solution:
    • Service Model refacoring
    • Completion of all solution Use Cases
  • Develop User Experience Design
  • Develop Technical Solution
    • Develop Technical Architecture: This activity "structures" the solution via three key work products. The Component Model defines how the application as a whole is structured, across the releases, and how these major parts of the application communicate with one another. This ensures the releases are developed with a consistent view of the overall structure of the application. The Operational Model defines the infrastructure required, again with a cross-release view, and how the components are distributed to the various infrastructure nodes. The Operational Model (and the surrounding work products) ensures that the releases are able to share a consistent infrastructure. Architectural Templates represents an application architectural framework that defines the major application design standards. It ensures that the release are agile and cost-effective to maintain by providing a consistent and appropriate approach to common design problems. In addition, products and technologies required for all the releases are evaluated and selected.
    • Develop Data Design: 
  • Conduct Static reviews
  • Develop Solution Development Plans: This activity builds upon the engagement strategy developed in Solution Outline by driving the various plans (e.g., Test, Training and User Support, Deployment) from outlining general approaches to specific executable plans (i.e., cost, tasks, resources, schedule). The Release Plan defines the number of releases and assigns use cases and application components to the releases.
  • Setup Development Environment: This activity ensures an adequate build environment is in place for the Build Cycle phase.
