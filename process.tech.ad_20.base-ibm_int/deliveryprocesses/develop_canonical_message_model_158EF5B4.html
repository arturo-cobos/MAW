Task Descriptor: Develop Common (Canonical) Message Model
Develop canonical message model that includes defining the structure, rules and semantics of the data being passed in service messages.

Steps
Obtain industry or other canonical data models
Identify an existing industry message model of an existing canonical data model that may be leveraged to define the enterprise message format. Align the enterprise message format with an industry standard message model or a data model.

Research to find out whether there is any existing and standards-based industry model for the industry vertical for which this project is being initiated. If a standard industry model does not exist then determine the existence of a canonical data model. Tag the industry model or the canonical data model to be leveraged in SOMA. The standards based message schema or the data format assists in defining the specifics of the service operations. Aligning to an industry model enables flexibility to work in a cross enterprise SOA ecosystem. Examples of such industry models would be ACCORD for Insurance, OTA for Travel and Transportation, ARTS for Retail, etc. If the project is for an industry which has an industry model then it is recommended to be adopted. If an industry model does not exist for the industry vertical then we look for a canonical form of the data model in the enterprise (which may be an existing asset). The canonical data model may be then used as a starting point to define an enterprise message format.
Create or refine enterprise message format

Identify, create, and refine candidate message formats to be used downstream to define the service interface specifications.

  • From service model
  • From existing assets
  • From industry standards
  • From variation oriented analysis
  • Design canonical message set
  • Extension vs. restriction for complex types
  • Type hierarchies vs. composition
  • Managing relationships


Common messages refer to messages that are transferred across the tiers of a n-tier architecture. During each of these transfers, a message is being exchanged between tiers where each may have a different format. The key thing is to agree on one standard for a common message format within the enterprise so as to overcome any format translation overhead where an Enterprise Service Bus (ESB) is not used or when its use is deemed expensive in terms of format translation. The use of an ESB will take care of many of these mediations, transformations and routing.

In some cases, it may suffice to agree on the outgoing and incoming message formats. The issue of a common message format is a key architectural decision: you may choose to “roll your own” as specified here, adopt industry models such as travel industry’s OTA XML or adopt non-industry specific models such as those defined by OAGIS. In some cases, the decision will be to use a common enterprise message format that updates fields in the message and passes it on to the next tier for further processing. If such a common scheme cannot be achieved due to political factors, then adapters that translate messages to an internal common message format can be designed. They can also be leveraged as part of an ESB.

If an enterprise message format exists then it should be identified and tagged to be used (with potential specialization or enhancement) to define the service semantics, otherwise a new enterprise message format needs to be initiated.

Once the proper source is determined which can be either the Industry model or a canonical data model, it is used to create an enterprise message format. An enterprise message format is usually a specialization of the industry model so as to use only those model elements that are pertinent to the enterprise domain / line of business that is under consideration. The enterprise message format will be used to define the input and output messages for the service operations.
Develop or customize message schema to define service interfaces

Identify simple and complex types from canonical data model including

  • Naming standards
  • Defining namespaces
  • Schema design style
  • Using attributes vs. elements
  • Choosing data types
  • Adding metadata and annotationsAdding constraints

If an organization is adopting enterprise wide message standards then these standard formats must be mapped to existing data models. Since executing services depend on existing applications that are based on their own data models, this mapping of enterprise message formats to existing data models is a very important step to be done.

Input and output messages for individual services are normally based on existing application data models. If enterprise message formats are adopted then they must be reconciled with the input/output messages of individual services, so they are related and assigned to allow appropriate services to use and update them as needed. Services may need to extract information from or expect output from the Common Enterprise Message format.
