Deliverable: High-Level Test Plan and Specifications (DCD 078)
This deliverable provides high level recommendations for the test architecture, elements, approach, data and expected results for testing the solution against its identified requirements.
Purpose
The purpose of this document is to provide a basis for mutual agreement as to the proper and sufficient test architecture, elements, approach, data and expected results for testing the solution against its defined requirements. High-level definitions of testing are established during the Macro Design phase and are further detailed as applies to each specific release of the solution as they undergo Micro Design.
Main Description
This deliverable extends and enhances the existing test strategies outlined earlier in the project. This deliverable is initially drafted in the Macro Design phase and further detailed during subsequent Release phases (Micro design, Build Cycle, and Deployment).
 
This document contains:
  • The recommended testing plan to verify the solution satisfies its defined requirements as well as how that testing will be structured to provide the basis for client acceptance of the solution.
  • The recommended testing plan for the solution defining the timing; functions to be tested; any early hardware, software, and security environment requirements; and resource needs for the testing efforts.
  • Definition of the test scenarios, test matrices, test conditions, test cases, test data, test environments, and test scripts as apply to each level of testing to be conducted.
  •  Awalkthrough of anti-use cases to prepare for penetration tests and anomalous flows
Deliverable Properties
External DescriptionThis deliverable serves to document the design and prioritization of the tests to be used in verifying that the solution’s objectives are met. It covers:
  • Definitions for testing the solution. This includes refining the test strategy to include security, as well as describing the test architecture, elements, approach, data, environment, and expected results for each test level to be conducted.
  • Refinement of the Acceptance Test Plan developed per client specification and definition in Macro Design containing an initial version of the test scenarios comprising the acceptance test. The plan will be detailed further in subsequent phases. The acceptance testing and acceptance criteria will be based on the functional requirements (use cases and scenarios), nonfunctional requirements, and usability requirements defined for the solution.
  • Development of anti-use cases to ensure the functionality developed does not leave security holes.

Packaging InstructionsThe following table outlines the various work products that contribute to the contents of this deliverable’s topics or sections.
 
The work products identified for each testing level are defined at a high-level for all known releases during Macro Design. They are each more fully articulated for a specific release during Micro Design. They may each be revised, as needed, during the Build Cycle for a specific release. Modifications may also be made while developing one release that may apply to the solution as a whole or to its future releases.
 
This deliverable is initially drafted in Macro Design in support of the solution as a whole and in preparation of the solution’s first release. At the conclusion of each release’s Micro Design phase, it is completed to the level its work products attain in support of that specific release.
 
If an existing test environment is to be,used, it should be referenced and briefly overviewed as to its specifications, composition, and characteristics, along with any necessary modifications including security. These characteristics should be described for all the known solution releases during Macro design and more fully detailed as required per subsequent project phases.
 
Note the levels of testing conducted for a solution and per each of its releases are dictated by the project scope; the complexity of the solution as a whole and as pertains to a specific release of the solution; and the solution’s requirements. Consequently, testing defined and conducted for the overall solution and per each of its releases may entail all or some of the testing levels listed in the table below and may for some projects be expanded to include those not listed, for example, Stress Testing. From this perspective all test levels listed in the table can be considered optional as their specific applicability varies per each project and its defined scope.
 
Also note that the testing listed in this table is not for projects using a Rapid Application Development (RAD) approach.
