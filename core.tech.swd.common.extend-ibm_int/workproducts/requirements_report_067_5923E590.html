Deliverable: Requirements Report (DCD 067)
This document will be used to gain consensus between the client and IBM on the requirements and it will provide the foundation for all future project efforts.
Purpose
The purpose of this document will be to define and document the requirements that must be met by a new Solution.
Main Description
This document will contain the following:
  • A system context diagram that identifies the planned information exchanges between the solution and the external entities that will need to interface with the solution.A high-level use case model that lists the names, actors, and high-level descriptions for the use cases.
  • A list of the identified service level requirements, system constraints, and other technical properties for the solution. Specific measurements and/or criteria may not be defined for these requirements in this phase.
  • The list of identified usability requirements defined for the solution. Specific measurements and/or criteria may not be defined for these requirements in this phase.
  • A list of the business rules and/or policies that must be supported by the solution.
  • A set of user profiles that define the different types of user groups for the planned solution and the key characteristics of each group
 
Deliverable Properties
External Description

The objective of this deliverable is to outline the requirements that must be met by the new solution.  The IBM team will perform the following:

  • Identify the different types of intended users for solution and the key characteristics for each user type. 
  • Identify the planned information exchanges between solution and the external entities (systems/applications) that will need to interface with the solution, including the primary inputs and outputs for the exchanges. 
  • Identify the key requirements for the solution:
    • Identify the list of use cases that describe the functional requirements.
    • Identify the list of usability requirements.
    • Identify the list of service level requirements, system constraints, and other technical properties for the solution.

Packaging Instructions

This document will be comprised of information formatted for delivery to the client extracted from the following work products:

Business Rules Catalog – used to define the business rules that must be implemented in the system.  This is typically provided as a list of rules with at most a short description at this point.

Change Cases – are optionally included and are typically identified as out of scope requirements so there is no confusion if the circumstances arise where they will actually have to be implemented.  Not all change cases need be included if client involvement will not be needed to resolve them.

Non-functional Requirements – identifies the basic requirements related to performance, scalability, availability and other technical issues that the system is required to meet.  Additional business constraints may also be included in this section.  This section consists of a list of the non-functional requirements and a short description, measurable criteria is not required at this point.

System Context – provides a graphical representation of the various other systems that the new system will need to interact with in order to function.  All other systems need to be identified as well as the types of information that will be passed back and forth.  Optionally different user types may also be shown on this diagram.

Usability Requirements – identifies requirements related to usability which may impact system design or cause other non-functional requirements to be established in order to meet these requirements.  The usability requirements are represented as a list that mirrors what the users have been requesting regarding the user interface.  Details should be provided beyond “the system is easy to use” but need not provided down to a measurable level yet.

Use Case Model – identifies the functional requirements the solution needs to provide.  The initial set of use cases is typically developed as a list showing subject area, actor, use case name and description.  Definitions for all actors should also be included and the scope of the overall list should be as complete as possible.  The use case model may optionally include a graphical representation showing the relationships between uses cases as well as which actors can execute each use case.

User Profiles – provides details on the various types of typical users of the system.  Categories should be established that cover the entire expected user population with general characteristics identified for each group.  Although it not necessary to fully document each group at this time the minimum of computer experience, subject matter expertise, language and other special needs should be document at this time.

The final document will be delivered to the customer in a reproducible format.  The final report may also be delivered in hard copy format printed in a single, bound document.
