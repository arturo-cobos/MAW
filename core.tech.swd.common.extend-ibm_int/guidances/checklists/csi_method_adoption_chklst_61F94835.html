Checklist: CSI Method Adoption
Provides a means for teams performing method adoption on Complex Systems Integration scenarios to validate that they have covered most common items of concern.
Relationships
Related Elements
  • CSI Method Adoption
Main Description

The items in this checklist provide a means for teams involved in adopting methods in CSI programs to review their coverage of commonly encountered pitfalls. The check items may be manually reviewed during the method adoption activity, or alternatively the attached spreadsheet can be completed along with documentation of each check item to provide a record of these decisions and potentially identify any issues remaining to be resolved.

Check Items
General items
  1. A Method Adoption Workshop (MAW) has been held
  2. Adopted method approach has been documented and communicated to team
  3. Method artifacts have been identified, mapped to client deliverables and allocated to corresponding delivery teams
  4. Allocation of method artifacts and deliverables has been communicated to the delivery teams
  5. The adopted artifacts and deliverables are reflected in the Work Breakdown Structure (WBS) and plan
  6. Global delivery (GD) participation is clearly defined and the corresponding GD organizations engaged
  7. If multiple releases are planned, the method includes the production of business and solution roadmaps
  8. If multiple releases are planned, the method includes the production of delta and baseline designs for releases
  9. If multiple releases are planned, Detailed Design of release n is planned to complete before Solution Outline of n+1 starts
  10. If multiple releases are planned, Test of release n is planned to complete before Build of n+1 starts

Client Relationship items
  1. Client is aware of, and understands the key aspects of, the method approach being adopted
  2. Client clearly understands what method deliverables are being delivered when
  3. Client clearly understands the dependencies on them producing deliverables
  4. Client technical governance has an agreed interlock with program technical governance
Delivery Governance items
  1. Program delivery management understand the key aspects of the method approach
  2. Program technical governance (Design Authority) are driving the E2E aspects of the method approach
  3. Program technical governance are assuring the component-level aspects of the method approach
  4. Change assessment takes into account the need to update already-produced method artifacts
  5. The method includes a Business Requirements Review (BRR) or equivalent
  6. The method includes a Preliminary Design Review (PDR) or equivalent
  7. The method includes a System Requirements Review (SRR) or equivalent
  8. The method includes a Critical Design Review (CDR) or equivalent
  9. The method includes a Test Readiness Review (TRR) or equivalent
  10. The method includes a Production Readiness Review (PRR) or equivalent
  11. The method deliverables for each delivery plan milestone are clearly understood
  12. If GD is involved, additional governance is included to cover onshore / offshore method interlock
  13. If IBM is contracted to support once live, the method caters for transition of governance to support team
End to End Solution Delivery - Requirements items
  1. Requirements definition and management approach is clearly understood and documented
  2. Requirements (or equivalent documentation) are being assured for quality and completeness
  3. Both functional and non-functional requirements are catered for in the method
  4. The team are clear that requirements must be baselined before a commercial delivery commitment is made
  5. If requirements are being identified or elaborated via e.g. BPM, a single consistent approach is being taken
  6. Requirements or subsequent elaboration must provide sufficient detail to support test activities
End to End Solution Delivery - Solution items
  1. Major components of E2E solution have been identified
  2. Method produces a single view showing all major application components and a static view of their key interactions
  3. Method produces dynamic application component interaction views for key E2E scenarios
  4. Method includes an E2E data model at some level plus corresponding data placement matrix
  5. Method drives early specification and tight management of interfaces between major application components
  6. Method supports the update and maintenance of these interface definitions through high-level design
  7. Method produces a single view showing all major production hardware and placement of application components
  8. Method produces infrastructure views for development and test environments in addition to production
  9. Method produces a performance and capacity model
  10. Method produces a design for system and service management of applications and infrastructure
  11. Method caters for "constraining" (i.e. hard to change) components such as packages or external interfaces
  12. Method includes allocation of requirements (or equivalent) to major components
  13. If GD is involved, the corresponding GD teams are included in the E2E solution activities
  14. If IBM is contracted to support once live, method delivers E2E solution documentation needed for support
  15. If IBM is contracted to support once live, method specifies necessary support environments
End to End Solution Delivery - Test items
  1. Method includes a test strategy setting out how all component types will be tested and integrated
  2. Method requires early test planning - as soon as E2E solution is established
  3. Method includes E2E functional testing
  4. Method includes E2E performance testing
  5. Method includes other E2E non-functional requirement testing, e.g. availability, systems management, etc
  6. If data is to be migrated, method includes data migration testing
  7. Method includes required client acceptance testing
  8. If GD is involved in E2E test, the GD teams are provided with sufficient education in the E2E requirements and solution
  9. If IBM is contracted to support once live, method includes E2E operability testing
Component Solution Delivery - Application Component items
  1. Major components are classified and appropriate component method selected in each case
  2. Component delivery teams understand how their component delivery method dovetails with the E2E method
  3. Method requires component designs to show compliance with allocated requirements (or equivalent) from E2E solution
  4. For components taking an agile approach, the method imposes E2E design constraints on the allowable agility
  5. For components taking a package approach, the method ensures the E2E solution is informed of any design limitations
  6. Method includes development standards
  7. Method defines application configuration management approach and processes
  8. Method includes unit test guidelines
  9. Method requires each component delivery team to test the corresponding component prior to E2E integration
  10. Method caters for any necessary data migration
  11. If GD is involved in a component delivery, method hand-off points and corresponding artifacts are clearly identified
  12. If IBM is contracted to support once live, method delivers component solution documentation needed for support
Component Solution Delivery - Infrastructure Component items
  1. Method caters for early definition of development environment to support its build in time for development activities
  2. Method uses a performance and capacity model to inform early infrastructure component sizing
  3. Method produces necessary application input to infrastructure design, e.g. data sizes, performance capacity
  4. Method covers all aspects of infrastructure, including network, servers, storage and systems management
  5. Method caters for the specification of all infrastructure, in the order in which the environments must be built
  6. If GD is involved, GD Center connectivity requirements have been factored into the infrastructure solution
Delivery Environment items
  1. Method includes early identification and planning of all environments needed for delivery
  2. Method includes environment configuration standards
  3. Method defines environment configuration management approach and processes
Business Readiness items
  1. Method identifies necessary business readiness activities
  2. If IBM is contracted to support business readiness, method includes production of education and training material
