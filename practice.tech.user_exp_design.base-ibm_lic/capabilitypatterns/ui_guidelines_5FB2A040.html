Work Product Descriptor (Artifact): User Interface Guidelines
The User Interface Guidelines work product provides a consistent set of standards to assist developers in constructing work products related to user interfaces.

Main Description

The User Interface Guidelines work product usually comes in the form of a (set of) document(s), with separate entries for:

  • Company-wide user interface design guidelines
  • System-specific user interface design guidelines
  • Project-specific user interface design guidelines

It typically contains a reference to (not a copy of) certain widely accepted industry standards (such as the Microsoft Windows User Interface Guidelines or the IBM Common User Access guidelines).  The industry standard reference may also be supplemented with a set of external regulations and standards, such as ISO (International Standards Organization) concerning display and keyboard standards for character width and height, display brightness and contrast, and keyboard key placement.  In addition, the User Interface Design Guidelines work product explicitly documents mandatory, recommended, discouraged, and bad practices based on the most frequent types of errors that are made against this standard and the specific types of systems that are being built across the organization.

Within the same company, different classes of applications may require a further refinement of the User Interface Design Guidelines.  For example, retail kiosk applications in a bank will be used by actual customers and will therefore have higher usability and accessibility requirements than traditional transaction-oriented applications used by the retail bank’s employees.  The latter can be trained; the former cannot.  Simply because the retail kiosk is a different style of interface, there may be some different or additional guidelines.  Thus, User Interface Design Guidelines may be applied across projects or across an application suite as a means to ensure enterprise-wide user interface design consistency.

Key Considerations

Illustrate your guidelines with examples showing compliance with a particular guideline. Show examples of noncompliance if the guideline is easy to violate or difficult to communicate easily. People sometimes learn more from bad examples than from good examples.

Be judicious in deviating from common, accepted industry standards, especially in the first release of an application. If you choose to deviate, provide a paper trail documenting the reasons for the deviation. Ideally, the reason for the deviation is based on specialized usability or data element requirements, and its tailoring is justified on the basis of usability test findings.

Appoint a style guide “czar” who is responsible for developing the User Interface Guidelines, reviewing user interface designs before they are completely implemented, and ensuring compliance with the guidelines. This person is sometimes not all that popular, so he or she should have the fortitude to ensure that the guidelines are adhered to, even in the face of strong opposition.

Make the User Interface Guidelines usable for the developer. Do not put a 300-page document on the developer’s desk that he/she is expected to read entirely before designing the first screen. Consider placing the style guide on the intranet, providing hyperlinks to allow developers to download style guide demos if they exist. If you use the document format, make sure to provide an index at the end of the document.

Depending on how much tailoring is required, a solution-specific set of guidelines may be anywhere from 20 to 200 pages in length.

Check that all guidelines are either very explicit or have references to documents or standards.

Involve an accessibility expert and users who have disabilities in the review process.

Involve the user through user reviews, discussions of examples, and/or prototyping of standard layouts and interactions.

Involve the developers by means of developer reviews. You should aim to create a feeling among the developers that the User Interface Guidelines are their User Interface Guidelines. If developers are not involved and/or disagree, you will encounter problems enforcing the use of the guidelines.

Call for a review by an expert from outside the project team. Getting the view of someone outside the team gives you some other approaches and perspectives. This should be someone who has expert knowledge of the industry standard and who has a background in usability engineering.

Verify external regulations and environmental style guides such as ANSI (American National Standards Institute) or ISO. Are there any legally imposed threshold values on the size of a character on a screen, the brightness of a display, the use of certain terminology?

Ensure that the guidelines are technically achievable in the target environment. If a certain guideline currently cannot be implemented in a straightforward fashion, if possible, describe a workaround to the guideline. If not, document the fact that the guideline does not apply to this release.

This activity is often “meeting-intensive” especially if a task force is set up to adopt “company-wide” standards. Try to handle as much task force communication via e-mail as possible, and try to limit the occurrence of time-consuming, potentially unproductive, in-person meetings. Be very selective in staffing this effort if the client insists upon creating the task force.
