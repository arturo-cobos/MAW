Main Description

The Usability Requirements work product defines and describes various aspects of usability for either a new solution under development or refinements and extensions to an existing solution. These usability attributes and measurable requirements (measurable requirements are user performance or productivity needs (e.g., 2 minutes to complete a transaction)) will be displayed in text or tabular format within the document.

The usability attributes are those features that facilitate ease-of-use with the system. These include the attributes of interaction (e.g., navigation), display (e.g., screen layout), affective (e.g., aesthetic) and can include accessibility considerations.

Accessibility requirements ensure that persons with disabilities can use the system. Therefore, the usability of accessibility features is also measured (because sometimes accessibility requirements can influence usability requirements). Refer to the work product describing and defining accessibility attributes.

The Usability Requirements work product should document the usability requirements of the features and implementation techniques chosen to meet accessibility standards. It does not need to document the technical accessibility requirements or the methods used to evaluate the implementation of technical accessibility requirements.

Accessibility and usability are two of the many important concerns involved in developing Web sites. Accessibility means that Web sites are barrier-free to everyone and open to individuals regardless of their disabilities. Usability means that Web sites are easy to use and demonstrate inclusiveness, intuitiveness, and common sense, like elevators that both announce and display the floor numbers. Both accessibility and usability are powerful indicators of an understanding of universal design. They reflect positively on Web developers and their clients, as well as lead to increased numbers of satisfied Web customers.

Key Considerations

Use representative current and potential future users and subject matter experts.

Map out the users’ requirements according to their classification. For example, an analyst will have different requirements than a data entry clerk.

Ensure that technical accessibility (the feature/function is enabled and can be performed) is separated from the usability of the feature. Consider including a subject matter expert and ensure the various stake holders (the user’s training, the capabilities of the assistive technology, and the technical accessibility enablement of the UI itself) are all considered when identifying requirements.

Allow actual users to supply all measurement criteria.

Verify measurement criteria with users’ management for “buy-in” and to reduce potential conflict (e.g., user must answer x number of calls a day, however user is rewarded only for the quality of calls).

Examine requirements for objective measurability.

Pretrain observers to achieve maximum interobserver reliability during testing sessions.

Plan requirements gathering sessions so listings of requirements are as exhaustive as possible, are objectively defined, and a consensus is reached.

Plan users’ testing procedures, including people who have disabilities.

Use managed iterated prototypes to provide early feedback. This improves communication between the developer and users because users react to a tangible asset.

Plan for and provide documents in alternate formats for people with disabilities. Note that Web accessibility requires that Web designers not only use alt tags, but also carefully craft the tag text so that users will be alerted to the fact that the graphic also serves a navigational purpose.

Engage domain experts, application designers, human factors experts, and graphic and media designers for development capabilities.
