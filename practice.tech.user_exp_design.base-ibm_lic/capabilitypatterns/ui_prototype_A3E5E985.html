Work Product Descriptor (Artifact): User Interface Prototype
A User Interface Prototype work product is a simulation of the user interface of a system or application with which a user can interact.

Main Description

A User Interface Prototype work product is built early in the development cycle with the express purpose of being changed and improved as part of the iterative design approach. 

Prototypes can be rendered in a range of fidelities, (e.g., low fidelity (lo-fi) and high fidelity (hi-fi)). They do not necessarily need to be working software.

A user interface prototype can be classified along the following dimensions:

  • Horizontal versus vertical: Horizontal prototypes represent a broad but superficial sampling of the system’s function. Vertical prototypes represent a narrow but in-depth range of the system’s function.
  • Evolutionary versus throwaway: Evolutionary prototypes are evaluation vehicles that incorporate evaluation results as adjustments and refinements to the prototype. In this fashion, the prototype may eventually evolve, or part of it may be incorporated into the production system. Throwaway prototypes are used to test a design concept with no intention of reusing the prototype.

Key Considerations

One of the difficulties in the development of User Interface Prototypes is knowing when to stop iterating. The following should provide some useful tips for creating prototypes:

  • Set deadlines at which point the increments of the prototypes will be tested.
  • Test each prototype with at least six users; if there is more than one user then plan to test with three to four users per segment.
  • Use scenario-based testing. Use the scenarios from the draft-level use cases as a starting point during lo-fi prototyping. Use full description use cases as you move into hi-fi prototyping. The full description use case models will provide critical information on data elements, which will affect the eventual detailed layout and flows of the views.
  • To adequately test the user interface, the prototype must adhere to the underlying information model targeted to the specific types of users.
  • Iterate until all usability problems are resolved.
  • Consider engaging with an accessibility expert in the development and/or description of the features. Resources with accessibility skills will be better able to explain the accessibility considerations when the prototype software doesn’t handle accessibility requirements.
  • Be diligent in specifying details of the user interface in both the prototype and the accompanying text specification. Otherwise, developers may design ad hoc, and ad hoc design is rarely successful.
  • Limit the scope of the prototype to user interface issues and concerns. Do not overload the prototype with other issues like performance, persistence, business requirements, or distribution.
  • Depending on the circumstances and the customer, sometimes it is advisable to develop the User Interface Prototypes using different hardware from that used for the final application. This prevents the customer and users from seeing great looking user interfaces, consequently assuming that development is nearly complete and demanding the system to be delivered immediately.
  • Lo-fi prototypes have the advantages of being easy and inexpensive to create and discard, thus allowing the early exploration and testing of design alternatives with continuous user involvement. They are limited in that they do not depict fully certain aspects of the system or application such as visual treatment, response time, and interaction “feel.” However, there are several compelling reasons to use lo-fi rather than hi-fi prototyping during the high-level design. Users tend to give more honest feedback to designs that do not look “finished;” alternatively, they often hold back when reacting to designs that look finished. There is usually reduced emotional investment on the part of the creator of the lo-fi prototype, as it is much quicker and easier to change or discard a paper-and-pencil rendering than a coded solution. Finally, changes can be made “on the fly” to enhance a design based on user feedback.

Any given design is not to be “frozen” until adequate prototyping is performed.  The following is a typical list of problems that can be uncovered during the investigation of a prototype:

  • Poor window navigation: Task performance may be made difficult by the nature of the navigation among windows. Often the usability of the interface can be significantly improved by tuning the flow of control through the windows and the allocation of functions between windows to a particular scenario.
  • Terminology and labeling: User interaction with prototypes often exposes problems in terminology. For example, some users may misinterpret a label because they use the same term to mean something different.
  • Window layout: Users often respond to a prototype by pointing out ways in which the layout of the controls or fields in a window could be more meaningful or more convenient to use.
  • Menu and tool bar organization. User interaction with prototypes can help assure that the most important and frequently used functions are those most accessible through menus, toolbars and other controls.
  • Setting defaults: It is often possible to save substantial user effort by careful selection of defaults. Prototyping offers the user the opportunity to validate default settings and to suggest additional defaults.
  • Controls usage: User interaction with prototypes often exposes problems in the selection of controls and widgets. For example, users may have difficulty in the use of a catalog or notebook, prompting the selection from a list box, or in the use of a catalog or notebook, prompting selection of a more meaningful or applicable control.
  • Accessibility: Considerations for how accessibility issues will be addressed for disabled users (eg.low vision, blindness, hearing impaired, mobility issues) and interface designs to assistive technology (See References).

  The User Interface Prototype should have the following characteristics:

  • It should enable user interaction and useful feedback
  • It should be detailed and complete enough to support some kind of evaluation.  Examination of notes taken during lo-fi testing sessions and data gathered, such as task completion times, errors, and subjective data during prototype evaluation sessions, can indicate whether the prototype was adequate for the intended purpose.
  • It should be based on available data
