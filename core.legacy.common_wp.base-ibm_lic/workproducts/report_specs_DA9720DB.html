Artifact: Report Specifications (APP 027)
Report Specifications describe new reports and enhancements to existing reports to be produced by the application.
Domains: Requirements
Work Product Kinds: Specification
Purpose

The purpose of this artifact is to:

  • Document the specifications for each new and enhanced report that will be available from the application according to the specified standards
  • Provide the development teams with the details to develop and unit test each report
  • Provide a consistent communications vehicle among the development teams that have responsibility for developing the various programs comprising the target application. The format of the reports should be consistent regardless of which part of the application produces a report

Description
Main Description

A report is formatted data that results from running a query and applying a form to it. The subject of the report can be anything associated with the target application such as:

  • An aggregation of the application’s data
  • The application’s settings or configuration
  • The status of the application’s communications facilities

Reports may exist for the current application that are desired in the new application. The new application may provide features and functions that require new reports. Reports may be produced through several means such as on a display screen, printed on paper, or stored in a file for archival purposes. Reports may contain textual or graphical information.

The work product also documents:

  • The format and use of any intermediary file used in generating reports
  • How each report will be unit tested?
  • How security will be maintained for each report?

In essence, the Report Specifications work product gathers together in one place all the information necessary for developing and unit testing new reports and enhancing existing reports related to the target application. They are particularly important:

  • In large or complex projects where several teams may be developing different parts of an application
  • When the specifications are written by one team and the reports are developed by others
Brief Outline

Report Specifications are text documents that contain the following information for each report to be generated:

  • Report Name: a name defining the report to be specified.
  • Report Definition: a description of the report including:
    • How and when the report will be invoked?
    • What data will be used?
    • What calculations and aggregations will be made
    • What logic will be used?
    • What format the information on the report will take?
    • How the report will be presented to the user such as printed or displayed?
    • What error handing mechanisms will be employed for the report content and structure?
  • Tracability: identifying the requirements implemented by the report.
  • Performance Requirements: a description of any execution time or invocation frequency requirements that must be considered while developing the report.
  • Testing Considerations: a description of the unit testing required to verify that the report performs as expected including error conditions.
  • Security Considerations: a description of the security requirements for each report.

Key Considerations
  • Once the client has approved a specification, it should be placed under strict change control. Changes to specifications can cause significant effects in the project, particularly in schedule and cost estimates.
  • The level of detail for the report specifications should suit the project, its staff, and the application. Project size, schedule, cost, staff turnover, designer and programmer experience with both the application and the implementation language, program complexity, and level of formality required by the contract all affect how reports are specified.
  • Ensure that the application has the data to produce the reports specified.
  • Ensure that the report support the logic of the client’s application, external systems, and business processes.
  • The report specifications form part of the basis for the test plans for the application. The testing considerations should be well documented so that verification can be completed as thoroughly and expeditiously as possible.
  • Ensure that error handling is specified and tested for each report.
  • Ensure that performance and security testing is included in the application test plans.
  • Consult the client to ensure that the business processes to be used by the intended application are supported by the report specifications. In other words, does the information presented in the reports support the intended business processes?
  • While it is important to stabilize reports as early as possible to allow separate teams to work independently, the report specifications’ level of detail should match the needs of the project. It should tell the programmer what he/she needs to know, not what he/she already knows. The designer’s judgement must be applied here. Sometimes the details of an algorithm must be specified; more often a simple, short text description of what the report is supposed to accomplish is appropriate.
Tailoring
Impact of not having

Not producing this work product presents several risks:

  • Incomplete or inaccurate reports may be produced in terms of content and layout
  • Some reports may be omitted altogether
  • Some reports may be produced that are not required
  • Reworking the reports may expose the project’s schedule and costs
  • Detailed documentation on the application’s reports may not be available when future development is required or application upgrades are scheduled, this may lead to the loss or corruption of previously developed reports. This applies especially to amendments to existing reports
  • The application may not function as expected
Reasons for not needing

.

This work product is not needed when:

  • Reports are not required
  • The client may elect to use standard reports produced by the application and no new or enhanced reports are required
  • The reports may have been specified already through other activities

If standard extensions are used or if the extensions are described by means other than the Report Specifications, the project team must be informed of the nature and location of these entities
