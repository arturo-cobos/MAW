Artifact: Target Action Report (ENG 366)
The Target Action Report is a list of prioritized perfective maintenance actions to improve (or create) software components or documents (generically, work products).
Domains: Project Management
Work Product Kinds: Plan
Purpose
This work product is the primary source of reference to plan, implement and verify perfective maintenance actions.

Description
Main Description

Software components may be selected from the Candidate Report or added based on business requirements. Actions on work products, such as documentation’s, are added to meet the business objectives.   

This work product contains:

  • the work product to be worked on,
  • a description of the action,
  • the exit criteria for the action,
  • an estimate of effort to complete, and
  • the relative implementation priority for the action.
Brief Outline

The Target Action Report has the following sections:

  • Context - Application Name, the software version analyzed, and when the analysis was done.
  • Business Objectives – a statement of what the business objectives are in pursuing perfective maintenance.
  • A list with each entry containing:
    • Work Product Names – names of specific work products (such as software component names) to be worked on.
    • Perfective Maintenance Actions - actions to be taken.
    • Exit Criteria - the minimum goal for the actions to achieve, generally based on the selection criteria.
    • Effort Estimate – an estimate of the time required to implement the actions in days or hours.
    • Priority - Urgent, High, Medium or Low; implementations priority for the action.

Key Considerations

Explore different perfective maintenance approaches, e.g. restructure, modularize, redesign, reengineer etc. or a combination of these, particularly when very complex software components are involved.  Consider options relevant to specific coding languages and environments.

Application developers with experience on the software components should be consulted when deciding on the perfective maintenance actions.  Books or references on code quality can be consulted too.

All the candidate components should be reviewed and considered. 

The inclusion of software components marked as potential candidates but not targeted for perfective maintenance is optional.  When they are included with reasons of why they are not targeted, it will provide a good reference for future when perfective maintenance is revisited.  For example, a software component may depend on database design.  It requires redesign of the database to improve the code.
Tailoring
Impact of not having

There is a risk of performing perfective maintenance on software components that may not be relevant to the business objectives.

Communication on possible perfective maintenance work needs to be done in some other way.
Reasons for not needingIt is recommended that this work product be created in all circumstances. At least a cut-down version of the report should be created even if little or no perfective maintenance is recommended as a result of the analysis, for example, only one or two software components require perfective maintenance actions and that can be done easily and quickly.
