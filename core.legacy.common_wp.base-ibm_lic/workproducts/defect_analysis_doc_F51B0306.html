Artifact: Defect Analysis Document (APP 320)
This work product is a detailed description and record of a defect identified during the project lifecycle, including its analysis and recommendations to rectify it.
Domains: Test
Work Product Kinds: Assessment
Purpose

The purpose of the Defect Analysis Document is to provide a permanent record of the analysis of a defect, the outcome of the analysis and the recommended action to correct the defect and all impacted associated system and test components. It is used to expand information on the defects, which are identified in the project lifecycle and logged, in summary form, on the Defect Log. The Defect Analysis Document ensures that any defects raised during a testing project are formally documented in a correct and consistent format and given a unique identifier. It provides an audit trail for evaluating analysis and resolution of a defect and supports metrics related to the quality and defect management objectives for the organization and the testing project. It also serves:

  • to provide direction/instruction/aid to the application development, testing or user management team on how to correct the defect generated during the project
  • to track the effort consumed in resolving a defect (and support collection of defect management metrics and estimating guidelines by complexity, work product type, phase of project, and skills)
  • to facilitate the decision on whether it is worthwhile, in terms of return on investment (ROI) and/or quality objectives to deploy resources to address the defect
  • to support the decision against resolving the defect (close without action within acceptable quality parameters) or the decision to postpone the resolution to a later date
  • to facilitate tracking of the source of the defects by complexity, work product type, phase of project, technology and skills in order to identify, both during and after the project, where defects originate and the possible root causes
  • to support audit trails for project or organizational quality reviews if problems occur with the application once implemented in Production; and
  • to facilitate a root cause analysis program or ODC (see References Section) which may be performed to improve the quality of the application during preventive or perfective maintenance.

Description
Main DescriptionThe Defect Analysis Document describes a defect found during the execution of test management, static or dynamic testing activities, or other project activities. It includes a thorough description of the defect and how it was caused, brief description of the steps followed/techniques used to analyze the defect, the findings and results of the analysis, and the prescribed action to be taken to address the defect. Also documented is a listing of all the project domain components or program modules that are impacted by the defect that may have to be examined and/or corrected in the resolution process. The work product may also include the approval of the recommendations, details of the resolution activity and a formal sign off of the defect resolution. The Defect Analysis Document can be used throughout the project lifecycle, although a document is required for each defect for which analysis is deemed necessary.
Brief Outline

The Defect Analysis Document includes the following data (although the form for capture could be modified to accommodate a particular organization’s requirements):


Brief Description:

Some general summary information about the defect, including how it was found and diagnosed as a defect:

  • Defect ID: a unique alpha numeric code following the standards for the project/organization
  • Status: one of the standard ones set by the project.  Some common ones are raised, assigned, analyzed and estimated, recommendation approved, ready for re-test, signed off, closed.
  • Submitted By: Name, Title, Contact Information
  • Date Submitted:  yy/mm/dd
  • Severity (1/2/3/4):     definitions of severity are preset by the master test plan or defect management process;  levels should be defined by the organization
  • Priority (H/M/L): separate priority may be set using other project factors to provide weighting information against severity.  These too should be predefined
  • Project or Application/System Name: as formally defined
  • Project ID:  standard to the organization
  • Project Phase:  and/or test level name
  • Work Product/Deliverable: for static testing 
  • Inspection or Test Name: use standard nomenclature
  • Inspection/Test #:  use unique standard codes

Defect Analysis Information

Provides a record of analyzed components and findings

  • Description of the Defect (include attachments if required):  additional information confirming or refuting status as a true defect  
  • How Defect Identified (method of identification – testing, inspection, etc.)
  • Attachments/Cross references: actual results, diagnostic results, code, document references and other supporting materials; web-sites, screen ids, etc.  with file locations or actually attached, how to cause the defect
  • Analyzed by: name/title/contact information
  • Date defect detected: yy/mm/dd
  • Environment/release:  (e.g., Production, System Test Environment, Release 2.3 System Integration test environment)
  • Brief Description of Analysis Approach: enough to allow someone to reproduce the analysis results
  • Analysis Findings: conclusions in text format
  • Components Affected: list what else is impacted – other work products, functions, technical components, code, tests, etc.
  • Recommended Resolution Action: high level solution design
  • Planned Resolution Action: tasks 
  • Estimate (days): work effort, skills and resource requirements
  • Estimated by: name/title/contact information
  • Estimated Date: schedule
  • Defect Resolution Information: record of what actions were taken, when and by whom
  • Date Resolution Required: target and dependencies and constraints
  • Escalation Path (if necessary): standard or special
  • Submit To Group/Person: who will approve recommended action and estimates and/or sign-off retest and or implementation 
  • Assigned to:   name/title/contact information
  • Assigned Date: yy/mm/dd of assignment

Final Resolution Information

Record of what actually was executed

  • Retested by:  name/title/contact information
  • Resolved by: name/title/contact information 
  • Complete Date: yy/mm/dd
  • Closing code
  • Comments and Final Resolution: any post – resolution activity or cross reference to other processes if the defect was closed as an issue or some other items
  • Resolution Accepted by: name/title/contact 
  • Acceptance Date: yy/mm/dd
  • Name:                     Signature:                Signed Date: (for official acceptance)

Key Considerations

Complete all sections of the Defect Analysis Document as thoroughly as possible.  The description of the defect may appear simple and obvious to the author but confusing and complex to the recipient who will be doing the analysis, and similarly the analysis and estimate may appear simple, but be misinterpreted by the resolution team.  Avoid ambiguous terminology, abbreviations and messy diagrams. 

Be cautious when capturing information on the defect.  Focus first on describing the defect, not what the possible causes may be; this will be the purpose of the analysis step.   It is considerate to attach supporting documentation in an organized manner, explaining its purpose, e.g. this core dump was taken when program XXX3363, module aaa-bb-cd abended on an OC4 whilst running Unit Test #4 using the Customer and Product databases at 14:15:23 on 23/09/2000. 

The Test Manager should facilitate the efficient flow of defect analysis documents and activity.  Any originator of Defect Analysis Documents should complete them as the defects are discovered and then batch them for each test execution cycle and submit them to the Test Manager for prioritization.  For iterations around the same test, do not hand over one document at a time for analysis as one defect may trigger another and it is then more advantageous to analyze the thread of a series of defects to derive a resolution. 

In order to ensure that adequate resources are scheduled to resolve defects according to priority, it is beneficial to have regular defect review meetings daily or weekly.  At this meeting, development management, test management, and business and technical test specialists would

  • review all new defects, and assign them or reclassify them as issues, enhancements, etc
  • review all closed defects and their resolutions
  • discuss how defects should be batched or prioritized;  and
  • review the metrics such as number of open/fixed/closed defects, and trends in debug/fix time and project impact.

Any critical impact on project deadlines or budget, other dependent projects and high-risk items are escalated to the appropriate stakeholders such as the project sponsor for resolution. 

At the end of the project, the Defect Analysis Documents should be collated into logical batches (e.g. by project/sub-project/test levels/test cycles/test types/defect severity/application impacted) and handed over (with the Defect Log)  

  • the test organization for trends analysis on defect management effectiveness;  and 
  • the application maintenance team (a) for reference in case the same error arises in production or during testing of enhancements and (b) for root cause analysis in preventive or perfective maintenance. 

In a mature test organization, where an automated defect management toolset is used, there may be some overlaps in information captured on the Defect Log and the Defect Analysis Document and the tool will dictate where the data is captured.

Tailoring
Impact of not having

The omission of a Defect Analysis Document could impact the integrity and quality of the applications and the data it outputs, depending on the severity of the defect. This is true whether the defect is with the deliverable produced from test management, static or dynamic testing activities or whatever the phase and size of the testing project. The impact of not having defects documented in detail, analyzed and addressed could be very costly if they are neglected or forgotten due to project pressures, etc. A few sample defects which if lost, could cause the project to fail or increase the risk of failure are:

  • During Test Management: error in the scoping of the test project could cause an overrun of schedule and budget, necessitate submission of change requests too late in the testing lifecycle, and shortage of resources to achieve the quality and test objectives. This would impact the finances and deadlines for the project adversely, resulting in a cascading of negative effects throughout the entire value chain of the organization.
  • During Static Testing: error in documenting a functional requirement (e.g. capability of software (systems and application) to process 10000 vs. 100000 transactions per hour, or user response time required is 1 second/transaction vs. 10 seconds/transaction) would result in analysis, design, build and testing effort being wasted on meeting a misdocumented requirement. If found during stress and acceptance testing (or not found until the application is implemented live), this would result in substantial costly rework and user dissatisfaction.
  • During Dynamic Testing: a code error in one program module that produces incorrect output is not thoroughly analyzed for the impact on other applications resulting in wrong data being processed and transmitted throughout the organization (e.g. in calculating the interest on a savings account: savings-account-interest = savings-account-amount / interest rate / 100 vs. savings-account-interest = savings-account-amount / interest rate x 100). If this defect is bypassed in the Defect Log and not analyzed and documented in the Defect Analysis Document for correction by the project, it could cost the organization loss of finances and credibility with customers.
Reasons for not needingThe Defect Analysis Document is required for capturing the details of a defect, its analysis and resolution for all testing activities throughout the project lifecycle. If it is a minor defect or the details of the defect and its resolution are negligible, then the information in the Defect Log may be sufficient and this Work Product may be omitted. If during analysis, it is found that what is recorded is not in actual fact a defect (e.g. it may be an incident, issue or problem which should be addressed in a different process), then the Defect Analysis Document should be used to record that the “defect” has been transferred to another process, and the unique identifier for that process be documented. This will enable test management to trace the series of activities and outcomes relating to this particular incident, issue or problem, even if it has been deferred to a later date for the purposes of, for example, root cause analysis or to be corrected in the next release of the application.
Representation Options

It is preferable to capture this information in an automated tool, where an automated tool or process is not available to in can be captured in the equivalent form provided.  The defect management software may also facilitate the moving of defects from one status to another, in which case the process enforced by the template would be automated, eliminating the necessity to capture this logistical information.
