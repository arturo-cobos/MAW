Artifact: Database Transaction Descriptions (APP 115)
The Database Transaction Description work product documents the transaction and also contains its implementation in Data Manipulation Language (DML) statements.
Domains: Data
Work Product Kinds: Specification
Purpose

The purpose of Database Transaction Descriptions is to ensure the system adequately enforces data integrity. Specifically, the work product ensures that the developers and database designers understand which database accesses must be grouped together into atomic units. This information is used:

  • To develop commit and rollback logic in the detailed program specifications.
  • As performance constrains during physical database design.
  • To prioritize relative performance of the transactions.

Description
Main DescriptionThe Database Transaction Descriptions work product groups database operations into logically related groups, i.e., transactions, such that the successful application of a transaction does not result in a database state that violates any known data integrity or business rules.  Thus, a transaction is an atomic unit of work that causes the database to advance from one consistent state to another consistent state. Transactions are implemented as a collection of database accesses. Either all of the database accesses should complete successfully or none of them should be applied to the database. Should an unrecoverable error occur during the execution of any one of the accesses in a transaction, the database must be rolled back to its original state. The mutual dependency of these accesses, necessitating their grouping, can arise as a result of the business rules and requirements or the design of the physical database.
Brief Outline

The Database Transaction Description is a text document containing the following:

  • Descriptive name
  • Description: no longer than a few sentences, it should describe what the transaction accomplishes, not how it is accomplished. The description should be understandable by end users so it can be reviewed.
  • Performance data: average frequency, peak frequency, volume, and execution time specifications are necessary for physical database design.
  • Priority ranking: transactions should be ranked to give database developers a basis for making transaction sequence and arbitration decisions.
  • Implementation: the actual DML implementation of the transaction completes the work product.

Tailoring
Impact of not havingModule and database design and implementation will be affected by not having Database Transaction Descriptions. During module design and implementation, designers and programmers will create the commit and rollback logic in an ad hoc manner. The project will lose the ability to ensure that this logic correctly implements business requirements. Errors that may occur in the implemented logic will result in databases that violate the system data integrity and business rules. In addition, since transaction performance and relative priority are established while developing the Database Transaction Descriptions, database designers and administrators will not have criteria to tune their designs and implementations. Consequently, database performance will suffer.
Reasons for not needingSmall projects with relatively simple and intuitive transactions may not need to formally define transactions. In addition, systems implementing database accesses where each single database access forms an atomic unit of work do not need Database Transaction Definitions. In practice, however, these systems are rare.
