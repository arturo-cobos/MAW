Artifact: Network Node Design (ARC 317)
A Network Node Design Work Product Description contains information about the physical nodes (i.e., vendor products) of a network.
Domains: Operational
Work Product Kinds: Specification
Purpose

The purpose of the Network Node Design is to provide enough information that the development team, or any subcontracted team can take the schematics created and actually implement the network. There should be enough detail that any trained member of this team can take the information and specifications provided and correctly setup, test, and get to working order the devices of the physical network. As such, this artifact becomes highly valuable as a learning tool and even more so as an ongoing problem determination aid in case of outage within the network.

It can also be used to order the network and the communications equipment, and the services specified in the design. It will allow the team to work with network and telecommunications vendors to price the proper equipment and services, plan for appropriate technical training (if needed), and provides a basis for the rollout / implementation plan.

Description
Main Description

The Network Node Design Work Product includes the following:

  • One or more diagrams that depict a detailed topology of the network
  • A diagram that depicts the recommended network management tools
  • The network management coverage matrix that provides a view of the tools used to manage each component recommended in the network node design
  • Detailed description of every node and link, including:
    • Physical representation of the nodes (router or hub ‘pictures’ supplied by the vendor in ‘computer graphical’ form; suitable for operations training, trouble-shooting, etc.) 
    • Addressing structure of the nodes 
    • Speeds and capacities of the links 
    • Physical location of the node, with power/cabling/communications attachments; rack location/address; switching elements 
    • Any other vendor specific feature/function required to support the Conceptual and Specified Designs
Notation

The following diagram shows the types of special notation used for this work product.  Although standard flowcharting techniques have historically been used to construct the ‘higher-level’ network diagrams (the use of ‘lightning bolts’ to represent network links, circles for nodes, other ‘boxes’ to represent network generic devices such as hubs, routers, servers, & switches), it is much easier to use a product like Microsoft Visio when producing these diagrams since there are vendors that make specific Visio-compatible libraries of a wide range of popular vendor networking equipment and will sell it on a CD-ROM image.  Actual device ‘images’ can be used and simple lines can be used to represent connectivity between these specific devices.  Any number of node and/or link attributes can be described using either a tabular or matrix format as shown in the examples below.  This method also makes it much easier for the client to keep updated and accurate inventory databases and assists in network problem determination.

Key Considerations

Key points to keep in mind when creating a Network Node Design:

First, the terminology used when selling a Network Strategy/Architecture/Design engagement MUST be clearly articulated and defined - initially to the client and then also in the proposal. These terms can very easily be misrepresented, misunderstood, or misconstrued, and it is very difficult to complete a successful engagement if the client thinks they are getting a 'Detailed Design', when in the Network Consultants' minds they are only planning to do a 'Conceptual Design' with no intent of getting to a deeper level of design. In particular, be very careful when using terms such as 'high level design' or 'low level design' without doing an additional level of qualification on what deliverables the client actually will be getting. An effective technique to use here is to actually show the client sample sections of prior engagement deliverables that help to distinguish between the terms 'high level' and 'low level.' Or better yet, use the definitions and examples found in the Networking Methodology and in this Work Product Description (WPD) and in the Conceptual Network Design WPD.

Second, it is key to include experienced Network Architects at this part of the engagement. The networking industry is changing at such a rapid rate that to attempt to create a Network Design without this level of experience and skill in networking standards, protocols, products, industry directions, etc. may result in a less than satisfactory engagement.

Finally, it is key to continually revalidate that the Network Design being constructed is actually in alignment with the client Networking Strategy and Architecture. It obviously does not make sense to create a Network Design that either can’t or won’t be implemented in the client’s environment.

Tailoring
Impact of not having

It would be difficult to create an implementation plan (and choose vendor products) without this artifact, as the previous network design artifacts are not implementable at the component level. The required detail for component level implementation is contained in this artifact.

The IBM Network Design Methodology will not be adhered to. A critical phase of the design process will be omitted, which may lead to a flawed overall network design.

Reasons for not needing

The Network Node Design Work Product would not be needed if:

  • The project plan called for just a network assessment or strategy - in other words, if this level of detail was not required by the proposal or contract terms
  • This level of design was being provided by the selected hardware or telecommunications vendor

 This artifact is not needed if the client took responsibility to produce and document this level of design detail.
