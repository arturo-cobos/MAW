Purpose

This work product provides a structured definition to be used during document creation and maintenance. It describes the document type definition to be designed, in whole or in part based on the document class definitions and is a necessary artifact for SGML/XML implementations.

SGML and XML are technologies that provide a framework for maximizing both the value of electronic documents and the ability to manage and access them. Computer systems can then use this framework in a variety of ways:

  • Arranging, formatting, and excerpting the same electronic source files for different online and paper forms.
  • Searching for information based on characteristics of its context within a document.
  • Using hyperlinks to connect cross-references with the actual information to which they refer.
  • Treating archived documents as a database that can be searched and analyzed.

Description
Main DescriptionThis is a code-level artifact that is specific to some extensible Markup Language (XML) and all Standard Generalized Markup Language (SGML) implementations.

The DTD defines the content of a document application, and the structure of that content. More precisely, a DTD defines:

  • The names of the elements that are permissible;
  • How often an element may appear;
  • The order in which elements may appear;
  • Whether markup, such as the start-tag or end-tag may be omitted;
  • The contents of elements (i.e., the names of other elements that are allowed to appear inside them, down to the character data level;
  • Element attributes and their default values;
  • The names of all entities that may be used.

A DTD does not contain information on how to process a document or what the composed document should look like (screen or hardcopy rendition). A DTD file is merely an ASCII file defining the content of the DTD using XML or SGML notation. Use of the Near & Far tool can assist in viewing the DTD file.

Notation

The notation used to create a DTD is very robust and complex. A minimum subset is introduced here. Further detail may be obtained from several of the references in this document.

Essentially, the notation used to define a DTD consists of the following SGML/XML syntax:

  • SGML element declarations:XML element declarations:
  • XML element declarations: Note: there is no omitted-tag-minimization in the XML standard.
  • Attribute declarations: : Note: the attribute-name and declared value are a repeatable group.
  • Entity Declarations:
    • Internal Literal: <!ENTITY entity-name "literal string">
    • External Entity: <!ENTITY entity-name external identfier>
    • Parameter Entity: <!ENTITY % entity-name parameter data>
  • Comments: <!--- comment text --> or --- comment text ---

Key Considerations

Document analysis performed by an experienced analyst is critical for successful DTD development, seek practitioners with XML/SGML experience to design and develop DTDs. 

Use Near & Far Designer software to assist in designing DTDs. and syntax checking.  If Near & Far is not used, another parser will be needed to check syntax. 

Obtain test data samples from the client or use previously gathered sample documents to verify the DTD design.  The more permutations of the data that you can test will insure a higher probability of success.  If the DTD can successfully be used to render client documents, the design is successful.

Tailoring
Impact of not having

If the ultimate solution is SGML based, defining the DTD structure is mandatory. The system cannot be completed without this work product.

XML may be well formed or valid. Well-formed XML does not require a DTD, but without a DTD, validation of the data is not possible in the final system. In addition, there could easily be miscommunication among team members concerning the intent of the XML. The development team would have to create, at a minimum, a "mental DTD" and allow more time for communication in order to work effectively together.

Valid XML, on the other hand, requires a DTD. It is highly recommended that for mission critical data where structure and validity of the data is important, only valid XML be used. Please see the listed XML references for a complete discussion of well-formed and valid XML.

Reasons for not needing

This artifact will only be used for XML or SGML implementations.  This artifact is not needed unless the solution involves decomposing the documents into document components to determine component reusability.

If the client can utilize well-formed XML, a DTD is not required. Well-formed XML may be used in cases where the structure of the data is dynamic and may change, "on-the-fly," over time.
