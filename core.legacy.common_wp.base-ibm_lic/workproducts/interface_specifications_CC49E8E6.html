Artifact: Interface Specifications (APP 030)
This artifact describes the interfaces between the various programs that comprise an application as well as between that application and external systems.
Domains: Requirements

Purpose

The primary purposes of this work product are to:

  • Ensure that the programs comprising an application operate together as required
  • Document the specifications for each internal and external interface according to the specified standards
  • Provide the development teams with the details to develop the interfaces

The work product is also used to:

  • Provide a consistent communications vehicle among the development teams that have responsibility for developing the various programs comprising the application
  • Stabilize and publish the systemâ€™s interfaces
  • Act as documentation for the system
  • Develop unit test cases

Description
Main Description

The work product documents the specifications required to develop each interface:

  • Within the application.
  • Between the application and external systems.

The work product also documents:

  • The format and use of any intermediary file used in transferring information across these interfaces
  • How each interface will be unit tested
  • How security will be maintained for each interface

In essence, Interface Specifications gather together in one place all the information necessary for developing and unit testing internal and external interfaces related to the application. They are particularly important:

  • In large or complex projects where several teams may be developing different parts of an application
  • When the specifications are written by one team and the interfaces are developed by others

Note: The focus of Interface Specifications is on program to program interfaces rather than user to application interfaces.

Brief Outline

Interface Specifications are text documents that contain the following information for each interface to be implemented:

  • Descriptive Name: a name defining the interface to be specified
  • Interface Definition: a description of the interface including what information and entities are passed across it, semantic meanings, structures peculiar to this particular interface, intermediary files, and error handing mechanisms
  • Pre-Conditions: a detailed description of such items as global data requirements, hardware and software requirements (including memory), operating system requirements, APIâ€™s, database views used, and the system state prior to invocation
  • Post-Conditions: a description of the interface after a information transfer has been completed including the state of the system and the possible error handling conditions and their effects
  • Tracability: identifying the requirements implemented by the interface
  • Functional Specifications: a description of the logic of the interface including details of any data transformations, database transaction definitions, signals, and error conditions
  • Performance Requirements: a description of any execution time or invocation frequency requirements that must be considered while developing the interface
  • Testing Considerations: a description of the unit testing required to verify that the interface performs as expected including boundary conditions
  • Security Considerations: a description of the security requirements for each interface

Key Considerations
  • Once the specification has been approved, it should be placed under strict change control. Changes to specifications can cause significant effects in the project, particularly in schedule and cost estimates.
  • The level of detail for the interface specifications should suit the project, its staff, and the application. Project size, schedule, cost, staff turnover, designer and programmer experience with both the application and the implementation language, program complexity, and level of formality required by the contract all affect how interfaces are specified.
  • Review the interface from both directions. Ensure that the information requirements across an interface are satisfied. That is, ensure that all the information sent across and interface is used and that all information required is available.
  • Ensure that the interfaces support of the logic of the application, external systems, and business processes.
  • The interface specifications form part of the basis for the test plans for the application. The testing considerations should be well documented so that verification can be completed as thoroughly and expeditiously as possible.
  • While interfaces are boundaries between functional units, they also possess boundaries within themselves. These boundaries are the edge conditions between success and failure, the latter meaning that an error condition has occurred. It is essential to test the boundary conditions of the interfaces to ensure that they are correctly interpreted. This, in turn, means that error handling must be well defined so that a robust application is delivered.
  • Ensure that performance and security testing is included in the application test plans.
  • Consult the sponsor to ensure that the business processes to be used by the intended application are supported by the interface specifications. In other words, does the information flowing across the interfaces support the intended business processes?
  • While it is important to stabilize interfaces as early as possible to allow separate teams to work independently, the interface specificationsâ€™ level of detail should match the needs of the project. It should tell the programmer what he/she needs to know, not what he/she already knows. The designerâ€™s judgement must be applied here. Sometimes the details of an algorithm must be specified; more often a simple, short text description of what the interface is supposed to accomplish is appropriate.
Tailoring
Impact of not having

Not producing this work product presents several risks:

  • The project may fail
  • The application may not function as expected
  • Information passing between constituent programs of the application or between the application and external systems may be corrupted or lost
  • The teams developing the constituent parts of the application may not know what information is passed across the interfaces and may make erroneous assumptions
  • Schedules and costs may be exposed due to the possibility of having to rework the interfaces
Reasons for not needing
  • There may be no internal or external interfaces to develop
  • The interfaces may be standard or have been specified already through other activities
  • External interface diagrams, detailed requirements documents, or use cases may already exist. These, supplemented by reference to APIs, may be sufficient to document external interfaces

If standard interfaces are used or if the interfaces are described by means other than the Interface Specifications, the project team must be informed of the nature and location of these entities.
