Artifact: User Interface Design Specifications (APP 148)
The User Interface (UI) Design Specification work product describes the overall user interface architectural framework and details the sequence of screens (such as windows, dialog boxes, and prompts) the user will see and interact with in the user interface.
Domains: User Experience
Work Product Kinds: Specification
Purpose

The primary purpose of the User Interface Design Specification is to communicate detailed guidance to the developers concerning the user interface to be implemented. In addition, the User Interface Design Specification:

  • Will serve as a means to facilitate agreement and shared understanding of application requirements between the design team and  stakeholders
  • Will serve as a means to describe UI system functionality
  • Will serve as a “contractual” agreement between the design team and the development team.
  • Will serve as a contractual specification between the design team and a subcontractor team, as applicable.
  • Will serve to illustrate how business requirements involving user interaction with the system are to be addressed.
  • Is used to document the final version of the user interface. This documentation can be used as a basis for developing future enhancements to the system and documentation and training materials for end users.
  • Is used to define system test cases

Description
Main Description

The User Interface Design Specification also includes detailed screen design, objects, widgets, options, menus, buttons, visuals, dialog boxes, button states, screen behavior, user exception and error handling, and reference IDs for content, error messages and help text. The User Interface Design Specification includes references to all (internal and external) UI Standards that the UI must follow, as well as any deviations/design constraints to those standards and why it deviates.

A User Interface Design Specification is usually communicated in the form of text, graphics, and tables in a written document or series of Web pages and sometimes augmented by a high-fidelity prototype, which shows particular interface nuances not easily communicated via the written word or through screen captures. The user interface prototype often contains screen layout and widget interaction code in a format that ideally can be delivered to the development team and reused in the development effort.

A user interface design specification may be delivered via a written document or integrated into a live prototype by illustrating the design using actual development environment code (e.g., HTML, JavaScript, Java) and augmented by user interface design “notes” embedded in the prototype.

The user interface design specification is a living document which is often updated throughout development and test.

Brief Outline

A User Interface Specification contains the following sections.

Design Overview

This section provides a general design overview of the user interface, in terms of the conceptual model, presentation and interaction techniques, recurring interaction patterns, prototype, guidelines, page level entitlements, views, menus, and usage.

  • * Background
  • * Prototype
  • * UI Guidelines and Coding Standards
  • * Accessibility considerations
  • Screen Transition Diagram/Site Map 
  • * Usage
  • Entitlements
  • Common/Reusable elements
  • Framework specifications

Detailed Design

This describes the detailed design of every screen in the system which should include the following information.

  • * Overview
  • * Traceability
  • * Screen Captures
  • * Tabbing Order
  • * Control Specification
  • Programming interface
  • * System Event Handling
  • Help Information
  • * Open and Closed Design Issues (Highly recommended)

Key Considerations

User involvement is key to the development of this work product. Remember that the ultimate goal of this effort is to specify a user interface design that meets or exceeds end user productivity, satisfaction, accessibility, and ease of use goals. Thus, it is critical that users help inform the design at all stages of the development of this work product.

Use the prototyping, evaluation, and refinement process to guide the detailed design activities that are required to complete this work product, especially in those cases where a clear direction for how to approach a particular navigational or interaction problem is absent. Good designers realize that the best design ideas come from the end users themselves.

Many programming environments today are well suited for rapid user interface prototyping. When appropriate, it is highly recommended that the engagement development environment of choice be used to build the user interface prototypes.

It is strongly recommended that during this design effort, all prototype code be built in a format that can eventually be reused by the implementation team. To carry this out, prototype developers must be highly familiar with the chosen development environment coding and class and part naming conventions.

Lack of attention to detail is one area that separates a well-specified design from a poorly specified one. There are a number of things that are typically overlooked.

    • Designs often do not detail all the potential exception conditions that may be triggered and how they will be handled. Ideally, all exceptions will be identified in the use cases, but that is often not the case. Try to brainstorm all possible exception conditions, validate them with the business analysts and use case modelers, and then specify in as much detail as possible those cases where and how error, information, or warning messages will be displayed to users.
    • Failure to specify all possible behaviors that a widget may support. For example, in a list box, a designer may forget to specify that a list can support multiple selection or extended selection. Without clear design direction and specification, implementers often make ad hoc design decisions as they code.
    • Failing to anticipate possible interaction effects if there are dependencies between selections of different widgets on a particular view.

Often usability testing is not feasible for identifying each and every design oversight. A well-documented set of use cases and business rules will go a long way in helping address these problems. In the absence of complete use case exception and business rule documentation, try to brainstorm everything possible that can go wrong as a user interacts with a particular view and then work out presentation and interaction scenarios that address these problems.

Start simple. Use an incremental and iterative approach to writing detailed User Interface Design Specifications, and get frequent feedback on interim designs from the eventual end users of the system.

Use the 80/20 rule to your advantage. Probably 80% of user interface navigation runs through only 20% of the user interface. Focus on the few screens with which the users will spend most of their time. Proportion your effort on window specifications according to the importance that interface has to the user.

Tailoring
Impact of not having

Not having a design specification results in ad hoc design by programmers. Ad hoc design is rarely successful and undermines the role of design in the development process. Specifically:

  • If there is a separate development team, the UI designer and UI development may not have a clear agreement on the user interface specifications. The User Interface might be very clear to its designers, but it might not be clear to the team tasked with its implementation. Furthermore, last-minute improvements that were suggested for the last version of a user interface prototype may be lost during the transfer of responsibility.
  • Usability requirements may not be met
  • There may be inconsistencies in the design and UI architecture
  • UI designers may need to continue on the project to support questions during development
Reasons for not needing

This work product is necessary in every case where there is division of labor between designers, developers, testers, trainers, and other stakeholders, and the design of the user interface must be communicated in a clear, succinct, efficient manner.

This work product is not necessary in systems for which there is no direct user interactions, (e.g software interfaces between two systems).
