Artifact: Program Module Specifications (APP 114)
The Program Module Specifications work product specifies the interface and function of discrete units of design or code referred to as modules.
Domains: Design
Work Product Kinds: Specification
Purpose

The primary purpose of the Program Module Specifications is to ensure that the program modules are of high quality and meet the design requirements by documenting the detailed functional and interface design decisions. They are necessary to communicate the design to the programmers.

The Program Module Specifications also facilitate concurrent program module development. The specifications completely define the inter-module interfaces and the module intended functions. This allows implementation teams to work on different parts of the invocation hierarchy, knowing exactly what to develop in their modules as well as knowing that the interfaces to modules on which their implementations depend are stable.

The Program Module Specifications are used to:

  • Explicitly guide the development of program logic by programmers
  • Stabilize and publish the system’s interfaces
  • Develop unit test cases
  • Act as documentation for the system

Description
Main DescriptionThe modules in Program Module Specifications work product display four basic attributes: input and output, function, mechanics, and internal data. The Program Module Specifications define these characteristics of a module and are the precursor to its implementation.
Brief Outline

Program Module Specifications are text documents that contain the following information:

  • Descriptive Name–This is a name tailored to suit implementation language considerations.
  • Interface Definition–The formal parameters of the parameter list including type, semantic meaning, structure, units, and domain must be specified. If the implementation language supports an error handling mechanism, these details must also be specified. These can be specified using the target language, whether it is compiled or a script.
  • Pre-Conditions–These include global data needs, system memory requirements, operating system requirements, presence of external modules, presence of external application programming interfaces (APIs), database views used by the module, system state prior to module invocation, and the presence of any hardware.
  • Post-Conditions–Any unusual, nonexplicit side effects from executing the module should be noted.
  • Traceability–To ensure requirements traceability and to verify that the module specification implements its portion of the user requirements, the relationship of the module to the process model.
  • Functional Specification–The internal logic of the module can be specified structured English or even descriptive text. This logic may include details of any data transformations and database transaction definitions. While it is important to stabilize interfaces as early as possible to allow separate teams to work independently, the functional specification’s level of detail should match the needs of the project. It should tell the programmer what he/she needs to know, not what he/she already knows. The designer’s judgement must be applied here. Sometimes the details of an algorithm must be specified; more often a simple, short text description of what the module is supposed to accomplish is appropriate.
  • Performance Requirements–Any execution time or invocation frequency requirements that must be considered while developing the code should be included.
  • Module Size Estimate–Very accurate code and labor estimates can be derived from the module specifications. These should be used to update the project estimates.

Key Considerations

When specifying the implementation of database transactions, the goals of procedural and logical cohesion will tend to align module boundaries along transactional ones.  Encapsulation of the transactional details in modules retaining the business semantics yields a more understandable and maintainable system.

The level of detail for the functional specification should suit the project, its staff, and the product.  A short paragraph describing what a module does may be adequate; structured English notation for every functional detail may be appropriate in other circumstances.  If the structured English standard is adopted by the project, the designer will be bound by the principles of good, structured programming.  Project size, schedule, cost, staff turnover, designer and programmer experience with both the application domain and the implementation language, program complexity, and level of formality required by the contract all affect how modules are specified.  Excessive module specifications will provide little help to the programmer who already knows how to implement specifics of the logic.  It is also has no maintenance value.  Structured English at the level of detail of the code itself is never maintained and contributes no aid to understanding the code.
Tailoring
Impact of not havingNot having Program Module Specifications leaves the structure chart of the program module invocation model as the only means to communicate software design decisions to the development teams. While the module and data interface names give some indication of what they do and are, this is not sufficient to guide parallel development. Expected module function and interface details will have to be communicated via word of mouth, greatly increasing communication overhead and the chances of error. Integration will be made more difficult because there will be ambiguity over what function a module is supposed to provide. Likewise, many implicit assumptions about the data interfaces may be proved erroneous at runtime.
Reasons for not needingThe module specification is a vehicle of communication between designer and programmer. Projects where the same personnel perform both roles do not need module specifications as a separate work product. This does not, however, remove the necessity of having proper documentation in the source files themselves as an aid to debugging and maintenance.
