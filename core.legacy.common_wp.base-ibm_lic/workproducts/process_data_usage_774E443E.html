Artifact: Process Data Usage (APP 112)
The Process/Data Usage work product is a bi-directional mapping between: the enterprise processes and enterprise information and/or, each entity in the data model and all the processes in the function model accessing it.
Domains: Design
Work Product Kinds: Model
Purpose

The purpose of the Process/Data Usage is to make explicit all the dependency relationships between processes and entities. This ensures that the:

  • Processes within the enterprise / engagement scope and the key information have been identified
  • Function model provides the necessary data support functions and that the data model provides the appropriate data to the functions
  • As these models evolve, the Process/Data Usage helps to ensure that changes in one model are appropriately reflected in the other

Furthermore, it provides the basis for:

  • Defining the boundaries of the overall application portfolio
  • Data and function distribution and placement decisions
  • Detailed system design

Description
Main Description

The Process/Data Usage work product can be defined at different levels, ranging from an enterprise level to a specific, single business system level.  The matrix is usually known as a CRUD matrix because it documents the nature of the action that a particular process has on each entity.

Processes / functions may:

  • Create new instances
  • Read existing instances
  • Update or modify existing instances
  • Delete existing instances

of enterprise information / data entities.

Both current and future versions of the matrices can be produced and, if it is appropriate, a gap analysis will highlight any major business implementation issues and considerations.

Notation

The Process/Data Usage is conceptually a matrix where one axis lists every process in the Process Model and the other axis lists every entity in the system. Each cell in the matrix identifies how the process corresponding to that cell uses the data corresponding to that cell:

  • (C)reate
  • (R)ead
  • (U)pdate
  • (D)elete

Any one cell may exhibit several or even all types of usage. This is more common the higher the level of the activities and information classes used. Its implementation may be in a variety of forms, including spreadsheet for small systems or database for larger ones.

Key Considerations

If the Process / Data Usage is being created at a high level to identify the boundaries of a portfolio of applications, then a 100% accurate model is not necessary…an 80% accuracy is sufficient.

At the more detailed level, changes in the function and data models may be required following the reconciliation between them that occurs in Process / Data Usage. In particular, the function model is important that the teams who own the models being affected are the ones to effect the change. The function model quality standards of cohesion and requirements traceability can only be maintained by analysts skilled in this activity. Likewise, a change in the data model may affect its normalization. The data team should control these changes.

To minimize versioning problems, the relevant models should be under change control before a reconciliation is attempted.

Tailoring
Impact of not having

Without the Process/Data Usage work product:

  • It would not be possible to recognize logical groupings of business activities based on their usage of business information. Overlap in future project boundaries may occur when it is unclear which activities define the data within shared information classes
  • The systems may be poorly designed and unable to deliver the necessary performance and availability operational characteristics. Furthermore these systems may be difficult and / or expensive to manage
  • Each change in either the function or data model will require a complete audit of the other for impact due to the change, the only way to tell if a particular entity is correctly initialized, accessed, or destroyed will be to conduct a complete audit of the function model, to ensure that a particular process has available to it the correct data in the correct format, the complete data model will have to be audited
Reasons for not needing

The Process/Data Usage is not required

  • Where a function model that outlines the boundaries of the applications has already been developed
  • Where the sponsor is committed to buying packaged software solutions, there may not be a need for this work product
  • Where the system’s non-transient data is not complex enough to warrant a data model, then Process/Data Usage will not be required, this is not the case for most business applications
  • Likewise, systems, where the processing is simple enough, not warranting a function or even a formal design model will not need a Process/Data Usage. However, database views will be necessary to maintain any parallelism between the user interface and the data model
  • Integrated CASE tools, where the function and data model are automatically cross-checked, eliminate the need for a Process/Data Usage. The CASE tool itself performs the role of the matrix
