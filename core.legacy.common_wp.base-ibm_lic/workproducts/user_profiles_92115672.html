Artifact: User Profiles (APP 150)
The User Profiles work product defines the characteristics of a typical user in each user category.
Domains: User Experience
Work Product Kinds: Inventory, Model
Purpose

Understanding the intended users of a new system and having knowledge of their characteristics is the first step in designing a user interface that meets users’ needs.  User Profiles help identify and describe the intended categories of users of the system so that the developers can build meaningful ease-of-use and accessibility features into their design. The User Profiles work product also serves as the basis for selecting users to participate in usability and accessibility evaluations. Test results are valid only if the participants in the evaluations are typical and representative users of the solution and User Profiles provide the benchmark by which potential usability test participants are identified. 

In addition, the creation of realistic personas based on these user profiles help “bring to life”  the representative characteristics and needs of the various targeted users, thereby enabling the user-interface design to be more effectively modeled. For example, many design teams use personas as a tool to supplement design storyboarding when presenting initial designs to business representatives.

Description
Main Description

Each user profile includes a description of the user’s prior knowledge and experience, physical characteristics, social and physical environment, cognitive characteristics current job characteristics, and goals, as they relate to how the user will interact with the new system.

Optionally, the User Profiles work product can include one or more personas. A persona is an instantiation of a user profile in that it “brings to life” the information in the user profile. These personas come to life through the creation of one or more fictitious persons who reflect the user characteristics described in the user profile.

According to Alan Cooper, the inventor of the persona method, a persona turns the data of a user profile into a 'real person', and helps the design team focus concretely on the intended end users of the system and their needs, especially in cases where a design team does not have significant day-to-day contact with the user population. Besides a fictitious name and photograph, personas include personal details such as behavior patterns, goals, skills, brief context scenarios and attitudes that will have influence on, and therefore should be considered in, the design of the system.

It is important to note that Use Case Actor, roles, personas and user profiles are not the same thing. For instance, a user described by a user profile could have one or many roles in interacting with a system. Likewise, there is not always a one-to-one correspondence between a user profile and a persona, as user sets defined by the User Profile may be distributed unevenly along one or more key user profile elements. For example, ”College Joe” and “Suzie Seasoned-Citizen” may be personas who belong to the same user profile (a Customer), but represent distinct sub populations of that user profile having non-intersecting values for key user profile elements - in this case Age (Joe is young, Suzie is older) and Computer Experience (Joe is tech savvy, Suzie isn’t).

Often user profiles are used in conjunction with the Use Cases, User Scenario, and Usability Requirements to better understand users and their needs.

Brief Outline

The User Profiles work product should include the following information

Overview

This section should describe an overview for each identified user profile including title and several summary sentences describing key attributes of each of the identified user profiles to be described in detail in subsequent sections.

User Profiles

Included in this section are:

  1. A one-sentence summary definition of the User Profile and
  2. Attributes for each User Profile. These six attributes are:
    • Prior knowledge and experience
    • Physical characteristics
    • Social and physical environment
    • Cognitive characteristics
    • Current job characteristics
    • Goals

Personas

Depending on the team’s familiarity with and access to the end user population, a persona may also be developed. Typically, personas are developed when the attributes of the end user population are not well known and/or not easily accessible by the design team -- typically in cases where a web development team is developing for the Internet or a team is developing shrink wrapped software to be sold to consumers.

Personas also can be useful in corporate environments even where a development team has easy access to end users (for example, an internal development team developing an employee portal) in helping keep design team members and other stakeholders on track during design sessions.

Personas are also a good way to highlight the broad range of users within the same user profile. As mentioned earlier, in the example of “College Joe” and “Suzie Seasoned-Citizen” – both may be personas who belong to the same user profile (a Customer), but represent distinct sub populations of that user profile because of non-intersecting values for a key user profile element - in this case Age (Joe is young, Suzie is older) and Computer Experience (Joe is tech savvy, Suzie isn’t).

Open and Closed Issues

Key Considerations

Coordinate with the Human Resources and Marketing departments and review your work with them. It is advisable to contact them even before you begin, because they might already have a great deal of information. In addition, they may also get the impression that you are invading their territory if you do not alert them to your data gathering activities.

One should develop user profiles and personas early on, prior to any user interface design work.

One rule of thumb for determining whether the user profile matrix is completed properly is that it clearly identifies users sets for usability testing when provided to a recruiter.

Follow systematic, structured processes when collecting information from and about users. Train those who will gather information.

Be sure to differentiate between the purchaser of the system and the actual user. Sometimes the user profile is improperly slanted towards the person who makes the buying decision instead of the person who will use the system. Sometimes it make sense to create a limited user profile and negative persona for the Buyer to help make this differentiation.

Note the numbers of users in each user profile. Neglecting to do this may result in the development team focusing excessively on a small population of end users at the expense of the majority. Note: Users with disabilities may be small in number, but the accessibility requirements need to be implemented for other reasons, including business direction, benefits to others, and standards and regulations.


If supporting a large project, with many different user groups and roles, use the role modeling technique to identify unique user profiles. The role modeling technique uses UML modeling to distinguish actual subgroups of users by clearly showing relationships between roles and facilitating the analysis of relationships between roles and goals. As mentioned earlier, see Role Modeling - A Technique for Defining User Profiles for more information.

Probably the most important guidance to convey in developing personas is that if an aspect of the persona description can't be tied back to real data, it's no longer a persona—it's at best a creative writing exercise Goodwin, K (2001, July). Perfecting Your Personas. and in its worse case, an invalid and misleading stereotype. In either case, it will lead the design team astray. Work to validate personas through inspection – compare persona drafts against the data captured in the user profile matrix or in information gathering notes and data.

Tailoring
Impact of not having

Designing a solution without understanding for whom it is being designed usually results in a poor design that frustrates users. This makes it more difficult to:

  • Lay the groundwork for establishing usability requirements
  • Understand potential use case actors
  • Understand how the requirements vary depending on user characteristics such as
    • Type of user (including accessibility considerations)
    • User role
    • Work environment
  • Understand how the application security should be modeled
  • Identify representative participants for usability evaluation

For example, if the intended users for a kiosk include naïve “walkup and use” customers but only experienced people were profiled in the design of the system, then the deployed system may fail because a large segment of potential users were ignored during the design modeling phase of the effort.

Reasons for not needingUser profiles may not be required if this work product already exists for a prior version of the system, though even in this case, the target user categories must be verified to ensure that the original User Profiles are still valid.
