Artifact: User Groups (ARC 311)
The User Groups work product combines business roles into categories of users with similar characteristics, IT experience and usage requirements.
Domains: Enterprise
Work Product Kinds: Inventory
Purpose

There are typically a large number of business roles. Consolidating these into a set of key User Groups simplifies the planning of suitable IT environments.

The work product is used to:

  • Provide a basis for the development of logical models of enterprise-wide usage of the technical environments (hardware, software and management).
  • Understand who will use a system, their characteristics and types of business locations - this is the first step in defining a common user interface and environment that meets users needs.
  • Avoid the duplication of function and infrastructure in the workplace.
  • Provide views of the different constituent parts of an Enterprise Architecture (applications, data and technology) that will highlight the similarities between many existing organizations/parts of an organization with different I/T systems, possibly leading to consolidation and savings.

Description
Main Description

User Groups are categories of users who have similar application and data access requirements. 

User Group characteristics include prior knowledge and experience, physical characteristics, social and physical environment, tasks, application and data access requirements (including interface requirements).

Illustrations
Examples
  • User Groups for Banking and Financial Services
Key Considerations

To avoid drowning in the detail of every organizational group in the enterprise, the user groups should be defined around anticipated similarity in their need for application and data access. For example, even though a Bank Teller and a Bank Customer Service Rep have a very different way of interacting with clients, both require access to similar information and IT function. You may decide to lump them into a single user group, realizing that the Teller has some unique requirements that can be dealt with when designing systems involving them.

Be sure to document which roles are included in each user group so that this differentiation can be done at the appropriate time.

It is important to view the user groups as collections of individuals/roles requiring the same or similar I/T support, not necessarily along organizational boundaries.

Tailoring
Impact of not havingUser Groups provide the basic foundation for the deployment of infrastructure within an enterprise. Without clearly defined User Groups, the implementation of function and technology to users tends to be driven by business processes only - which often leads to duplication and inefficiencies in the workplace environment.
Reasons for not needingThis work product is not needed if a base set of user groups has already been defined and is being used as the basis for implementation planning.
Representation OptionsA list of User Groups with their key characteristics is often held in tabular or matrix format.

The relationship between identified business roles and user groups is usually held as a matrix.
