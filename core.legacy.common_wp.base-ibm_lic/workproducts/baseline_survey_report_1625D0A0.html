Artifact: Baseline Survey Report (ENG 367)
This artifact captures the result of documenting, reviewing or confirming the current environment. This could be the current business environment, application environment or technical environment depending on the scope of the particular project.
Domains: Environment
Work Product Kinds: Inventory
Purpose

The purpose of the Baseline Survey Report is to establish a level of certainty about the current state of any business, organization, processes and technology components that would be relevant to the project scope. For example, in the case of an outsourcing, it is important to determine the existing state of the application software, system components, existing supporting documentation and quality assessment, current performance levels, known problems, work in process, and unfulfilled requests.

The Baseline Survey Report work product confirms the existence and state of the reviewed documentation based on source information availability, completeness, comprehension and relevance.

This work product is used as input for determining the constitution and boundary of the engagement, application or set of systems under consideration to enable informed decision making with respect to an assessment or a proposed solution and work estimates. The existence and quality of the above components will impact the ability to do subsequent work.

Description
Main Description

The Baseline Survey Report helps each project to establish a context for the project and a starting point for any subsequent evaluation effort.

Depending on the engagement scope, this work product would capture the current existence and state of documentation including but not limited to:

  • Corporate strategy
  • Mission, vision, value statements
  • Annual reports
  • Business process documentation
  • Organization charts and missions
  • High level technology architecture diagrams
  • System context - all the documentation which describe the application system with respect to the requirements
  • Logical design components - all the design documentation which generically describe the application system
  • Physical design components - all the design documentation which describe the actual representation of the application system with respect to its generic description
  • Hardware and software inventory - a listing of all the technologies in use and any known information regarding its selection, long term viability, maintenance, and impact on the project objectives.
  • IT standards - all the support documentation which direct the development of the application system
  • Operating processes and procedures - all the documentation which assist users in utilizing the application system and operators to keep the application system available
  • Operating environment - all the assets which support the running of the maintenance and enhancement function
  • Training and support materials - all the documentation and structures that are used in the provision of the required readiness needed by both operators and users to function properly.

Key Considerations
  • Consider the contextual knowledge of project team members who will need to interpret and make decisions based on the available business or application system documentation.
  • Consider the timeframe within which the problems identified in the requirements are expected to be completed.
  • Indicate the location of all information of relevance.

Development of this work product is very dependent on the scope of the engagement. A typical consulting engagement focused on the identification of capabilities and enablers would most likely need only business and high level process and technical documentation in order to establish their baseline of analysis. An outsourcing engagement, however, would use this work product to establish a very detailed understanding of the current client systems covered by the outsourcing agreement.

Tailoring
Impact of not havingWhen a Baseline Survey Report work product is not produced, there is a significant risk of the project team making uninformed decisions regarding the proposed solution thereby leading to poor quality or inappropriate solutions and ultimately poor customer satisfaction. This could also increase the likelihood of rework leading to a significant variation between estimates and actuals.
Reasons for not needingThis work product is always required whenever there is no relevant and consistent document that describes the current state of the application system. The report should still be produced even where an application has little or no supporting documentation so that its status could be accurately reflected.
Representation OptionsThe Baseline Survey Report is documented in a textual form, illustrated by diagrams where necessary and strengthened by references to relevant work products of the existing application system.
