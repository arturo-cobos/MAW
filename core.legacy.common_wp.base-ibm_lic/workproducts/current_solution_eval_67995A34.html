Artifact: Current Solution Evaluation (APP 142)
A Current Solution Evaluation is the documentation of an old application or existing business process that is related to the new application under development.
Domains: User Experience
Work Product Kinds: Assessment
Purpose

A Current Solution Evaluation is made to establish a baseline for the design of the new application. The end users of the new system will be satisfied and dissatisfied with various aspects of the existing system. In addition, they will expect the new system to maintain the aspects with which they are satisfied and improve upon the ones they do not like. The Current Solution Evaluation will help ensure that the users expectations of the new system, as formed from his perception of the current solution, are met. In addition, the Current Solution Evaluation can be used to convince users to migrate to a new solution by providing compelling reasons to change and provide input into the usability requirements and usability evaluation.   

In a market environment, competitors’ solutions may have set minimum standards for use. Providing those functions in the new application may be necessary just to be a viable competitor. With knowledge of competitive solutions, key-differentiating factors can be established.

The Current Solution Evaluation can be compared with the clients stated goals or perceived competitive differentiating factors. This comparison is useful in targeting the key areas of the application’s design that will require the most enhancements to succeed.

Description
Main Description

A Current Solution Evaluation can be a competitive solution, a solution that is to be replaced by the new application, or a solution that must work closely with the new application. This documentation may include analysis, interpretation, or assessment of effectiveness as needed. The following are examples of Current Solution Evaluations:

  • A Competitive Analysis compares multiple solutions in the marketplace against each other to find common functions and note differentiating functions.
  • A Gap Analysis shows the existing solution and how it compares to industry benchmarks, the project goals, or competitive solutions.
  • In a User-based Subjective Walkthrough, a user performs his/her current task while providing observations and subjective feedback to the observer. To get feedback about accessibility features that should be carried over or need to be incorporated in the new solution, be sure to include one or more users who have disabilities in the walkthrough.
  • With a Performance Benchmark, the focus is on establishing time on task and error criteria to establish new solution design priorities and productivity baselines.
  • A Heuristic Evaluation involves a human-interface specialist to conduct a self-assessment of a solution. The human-interface specialist uses a set of guidelines to gauge a solution’s usability, pinpoint areas for remediation, and provide recommendations for design change. This evaluation can also include global, national, industry-wide or enterprise-specific guidelines designed to assess the solution’s accessibility by users with disabilities.
Notation

User-Based Subjective Walkthrough

When providing a user-based subjective walkthrough, present current solution data in text and in graphical form. Compile text synopses of qualitative data (user comments). Construct tables for all quantitative data. Construct quadrant charts from user ratings of importance (or frequency) and satisfaction. The two key quadrants are the “quadrant of success” (high importance and high satisfaction) and the “quadrant of opportunity” (high importance and low satisfaction). Be sure that you also provide accessible formats for team members who have disabilities.

User-based subjective walkthroughs can also collect key objective performance data collected such as time on task (in minutes) and error rates (percentage of tasks that contained an error). Both recoverable and unrecoverable errors are defined and measured. Present the data in tabular and graphical forms, making sure that you also provide accessible formats for people who have disabilities.

Expert Heuristic Assessment

An expert heuristic assessment provides results that are usually presented in tabular form. Tables can include these headings:

  • Problem identification number
  • Screen name or number
  • Usability problem
  • Guideline violated
  • Impact/severity of this problem
  • Suggested design change to remedy this problem

Key Considerations
  • Some analysis is very useful, but be careful not to over-analyze and inhibit the design teams innovative thinking
  • Consider the attributes of the proposed new system when designing an evaluative study
  • Comparative studies are only valid if conducted consistently (i.e., the same performance benchmarks are used, the same ease of use questions asked)
  • Stakeholder reviews are useful to ensure accuracy and completeness of the evaluation
  • A heuristic review identifies issues and bottlenecks to avoid in the new system
  • A performance benchmark establishes firm criteria to measure the new system
  • A user-based subjective walkthrough defines substantive features in both the “quadrant of success” (high importance and high satisfaction) and the “quadrant of opportunity” (high importance and low satisfaction)
Tailoring
Impact of not having

The design team must rely on other sources, requirements or specifications, to ensure that none of the good aspects of the existing solution are left out. Often these sources are not complete, resulting in omissions that dissatisfy users. If the current solution is not evaluated, the new system may not meet the users expectations based on their current solution, resulting in customer dissatisfaction.   

Without reviewing competitive solutions the design team may miss good ideas that can be improved upon, creating a competitive advantage. In this case, the application will not be as successful as it could have been, and the client will be less satisfied.   

The application may not meet the minimum standards of the marketplace as dictated by the competition causing it to fail to attract users, dissatisfying the client.   

Furthermore, if the existing system is not evaluated, the new system may not meet the user’s expectations, resulting in customer dissatisfaction. The design team may overlook specific design ideas emanating from an examination of the existing system. Objective user performance data will not be available to evaluate the new solution. This may result in a new system that misses the mark in the client’s eyes without objective data to counteract that impression.

Reasons for not needingOn some projects, it is difficult to identify the current and/or competitors’ solution(s). The most obvious example is an organization developing a new system for a new area of business. Another example is an organization automating a manual set of business processes. In either case, for large projects developing systems with a large user base and a long life expectancy, usability is a critical success factor. The project team should gain as much insight as possible into the critical usability factors. Propose creative alternatives for situations involving difficult identification of the current solution or for large projects (e.g., evaluating the manual procedures). If there is no existing technical or traditional solution and no competitive solution(s) can be identified, the Current Solution Evaluation may not be needed.
Representation OptionsCurrent Solution Evaluations take many different forms. Any way that a solution can be described could be appropriate, depending on the situation: lists of functions, user scenarios, brand concepts, content plan, interactive models, functional models, screen shots with descriptions, process models, etc.
