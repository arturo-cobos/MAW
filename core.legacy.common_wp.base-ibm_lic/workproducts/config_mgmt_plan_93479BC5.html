Artifact: Configuration Management Plan (ENG 101)
This artifacts describes details for identifying, organizing, and controlling modifications to the configuration items.
Domains: Environment
Work Product Kinds: Procedure
Purpose

The purpose of this artifact is to get a shared understanding about the approach for configuration management and to ensure that configuration management is maintained throughout the life cycle of the program.  This includes the identification of the items to be maintained under configuration management as well as identifying who is responsible for them, including as well as who has authority to create and promote baselines.  This artifact will typically identify a specific policies, schedules or triggers that weill result in configuration management activities being performed as well as any dependencies between items.

Description
Main Description

This artifact consists of procedures that define how to control:

  • The evolution of the system from the "base set" and manage changes
  • Architectural integrity and quality

A configuration item is the smallest component of a work product that is versioned and managed separately. Usually this is the whole work product; that is, the work product is usually versioned and managed as a whole.

Brief Outline

This artifact typically contains:

  • Project identification information
  • Configuration management (CM) organization, responsibilities, standards, procedures and directives
  • CM activities, such as identifying, naming and acquiring configuration items
  • Configuration control, such as accepting, evaluating, approving, disapproving and implementing changes
  • Configuration status accounting
  • Configuration audits and reviews
  • Interface control
  • Acquired software control
  • CM schedules
  • CM resources
  • CM plan maintenance

Key Considerations

The following advice is recommended:

  • Establish the Configuration Management Plan and Procedures as early as possible in the development planning process.
  • Ensure that everyone on the development team (and any contractors supporting the development effort) is trained in the procedures.
  • Follow the procedures strictly.
Tailoring
Impact of not having

By not having this artifact, you risk not having a shared understanding of the configuration management and of causing problems with implementation or maintenance throughout the life cycle of the project. These may be problems with wrong versions or inability to identify or recreate specific versions, inadequate safeguards on access control, poor management of change requests, or with the wrong appreciation of impact changes.

Reasons for not needing

If  the configuration management procedures are very obvious and there is a shared understanding of them within the project staff, this work product can be omitted.  This typically only applies to very small projects where the staff is co-located.
