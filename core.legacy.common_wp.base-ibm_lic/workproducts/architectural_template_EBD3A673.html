Artifact: Architectural Template (ARC 107)
An Architectural Template describes any common design approach that many developers need to know about, that is, one that cannot be localized to a single component or module.
Domains: Functional
Work Product Kinds: Model
Purpose

The purpose of the Architectural Template work product is to:

  • Ensure that architectural issues, such as transactions and persistence, are dealt with consistently across the project.
  • Promote consistency across the project by ensuring that common problems have a single solution that everyone can use.
  • Communicate chosen Architectural Templates to all developers.
  • Provide developers with all relevant information concerning the use of these Architectural Templates.
  • Increase the understanding of the intended system by reducing the complexity of its interactions to a set of fundamental ones.
  • Provide input to the developers of any frameworks that underpin the Architectural Templates.
  • Avoid repetitive documentation of generic structures and interactions.

Description
Main Description

The Architectural Template work product documents the essential collaborations and behaviors that underlie solutions to the typical problems (i.e., persistence, transactions, event notification, etc.) inherent in most projects of any size and complexity. For each of these areas, a solution must be constructed or reused. These collaborations are representative of those occurring in the system and can be used to structure all or part of the system.

An Architectural Template in a structured design consists of program modules or skeleton modules, interaction diagrams and explanatory text. An Architectural Template in an object-oriented design consists of abstract use cases, interaction diagrams, and class diagrams and may represent collaborations between components or collaborations between objects.

Often an overview of the system, in the form of a layered diagram, can be derived from the classes that participate in these typical collaborations.

Two important types of Architectural Template are transaction strategies and persistence strategies.

Key Considerations

Possible Organizing Principles

For many business-dependent Architectural Templates, inspiration can be found among the following organizing principles (i.e., the basis of the definition of the stereotypical interaction):

a) Layering (i.e., the emphasis is put on the differing levels of abstraction that exist in the system, lower layers reflecting less abstract layers than higher ones. Any layer only provides services to its immediate higher layer and only uses the services of its immediately subordinate layer). The following layers and responsibilities are common in business systems:
  • Presentation Layer
    • Displays data formatted for ease of use
    • Provides user interface “look and feel”
    • Manages all user interface issues, so the rest of the system isn't affected by user interface changes
  • Application Layer
    • Provides business objects and application logic to the user interface
    • Manages data so changes in communication or host services don’t break the user interface
  • Data Layer
    • Provides long-term central storage of business data
    • Shares data with legacy applications and batch process

b) Service based
c) Domain centric (e.g., VisualBanker)
d) Process centric (e.g., Use Case based)
e) Workflow centric (e.g., P(rocedure)-T(ask)-A(ction), Sextet)

General Advice

The development of Architectural Templates should take account of the following:

  • Existing patterns (e.g., consult the literature about design patterns before inventing one).  Compare the Architectural Template with other projects and look for references, especially pattern libraries and relevant OMG services (e.g., CORBA Transaction Service).
  • Important change cases for the system and their impact on the list of candidate Architectural Templates

As the system evolves, it is very likely that either new or additional stereotypical behaviors will emerge. These should then be fed back into new or existing Architectural Templates.

The chosen Architectural Templates will have an influence on the software components and their structure. An Architectural Template may apply to both the business dependent (application) and business independent (technical or infrastructural) parts of the software.

If the integration with a legacy system is required, verify theat your Architectural template allows for integration with this legacy system.

In order to assess an Architectural Template, two kinds of prototypes are extremely helpful:

  • One that measures performance. For this kind, set up a high number of transactions, execute them, and review whether transaction rate requirements are met. Also evaluate the results against the requirements of the Operational Model. Adjustments may have to be made there.
  • One that measures the ease of use for the application developer. For the second, implement a use case with a nontrivial behavior (e.g., with nested transactions). Make sure that the use of the Architectural Template is easy to understand for the application developer and no conflicts occur at execution time in a multi-user environment.
Tailoring
Impact of not havingWithout this work product:
  • Architectural Templates are implicit and undocumented.
  • Developers fail to understand and implement the intended architectural patterns.
  • Increased design complexity and defects result due to lack of architectural consistency.
  • Distributed systems with multiple users would end up inconsistent without a transaction strategy.
Reasons for not needingThis work product may not be needed when:
  • The project is very small
  • A standard approach to an issue has already been developed
  • The development tool or environment has a built-in approach (e.g., persistent Smalltalk defines and implements a persistence strategy)
  • The project does not deal with some issues (e.g., no updated persistent data)
  • An Architectural Template is derived without adaptations from a large asset, in which case the accompanying documentation should be sufficient. In most cases, though, some customization will have to be made to the frameworks, and these should be documented in an Architectural Template.
Representation Options

Architectural Templates are usually documented using one or more of the following:

  • Informal Picture: A system can be depicted in an informal picture showing different abstraction layers. Such a picture shows an overall partitioning of the system based on the components or classes that participate in the collaboration (e.g., based on the MVC framework, we can define three layers: model, view, and controller layer).
    Care should be taken, however, in interpreting these pictures. Adjoining layers do not necessarily exchange information. The interactions between classes belonging to different layers can only be precisely understood from accompanying interaction diagrams.
  •  Free Format Text: A detailed textual description is very important to explain facts that cannot be depicted. This includes rules and policies that define how key aspects are handled. For example, in the transaction service, rules and policies include strategic decisions on subjects such as how to share objects in concurrent transactions (see below), optimistic vs. pessimistic locking, long and nested transactions, etc.
  •  Class Diagrams: Class diagrams may be drawn to show some of the most important classes and their relationships. Note that this description is by no means complete. It only details the behavior of the participating classes that are relevant for the collaborations at hand. The conventions for representing class diagrams can be found in the UML notation documentation, available via http://www.uml.org/
  •  Interaction Diagrams: Interaction diagrams can be used to document the stereotypical interactions between the main actors of the collaboration. For notation conventions, refer to the UML notation documentation, available via http://www.uml.org/
  •  Interface: This section describes the interface of the service for the application programmer (i.e., the API). 
  •  Source Code: Source code can illustrate how the Architectural Template is used in practice, i.e., sample fragments of calls to the API.
