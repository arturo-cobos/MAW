Artifact: Application Function Model (ARC 302)
The Application Function Model identifies and defines the major groups of business function that are required in order for the enterprise to meet its business objectives.
Domains: Enterprise
Work Product Kinds: Model
Purpose

The Application Function Model is used:

  • To ensure future applications are aligned with the business strategy.
  • As a basis for the planned integration of all the architectural elements, each element being designed and scoped within the context of previous and future projects.
  • To provide an architectural framework against which the function of the existing legacy systems can be mapped - this will highlight certain strengths and weaknesses in these applications and will govern the evolution of these systems.
  • To define and develop reusable "components" (Component-based development) and promotes the enterprise-wide sharing of processes - by describing the major groupings of business function in a non-redundant representation. The Application Function Model is therefore an important input to the definition of component models.
  • As a basis for the evaluation of packaged solutions.
  • As a basis for partitioning business function - this helps development teams deal with the complexity of the enterprise, whilst maintaining a single, comprehensive view of the enterprise.

Description
Main Description

The Application Function Model identifies the natural boundaries and partitioning of these business functions, and shows how these different partitions, or groups, are related to each other. As such, the model represents the optimal set of applications (or application groups) required to support the enterprise.

These groups will be independent of the current or future organizational structure and will owe their structural integrity to the affinity they have with other functions in the same group. In a large, complex enterprise, the model will be composed of several levels, or layers, each level representing a decomposition of the one above and showing increasing granularity. Each group represents a discrete set of business function, with each business function belonging to one and only one grouping.

The Application Function Model is the "Application" component of the Enterprise Technology Framework (a summary work product). Both these work products are typically part of an Enterprise Architecture deliverable.

In the terminology of the IBM Architecture Description Standard, each application is a ‘subsystem’, a set of related software ‘components’.

The set of applications (or application groups) in an Application Function Model are optimal in that they display the following characteristics:

  • The business functions contained within an application are closely related.
  • The scope and boundaries of each of the applications is clearly defined and do not overlap.
  • Specific functionality is supported by a single application.
  • The interface / interaction between the applications is well defined.
  • There is minimal data sharing between applications and maximum sharing within.
  • The applications represent "reasonable" pieces of work that could be tackled in one project.

Key Considerations

There is no commonly accepted standard for the terminology used in this work product.  For the purposes of this work product description, an Application is defined as:

"...a grouping of logical business activities that have a high affinity for each other in terms of the business information they manage and that address the processing requirements of a recognizable area of the business" 

  • Many customers may have alternative definitions that are in common usage in their organization.  In such cases, some judgment will have to be exercised and it may be prudent to use different terminology than that used here. This is not important and should not affect the development or benefits of the work product.

The Application Function Model shows the partitioning and structuring of the business function.  This is not to be confused with the generic partitioning and structuring of the operational functionality of the applications themselves, (e.g. the division of executable code and data into layers such as the "Presentation" layer, "Data Access" layer, "Business Object Server" layer etc. This is still part of the overall Application Architecture and is found in either "Reference Models" or included in the guidelines for the placement of data and functions.

Tailoring
Impact of not havingThe complete Enterprise Architecture represents a "blueprint" and set of standards that will govern the evolution of the current IT environment towards some desired, beneficial future state. The Application Function Model will govern the evolution of the business applications. Without this model, there is no enterprise-wide view of the optimal set of applications and the functionality that they contain, making it difficult (if not impossible), to adopt a "component based" development approach. In addition, with no single enterprise-wide view of the required functionality, there can be no effective control or management of functional duplication and redundancy.
Reasons for not needingSome projects may be limited in scope and may not include the structure and partitioning of business applications.
Representation Options

The Application Function Model will usually be represented by a combination of text, matrices and diagrams. Several diagrams may show the overall view of the applications, or groups, and their inter-relationships, as well as their decomposition into smaller units. Definitions of the applications will be provided textually. 

Different engagement objectives, scope and customer preferences may demand variations on the notation. A degree of flexibility is therefore required.
