Artifact: Requirements Matrix (BUS 011)
The Requirements Matrix is a document detailing the functionality required by the client in its software solution.
Domains: Selection
Work Product Kinds: Requirement
Purpose
The Requirements Matrix is a matrix that is used to capture client requirements for software selection and to evaluate the initial functional "fit" of a vendor's software solution to the business needs of the client. It allows the vendor to highlight their ability to satisfy the client's needs and provides an initial means to compare multiple vendors against each other. In addition, vendor response results can be used as elimination criteria in determining the vendor short list.
 
The Requirements Matrix also is used to identify initial functional gaps or special software enhancements needed to enable each vendor's software to fulfill the client's desired system capabilities.

Description
Main DescriptionThe Requirements Matrix document is generated from the results of the business process definition task and details the system requirements for the software being evaluated. This work product consists of business, general, and technical software functionality requirements.
Brief Outline

The Requirements Matrix should consist of the following sections displayed in the spreadsheet (either a single sheet with all requirements divided by functional area or separated out on multiple tabs):

  • Title Page/Section - A title page/section should be created to display the organization name, logo and project name as well as the date of release for the Requirements Matrix.
  • Table of Contents - A table of contents should be developed as a pointer to each worksheet within the matrix to assist ease of use for the vendor.
  • Description and Instructions Worksheet/Section - A brief description of the Requirements Matrix, its purpose and use should be included as well as instructions to the vendor on how to respond to the matrix.
  • General Requirements Worksheet/Section - A general requirements worksheet or section should be included in the Requirements Matrix to address system requirements for such areas as support, documentation, training, user interface, navigation, etc.
  • Business (or Functional) Requirements Worksheets/Section - The business requirements should be separated into worksheets or sections by business areas defined during the business process definition task.  Each worksheet/section should cover the business process requirements defined and validated for the business area represented by the worksheet.
  • Technical Requirements Worksheet/Section - The technical requirements worksheet or section should be included in the Requirements Matrix to identify technical features and functionality desired in the new software.  Technical requirements should cover such areas as technical infrastructure, application architecture, technical performance, system management and support.

Vendors respond using a scoring model to indicate their ability to satisfy the functional, technical, and general requirements with the current release of their software.  Vendors score their software using the following model:

2 - Now Standard: Supported in Current Version
1 - Planned as Standard: Definitely Supported in Next Release
0 - Not Supported or Planned: Requires Custom Modifications

A comments section is included to describe anticipated enhancements (release dates), customizations, etc.  Vendors are able to provide additional detail to each requirement to clarify their response using this column within each functional area.

Key Considerations

The Requirements Matrix should be distributed to vendors and returned to the project team in an electronic format to reduce error and workload.  A decision or scoring model should be determined based on the business process definition task and subsequent meetings with client management.

The Requirements Matrix should be confined to the scope of the project Statement of Work (SOW).

Tailoring
Impact of not havingThe Requirements Matrix is required to perform software selection. Without the Requirements Matrix, it is difficult to determine whether or not each vendor offers enough functionality to satisfy the client's goals and objectives for their software selection engagement.
 
By eliminating the Requirements Matrix, you eliminate valuable information used throughout the project and you increase the risk of selecting a software product that does not satisfy the client's business needs.
Reasons for not needingThis work product would be optional during a software validation project, where the client has already performed many of the selection tasks facilitated by the Requirements Matrix.
