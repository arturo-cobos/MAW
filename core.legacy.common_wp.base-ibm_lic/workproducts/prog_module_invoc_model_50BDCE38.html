Artifact: Program Module Invocation Model (APP 113)
The Program Module Invocation Model work product describes the calling hierarchy of the application by mapping the operations of a functional model onto modules that are invoked in program code.
Domains: Design
Work Product Kinds: Specification
Purpose
The purpose of the Program Module Invocation Model is to map the processes of a functional model onto an invocation hierarchy that can be efficiently implemented as called functions in a programming language. This mapping is necessary to implement the system in a structured programming language.
Description
Main Description

The Program Module Invocation Model describes the application call tree, the transient data passed between invoked modules, and, depending on the particular notation used, the timing dependencies between modules.   

The Program Module Invocation Model, when documented by structure charts, is horizontally partitioned. Each diagram contains a tree of modules and is connected to other diagrams by "off-page" connectors. There is a single root module, which branches out in a tree structure of invoked nodes until the leaf nodes are reached. The modules of the model are connected by bi-directional data, and are invoked roughly starting at the root and moving through the tree from left to right. Additional notation allows more precise calling sequences to be specified. While a functional model A functional model, when documented by data flow diagrams (DFDs), is top-down partitioned, with each diagram containing a network of processes. Each process in a diagram may be documented in greater detail by another DFD. Semantically, processes are peer to all others. The processes are connected to other diagram elements by unidirectional data flows. Notation

The Program Module Invocation Model is normally represented as a set of structure charts. Each chart is a tree with the root node at the top. Leaf nodes and off-page connectors to other charts occur near the bottom. The entire set of charts combines to a single root node, the entry point to the program.

  • The nodes are boxes representing the software modules. Modules closer to the root invoke modules closer to the leaves. Boxes with an extra vertical line at each side represent predefined modules.
  • Modules are connected by lines indicating invocations. Data and control passing across invocation interfaces are indicated by arrows whose base is an empty or solid circle, respectively.
  • The arrows are labeled with a name descriptive of their parameters. A module with connections below it to other modules indicates an invocation sequence with the lower left-hand module being invoked first.
  • A small diamond at the bottom edge of the invoking module with connectors emanating from it indicates that selection logic determines which of the lower modules are invoked.
  • A small circular arrow grouping one or more connectors indicates that that the associated group of lower modules is being invoked iteratively.
  • Each module in the Program Module Invocation Model is documented in the data dictionary as a Module Specification.
  • The data and control connections are also documented in the interface sections of the Module Specifications.

Key Considerations
This is a job best accomplished by a team of a few experienced developers.  Given that interfaces will be unstable at this point in the design, team-based work is best.  The design activities for this output will require multiple iteration.  The approach will be to work from the top level of the data flow diagrams down, starting with the main initiation processes of the application.  The design team should work in an environment where the structure chart can be pieced together as it is developed and easily rearranged.  A large conference room with clear wall space, on which the charts can be pieced together, works well for this type of task.
Tailoring
Impact of not having

Not having the Program Module Invocation Model requires that the design be performed by the programmers, most probably at implementation time. When pursuing this approach, experienced programmers/designers are necessary to ensure that a well-formed system is developed from the functional model. In addition, high-level module interfaces have to be established early to allow independent teams to develop in parallel. In any event, the risks to quality program structure and properly documented data interfaces remain high.

An alternative to a formal Program Module Invocation Model, but one that reduces the risks cited above, is to build an informal model out of index cards positioned on a large surface. The index cards can be arranged in the tree structure, module names and specifications can be documented, and transient data interfaces can be documented. The design teamâ€™s decisions can be easily displayed, weighed, modified, and disseminated to the development teams.

Reasons for not needingSmall projects with simple calling hierarchies do not need a formal structured design step. A project that is architecturally a graphical user interface (GUI) front-end to a database will frequently not need a formal structured design step, particularly if the GUI was built with a visual tool. An object-oriented system project does not need a Program Module Invocation Diagram because the call hierarchies inside a well-designed object are small. Inter-object invocations are designed with object interaction diagrams.
