Purpose
An API is defined so a system or component can be used without any need to understand all of its internal details. Precise documentation of APIs allows detailed design work to be done in parallel. Component interfaces allow you to separate the development of interdependent components. If component interfaces are not defined, no independent component development is possible because each developer needs to know too much about the internals of other components.

Description
Main Description

APIs are used to describe the interfaces of the full range of software components (modules), implemented in any programming language.

In object-oriented terms, an API is simply a set of visible classes and their interfaces. It also includes associated global types, data, and functions, but these should be kept to a minimum or eliminated entirely. The classes that appear in an API are a subset of those contained in the component to which the API is an interface. When a class does appear, it is possible that only a subset of its interface is part of the API. An API can, therefore, be considered to be a particular view or filter of a componentâ€™s classes.

In non object-oriented terms, an interface consists of a set of operations (functions and/or procedures and/or subprograms), each of which has parameters (with appropriate types) and a description.

Whole systems and individual components can each have an API.

A Component Model defines components, their dependencies and their interfaces. Often the interface is only defined at a high level, perhaps in terms of a more abstract contract. The API defines the interface in full detail, at a programming level.

Brief Outline

The structure of a document describing an API might take the following form:

Section 1: Purpose of API
The motivating use case should be used to guide the content of the introductory chapter.

Section 2: API Structure
The API structure chapter might contain an object model together with a textual walkthrough of the model.

Section 3: Class Interfaces
See the example for a suggested format for documenting class interfaces.

Section 4: Usage Scenarios
Each usage scenario should contain a description of the scenario, statements of assumptions and outcomes, and an OID and/or sample code.

Key Considerations
  • Define APIs in development increments as early as possible.
  • Consider using the facade design pattern to encapsulate the interface or to provide alternative, simplified interfaces.
  • Exploit the capabilities of your tool set to generate as much of the API documentation automatically as possible.

A minimal API document obviously lists just the type declarations (method prototypes) relevant to client programmers.  In addition to bare-bones class definitions and method prototypes, API documentation should include information and references to permit the API to be undertood independently of the system design. The degree of independence will depend, of course, on the intended audience for the documentation. It might be appropriate for API documentation to assume knowledge of the system, or it might not.

Tailoring
Impact of not havingIf API documentation does not exist but is needed, programmers must spend valuable time searching for the information they need in order to program. The application designers and programmers have to look at program source files for this information.
Reasons for not needingIf your application or system has no programming interfaces or components with programming interfaces, an API work product is not needed.
Separate API work products are not needed if the detailed information is held in the Component Model or some other work product.
Representation Options

If the API documentation is to be stand-alone and is complex, then you should provide an Object Model and perhaps a set of Object Interaction Diagrams (OIDs) to help the reader understand the semantics of the interface. Sample client code is often a good way of illustrating usage while hiding the internal details that OIDs would expose. If the audience for the API documentation might customize the interface, include customization OIDs and/or code.

APIs for non object-oriented systems should help the reader understand the semantics of the interface, either by a detailed description of the effect of each operation or, ideally, by specifying the pre-condition and post-condition for each operation. See, for example, the specification of system operations in The Fusion Method by D. Coleman.

Adjust the amount of additional documentation, such as usage scenarios or method descriptions, according to the intended audience and use of the API.
