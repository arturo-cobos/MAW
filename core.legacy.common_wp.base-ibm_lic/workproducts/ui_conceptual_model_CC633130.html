Artifact: User Interface Conceptual Model (APP 146)
The User Interface Conceptual Model is to ensure that the user interface matches the users’ mental model and, by presenting familiar concepts and relationships, is easy to learn, adapt to, and use with little or no training.

Description
Main Description

The User Interface Conceptual Model forms the foundation upon which the entire interface is built. It provides a visual and logical framework for the overall user interface design. This includes the high-level look (visual paradigms, aesthetics) and feel (interaction paradigm) and the metaphors (e.g., icons, images) used to represent real-world objects and their relationships. It can be presented in a number of different ways, including prototypes, diagrams, drawings, or pictures. Note that both the overall user interface design and the preliminary presentations of it should provide accessible alternatives for people who have disabilities.

The model contains visual images and text describing how users interact with them to perform tasks and an explanation of how the objects and metaphors were selected and in what way they relate to mental models or real world objects.

Brief Outline

A conceptual model can be depicted via prototypes (low- or high-fidelity) plus an accessible text document containing the following standard headings:

  • Metaphors
    • These include text descriptions and sketches of interface metaphors used to communicate the conceptual model to the user.
  •  Look
    • This includes samples of the visual treatment rendered using graphic design tools.  These would include alternate screen designs, colors, backgrounds, splash screens, and other visual elements.
  •   Feel
    • This includes text descriptions of interaction paradigms.  These are rather like task flows that describe how users interact with the visual elements to accomplish a task.  For example, the user views the notebook, finds the desired tab, clicks on it, views the page, selects an element on it, etc.

Make sure that accessibility is a consideration in each of these sections.

Key Considerations
The following guidance is recommended:
  • Review all prerequisite data (tasks, current/competitor solution evaluation) as part of any design activity.
  • Employ a multidisciplinary team to create the conceptual model.
  • Ensure that accessibility is considered as part of the conceptual model.
  • Allow users to suggest alternative metaphors and visual treatments.
  • Use more than three alternatives if test results dictate.
  • Use screens more than three levels deep if required for users to complete a simple task.
Tailoring
Impact of not having

If the User Interface Conceptual Model is not developed, the user interface engineers are forced to design the user interface with no explicit insight into the user’s mental model or overall conceptual framework for the look and feel of the user interface. Therefore, the user interface may not match the user’s mental model. Furthermore, the visual paradigms, aesthetics, interaction paradigms, and metaphors may be inadequate to produce pleasing and productive screen designs and flows.

The consequences of the above risks are that the solution will not meet users’ requirements. It will be more difficult to learn and use, thus increasing the overall system costs. Furthermore, it will be less consistent, creating errors as users navigate through transactions.

Reasons for not needingIn cases where an effort consists of making modification to an existing solution and the conceptual design of the existing solution is not to be changed, there would be no need to generate this work product “from scratch.” However, the modifications may result in some changes to the existing conceptual design (e.g., new objects) and the possible inclusion of new metaphors. Some applications that have a very simple user interfaces or use a technology that may not fully support the use metaphors (e.g., a “green screen”) may not require a conceptual model.
