Artifact: Content Plan (APP 310)
This artifact is a design document that describes the details about the content that will be required for an interactive application. Additionally, it can be used to describe the best ways to create new content and incorporate it into the application or its publication tools.
Domains: Design
Work Product Kinds: Plan
Purpose
Content Plans are intended to catalog all content for an application so that the scope of the content development can be more readily defined and to ensure no content is missing from the completed application.
Description
Main Description

Content is the information that an application imparts to its user. It exists in the form of text, images, audio, and video, and motion graphics. Content exists separately from functionality, but due to advanced interactive technologies, at times it can be difficult to distinguish the two; however, one differentiating factor is that content is not interactive. How a user interacts with the content must be described via Interactive Models or Interactive Concepts, in addition to the Content Plan.

Brief Outline

Content Plans are usually presented as a cross-reference between content types and the data that is to be collected about each piece of content. Some common data is:

  • The typical or average size of the each content type
  • The minimum, usual, and maximum number of instances for each content type
  • Relative position in the new application
  • Format in which it will appear
  • Source and intermediate formats
  • Content owners and developers
  • Content creation process
  • Frequency of update

Key Considerations
  • Content Plans can be created to represent existing or future content. They can be executed at different levels of detail depending on the phase of the project. 
  • Start with a Content Plan that only describes existing content and its value as part of the user experience. This benchmark will ensure the new application does not leave any content out. Documentation of the existing content is valuable for the interactive modeling process.
  • Content Plans frequently contain recommendations about the development of new content. These recommendations can suggest the kinds of content strategies that will successfully engage particular audiences. Recommendations can also be made about the process of developing the content itself, including:
    • Workflow
    • Editorial review
    • Legal review
    • Brand consistency review
    • Approval processes

  • Create a high-level sketch of the Content Plan early in the project to use as a tool to define the scope of content development work with the client. Point out dependencies on client resources for content, and begin to establish key hand-off deadlines. These deadlines are often on the critical path of the project. Much of the brand translation, creative development and content integration work is better accomplished with content and examples in hand.
  • Sometimes the jobs with the smallest content requirements are the worst, because no one thinks it is necessary to go through the effort of developing the content plan, so we often do not realize what is really needed until it is too late – despite what the client tells us.
Tailoring
Impact of not having

The primary danger of not having a Content Plan is that the scope of content development and publishing is so specific to individual projects it would be difficult to fully understand the magnitude of the work involved without the detail inherent in the plan. Without this understanding it is difficult to accurately set expectations regarding the resources and time required to create and integrate the content for the application. Furthermore, with advanced interactive technologies, content is usually tightly coupled with the functionality, and the functionality cannot be defined properly without an analysis of the related content.

The content plan provides a benchmark against which the application’s final content can be measured. Without this benchmark, it can be difficult to ascertain if all of the content has been included, running the risk that content inadvertently may be left out.

Reasons for not needing

The application uses very little or no content. 

The client will create and integrate all content and has taken the responsibility to set up the content development schedule. The client’s plan must be incorporated into the project plan, noting key content delivery dates and related dependencies. Detailed specifications of the requirements that the content makes on the user interface and the publishing tools must already be documented and all examples should have been provided for each content type in order to effectively complete user interface, functional, and technical design.
