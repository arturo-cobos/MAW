Artifact: Interactive Model (APP 160)
The Interactive Model is the blueprint for the user interface of an application.
Domains: User Experience
Work Product Kinds: Model
Purpose

Interactive Models serve to establish the scope of functionality and the framework of the user interface. It can be executed at varying levels of detail to provide a structure from which to develop the user interface for the application. 

  • During application design, this artifact is used to communicate the scope and general logic of the interface design in a format that is easy to understand by the whole team.
  • More detailed instances of this artifact precisely specify the functional requirements for every aspect of the applications functional behavior and logical flow.  This artifact establishes the scope of functionality and the user interface for each view of an application and provides graphic design and technical design teams with a specification of the functionality that must be supported in the view. 

High-level iterations of Interactive Models can be used to provide assumptions for estimating the cost of building the application.

Description
Main DescriptionThe Interactive Mode illustrates the overall approach to a users experience with the application as it relates to navigation, content categories, grouping of functional elements, the logic of information layout, and high-level conceptual models of the application. Like architectural blueprints for a building, these work products provide a organizational framework that influences the technical architecture, and supports the appropriate media to enrich the users experience.
Notation

The Interactive Model is expressed as diagrams.  These diagrams are usually accompanied by text descriptions of the functionality they represent.

To show the relationship between functions and content of an application, flow charts are often used, accompanied by an exposition of each node in the chart.  Individual functions or views of an application can be represented by wire-frame user interface designs that illustrate the types of interaction possible: buttons, drop down boxes, radio buttons, content areas, etc. 

The Interactive Model is organized into a hierarchical structure that allows for the organization and logical naming of each element.

Key Considerations

Interactive Models are more difficult to execute without adequate user identification and requirements:

  • Audience characteristics of the application’s users found in user profiles.
  • Catalog of required content and examples of typical instances.
  • Functional requirements as captured in use cases.
  • Technical limitations of the interface.
  • Input and output requirements for key processes.

For each of the above requirements that the client is unable to specify to the senior information designer’s or the creative director’s satisfaction, add 25% to the estimated hours to allow the Information Design team to formulate assumptions that will replace the actual information. When any of the above information is missing, it is less likely the Interactive Model will meet the client’s needs, and additional iterations with the client’s feedback may be necessary provide a satisfactory work product.

If this is reviewed with user groups and clients, the abstract nature of the work products makes it necessary to carefully explain their meaning to the audience or present them in conjunction with work products that are more easily understood.

Tailoring
Impact of not havingWithout this artifact, interface design ideas must be communicated among the team and with the sponsor in some other way. This communication and specification is important to ensure the technical design, information design, content strategy, and graphic design teams all have the same understanding of the scope of functionality, the navigational relationships, and functional details of the interface design elements. Without this common understanding, the output of the technical design and creative design runs the risk of being incompatible.
Reasons for not needingAn existing application or prototype may already provide adequate documentation to communicate the user interface requirements for the application.
