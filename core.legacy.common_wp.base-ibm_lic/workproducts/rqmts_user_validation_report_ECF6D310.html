Artifact: Requirements User Validation Report (APP 145)
This artifact validates the functional requirements on the basis of user satisfaction and frequency of use.
Domains: Requirements
Work Product Kinds: Measurement
Purpose

The purpose of this artifact is to validate the managerial view with the actual end user view.

Business process models, that identify potential functional requirements are typically developed from interviews with first line managers and above. In addition, requirements are also typically elaborated and refined based on interviews with this same population. This approach satisfies the need to gain an “end-to-end” view of the business process without interviewing an inordinate number of individuals. As a result, the initial list of requirements is generated based on the input of those who may not actually be the eventual end users of the system this artifact ensures that the requirements are able to:

  • Accurately capture the tasks that reflect the functional requirements, to ensure no tasks have been overlooked, and that all of the existing requirements represent tasks that are still performed.
  • Accurately represent user activities that contribute greatest to user satisfaction.

In addition, user interface designers use this information during design to help them focus on those components of the system that are most important to users. Satisfaction and frequency of use information is collected to help user interface designers identify and prioritize which aspects of the user interface design need particular attention during the later design efforts. When gathering this information, it is important to keep federal and state regulations on IT accessibility in focus.

Description
Main DescriptionThe metrics in Requirements User Validation work product are collected using focus group and survey techniques and the results are described in a table, where overall user satisfaction and frequency of use are combined to create a composite score representing importance.
Brief Outline

The Use Case Validation Report usually contains these sections: 

  • A description of the number and types of users, including users who have disabilities, who participate in the generation of the Use Case Validation Report.
  • A description of the procedure for collecting these data.
  • A summary of findings, presented in a tabular format, that provides ratings and rankings of the use cases in terms of frequency of use, satisfaction, and overall importance.

Key Considerations

Even if the functional requirements represent new system functionality previously unseen by users, they can still be involved in validating the requirements. Users can be asked how they would envision themselves or other end-users using the new functionality.

In rare cases, the fundamental nature would need to be changed to accommodate users with disabilities. For example, users with cognitive disabilities may not be able to respond to real time applications that require near immediate responses. Accessibility experts should be consulted in these cases.

Other measures may be added depending on the unique needs of the project. For example, these measures may include “difficulty in completing the required task” or “time consumed on the part of the user by the performing the task.” These two measures will aid user interface designers in focusing on those requirements that appear to need the most attention for improving usability. Sometimes a noncomposite measure of “importance” may be added if there are many tasks that are not performed frequently but are very significant to the business.

It is strongly recommended to automate the focus group session by using electronic groupware, especially when there are many use cases in the use case list. One useful tool for collecting requirement validation data is from GroupSystems Corporation.

GroupSystems iprovides a suite of team-based, decision support software tools that allow collection of data via a local area network or through an Intranet. It is efficient because all participants can enter data at the same time. It includes a set of debating and voting tools that help participants create better ideas and reach consensus more quickly. Collection of data is anonymous, so individuals are encouraged to comment without fear of reprisal. It also instantly documents and summarizes all input so analysis is more thorough, and distribution of results is quicker and easier.

Tailoring
Impact of not havingIf this artifact is not produced, the development team must rely on the managers to select valid and accurate requirements. These requirements must include those that are specifically geared toward people who have disabilities. If the management team errs in their selection, then development of the initial user interface design may meet the requirements of management but fails to meet the needs of the users tasked with using the system day to day. A project team may compensate somewhat for not having this work product by heavily involving actual end users in the user interface design and conducting usability testing with actual end users on the initial user interface design. However, the project team runs the risk of discovering significant usability issues during early usability testing, which could lead to expensive rework of the initial user interface design, the functional requirements, and the resulting analysis models which are based on the functional requirements.
Reasons for not needingThis artifact may be omitted without significant risk on small projects or those with simple user interfaces. It may also be omitted when the functional requirements have been generated from or with significant input from the actual user community, for example, where the managers are, or have been, end users of the system.
