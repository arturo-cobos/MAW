Purpose
The purpose of this work product is to provide detail information regarding the external data references that exchange information with the system. These details will become necessary as the system moves through the development life cycle.

Description
Main Description

This work product details the information flows identified in the system context definition. Each information flow has a source (origin) and a target (destination). With respect to the system context definition, the external entity point of an information flow will be referred to as an external data reference. For each External Data Reference, it is necessary to define the following:

  • Environment - statement of the platform, owning application and access method, whether or not the reference is the actual operational data store or an extracted copy / unload of the data store
  • Other Related Systems - identification of other systems that may be using this reference, this may help to understand where this reference is used and possibly manipulated
  • Structure Description - layout of the structure showing format and where that description is stored / maintained, metadata and/or data model reference
  • Structure Content - information regarding the domains of the fields / columns contained in the structure, appropriate business rules and where that information is stored / maintained (metadata repository, data model, etc.)
  • Volume Statistics - information on the size of the reference, growth rates, business rules for purge / delete and archive activity, etc.
  • Volatility - how often and the magnitude of updates applied to the reference
  • History / Archive - how much history is maintained and what metadata history is maintained (structural changes log, etc.)
Brief Outline

The External Data Reference is mostly textual but should include tabular description of the structure. Information elements required for each External Data Reference are as follows:

  • Name – use name as it is known by organization supporting the data reference
  • Platform – description of the platform and operating system
  • Access Method – what access method manages the data content
  • Location – physical location of the data
  • Business Owner – business organization
  • IS Owner – organization in IS that supports the data reference
  • Row Length – length of row / record
  • Size – number of rows / records
  • Description – description of what the data reference is and what it is used for
  • Related System Comments – comments relative to how or where this information is used in the enterprise – on some occasions, data analysis work may already be done for this reference
  • Volatility Comments – narrative describing how rows are added, deleted, archived, etc.

For each data reference, a Structure Definition and a Business Rule Definition must be included which describes each element in the data reference, where the source for the definition is located and any business rules established for the data reference.

  • The Structure Definition should include for each element:
    • Sequence Number – position of the element in the row.
    • Element Name – actual name as used in the system.
    • Format – data description as identified by the language or DBMS.
    • Length – length as defined by the language or DBMS.
    • Domain Values – description of the range of possible values that can occur for this element.
    • Comments – any additional information that might be useful.
  • The Business Rule Definition should include:
    • Business Rule Type – a classification that helps establish the context of the business rule.
    • Business Rule Name – a descriptive name that may be optionally applied to distinguish one rule from another.
    • Business Rule – the business rule itself.
    • Business Rule Comment – comments relative to the conclusion or implementation of the rule itself.

Key Considerations
  • Business and IS owners must participate in the workshops to develop the content.
  • There are tools available to assist in the preparation of the External Data Reference Definition content. Several ETL (extract/transform/load) tools come equipped with a dictionary/repository function that can be used to capture this information.
  • Efforts spent on ensuring the completeness and clarity in this area will result in a reduced timescale for development and will diminish the ‘interruption’ of non-technical project team members to attain explanation in the later stages of the project.
Tailoring
Impact of not having

The level of detail will not be enough to create the technical specifications that enable development of the system. If an attempt is made to technically specify or develop solutions without accurate detail, then there will be an overhead caused by the necessity to correct or adjust misunderstandings or assumptions. This will obviously lead to a delay and increase the development costs within the project.   

This work product impacts assessment, data mapping and data migration activities.

Reasons for not needingThe only situation where this work product is not needed is where the system does not exchange information with any external data references or the content has been developed by the client or another third party vendor and the results have been shared with the current project team.
