Artifact: Configuration Parameters (APP 306)
Configuration Parameters describes the selection and setting of values and options that customize a package for an installation.
Domains: Implementation
Work Product Kinds: Specification
Purpose

The purpose of this work product is as follows:

  • To establish and document a set of rules and standards for setting configuration parameters
  • To set the configuration parameters for the package
  • To document the parameter settings and the decisions associated with choosing the settings
  • To obtain agreement that the configuration parameter settings match the business processes and organization addressed by the package

Description
Main Description

The customization rules and standards for configuration parameters are key elements that should be well known, agreed, and documented so that the package can be can be correctly configured with the appropriate settings to support the intended business processes and organization. There may have to be compromises made in the business processes and organization to accommodate the limitations of the chosen package. The selected package may provide some assistance with configuration tools and techniques.

Building on the rules and standards and the package characteristics, the Configuration Parameters work product identifies, documents, and sets the parameters for a client-specific version of a package. A variety of means such as configuration scripts or files containing lists of parameters may be used to set the parameters that typically include:

  • Global settings
  • Country settings
  • Languages
  • Currencies (such as exchange rates, decimal places, and rounding rules)
  • Units of measurement
  • Calendar
  • Chart of accounts
  • Company codes
  • Organization structure and hierarchy elements

While it is essential to configure the package parameters themselves correctly to allow the intended business processes and organization to operate, it is vital to document the background decisions made in choosing the parameter settings. This documentation should include:

  • Who performed the configuration
  • Change management history
  • Who approved the changes and the configuration
  • Scenarios required to test the configuration
  • How the configuration meets the business process and organizational requirements

Key Considerations
  • Produce the Configuration Parameters work product as early as possible in the life of the project. Thereafter, elaborate the work product as the design progresses adding details, remembering to document the decisions made.
  • Package limitations may require the redesign of business process and organization details. Again, it is best to know the limitations of the package and the intended business processes and organization early in the project to avoid surprises, delays, cost overruns, and client dissatisfaction. Pay attention to changes as they are made. The limitations may become more severe than originally anticipated.
  • Ensure that the scope of the project is clearly understood with respect to the configuration.
  • Ensure that effective communications mechanisms are in place between the project team and the client so that problems can be addressed as early a possible.
  • Ensure that rigorous change management is executed subsequent to the client approval of the configuration parameters. This applies equally to the project team’s activities and to the client’s business process and organizational designs. Unexpected changes in either can have catastrophic results.
  • Ensure that the parameters are set in accordance with the project's overall design. Use a consistent format for documenting the parameter setting and the decisions taken in choosing them. Consider the following:
    • Who will be allowed to customize the different modules?
    • How and where will the customization be documented?
    • Where will the customization be done: on a sand box system, a development system, or somewhere else?
  • Document the configuration and background decisions clearly and preserve them in a known location so that future configuration activities may use them.
Tailoring
Impact of not having

Not producing the Configuration Parameters work product leads to the possibility of:

  • The package not performing as expected
  • The package not addressing the business processes and organizational requirements as expected
  • Reworking the parameter settings of the package causing extra cost and delays
  • Configuration management problems in the current and, possibly, future versions of the system

  • Client dissatisfaction
Reasons for not needing

The documentation of the rules and standards for customization may be part of the system itself. Otherwise, the rules and standards should be written down and agreed in the format most appropriate for the package and the project.

Representation Options
  • The package itself may provide integrated tools for on-line documentation and tracking of the configuration parameter settings.  Regardless of how the configuration parameters are set, the decisions made when choosing particular settings must be recorded for future reference.
  • The documentation of the configuration parameters may take any convenient form since it will vary considerably from package to package. However, the parameters and the decisions taken to select their settings should be documented together. Text, tables, or spreadsheets might be used here.
  • The setting of the parameters themselves depends on the system and may include configuration scripts, placing files in the appropriate directories, or using the systems on-line configuration management tools.
  • Organization hierarchies are usually shown graphically.

Scenarios for guiding testing should be documented during the comparison between the package constructs and the client's business processes and organizations. These will usually take the form of descriptions or use cases.
