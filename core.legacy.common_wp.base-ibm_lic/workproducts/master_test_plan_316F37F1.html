Purpose

The purpose of the Master Test Plan work product is to:

  • Address testing from a high-level system viewpoint.
  • Provide a big picture view, or a view of the forest, whereas the detailed test plans provide views of the trees in the forest.
  • Provide an early definition of the test environment requirements.
  • Ensure that the test plans are built systematically and completely and there is no more and no less overlap between detailed test plans than is necessary.

A master test plan provides planning information, ahead of time, to alert the organizations to get ready for testing by allocating the right set of resources at the right time. By providing high level requirements for hardware, software, and people resources as early in the lifecycle as possible, there is more likelihood that the resources will be planned, prepared, acquired, and trained in a timely manner.

Description
Main Description

The Master Test Plan documents the "what" (scope and objectives), the "why" (purpose), the "when" (key milestones), and the "who" (organizational roles and responsibilities) for all levels of dynamic testing. Full lifecycle testing is an iterative process. The development of the master test plan is started early in the development lifecycle, and is refined, as more information becomes available. It ties together all the detailed test plans and points to them. Each testing level represents a known level of physical integration and quality of the system solution under development.

The master test plan evolves from the test strategy and is driven by the quality goals contained in the quality plan. It uses the concepts or organization level standards, and applies them to the project to identify specific test focus areas, types and levels of testing as they apply to the project. As the master test plan is developed it is possible that the levels of testing, as prescribed by the standards, may not be sufficient to meet the test objectives or the quality goals set forth by the quality plan. In this situation, the base set of test levels may be modified or additional levels of testing defined to accommodate the project needs. For example, Performance Testing, which is a type of testing as per IBM’s standard testing methodology, could also occur over a number of different levels of integration (i.e., at unit, integration, systems integration, operability, etc). It is possible that there could be a separate detailed (and/or master) test plan for Performance Testing and that Performance Testing could occur, as a stand-alone test, at different levels of integration. It is also true for Usability testing that spans the full lifecycle and runs throughout the development phases. Thus the master test plan needs to be scalable and tailorable, and must be flexible enough to be used for the small project or the very large projects.

For smaller size projects the master test plan may combine the detailed plans for different levels of testing into one plan.

Some of the various detailed test plans that may be referenced are:

  • Unit Test Plan
  • Integration Test Plan
  • System Test Plan
  • Systems Integration Test Plan
  • Acceptance Test Plan
  • Operability Test Plan
Brief Outline

The table of contents of a master test plan includes the following sections, but does not include detailed testing tasks or tracking of them or test schedules:

  • Test objectives: describe the tangible goals of testing for this engagement based on solution objectives.
  • Risk assessment: detailed risk assessment driven by the business functions and technical requirements of the solution under development.
  • Test Focus Areas: the set that is applicable to the project and driven by the quality goals.
  • Levels of test: test levels defined for this project based on organization standards.
  • Types of test: relate to business functions (Functional Types) or to structural functions (Structural Types).
  • Organizational responsibility: defined so that boundaries for hand-off or hand-over are clear.
  • Entry/Exit Criteria: clearly defined for each level of testing selected for the project.
  • Test Data Strategy: approach for test data creation and setup.
  • Test Environment set up: includes test environment requirements at a high level for all dynamic levels of testing.
  • Tools: specifications, selection, evaluation, and implementation planning steps.
  • Testing Milestones: lists major testing schedule targets.
  • Test Management Procedures: describe how the testing activities and plan will be managed.
  • Test Reporting Procedures: describe how and when testing progress will be reported.

Key Considerations

The Master Test Plan is gaining acceptance as a standard deliverable in most major projects.  In fact it is perceived to be a major catalyst and facilitator to achieving the test objectives in a cost-effective manner.  It is therefore very important that this is verified and validated before embarking on any construction activities, such as coding and unit testing.

The quality plan for development and testing would have defined the quality targets for static and dynamic testing.  Using these quality goals, the master test plan is developed that balances the level of quality against the cost of quality, so as to define cost effective testing that reduces the project and testing risks. 

If the engagement involves a number of projects that correspond to portfolios of applications, it may be prudent to create a template for the master test plan (similar to the one done for test strategy).  After review by the client and refinement, it can be used as a standard template for creating the master test plans for subsequent projects. 

In situations where the project is very large, and multiple releases and multiple implementations may be running concurrently, there are a couple of suggested approaches.  Please refer to the test strategy work product for examples.

Tailoring
Impact of not having

Without a master test plan, the various stakeholders may be working independently and coming up with test plans with conflicting priorities. The master test plan brings the stakeholders and their representatives to work together, collectively and collaboratively as a team, to agree and work toward a common set of goals. Absence of a master test plan can lead to duplicate effort and result in missed plan components. Without an overall and common understanding of testing, it may be difficult to comprehend the complexities that testing entails, and the effort may be underestimated.

If this work product is not produced, then the detailed testing plans have to be stand alone, and each detailed plan will have to go through the test focus areas and identify the types of tests needed for that level of testing.

Reasons for not needingIf the project is small, self-contained, low risk, and all team members are co-located, then instead of a separate master test plan, there may only be one test plan that combines the master test plan and all the detailed test plans into one. The test plan should be documented whether or not separate master test plan and detailed test plans are created.
