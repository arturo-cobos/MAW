Artifact: User Interface Guidelines (APP 147)
The User Interface Guidelines work product provides a consistent set of standards to assist developers in constructing work products related to user interfaces.
Domains: User Experience
Work Product Kinds: Standard
Purpose

The primary purpose of the User Interface Guidelines is to ensure user interface design consistency across solutions and projects. Users interacting with a similar “look and feel” within and across solutions and systems provide consistency and give the user the advantage of not having to learn a new way of interacting with the system. The User Interface Design Guidelines are also used:

  • Early in design as a reference for guiding broad high-level design decisions pertaining to the use of metaphors for capturing the user’s mental model of tasks and processes in the user interface conceptual model.
  • Later in design as a reference for guiding detailed low-level design decisions pertaining to the selection and use of windows, menus, and other widgets and user interface controls (e.g., entry fields, window titles, group boxes, and push buttons).
  • During the development of user interface code by programmers as a reference for guiding the development of user interface design intricacies that are sometimes not captured in a user interface design specification. These design specifics may include the selection of mnemonics for push buttons and menu items and the identification of accelerator keys for lists, buttons, and menus.
  • After the code has been written and testing is underway by systems assurance to perform an objective quality review aimed at assessing the consistency of the user interface.

Description
Main Description

The User Interface Guidelines work product usually comes in the form of a (set of) document(s), with separate entries for:

  • Company-wide user interface design guidelines
  • System-specific user interface design guidelines
  • Project-specific user interface design guidelines

It typically contains a reference to (not a copy of) certain widely accepted industry standards (such as the Microsoft Windows User Interface Guidelines or the IBM Common User Access guidelines).  The industry standard reference may also be supplemented with a set of external regulations and standards, such as ISO (International Standards Organization) concerning display and keyboard standards for character width and height, display brightness and contrast, and keyboard key placement.  In addition, the User Interface Design Guidelines work product explicitly documents mandatory, recommended, discouraged, and bad practices based on the most frequent types of errors that are made against this standard and the specific types of systems that are being built across the organization.

Within the same company, different classes of applications may require a further refinement of the User Interface Design Guidelines.  For example, retail kiosk applications in a bank will be used by actual customers and will therefore have higher usability and accessibility requirements than traditional transaction-oriented applications used by the retail bank’s employees.  The latter can be trained; the former cannot.  Simply because the retail kiosk is a different style of interface, there may be some different or additional guidelines.  Thus, User Interface Design Guidelines may be applied across projects or across an application suite as a means to ensure enterprise-wide user interface design consistency.
Brief Outline

Typical contents include the following:

Industry Standard Adherence
This section specifies the industry standard that will be adhered to by default.  It should include details about where to find and how to access this information (books, CD-ROMs, Web addresses, and so on).

General User Interface Principles
This section contains some general user interface principles that are applicable for any type of user interface on any platform.  Principles of accessibility for alternative access to information may be considered in this section.

Design Methodology
This section describes a user-centered design process.  It includes discussion of requirements gathering, designing through low- and high-fidelity prototyping, and iterative refinement based on usability evaluation.  It will discuss, at a high level, techniques for usability, accessibility, and other evaluations including cognitive walk-throughs, heuristic review, and other engineering approaches to usability evaluation.  It may also discuss laboratory-based usability testing, accessibility testing, and measurements, such as ability to complete task, time on task, and measures of error.

Component Guidelines
This section discusses the tailoring of the industry standard style guide for support of the solution.  It will also describe specific guidelines for new components developed for the solution.  For each component, two categories of compliance are often identified.

The first, typically called Required, denotes guidelines that absolutely must be observed to comply with corporate standards. These may also include government regulations and the client’s policies. The second category, typically called Recommended, contains guidelines that should be adhered to if possible, but do not violate the standards if ignored.

Key Considerations

Illustrate your guidelines with examples showing compliance with a particular guideline. Show examples of noncompliance if the guideline is easy to violate or difficult to communicate easily. People sometimes learn more from bad examples than from good examples.

Be judicious in deviating from common, accepted industry standards, especially in the first release of an application. If you choose to deviate, provide a paper trail documenting the reasons for the deviation. Ideally, the reason for the deviation is based on specialized usability or data element requirements, and its tailoring is justified on the basis of usability test findings.

Appoint a style guide “czar” who is responsible for developing the User Interface Guidelines, reviewing user interface designs before they are completely implemented, and ensuring compliance with the guidelines. This person is sometimes not all that popular, so he or she should have the fortitude to ensure that the guidelines are adhered to, even in the face of strong opposition.

Make the User Interface Guidelines usable for the developer. Do not put a 300-page document on the developer’s desk that he/she is expected to read entirely before designing the first screen. Consider placing the style guide on the intranet, providing hyperlinks to allow developers to download style guide demos if they exist. If you use the document format, make sure to provide an index at the end of the document.

Depending on how much tailoring is required, a solution-specific set of guidelines may be anywhere from 20 to 200 pages in length.

Check that all guidelines are either very explicit or have references to documents or standards.

Involve an accessibility expert and users who have disabilities in the review process.

Involve the user through user reviews, discussions of examples, and/or prototyping of standard layouts and interactions.

Involve the developers by means of developer reviews. You should aim to create a feeling among the developers that the User Interface Guidelines are their User Interface Guidelines. If developers are not involved and/or disagree, you will encounter problems enforcing the use of the guidelines.

Call for a review by an expert from outside the project team. Getting the view of someone outside the team gives you some other approaches and perspectives. This should be someone who has expert knowledge of the industry standard and who has a background in usability engineering.

Verify external regulations and environmental style guides such as ANSI (American National Standards Institute) or ISO. Are there any legally imposed threshold values on the size of a character on a screen, the brightness of a display, the use of certain terminology?

Ensure that the guidelines are technically achievable in the target environment. If a certain guideline currently cannot be implemented in a straightforward fashion, if possible, describe a workaround to the guideline. If not, document the fact that the guideline does not apply to this release.

This activity is often “meeting-intensive” especially if a task force is set up to adopt “company-wide” standards. Try to handle as much task force communication via e-mail as possible, and try to limit the occurrence of time-consuming, potentially unproductive, in-person meetings. Be very selective in staffing this effort if the client insists upon creating the task force.
Tailoring
Impact of not having

If this work product is not developed:

  • The user interface is likely to be inconsistent in look, feel, and conceptual model, and end users may make more errors or take longer to perform tasks because of these inconsistencies. Users will also require a longer training period to become proficient using the system because inconsistencies in the user interface require users to learn more procedures for performing tasks.
  • Electronic performance support aids such as online tutorials, coaches, and wizards may need to be developed to compensate for the inconsistent design and to help users learn how to master the inconsistencies of the system.
  • Software engineers will develop conflicting designs for similar functions, and time and resources will be needed to resolve these inconsistent design approaches. These inconsistent implementations frequently cause accessibility problems, requiring further resources and time to resolve.
  • Rather than reuse approved design and code, developers will create their own idiosyncratic designs and code, resulting in the need for more testing (since more test cases have to be developed) and potentially greater defects in the code.

The solution team can compensate for the absence of this work product by scheduling more up front training for users and developing electronic performance support systems to augment the user interface. Without a user interface design guide, the development team can compensate by building more time and resources into the schedule to resolve inconsistent user interface designs and respond to defect reports associated with the inconsistencies in the user interface.

Reasons for not needing

There are situations when this work product may not need to be created.

  • This work product is not necessary in cases where the system does not have a user interface.
  • An existing style guide may already exist and is applicable without the need for tailoring.
Representation Options

Other ways of documenting the User Interface Design Guidelines could include:

  • A demo system that presents the guidelines in an interactive way
  • A set of Intranet Web pages that provide hyperlinks between pages
