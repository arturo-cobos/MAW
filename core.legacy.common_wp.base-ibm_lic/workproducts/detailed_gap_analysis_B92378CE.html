Artifact: Detailed Gap Analysis (APP 303)
Detailed Gap Analysis is a series of documents that describe the analysis of a package in relation to the future business processes.
Domains: Design
Work Product Kinds: Assessment
Purpose

The purpose of this work product is to document:

  • A detailed comparison between the package and the future business processes
  • Strategies for resolving the gaps between them.

Description
Main Description

The Detailed Gap Analysis work product includes:

  • Analyzing the package and business processes
  • Options for resolving the gaps between package and processes including estimates of effort to implement them
  • Implications of implementing the options
  • Selection of strategies based on the options
  • Documentation of the analysis and strategies

Changes may be considered for either the package or the business processes. The choice of which changes is a function of the cost and schedule impacts to the project.

Brief Outline

The Detailed Gap Analysis is structured to mirror either the business process or the module structure of the package. In either case, the other needs to be cross-referenced.  This helps to identify the level within the structure (process or module decomposition) that is appropriate.

The following information should be present in the analysis:

  • Name: The business process or package module being analyzed
  • Identifier: A unique number or title that is descriptive of the gap
  • Summary Results: A scale or narrative conclusion as to the suitability of the package to the intended purpose. From reading this section, it should be clear whether or not significant work is needed to customize the package or process.
  • Detailed Results: A series of points, questions, or capabilities that were tested against the package and the results of these tests. The results should be detailed enough to be self-explanatory. The results should clearly articulate any issues that will result from the implementation of the package such as the need for new skills, a change in the process flow, or a change to the package.
  • Gap Analysis: Each gap is fully described and the shortcomings noted.
  • Gap Resolution Strategy: The options for resolving the gap are described. They need to be of sufficient detail to be used by other project team members who are responsible for resolving the gap. The implications of using each option are documented. This includes an estimate of the implementation effort (time and resources). From the options the strategy for resolving the gap is selected and documented.
  • Security Implications: The resolution of the gap may require new or modified security arrangements within the application. These should be documented so that they can be addressed during the implementation phases of the project.

Each individual document should be limited to 1-2 pages.

Key Considerations

The discussion of options and the selection of gap resolution strategies can be very contentious if not carefully managed. It is critical those options are explored fully and that resolution strategies are supported by the underlying data, i.e., the analysis of the processes and a strong knowledge of the package. Avoid the temptation to put the old processes into the new package by dismissing the package capabilities too quickly and relying on custom programming to fill the gap.

The consideration of options has to be done with the project time schedule and resources in mind. The selection of resolution strategies impacts both schedule and resources, and has to be balanced with the function that is required to meet the business requirement. However, keep the following items in mind:

  • Packages are typically bought to use the functions they contain. Modifying the source code costs time and resource during the project’s life cycle, it can also create a maintenance problem in the future.
  • Assuming that source code has been tested prior to delivery, modifying it creates the need for regression testing.
  • More skill is required on the project team to modify the software than to configure it.

If there is doubt that the package can be used without modification, consider the following:

  • Ensure that the modification of the package is possible and that the proposed changes are practical within the schedule and costs of the project.
  • Long-term maintenance costs can be minimized if the appropriate development standards are used for the modifications.
  • Modifying code may be a better option than incurring a major negative impact on the productivity of a business process.
  • As modification recommendations get more extensive, the original decision to use the package for the particular business function should be revisited. If this is the case, revisit the decision at the earliest opportunity.

Security considerations should be included in this analysis.

Document and consolidate issues and opportunities as they occur during the project’s phases. Of course, documentation and consolidation is useless without some action being taken either in the project team or elsewhere.

Since they will affect the future business process and application operations, the client should approve the actions proposed by this analysis including the security changes proposed.

Other points to consider:

  • Are the resolution strategies documented sufficiently for their implementation and testing?
  • Has a cost benefit analysis been performed on the options available for resolution strategy selection? Has it been documented?
  • Have policies and practices been defined for business process modifications?
Tailoring
Impact of not having

Not performing this analysis presents several risks:

  • The future process may not be adjusted to accommodate the limitations of the package
  • The gaps which are to be addressed by modifying the package may not be understood in the context of the overall business process
  • The gaps between the business process and the package may not be identified till late in the project causing cost and schedule exposures
  • The amount of project work required to address process and package modification may not be understood or planned
Reasons for not needingProcess designs supplied by the package meet the organization's needs without modification.
