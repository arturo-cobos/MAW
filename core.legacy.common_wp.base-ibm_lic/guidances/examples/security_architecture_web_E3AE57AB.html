Main Description

A General Web Security Functional Architecture

This architecture is built on a structure that employs end-to-end security between the browser and resource manager (Web server, Data base manager (DB2, Sybase), Transaction monitor (CICS, IMS, ENCINA), DCE application server) in addition to transport level security (see Figure below). The end-to-end security ensures that only authorized requests are honored and that the data that flows between the browser and the resource manager is private and has integrity. In addition, the two secure transports (one between the Browser and the Web Server, and the other from the Web Server to the Resource Manager) are used to ensure that only valid requests are allowed to the adjacent tier. This architecture eliminates the potential security gap that can exist if the Web Server is compromised.

The end-to-end security is specifically accomplished by having the browser’s request digitally signed and enciphered by the user. This request is then deciphered and signature validated by the resource manager. In addition, the response from the resource manager is signed and enciphered so that only the user at the browser is permitted to obtain the results.

The diagram below overviews the architecture. Its description is as follows:
1. Browser establishes a secure transport with the middle tier server. This accomplishes following security services:

- Mutual identification and authentication between the two parties.
- Data Confidentiality and Integrity.

2. Browser sends a request to the middle tier server.
3. Middle tier server makes an access control decision based on the identity of the user that was used in step 1. It also can optionally log the users activity. This accomplishes the following security services:

- Access Control
- Logging for Audit

4. If the request is not permitted, it is rejected.
5. If permitted, the middle tier establishes a secure transport with the resource manager handler. This accomplishes following security services:

- Mutual identification and authentication between the two parties.
- Data Confidentiality and Integrity.

6. The browser request is relayed to the request handler.
7. The request handler translates the Internet request into an enterprise request and forwards it to the resource manager.
8. The request is processed and the response is returned to the request handler.
9. The request handler translates the enterprise response into an Internet response.
10. The response is forwarded to the middle tier web server.
11. The response is returned to the browser.
12. End-to-end security is accomplished between the user and the request handler by digital signing and enciphering the request and requests and responses. This accomplishes following security services:

- Mutual identification and authentication between the two parties.
- Data Confidentiality and Integrity.


This architecture provides authentication, data confidentiality and integrity, access control and audit functions for the applications regardless of whether the application is high risk or low risk.

Security Architecture for the Resource Manager / Request Handler for Component shown above

When the resource manager is a DCE enabled application server (SYBASE, DB2, CICS, IMS, Encina), the user’s signed and enciphered request is relayed by the middle tier web server to a resource manager request handler that is a JPM customized component. In this case the request handler

1. Accepts enciphered requests from the middle tier server.
2. Initiates a separate process / thread to handle the request.
3. Verifies the signature and deciphers the request.
4. Ensures that the user’s privilege to sign request has not been revoked.
5. Establishes an enterprise identity that is to be used to process the request (based on the Internet credential, certificate).
6. Establishes a security context for the request that is based on the enterprise identity.
7. Performs any necessary business logic including access to enterprise resource managers and the translation of requests and responses to/from Internet and enterprise formats.
8. Signs / enciphers the response.
9. Returns the response to the middle tier web server.
