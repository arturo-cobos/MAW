Checklist: Defect Analysis Document Validation and Verification
Relationships
Related Elements
  • Defect Analysis Document
Check Items
Accurate project details are captured

So that the failed test, test type, level and sequence is easily identifiable, as are all the impacted/associated system, test or project components.

The defect information is easy to understand
The information is comprehensive enough to allow analysis and resolution of the defect.
The severity, and the priority assigned are consistent with the guidelines in the master test plan
The unique identifier for this document matches that captured in the Defect Log

This is for the purposes of test management and the monitoring of the progress and status of resolving this defect.

The defect is described succinctly

Whatever approach or technique used to analyze the defect should be described succinctly but fully enough in order to clarify how the recommended resolution is derived.

The Defect Analysis Document has been completed as soon as the defect is found and/or the analysis is conducted
The salient points are documented before they are forgotten and for speedy expedition to the next process/team to resolve. All information should be consistent with the previous input and deviations should be explained and justified.
Defect has not been closed without some action being taken to resolve it

No Defect Analysis Document may be considered as closed unless the action taken to resolve the defect is documented and the appropriate stakeholder has signed it off as accepted  This is true even if the action is “None – within quality parameters”.
