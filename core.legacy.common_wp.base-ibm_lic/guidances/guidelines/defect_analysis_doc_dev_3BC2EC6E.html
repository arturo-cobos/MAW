Guideline: Developing the Defect Analysis Document

Main Description

The creation of the Defect Analysis Document may be during any of the test management, static testing or dynamic testing activities. It may be developed before, during or after the creation of the Defect Log, but should always be synchronized with the Defect Log. The steps to creating the Defect Analysis Document are outlined briefly. Use the template as a guide:

  1. Create one Defect Analysis Document for each defect or variance found during test management or static or dynamic testing. If the defect is negligible (use caution here in assessing this) and no resolution is required except to log it, then this work product may be omitted. 
  2. Document the project and defect information as accurately as possible 
  3. Assign a unique defect tracking reference number or code following the sequence in the Defect Log and the naming standards for the organization. 
  4. Classify the defect: Using the severity rating process from the master test plan, assign a severity level to the defect. Review with the testing stakeholders for agreement and prioritization, and invoke the defect management process for that severity level as defined in the master test plan. 
  5. Assign the defect for analysis. 
  6. Confirm that the defect is not caused by testing errors or omissions. If the testing process is the problem, return for retest, and document the defect in the testing process for correction and as input to improving the testing process. 
  7. Confirm that the defect is not a new or changed requirement. These may require a change request. Refer these to project and/or test management. 
  8. Analyze the defect to determine its cause. 
  9. Recommend remedial action, identify skills and resources required and estimate work effort. 
  10. Approve or reject recommendations. Rejects will be re-analyzed or withdrawn as defects. 
  11. Resolve defect and document activity (it may be more effective to batch these up for resolution)
  12. Submit defect resolution for re-test (it may be more effective to batch these up for re-test) 
  13. After successful retest, sign-off defect. 
  14. Unsuccessful re-tests should return defect to first step in defect management process.
