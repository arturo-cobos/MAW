Guideline: Developing the Database Transaction Descriptions
Migrated development approach from existing IBM Global Services Method work product.

Main Description
Use the following development approach:
  • Review the business requirements and process models to determine what business events affect the data.
  • Review the logical data model to determine which entities and attributes are accessed and which associations are affected as a result of the business events.
  • Group individual accesses of a business event into transactions.
    • Assign a name to and describe each transaction. The description should be no longer than a few sentences. The goal of this activity is to describe what the transaction accomplishes not how it is accomplished. The description should be understandable by end users (i.e., should not include technical jargon). The transaction names and descriptions provide end users and MIS staff with a means to distinguish one transaction from another. 
    • Document the average and peak frequency of each transaction (i.e., transaction volume statistics). The frequency information should include the rate and period involved (e.g., 100 times per hour). The transaction volume statistics provide critical input to physical database design. In physical database design, the logical data structures are mapped to the physical constructs of the target DBMS. The database designer chooses which physical constructs to use based on the performance requirements and the transaction volume statistics of the transactions. 
    • Document the performance requirements of each transaction (i.e., in what timeframe must the execution of one instance of the transaction complete?). The transaction performance requirements are an input to physical database design. One objective of physical database design is to choose the physical constructs from among the various options provided by the target DBMS such that all transactions meet or exceed their transaction performance requirements. Hence, these requirements must be defined prior to beginning physical database design. 
    • Rank each transaction in terms of execution priority (i.e., how important is it for this transaction to meet or exceed its performance requirements?). This can be accomplished, for example, by assigning the priority of 1 (very high) to 10 (very low) to each transaction. One objective of physical database design is to ensure that each transaction meets or exceeds its performance requirements. This objective cannot be accomplished all at once for all transactions as a group; the transactions must be evaluated one at a time. This ranking provides the basis for the sequence in which transactions are evaluated. In addition, tailoring the physical database design to one transaction so it executes quicker sometimes negatively affects the performance of other transactions. This ranking also provides the basis for any transaction arbitration scheme.
