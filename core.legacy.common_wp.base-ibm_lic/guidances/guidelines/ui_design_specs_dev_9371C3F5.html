Main Description
Based on the use case model, a set of use cases and their associated scenarios, an analysis object model, a user interface conceptual model, early design low-fidelity user interface prototype, the User-Interface Design Guidelines (if they already exist) branding, and visual design collateral (as applicable) and a set of early usability test scenarios (cognitive walkthroughs), you can follow a systematic approach for developing a User Interface Design Specification. These generic steps can also be followed for projects which do not have uses cases or used an Object Orientated development approach (i.e. projects based on business, functional, and non-functional requirements). You can follow these steps more or less sequentially. Each is described in more detail below.
  1. Identify the audience for the work product.
  2. Identify the level of detail of specification.
  3. Create a User Interface Specification template to be shared among all User Interface Specification designers.
  4. Determine the division of labor among designers (only needed if more than one writer).
  5. Determine the review cycle for the specification.
  6. Identify the reviewers and approvers of the work product.
  7. Create the initial draft.
  8. Begin the review cycle.
  9. Update the User Interface Specification with the agreed upon changes and document the changes in the revision history.
  10. Obtain approval.
  11. Update the User Interface Specification during development and testing as needed.
Identify the audience for the work product
 
Depending on the client you may have different consumers of the User Interface Design Specification. The technical team uses the User Interface Design Specification to develop the UI interaction. In contrast, the business and test teams often use this work product to validate the UI.
 
Identify the level of detail of specification
 
Depending on the target audience, the control specification may differ in level of detail. Use the User Interface Development Guidelines to select which control attributes are required for business approval. In the ideal situation, this document will be used to capture all the technical details including data source, initial value, enabling and visibility logic, maximum length, event handling and error handling details. This assumes that all required supporting technical documents such as data and object models are available. Decide what level of detail is needed for the first draft and add these additional details through iterations of low-level design as the required supporting technical documents become available.
 
Create a User Interface Specification template to be shared among all UI Specification designers.
 

Since there may be more than one designer, in order to ensure consistency it is advisable to create a template to be shared by all designers. Start with the template attached; but tailor it to better meet your customerâ€™s needs.

Determine the division of labour among designers
 
It is advisable that you determine the division of labor among designers prior to starting and then manage the plan along the way. Some screens will probably be more complex than others, and therefore will require more detail. For best results, have the original User Interface designer create the specification for a given screen since s/he will have more background to accurately capture the design interaction.
 
Determine the review cycle for the specification
 
Depending on the audience, you may need to have technical leads and business stakeholders review specification drafts. During this stage, reviewers and approvers for each draft should be identified. For best results, reviewers must be given adequate time to thoroughly review the draft by allotting them one half the time taken to create the draft.

Also, schedule an internal review after each writer has completed the first draft of one screen. This will help ensure consistency among writers and give writers an opportunity to compare experiences.

Lastly, include an approver stage since there may be feedback that you are given that you may wish to reject with good reason (such as out-of-scope functionality, previously agreed and resolved, etc.)

Identify reviewers and approvers of work product

Identify reviewers and approvers of the work product before creating the initial draft and confirm them, prior to contacting them, with the stakeholders and the project manager.
 
Create the initial draft
 
Use the User Interface Design Guidelines and the project specific User Interface Specification template to document all the controls and interaction for each individual screen.
 
Begin the review cycle
 
Once the initial draft is completed, distribute the document to the identified reviewers. Use the revision tracking mode of word processing tools to capture all changes.
 
Update the User Interface Specification with the agreed upon changes and document changes in the revision history
 
After each of the review cycles, capture the accepted changes in the revision history with as much detail as possible. Update the version number to reflect each change.
 
Obtain approval
 
It is advisable that the client be asked to authorize development based on the accepted specification. This ensures that the application to be delivered is based on a shared understanding. An acceptable form of authorization is an email stating that the included document (e.g. user interface design embedded in a high-fidelity prototype) is approved. Projects may use sign-off documents if desired.
 
Update the User Interface Specification during development and testing as needed
 
During development, there may be many reasons why a previously-documented screen may require changes. In such an event, the User Interface Specification should be updated and validated again by the approver.
 
During testing, the test team may find a discrepancy between the User Interface Specification and the application. If the development team did not follow the UI Design Specification, then it is their responsibility to update the code to reflect the specification. In the event that the user interface is found to be incorrect, incomplete, or insufficiently usable, the User Interface Specification should be updated. Again, the approver must be notified and provide approval.
