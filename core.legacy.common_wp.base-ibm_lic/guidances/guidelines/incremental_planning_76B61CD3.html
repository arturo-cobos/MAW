Guideline: Incremental Planning
Incremental development (also variously called iterative or spiral development) the important management checkpoints have working software among their deliverables.

Main Description

1 Introduction

Working software gives a really reliable indication of progress. There is little scope for error or over-optimism.

Decisions about what should be included in each increment, that is in each programming cycle, each Build cycle, and if appropriate each Release cycle, are consequently very important. On the one hand we want to make quick and visible progress on important parts of the system. On the other we want to ensure that the project is built on good foundations, that we do not just create the illusion of progress, and that we build first those parts that make the overall project go fastest.

Incremental planning is about the choice of what to build in each programming cycle, build cycle and release.

There is a fundamental tradeoff between the short and the long term, between:

  • Building the easiest, most valuable functionality (biggest bang for the buck), and
  • Building a base for the harder functionality and the harder nonfunctional requirements that demand more infrastructure and more components (mitigating risk).

The right balance depends on the circumstances of the project. Sometimes it is necessary to deliver or at least demonstrate valuable functionality to help justify the project. Sometimes the clear priority is to build a solid long-term base. It is almost always necessary to do some of each.

Incremental planning is done top down. First the contents of a release cycle are defined, at least in outline. Then as each release cycle starts, the number and goals of the build cycles are defined, and then as each build cycle starts the number and goals of the programming cycles.

A release (unlike a build cycle) has to be useful to some users, so functionality plays a bigger role. We give no further guidance here except to note the following:

  • Sometimes the date some functionality has to be delivered is more important than the exact contents of a release. In this case, plan a fall back position and define the early build cycles so that they establish the fall back.
  • Especially when under pressure, consider the needs of different categories of user separately. The three requirements with the highest priorities may be for different users. Yet supplying all three may not really satisfy anyone. Each needs their top priority plus some other lesser items. It may be better to gear a release to a single category of users.

Assuming the first release is defined, or there is only one release, here is a simple procedure, which gives some guidance in choosing what to build first. The first three steps can be done in parallel.

2 Steps

2.1  Risk Mitigation

  • List and prioritize the risks on the release (or the project if there is only one release).
  • For each risk, identify what could be built that would mitigate the risk.
  • In each case, identify prerequisite components.  For example, suppose risk A would be completely removed if X could be built and demonstrated.  But perhaps X cannot work unless Y and Z are in place.

2.2  Early Desirable Functionality

  • List and prioritize desirable early functionality.  (What would look good at the management review?  Which use cases? Which nonfunctional requirements?)
  • For each item, identify prerequisite components.

2.3  Team Learning

  • List and prioritize the gaps between the experience of the project team (both individually and as a group) and the needs of the project.
  • For each gap, identify what could be built early to close the gap.

(Even if everyone in the team is experienced they may not have worked together before.  It can be valuable to go through a small build cycle simply to practice working together as a team and to familiarize everyone with the development procedures.)

2.4  Choose the Increment Goals

  • Establish what it is possible to build early.
  • Taking the possibly conflicting priorities in points 1-3 into account, choose the goals for the first build cycle.
  • Try to meet several goals with each component.  For example, it is very rare to build a component solely to close a gap in the experience of the team.  Almost always this is a contributing factor in the choice of a component that has some other value.  Or it may result in an extension to the definition of a component that has some other justification.

Projects vary in the precision with which they can specify later build cycles.  In some the requirements are deliberately left open and this has an obvious effect on the goals of later build cycles.  In others, the requirements are defined early and put under change control.  The number and goals of the build cycles can be determined with fair precision.

Full incremental planning depends on the architecture of the system.  Only when the structure of the system is reasonably understood can detailed dependencies be understood and a sensible progression through successive subsystems be proposed.  The architecture also influences whether it is more sensible to:

  • Build breadth first, establishing skeleton versions of all the subsystems and then elaborating each in successive build cycles, or
  • To progress by successively adding more or less complete subsystems.

Both are possible within the managerial framework defined by the delivery process.  The architecture also determines how and to what extent large pieces of work can be done in parallel.  Generally though, given knowledge of the architecture and the relative importance of the functionality, an obvious order of construction suggests itself.

At all levels it is valuable to characterize the goals of an increment by a few short phrases, as well as giving more detailed references to use cases, nonfunctional requirements, subsystems, and error reports. The phrases help stakeholders and developers alike to grasp the essential goals of the increment.  For example:

  • Make the read-only data connection to the (legacy) customer information system.
  • Add a printing subsystem, so that complete invoices can be printed.
  • Extend the claims subsystem so that it covers health claims.
  • Build one use case end-to-end.
  • Fix all errors reported up to end January.

Sometimes the short phrases sum up the real target of the increment more honestly than a more detailed document.

Finally, ensure that the goals of build cycles and programming cycles are completed. Incremental development means we finish building part of the system, and then finish building another part.  It does not mean leaving everything half finished.
