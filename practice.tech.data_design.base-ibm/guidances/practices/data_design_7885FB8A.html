Purpose
The purpose of this practice is to guide the project team in outlining and detailing the logical and physical databases that will support the application under development. Any data migration that is needed should also be considered during development of the application and its deployment into the production environment.
Main Description

The information discipline in SOMA considers information from top down business perspective and incorporates information into business design in Information Analysis and Modeling practice. It aligns solution design to business needs from information perspective and helps identifying information services that the business requires. Furthermore, it introduces message modeling and message specification to existing data design practices to define service specification using a common message model. A common message model limits data transformation across and within the layers of the architecture.

Information Specification

In this practice, we begin to design information - both data design and message specification -  in Information Specification (IS).

We develop the Logical Data Model, defining the data structures and relationships for all the business entities within the scope. The Logical Data Model takes these high level entities in Conceptual Data Model and elaborates to include detailed definitions of their attributes and relationships for data design. Then we perform Data Quality Analysis to quantify the data quality risks, validate data matching rules, define detailed data integration mapping strategies, and begin evaluation of alternative service realization choices where providing acceptable data quality proves to be problematic. This may lead to technical feasibility studies.

We also develop Canonical Message Model that is used in services specification to define service messages. Each service contains one or more operations and each operation includes input, output, and error message definitions. To maximize service reuse and service interoperability, all service messages should comply with a Canonical Message Model. The Canonical Message Model defines the physical structures, rules, and semantics of the data being passed in service messages. It is normally implemented as an XML schema or set of schemas. Like the Logical Data Models which it is derived from Conceptual Data Model, the Canonical Message Model may be based directly or indirectly on industry standard models. Solutions are designed and deployed as multiple architectural and operational layers. Data flows exist both horizontally across an architectural layer—for example, through the activities of a business process—as well as vertically up and down through architectural layers; for example, from a business process down through the services to service component, to application and database layers. Establishing a standardized and shared data representation throughout the solution helps minimize data transformation required for the solution to operate. The Data Mapping activity defines the data transformation rules needed to integrate all solution elements to ensure data is shared correctly within and across architectural layers.

Information Realization

Next, we make realization decisions related to information elements in the solution in Information Realization (IR).

We make Realization Decisions and Architectural Decisions on realizing the information services. We select types of Information Services such as data validation services, data cleansing service, master data services; and also we Select Information Integration Patterns such as data consolidation patterns, data federation patterns, and so on.
How to read this practice

The best way to read this practice is to first familiarize yourself with its overall structure - what it is in it and how it is organized. 

The best place to start is with the Key Concepts for the practice - those concepts that are critical to understand in order to adopt the practice. Once you understand the key concepts, you can turn your attention to the Work Products produced by the practice.  Then you can review the practices Tasks. From the work products and the tasks, you can access applicable guidance - guidelines and tool mentors associated with each task provide details of how to perform the task. Templates and checklists associated with the work products guide you in their completion and evaluation. You can also access the guidance provided by the practice directly by using the Guidance folder.
