Task: Develop Physical Database Design
Define the detailed physical design of the database.
Disciplines: Technical Design
Purpose
Create an implementation-specific database design that will provide adequate performance and ensure database integrity and security.

      Main Description

      This task outputs a physical database design which is represented as a physical data model. This model is the source from which the schema data definition language (DDL) code for direct physical implementation of the target database is created.

      Typically an idealized design of the database is used as an input to this task. Primarily driven from a business context, this logical design is created without consideration for the target implementation environment. Progression to a physical design involves making adjustments to the model in order to account for various implementation considerations and performance requirements.

      Steps
      Map and Define Domains
      Map domains assigned in the idealized database design to a data type supported by the target database management system (DBMS). For cases when no acceptable built-in data type exists, create a user-defined data type.
      Create Initial Physical Database Design Elements
      Identify the initial set of tables, columns, and relationships between tables as described in Guideline: Physical Database Design.
      Create Primary Key and Unique Constraints
      • Define the one or more columns that uniquely identify a row in a table.
      • Define constraints on columns that guarantee the uniqueness of the data or collection of data.
      Define Data and Referential Integrity Enforcement Rules
      Ensure the integrity of the database by defining validation rules to enforce constraints on data values. Define foreign key constraints to enforce a relationship between two tables.
      De-Normalize Database Design to Optimize for Performance
      To eliminate the cost of computationally expensive joins, de-normalize the idealized design by introducing redundant columns.
      Optimize Data Access
      Define indexes on columns to improve query performance. Consider the trade-offs in longer insert and update processing as described in Guideline: Physical Database Design.
      Define Storage Characteristics
      Design the space allocation and disk page organization of the database.
      Design Stored Procedures
      Determine if stored procedures or triggers should be used to implement data access operations.
      Build or Generate the DDL
      Create the DDL that will implement the physical database structures.
      Key Considerations

      Before introducing complexity into the database design for performance or other reasons, check if there is some flexibility in the requirements that could allow a simpler design.

      Consider using tools that generate DDL from the model, to save development time and to ensure the model is consistent with the implemented database.
