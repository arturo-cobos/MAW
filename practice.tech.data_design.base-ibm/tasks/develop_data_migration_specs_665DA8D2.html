Task: Develop Data Migration Specifications
Describe how to migrate a legacy data source to a target database.
Disciplines: Technical Design
Purpose

To specify the data migration by defining its scope, its data profile, the mapping between the data sources and the target database, and any required data transformation processes.

Both one-time data conversions and ongoing data interfaces are addressed by this task.

        Steps
        Define Migration Scope
        • Identify the sources of data to be migrated and their location.
        • Identify the systems that use the current data sources and the systems that will be using the new database.
        • Identify data not currently held electronically and how to enter it into the new system.
        • Determine if the source data will be retired after migration. If the source data will continue to be used, then maintenance of the data in both its original source and in the new database can be an issue.
        Understand Data Sources Through Data Profiling
        • Gather metadata that describes the data sources.
        • Develop a map of the data sources showing how data fields are stored. Establish rules for dealing with redefines and repeating groups of data within the data structures.
        • Determine the content, domain, and quality of the data in each attribute and establish the semantics behind each attribute. It is important to perform this operation with the actual source data, as the documented metadata may be incorrect.
        Define Mapping Between Data Sources and Target Database

        Describe the mapping between the source database elements and the corresponding target database elements. Mapping between physical data models is more practical than attempting indirect mappings via logical data models.

        Develop rules for:

        • Selection
        • Cleansing
        • Transformation
        • Exception Handling
        Identify Manual and Automated Data Migration

        Identify which parts of the data migration can be automatically performed and which parts should be migrated through manual procedures.

        The Guideline: Designing Data Migration Subsystems gives some insights on how to design components to perform automated data migration.

        Key Considerations
        It is preferable to have legacy developers from the source data references as participants in the data mapping process. Business analysts who possess an intimate knowledge of the data that needs to be migrated and those familiar with the target systems should also be active participants in the process.
