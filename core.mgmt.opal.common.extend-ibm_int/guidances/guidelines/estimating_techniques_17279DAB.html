Use this guideline to provide organizations, project teams and estimating practitioners with suggested methods and techniques for performing the various types of estimates.

Main Description

Purpose

IBM's Worldwide Estimating Community has created a revised estimating procedure intended to lead to fewer troubled projects with estimation as a root cause. The procedure includes a formal estimation process with sign-off points, defined roles and responsibilities, and a database for estimates and results. If you have any questions about this information, please contact Project Services Europe/UK/IBM.

Use this guideline to:

Provide organizations, project teams and estimating practitioners with suggested methods and techniques for performing the various types of estimates.

Scope

This guideline applies to:

The production of a size, effort, cost or schedule estimate for a project.
The objective of this guideline is to provide suggested techniques and approaches for the estimating practitioner, or any project team member to estimate size, effort, cost and schedules for a project.

 Refer to IBM Global Solution Delivery Framework (IGSDF) Tools Guidance, IBM ICM AssetWeb (Project Estimation and Planning Practice) and the Estimating within IBM wiki - tools page for recommended estimation tools.

 Refer to the Estimating Considerations Checklist in the Checklist index.

Overview

The implementation of the Estimating Procedure (IMSP380) may trigger the use of this guideline, which can be used to generate estimates in many different development environments and for projects of all sizes.  Estimates can be recorded using the estimation report.

Using an estimating technique has many benefits for both your organization and the client, including:

  • Repeatability and gathering of historical data for use in future estimates
  • More accurate estimates that are produced more efficiently
  • Software releases that are easier to negotiate, track, control and deliver
  • Increased client confidence in the ability of your organization to deliver within budget and on time

Details

1. Estimate Size

Size estimates can be performed in several ways, all of which rely on counting some element(s) of the work in order to understand the magnitude of the project. 

1.1. Wideband Delphi Technique

This technique uses the experience and expertise of the project team to produce the estimate.

 Refer to Estimating with Variance Guideline for details

1.2. Pert Sizing 

The Program Evaluation and Review Technique (PERT) uses an assessment of low, expected (most likely) and high values to arrive at a most likely estimate for the variable in question along with a confidence level.

These values might come from the Wideband Delphi technique (see above) OR alternative approaches.
 Refer to Estimating with Variance Guideline and Choosing and Using an Estimating Approach Guideline for details.

1.3. Function Points

A function point is a unit of measurement to express the amount of business functionality an information system provides to a user. They can be used to size requirements and applications, and in prediction and analysis of application development and maintenance performance, but training is required to use them.

 Refer to IBM Global Account Delivery, Measurements: Function Points or the external website of the International Function Point Users Group (IFPUG) for additional information on Function Points.

1.4. Sizing by Analogy

This approach involves finding a previous project and its final size, and comparing this with the project being estimated, to arrive at a size by appropriate scaling of the previous project figures.

 Refer to Choosing and Using an Estimating Approach Guideline for details.

1.5. Sizing by Object Oriented Effort Point (OOEP) Methodology

In the Object-Oriented life-cycle, we model the business for which the application is being written, which leads to the discovery of Business Objects which provide the business-logic services that are required. The business objects can then be used to estimate the number of supporting objects that will be needed. Each object is then broken down into its component areas to determine the number of OOEPs that the object requires.

 Refer to internal webpage AMS NL Estimation Methods for details

2. Estimating effort

2.1. Top down

The Top-down estimating technique assesses overall project size in some appropriate unit, and converts this size into a cost (and schedule) using algorithms, historical information or rules of thumb.

 Refer to Choosing and Using an Estimating Approach Guideline for details. 

2.2. Bottom up

The Bottom-up estimating technique assesses effort figures for individual elements of the project e.g., activities, tasks or work products. The technique then sums up these individual effort figures to provide an overall project effort (and thus cost).

 Refer to Choosing and Using an Estimating Approach Guideline for details.

2.3. PERT

The Program Evaluation and Review Technique (PERT) uses an assessment of low, expected (most likely) and high values to arrive at a most likely estimate for the variable in question along with a confidence level.
These values might come from the Wideband Delphi technique (see above) OR alternative approaches.

 Refer to Estimating with Variance Guideline and Choosing and Using an Estimating Approach Guideline for details.

2.4. Effort Estimation Using Productivity

How to work out the effort required when you know the size and productivity of your project.

 Refer to Choosing and Using an Estimating Approach Guideline for details.

Overall effort is the sum of Total effort and Contingency factor

2.6 Effort Estimation For Support And Maintenance Projects

As the relationship between the customer and the servicing team expands and extends, it becomes imperative on the part of the servicing team to provide value addition in the services rendered to its customer.

For a maintenance team, the predictability is the best value-add. Thus, the first question that needs to be answered is 'Can we tell our customer that we will be able to solve a bug between 12 hours to 20 hours with 80% confidence level?'

The following procedure gives guideline for estimating an effort spent for maintenance & support project.
1. Identify the basic characteristics of the project. This includes Type of Application, Size of Application, Application launch date, Number of operating systems supported, Number of databases supported, Primary Programming Language, Predominate code & logic Structure, Predominate performance / execution frequency requirements, Database Size, User Details, Experience of Users, major customer requirements, Number of Interfaces to the application such as.
2.  Identify the team size, skill level of team, team volatility, availability of development and test environment, Test tools used, System availability, Configuration Management Mechanism, Flexibility in allocating resources.
3. Identify the characteristics of Service requests, which includes the Type of SR (for example. PMRs, Defects, User Queries, Production Problems Such as.), Size of SRs (For example. LOC impacted), volume of incoming SRs, Complexity of SRs.
4. Based on the identified information, if any past data from other similar projects or current industry data is available, estimation can be done based on that.
5. For a project with a given set of characteristics specified (in 1. And 2.) above, estimation of effort may be arrived at based on the project's actual data for resolving Service Requests categorized by SR Type and Complexity.

Terms and definitions

  1. Population
    Population would comprise of data collected for all bugs fixed or closed till date including the actual effort required to fix or to close the bug.
  2. Sample
    Sample would comprise of the data collected for the last 30 bugs that are fixed or closed, or the bugs fixed or closed in the last quarter, whichever is greater. The sample is subset of the population.
  3. Normal Distribution
    Mathematician-astronomer, Karl-Gauss, has been instrumental in the development of the normal distribution. Normal distribution is a continuous probability distribution. The normal distribution comes close to fitting the actual observed frequency distributions of many phenomena, including human characteristics (weights, heights and IQs), outputs from physical processes (dimensions and yields) and other measures of interest to managers in both the public and private sectors. It is important in situations where it is necessary to make inferences by taking samples from a large data source.
  4. Sampling distribution
    Sampling distribution of the mean various samples can be drawn from the population and each would have a different sample mean and a different standard deviation. A probability distribution of all the possible means of the samples is a distribution of the sample means, which statisticians refer to as sampling distribution of the mean.
  5. Central Limit Theorem
    For drawing inferences, samples can be drawn both from normal and non-normal distributed population. The central limit theorem states that the mean of the sampling distribution of the mean will equal the population mean regardless of the sample size, even if the population is not normal. Also, as the sample size increases, the sampling distribution of the mean will approach normality regardless of the shape of the population distribution.

    The significance of the central limit theorem is that it permits to use sample statistics to make inferences about population parameters without knowing anything about the shape of the frequency distribution of that population other than what we can get from the sample.

    Statisticians use a sample size of at least 30 for approximating the sampling distribution as a normal distribution.

  6. Estimate Interval
    The estimate interval is the range of the actual effort within which a bug can be fixed or closed.
  7. Confidence level
    The confidence level projects the percentage of bugs that can be fixed or closed within the calculated estimate interval. The confidence level is pre-decided to calculate the various estimate intervals.

  4. Calculate the mean of the sample data

    The mean of the sample data, here actual effort, is calculated using the following formula:

    Sample mean = Mean of the actual effort spent on fixing or closing the bugs in the sample = m = (x1 + x2 + ......+ xn-1 + xn) / n

    Where n = Total number of bugs fixed or closed in the sample.

  5. Determine the mean standard deviation, Sx

    The mean standard deviation or standard error of the mean can be calculated using either of the following formula: 

    sx  =  s / Ön, if n/N <= 0.05
     sx  =  (s / Ön ) * (Ö((N-n)/(n-1)), if n/N > 0.05

     Where,
    n is the number of bugs in the sample,
    N is the number of bugs fixed or closed till date

  6. Assuming the confidence level and determining the estimate interval

    The confidence level that has to be projected to the customer is assumed. Say, it is 95.5% confidence level.
    Use the following formula to calculate the estimate interval:

    Use the following formula to calculate the estimate interval:
    Estimate interval = m + zsx

    Where,

    z = Statistical value obtained from z-distribution tables (Appendix C) based on the a/2-value (= (confidence level)/2)   
    Thus, using the z-distribution tables of the statistics, the a/2-value (the unacceptable region) is 0.4775 (i.e. (0.955/2 » 0.4775). The corresponding z-value for a/2-value is 2.
    Hence,
    Estimate interval in which the bugs can be solved = m + 2sx with the confidence level as 95.5%.

 Measurements

a. Performance of the team and customer satisfaction level
Performance of the team will be defined as the improvement in the estimate interval (that is, a reduced interval range) over a period of time with the same confidence level. It can be measured every quarter taking into account the sample of bugs from last quarter or last 30 bugs, whichever is greater. This would imply that the normal distribution curve of the population would become steeper, which can be measured by reduced population standard deviation, s.
If the same data is shared with the customer, a steep curve (compared to the previous one) will indicate a higher customer satisfaction.

b. Defect Measurement
Every quarter if the percentage of bugs, whose actual effort is beyond the estimate interval, exceeds the unacceptable confidence level (for example., for 95.5% confidence level, unacceptable confidence level is 4.5%), a defect would be raised. The defect would imply that the confidence level would be high for a wider estimate interval. The objective is to optimize the confidence level and estimate interval. The preventive action would be to recalculate the estimate interval along with the confidence level.
A high confidence level is not desirable, as it would carry with it the uncertainty, as the estimate interval for actual effort is wide and the estimate will carry a very little value. Similarly, a low confidence level is also associated with uncertainty as the estimate interval for actual effort is too narrow and it might have lots of credibility issues. A simple example that can clarify the above concept is as the customer sets tighter and tighter estimate interval (that is, reduced estimate range), the developers agree to a lower and lower confidence level.

c. Advantages of using the model

  1. It is a general model that can be applied to any maintenance project that has sufficient data (at least 30 data points) available. Also, the data collected should be accurate (data collected by filling time sheets can be the best input).
  2. Being a standard statistical method applied to calculate yield or productivity in process-oriented industries, the credibility of the measures with respect to both performance and defects is high.
  3. There had been debates on the accuracy of the point estimates carried out for each bug in a maintenance project. The model presented in the paper gives an estimate interval within which a bug can be solved. The estimate interval is purely based on the performance of the team.
  4. The measure of the confidence level and the estimate interval can also help a maintenance team to analyze the impact of any changes either in the team structure or the processes adopted.
  5. Defects would be raised on the basis of a collection of bugs rather than on the basis of data for a singular bug. Some of the bugs, for example, memory leak bugs, tend to have exceptional conditions. As per the model, such bugs will lie in the unacceptable region. Only, when the number of bugs in the unacceptable region exceeds the desired limit, a defect will be raised. The exceptional conditions may not be a defect (which is treated as one in the current model).

d. Limitations and solutions with respect to the existing measures in this model

There is no demarcation of low, medium and high complex bugs. The demarcation, though, can be made on the basis of number of hours required to fix the bug.

Appendix C. Normal Distribution tables for Z values

Normal distribution tables tabulate the likelihood of a particular value (Z) being achieved, and have a key place in probability calculations. The tables are derived from graphical representations, and these representations take on characteristic shapes. The Normal shape is a bell curve.
