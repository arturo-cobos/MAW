Tool Mentor: Change Management Using Rational PM
This tool mentor provides guidance on how to perform Change Management when using the OnDemand Process Asset Library (OPAL) and Rational Portfolio Manager (Rational PM).
Tool: Rational Portfolio Manager
Relationships
Related Elements
  • Manage change request (IMSP610)
  • Raise change request (IMSP610)
Main Description

1. Purpose

This document provides guidance how Rational PM is used to support Change Management in accordance with the OnDemand Process Asset Library (OPAL). This document may serve as a template for creating a tailored procedure or guideline.

This guidance document contains three sections:

  1. Purpose
  2. How Rational PM supports the OPAL procedure
  3. Usage table for Rational PM data elements (fields)
The intent of this guidance is to provide sufficient direction on how Rational PM is used on a project to support Change Management, so that there are no ambiguities about the process and the team training can be concise.

The guidance assumes a basic understanding of how to use Rational PM and is not intended to be training material. Detailed training on the use of Rational PM can be found by selecting Education and Training from the GBS Rational Portfolio Management Enablement page.

Step 1 - Raise Change Request - performed by Stakeholder

Please note that Rational PM elements in BOLD text and highlights are considered required. Other fields and portlets are recommended for completeness. Refer to the table in section 3 for a complete description of all fields, standards and practices.

  • Receives Change from project stakeholder.
    • CREATE Change Request in Scope Management using the OPAL IMSP610 Change template and the OPAL IMSP610 Change layout. Key information that should be captured includes Rank, Description, Target Date, Proposed By, and Completion Criteria. A value for the Change Type should be captured in the Attributes portlet. State is set to IDENTIFIED by RPM
    • The OPAL IMSP610 Change template can be located by selecting Template from the Rational PM Scope Management View and expanding the Scope Templates / OPAL Scope Templates folder.


Step 2 - Perform Initial Assessment, Reject or Accept for Analysis - performed by Project Manager

  • If Change determined to be valid and in scope including Contract Change handled within the project
    • RECORD - assign Change owner in Staffing view or Resources portlet, enter Est Start, Est Finish, and Target Date. Additional information to capture includes Quantification/Impact, Order of Magnitude Cost/Benefits, Approved Cost/Benefits, and Background portlets. Change State to ACCEPTED FOR ANALYSIS.
  • If Change determined to be invalid reject Change
    • RECORD - document reason for rejecting Change in the Status portlet, update Closed By portlet with person closing the Change. Change State to REJECTED. After notification of rejection is made to requester, change State to CLOSED.
  • If Change determined to be deferred
    • RECORD - document reason for deferring Change in the Status portlet. Change State to DEFERRED.
  • If Change determined to be valid Contract Change outside of the project scope raise a Contract Change Request (CCR)
    • RECORD - document decision on transferring ownership to Change Control Manager in the Status portlet. State remains 'Identified' until CCR is accepted by the program management office. Then change State to CLOSED.


Step 3 - Manage contract changes - performed by Change Control Manager, or Project Reviewer

  • Project Manager submits Contract Change Request (CCR) to IBM Program Management Office (PMO) and monitor changes transferred to PMO as External Dependencies.
  • PMO performs initial assessment, and rejects or accepts the proposed contract change for analysis.
  • If a full assessment of the change is required, then a Service Request must be entered to perform the analysis and CCR is closed.
  • If the CCR is acceptable but will not enter the Service Request process, PMO analyzes impact and potential solutions, obtains formal agreement and schedule change orders, and then closes CCR.

Step 4 - Determine impact and potential solutions - performed by Project Manager, Business Process Analyst, Code Reviewer, Implementer, or Stakeholder

  • Review Change Request and assess impact
  • If the impacts and potential solutions have been determined
  • Record - Impact Assessment, Solutions Considered, Quantification/Impact, and Order of Magnitude Cost/Benefits. Update Status with record of activity. Change Stateto 'SUBMITTED FOR APPROVAL'
  • If the impacts and potential solutions have not been determined or require additional definition
  • Update Status with record of activity, State remains 'ACCEPTED FOR ANALYSIS'



Step 5 - Obtain formal agreement and schedule change orders - performed by Change Control Board or Project Sponsor

  • Review results of the assessment with Change Control Board or Project Sponsor.
    • If the Change is approved
      • Update the hours, cost, and schedule information in the Approved Cost/Benefits portlet. Record who approved the Change in the Approved By portlet. Update Status portlet. Change State to 'APPROVED FOR IMPLEMENTATION'
    • If the Change is rejected
      • Update actions and reason for rejection in the Status portlet. Change State to REJECTED. After requester is notified change State to CLOSED
    • If the Change is deferred
      • Enter reason for deferral and expected date to review in the Status portlet. Notify requestor of the decision. Change State to DEFERRED.
      • Re-enter this step when the reasons for the deferral have been resolved.
    • Use OPAL ENG319 Change Request Detail Report, OPAL ENG317 Change Log, and OPAL ENG319 Change Detail pivot to assist with review and provide information to non Rational PM users.

 






Step 6 - Implement change - performed by Project Manager

  • Verify all actions and tasks for implementing change have been completed
    • If completed
      • Update Solution Retained, Implementation, and Status portlets. Change State to IMPLEMENTED.
    • If not completed
      • Update Status portlet. State remains 'APPROVED FOR IMPLEMENTATION'.

Step 7 - Close change - performed by Project Manager

  • Close the Change.
    • Update Status portlet. Record person closing the Change in the Closed By portlet. Notify stakeholders that Change has been closed. Change State to CLOSED

Implemented: CR that has been implemented
