Tool Mentor: Defect Management Using Rational ClearQuest
This tool mentor provides guidance on how to perform Defect Management when using the OnDemand Process Asset Library (OPAL) and Rational ClearQuest.
Tool: Rational ClearQuest
Relationships
Related Elements
  • Identify, raise and log defect (IMSP520)
Main Description

1 Purpose

This document provides guidance how IBM Rational ClearQuest is used to support Defect Management Procedure (IMSP520) in accordance with the OnDemand Process Asset Library (OPAL). This document may serve as a template for creating a tailored procedure or guideline.

The intent of this guidance is to provide sufficient direction on how IBM Rational ClearQuest is used on a project to support Defect Management, so that there are no ambiguities about the process and the team training can be concise.

This document does not replace the procedure and does not cover all of the steps defined in the procedure. To fully understand this document, it should be read in conjuntion with the procedure.

Refer to the procedure and the associated references described in OPAL for details on the tasks, steps and techniques for performing this procedure. In particular, you should be aware that the procedure may contain additional steps that take place outside of ClearQuest and are not duplicated here.

2 Scope

The guidance assumes a basic understanding of how to use IBM Rational ClearQuest and is not intended to be training material. Detailed training on the use of IBM Rational ClearQuest can be found by selecting Education from the GBS Rational Lifecycle Tools Enablement page.

The process documented is dependant upon the use of the tailored ClearQuest Schema produced by the Global Services Rational Adoption Program (GSRAP) team.

The current customized schema has the following characteristics:

  • It was developed and tested using IBM Rational ClearQuest, ClearQuest Web v7.0.0 Label BALTIC_PATCH.D06.09.29
  • It does not provide automatic integration with Rational Portfolio Manager (RPM).
  • It does not provide automatic interface or integration with other tools or applications that an organization may use in accordance with its measurement program to support quality goals at an organizational level.
  • It doesn't cover problem management, including emergency fixes, which are covered by separate procedures within OPAL.

3 Tool implementation of the OPAL Procedure

3.1 The workflow (defect's lifecycle)

The following state transition diagram shows the lifecycle of a defect within ClearQuest. States are shown by boxes; Actions are shown by the connecting arrows. Each action (arrow) has a legend with the format [ActionName, Role] where: ActionName is the name of the action and Role is the role with the permissions to execute the action.


Reference: PM = Project Manager; O = Resolution Owner.

The normal lifecycle of a defect is shown through the centre of the diagram from left to right: Raised through Closed. Other states are exceptional.
The following table defines each of these states.

State

Definition

Raised

A defect is Raised if a Submitter has just identified and logged it into ClearQuest. No other action has taken place at this state.

Opened

A defect is Opened if it is considered as a valid defect and it is ready to be analyzed, so a Resolution Owner is assigned to work with it and proceed with its analysis.

Analyzed

A defect is Analyzed when the Resolution Owner has defined the schedules and planned actions required to solve the defect. The proposed solution is ready to be validated by the Project Manager.

Solution Approved

A defect is given a Solution Approved status when the Project Manager considers that the schedules and recommended actions defined by the Resolution Owner are appropriate; the defect's resolution is approved to be implemented.

Resolved

A defect is Resolved if the resolution of the defect is complete and it is ready for verification/validation.

Resolution Accepted

A defect is given a Resolution Accepted status if it passed the verification/validation process successfully.

Closed

A defect is Closed if it has been solved successfully and it no longer requires attention.

Cancelled

A defect is Cancelled if it should not be considered in any analysis after its submission.
Note: if a new defect has been raised the Submitter can cancel it to avoid any analysis probably because the defect was raised by error.

Deferred

A defect is Deferred if it is out of scope for the current release(s). Defects in the Deferred state are held and reconsidered for future releases.

Duplicated

A defect is given a Duplicated status when exists another submitted defect that encompass some or all of the new defect's symptoms.
Note: When the defect is placed in the Duplicated state, the defect number it duplicates is recorded in the defect's record into ClearQuest.

Rejected

A defect is Rejected if it is considered invalid.
Note: a rejection can occur for many reasons like:
- it is the result of a risk occurrence, change or issue
- it was caused by testing errors or omissions
- it was raised as the result of a user misunderstanding, etc.

Revision Required

A defect is in Revision Required state if it has been returned to the Resolution Owner for further analysis/actions.
Note: a defect may require this kind of revision during analysis or solution implementation.

ClearQuest may be accessed either directly via the ClearQuest Client, or via the web interface ClearQuest Web. If using the ClearQuest client similar actions are available and a table mapping these actions between both interfaces is provided below.

3.5 Evidence of Conformance to Defect Management Procedure

This section describes the features built-in to the ClearQuest schema to provide evidence for conformance to the Defect Management Procedure (IMSP520).

1. Each identified defect is represented by a defect record in ClearQuest.

2. The query Defect Log (APP414) contains the information in the Defect Log (APP 414) or in the Defect Management Documentation (DCD 325). Using this query, it is not necessary to maintain a separate defect log as the query can be run on demand.

3. Defect metrics can be collected from the individual defect records in ClearQuest or analyzed by building queries and charts as required. This information can be exported for inclusion in a separate database (for example ESSBASE) if required.

Step 1 Identify, raise and log defect

1.1 Submit a defect


When a defect has been identified, a defect record must be raised in ClearQuest. This will be carried out by whoever finds the defect (henceforth, the Submitter).
Connect to ClearQuest Web, using the instructions supplied by the deployment team. Using the options at the top of the ClearQuest Web interface select New: Defect and click the Go button. A Defect submission form will be displayed.



ClearQuest will automatically fill some fields like Defect ID, Date Raised, Originator (Submitter) and Status. Fill in all mandatory fields and any optional fields that you feel may aid in resolution. After completing all the required fields, save the form. Now the defect is identified and logged in ClearQuest and its status is Raised.


The Project Manager will be sent an e-mail telling that a new defect has been raised.
The decision regarding if the defect is valid or not is made in the following step.
1.2 Cancel a defect

If after raising a new defect in ClearQuest the Submitter considers the defect should not be submitted he/she can cancel it before the Project Manager makes any analysis on it. In ClearQuest select ChangeState.Cancel.


Optionally the reason or comments regarding this action can be added into the Actions/Comments field.




Then the form should be saved. The defect's status is now Cancelled.
The Project Manager will be sent an e-mail telling that the defect has been cancelled.

Step 2 Accept defect and assign responsibility

Once a defect has been submitted, a decision must be made upon whether it is valid.

2.1 Set defect as Duplicate

If there is an existing (reported) defect that encompasses some or all of the new defect's symptoms, the Project Manager marks the new defect as Duplicate. In ClearQuest select Duplicate. At this moment you will be asked to "Enter the Entity Id".

 

Complete this field with the Defect ID value automatically set by ClearQuest to the defect to which the new one relates to and click OK. Save the form after completing the defect's information with your desired comments in the Actions/Comments field.
The defect's status is now Duplicate.
Note that all defects which are in status Raised or Deferred can be set as duplicate at any time.

2.2 Set defect as Rejected (invalid defect)

A defect can be considered invalid for many reasons like:
  • It is the result of a risk occurrence, change or issue.
  • It was caused by testing errors or omissions
  • It was raised as the result of a user misunderstanding, etc.

If the defect is considered to be invalid it is rejected. In ClearQuest select ChangeState.Reject and complete at least the following field:

Resolution Code -Use the following criteria:

  • Fixed: the defect has been fixed (although not yet in production). Used when resolving a defect.
  • Change: the defect was introduced as the result of a change needed to amend the contractual obligations, baselined project plans, or project requirements.
  • Issue: the defect was introduced as the result of an issue where the defect is an internal matter of concern for the project.
  • ByPass: a workaround exists for the defect. Usually this resolution code is using when deferring a defect because a workaround exists, and not when rejecting a defect.
  • Risk: the defect was introduced as the result of an occurrence of a risk already identified in the project.
  • Invalid: the defect is invalid.
  • Fixed indirectly: the defect is/will be fixed as the result of fixing another existing defect.
  • Functions as designed: the application functions as designed.
  • Fix in future release: the defect will be fixed in a future release. Usually this resolution code is using when deferring a defect, and not when rejecting a defect.


Save the form after completing the defect's information with your desired comments in the Actions/Comments field explaining better the reason why the defect is being/ has been rejected.
The defect's status is now Rejected.



The Submitter and Project Manager will be sent an e-mail telling that the defect has been rejected.
Note that all defects which are in status Raised or Deferred can be rejected at any time.

2.3 Set defect as Deferred

If the defect cannot be resolved at the moment the Project Manager can defer its resolution to a future release.
In ClearQuest select ChangeState.Defer and complete the mandatory fields and any optional fields that you feel may aid in future analysis/resolution. At this point the mandatory field is Resolution Code and, although several values are allowed, probably the most suitable for this state will be ByPass or Fix in future release.
 
It is a good practice to complete other optional fields (like Proposed workaround, Additional comments, actions comments or those related to estimates/planned dates and/or planned actions), in order to provide more information for future analysis/resolution or to document why the decision of postponing the defect was taken.
After completing the appropriate information save the form. The defect's status is now Deferred.
Step
The Submitter and Project Manager will be sent an e-mail telling that the defect has been deferred.

2.4 Assign a defect

If the defect is valid, analyze the defect and determine who will work on it (the Resolution Owner). Once this has been decided, in ClearQuest select ChangeState.Assign, and complete all mandatory fields and any optional fields that you feel may aid in analysis/resolution.


After completing all the required fields, save the form.

The defect's status is now Opened.
The Resolution Owner will be sent an e-mail telling that a new defect was assigned to him/her.
Note that all valid defects which are in status Raised or Deferred should be assign at any time.

Step 3 Analyze the defect and record action for defect resolution

Analyze the defect's information to ensure that there is enough information to resolve the defect and that it can be resolved within the required timescale; also defines the schedules and planned actions required to solve the defect.
In the event that the Resolution Owner cannot implement all of the actions personally, the Resolution Owner assumes responsibility for enlisting others to implement the necessary actions.
If the defect's information is insufficient the Resolution Owner contacts the Project Manager, Submitter of the defect or any other stakeholder, as appropriate, for additional information. At this point both the Resolution Owner and the Project Manager are able to document the additional information using the Modify action.

3.1 Set defect as Duplicate

If the Resolution Owner finds an existing (reported) defect that encompass some or all of the new defect's symptoms, the defect is set as Duplicate following the same actions defined by the Project Manager is a previous section.

3.2 Set defect as Rejected (invalid defect)

If the Resolution Owner considers the defect is invalid, set it as Rejected following the same actions defined by the Project Manager in a previous section.

3.3 Analyze a defect

If the information of the defect is enough and the defect is confirmed -attempt to recreate the defect is necessary- determine the actions required to resolve the defect. If the defect's impact is complex or it is expected to have significant impact on project scope or on other projects, estimates are required. In ClearQuest select ChangeState.Analyze, and complete all mandatory fields and any optional fields that you feel may aid in resolution or it is important to document for future reference.
Note that some fields in the Estimate/Planned section will become mandatory if Estimate Required? is set to Yes.
After completing all the required fields -or change them as appropriate if reviewing a defect previously analyzed-, save the form.
The defect's status is now Analyzed.
The Project Manager will be sent an e-mail telling that the defect has been analyzed.
Important: This analysis could take some time; if you are analyzing a defect and don't have all the required information for the mandatory fields you can register your partial information using Modify action by a Save. When you are ready you should select ChangeState.Analyze to change the defect's status.
Note that defects which are in status Opened or Revision Required should be analyzed at any time using the steps described here.

Step 4 Review actions and decisions

All the defects analyzed by the Resolution Owners should be reviewed by the Project Manager in order to be approved and prioritized.

4.1 Return defect for further analysis

If the Project Manager considers that the recommended actions or estimates documented by the Resolution Owner are not appropriate, return the defect to the Resolution Owner for further analysis. In ClearQuest select ChangeState.Revise, complete all mandatory fields and describe why the defect requires further analysis including that in the Actions/Comments field.

After completing all the required fields, save the form.
The defect's status is now Revision Required.
The Resolution Owner will be sent an e-mail telling that the defect requires further analysis.

4.2 Accept recommended actions

If the Project Manager considers that the recommended actions and estimates are appropriate, approve, prioritize, schedule and assign the recommended action. In ClearQuest select ChangeState.ApproveSolution; if desired complete the Actions/Comments field.




After completing all the required fields, save the form.
The defect's status is now Solution Approved.
The Resolution Owner will be sent an e-mail telling that the analysis was accepted.

Step 5 Resolve defect

5.1 Resolve

Primary role: Resolution Owner carries out this step of the process.
The Resolution Owners may easily list all defects currently assigned to them by running the public query My Open Defects.
Once a defect's recommended actions and estimate has been approved by the Project Manager the Resolution Owner is able to start working on the required fix. These defects can be easily distinguished as they are in status Solution Approved. The defects in this status can be also found easily in the DefectsApprovedForImplementation query.
During resolution, record progress using Modify action and tracking information in the Actions/Comments field. When the work to correct the defect is complete, select ChangeState.Resolve, and complete all mandatory fields.

After completing all the required fields, save the form.
The defect's status is now Resolved.



The Project Manager will be sent an e-mail telling that the defect has been resolved.
Note that defects which are in status Solution Approve or Revision Required should be resolved at any time using the steps described here.

5.2 Accept or Reject Resolution

Primary role: Project Manager carries out this step of the process (note that this step could be carried out by a testing group if available).
Once the defect was resolved by the Resolution Owner the fix should be tested to verify the defect is really fixed as desired.

5.3 Accept resolution

If Project Manager accepts the resolution provided by the Resolution Owner the defect should be marked as accepted. In ClearQuest select ChangeState.AcceptResolution.


If necessary include actions/comments and save the form.
The defect's status is now Resolution Accepted.


The Submitter and the Project Manager will be sent an e-mail.

5.4 Reject resolution

If Project Manager does not accept the resolution provided by the Resolution Owner the defect should be return to de Resolution Owner as additional actions may need to be implemented in order to satisfactorily resolve the defect. In ClearQuest select ChangeState.Revise, complete all mandatory fields and describe why the resolution is being rejected including that in the Actions/Comments field. Then save the form.
The defect's status is now Revision Required.
The Resolution Owner will be sent an e-mail.

5.5 Close the defect

Primary role: Project Manager carries out this step of the process.
All the defects which resolution is accepted should be validated by the Submitter so the Project Manager can finally close the defect. In ClearQuest select ChangeState.Close, complete all mandatory fields and include any desired comment in the Actions/Comments field.






Then save the form. ClearQuest will automatically complete the Date Closed field with the current date.
The defect's status is now Closed.
The Submitter and Project Manager will be sent an e-mail.
Note that only defects in status Resolved or Resolution Accepted can be closed.

Step 6 Track and manage outstanding defects to closure

There is a public query called Open Defects which may be run in order to list all defects that have not been closed yes. This query should be run periodically and the report should be reviewed for any defects that may not be receiving the appropriate attention. Escalation procedures may need to be used in order to address any such defects.

Step 7 Analyze and report defects

Run the query Defect Log (APP414) periodically to create the equivalent of the APP 414 Defect Log or DCD 325 Defect Management Documentation.
Analyze the report and update the defects with any appropriate action planned or taken. Using this query, it is not necessary to maintain a separate defect log as the query can be run on demand.
Other public queries are available to get the appropriate information in order to analyze and report defects. Additional queries and charts can be created on demand based on needs (refer to ClearQuest User's Guide for more information). Refer to the section 4 for more details.
