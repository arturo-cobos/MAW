Task: Outline High Level Component Model
Identify the logical and physical components required to deliver the solution and define some initial/high level dependencies and collaborations between them.
Disciplines: Technical Architecture
Purpose

The purpose of this task is to outline the Logical and Physical Component Model and associated Component Specifications at a high level:

  • define the Logical Component Model in sufficient detail to enable creation of the initial operational model, and estimation of required capacities and throughputs.
    • develop an initial, high-level component model identifying logical application and technical components.
    • provide sufficient detail to enable realistic estimation of effort to develop and integrate components.
  • define the Physical Component Model that identifies physical application and technical components
    • identify candidate physical components (e.g. products, assets etc…) that will be used to implement the solution
    • identify physical components (e.g. existing applications) that must be used within the scope of the solution
    • illustrate the dependencies and collaboration between physical components

      Main Description
      In the steps describing this task, the first set are for developing the outline Logical Component Model (LCM), and the second set of steps address the outline Physical Component Model (PCM).
      Steps
      LCM: Identify logical application and technical components

      These can be identified based on an analysis of the requirements (e.g. business processes, use cases, existing systems, data models and business rules). Define each component in terms of its responsibilities, and ensure that the components can satisfy all the system responsibilities defined in the architecturally significant use cases (or alternative functional requirements specification).

      LCM: Allocate component responsibilities

      Allocation of responsibilities to components is guided by assets, reference architectures and architecture patterns.  For large or high risk components, decompose the component further until a scope has been arrived at which can be taken forward during design level activities. These responsibilities are documented in each components specification.

      LCM: Structure the components

      Structure the components according to the principles of cohesion, coupling, layering, isolation and granularity.
      Define the usage relationships between the components. Outline component collaborations for the architecturally significant use cases to validate the component model.
      The component model is not defined in detail at this point, e.g. component interfaces are not specified.
      PCM: Make technology decisions

      Decisions made about the technology, platform etc… that will be used to implement the system should be captured and modelled within the Physical Component Model. Identify physical application and technical components based on the constraints of the environment into which the system will be deployed or based on the technology standards governing the system’s development and implementation.

      PCM: Model collaborations

      Having identified the physical components, model the dependencies between these using a Component Relationship Diagram and model the collaboration between these using a Component Interaction Diagram.
