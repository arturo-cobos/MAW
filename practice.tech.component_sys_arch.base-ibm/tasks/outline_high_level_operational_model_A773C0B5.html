Task: Outline High Level Operational Model
Create a high level logical and physical Operational Model showing nodes, connections, and placement of components
Disciplines: Technical Architecture
Purpose

The purpose of this task is to:

  • Provide a basis for assessing the viability of implementing the system
  • Gain an understanding of the geographical distribution and operational complexity of the system
  • Provide a basis for an early effort and cost estimate
  • Create an outline logical operational model by describing how a set of application and technical deployment units should be placed across the geographic organization of the client whilst accommodating a particular set of constraints which the Operational Model will have to overcome
  • Create a technology-dependent physical operational model

    Main Description

    This task first creates the outline Logical Operational Model (LOM). The level of detail in the outline LOM would be expected to be greater for the application nodes and their associated components than for the technical nodes and components that work in a supporting role to the application.

    Secondly, this task also creates the outline Physical Operational Model (POM). The focus of this effort is on evaluating, selecting, configuring and validating physical products, which together satisfy the needs of the logical operational model.

    Steps
    LOM: Outline system topology

    Develop views of the system showing nodes and the deployment units that they run. Identify connection between nodes as point-to-point connections. Ensure that business user, locations and geographic sites are identified. Understand "givens" in the client’s environment. Develop the scope of external system interfaces. Use reference architectures as a source for specification when there is a good fit to the target problem domain. If diagrams become too busy, subdividing them into specific operational or systems management aspects of the architecture, for example physical views, system software views, location views, end-to-end application views, software distribution view, etc.

    LOM: Outline conceptual nodes

    Develop an initial description of each node, which usually includes a table or box diagram that identifies and classifies the software components that run on the node. For convenience, components (which may not all be software) are often grouped into deployment units for ease of placement. The description includes the node’s availability, performance, security and other non-functional characteristics. (A non-functional characteristic is what is achieved; a non-functional requirement is what is required.). Conceptual nodes ignore many technological limitations and focus on application software components, deferring treatment of middleware and other software.

    LOM: Outline conceptual connections

    Document the connections between the systems nodes at a conceptual level. This should include:

    • The end points (node names) of the connection.
    • The business traffic the connection supports.
    • A specification of the availability requirements for the connection.
    POM: Transform the logical node model into an outline physical node model
    POM: Develop outline physical specifications for nodes and connections
    POM: Evaluate and select products
    • for nodes
    • for technical components placed on nodes
    • for connections
    POM: Rationalize the Physical Operational Model
    Rationalize the Physical Operational Model based on products selected including outlining the  sizing of nodes and connections
    POM: Review and detail the Physical Operational Model
    Review and detail the Physical Operational Model including creating the configurations for nodes and connections
    POM: Identify technical components, nodes, and connections
    Identify technical components, nodes, and connections where detailed designs will be required and create outline designs as appropriate.
    POM: Summarize state of Physical Operational Model
    Summarize state of Physical Operational Model and estimate effort required to further develop the Physical Operational Model according to the context.
    Key Considerations

    The Logical Operational Model is developed in parallel with the Physical Operational Model and they are tightly coupled in both their development and also the development of the Component Model. Therefore decisions made when developing the Logical or Physical Operational Model may result in changes to both the alternative elaboration of the Operational Model as well as the Component Model.

    The state of the Operational Models which needs to be reached at the end of this task is dependent on the context in which they are being developed, the level of confidence that needs to be achieved in the architecture and design, and the degree of accuracy of estimates of resources to further develop the operational model.
